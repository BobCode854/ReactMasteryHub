        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q1">
          <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is React.memo?
        </button>
      </h2>
      <div id="performance-q1" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
        <div class="accordion-body">
          React.memo is a higher-order component that memoizes a functional component — it prevents unnecessary re-renders by shallowly comparing the previous and new props. If the props haven't changed (shallow equality), React skips rendering the component and reuses the last rendered output. It is the functional component equivalent of <code>shouldComponentUpdate</code> or <code>PureComponent</code> for class components. Use React.memo when a component renders frequently with the same props, especially in lists where a parent re-renders but many children don't actually change. The performance gain is only realized when the component's render cost is higher than the cost of the shallow prop comparison — wrapping every component in memo is not recommended.
        </div>
      </div>
    </div>

    <div class="accordion-item">
      <h2 class="accordion-header">
        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q2">
          <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is the difference between useMemo and useCallback?
        </button>
      </h2>
      <div id="performance-q2" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
        <div class="accordion-body">
          Both <code>useMemo</code> and <code>useCallback</code> are memoization hooks, but they memoize different things. <code>useMemo(() =&gt; computeExpensiveValue(a, b), [a, b])</code> memoizes the return value of a computation — the function runs once when dependencies change, and the cached result is returned otherwise. <code>useCallback((e) =&gt; handleEvent(e), [dependency])</code> memoizes the function reference itself — the same function object is returned on re-renders when dependencies haven't changed. A shorthand: <code>useCallback(fn, deps)</code> is equivalent to <code>useMemo(() =&gt; fn, deps)</code>. Use <code>useMemo</code> for expensive calculations; use <code>useCallback</code> for stable function references passed to memoized child components or used in dependency arrays.
        </div>
      </div>
    </div>

    <div class="accordion-item">
      <h2 class="accordion-header">
        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q3">
          <span class="diff-easy">Easy</span>&nbsp;&nbsp;When should you use React.memo?
        </button>
      </h2>
      <div id="performance-q3" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
        <div class="accordion-body">
          Use <code>React.memo</code> when: (1) a component is a pure function of its props (same props always produce the same output); (2) the component renders often with identical props (e.g., list items in a frequently-updating list); (3) the component has a non-trivial render cost. Avoid memo when: the component always receives new props on each render (memo overhead without benefit); the component is cheap to render (the comparison overhead outweighs savings); or the component has complex props that are always new objects (reference comparison always fails). Profile with React DevTools before adding memo — don't optimize preemptively.
        </div>
      </div>
    </div>

        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q4">
          <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is code splitting in React?
        </button>
      </h2>
      <div id="performance-q4" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
        <div class="accordion-body">
          Code splitting is the practice of splitting your JavaScript bundle into smaller chunks that are loaded on demand, rather than loading all code upfront. This reduces the initial bundle size, speeds up the first meaningful paint, and defers loading code that is not needed immediately. In React, <code>React.lazy()</code> combined with <code>&lt;Suspense&gt;</code> enables component-level code splitting. Webpack (and Vite) automatically create separate chunks for each <code>React.lazy()</code> import. Route-level code splitting (lazy loading each page component) is the most impactful approach since page components are the largest distinct chunks and each page is only visited by a subset of users at any given time.
        </div>
      </div>
    </div>

      <div id="performance-q5" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
        <div class="accordion-body">
          <code>React.lazy(() =&gt; import('./MyComponent'))</code> enables lazy loading of a component — the component's code is loaded on demand when the component is first rendered, not at initial bundle load time. The <code>import()</code> call returns a Promise that resolves to a module with a default export. <code>React.lazy</code> must be used inside a <code>&lt;Suspense&gt;</code> boundary that provides a fallback UI while the lazy component loads. Lazy loading is most effective for route-level components (entire pages), heavy feature components (rich text editors, chart libraries), and components below the fold that users may never see. In React 18, <code>lazy</code> also supports Server Components.

        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q6">
          <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is Suspense in React?

      </h2>
      <div id="performance-q6" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
        <div class="accordion-body">
          <code>&lt;Suspense fallback={&lt;Spinner /&gt;}&gt;</code> is a React component that catches "pending" states from lazy-loaded components (and in React 18, from async data sources via <code>use()</code>). When a child component is loading (a lazy import is in flight), Suspense renders the <code>fallback</code> prop instead of the children. When the lazy component finishes loading, Suspense replaces the fallback with the actual content. Suspense enables declarative loading states — you specify the loading UI in one place (the Suspense boundary) rather than having every component manage its own loading state. Multiple Suspense boundaries with granular fallbacks create a progressive loading experience.
        </div>

      <h2 class="accordion-header">
        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q7">
          <span class="diff-medium">Medium</span>&nbsp;&nbsp;How does React's reconciliation algorithm work?
        </button>
      </h2>
      <div id="performance-q7" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
        <div class="accordion-body">
          React's reconciliation algorithm (the "diffing" algorithm) determines the minimal set of DOM changes needed to update the UI when state or props change. When state changes, React creates a new virtual DOM tree and compares it to the previous one. The algorithm applies two heuristics to keep the comparison O(n) rather than O(n³): (1) Elements of different types produce entirely different subtrees — if a <code>&lt;div&gt;</code> changes to a <code>&lt;span&gt;</code>, the entire subtree is destroyed and rebuilt; (2) List items with stable <code>key</code> props are matched across renders — React uses keys to identify which items were added, removed, or moved. Without keys, list reconciliation falls back to index-based comparison, causing incorrect updates when items are reordered. Understanding reconciliation is essential for optimizing performance: avoid changing element types unnecessarily, and always provide stable keys for list items.
          <pre class="code-block">// Keys tell React which list item is which
// Bad: using index as key (React can't detect reorders)
{items.map((item, index) =&gt; (
  &lt;ListItem key={index} item={item} /&gt;
))}

// Good: using stable unique ID
{items.map(item =&gt; (
  &lt;ListItem key={item.id} item={item} /&gt;
))}

// Why it matters: if you prepend an item to the list with index keys,
// React sees all keys changed (0,1,2 -&gt; 0,1,2,3)
// and re-renders all items. With item.id keys,
// React sees only one new item added and keeps existing ones unmounted.</pre>
        </div>
      </div>

      <h2 class="accordion-header">
        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q8">
          <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is virtualization and why is it important?
        </button>
      </h2>
      <div id="performance-q8" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
        <div class="accordion-body">
          Virtualization (also called windowing) is a technique where only the DOM nodes for visible items in a long list are rendered — items that are off-screen are not mounted in the DOM. Instead of rendering 10,000 list items (10,000 DOM nodes), a virtualized list only renders ~20-30 items that are visible in the viewport. As the user scrolls, DOM nodes are recycled — off-screen items are unmounted and their DOM nodes are reused for newly visible items. This dramatically reduces initial render time, memory usage, and scroll performance for large datasets. The most popular virtualization libraries for React are <code>react-virtual</code> (by TanStack), <code>react-window</code> (lighter), and <code>react-virtualized</code> (full-featured). Virtualization is essential for rendering tables or lists with hundreds or thousands of rows.
          <pre class="code-block">import { useVirtualizer } from '@tanstack/react-virtual';

function VirtualList({ items }) {
  const parentRef = React.useRef(null);

  const virtualizer = useVirtualizer({
    count: items.length,
    getScrollElement: () =&gt; parentRef.current,
    estimateSize: () =&gt; 50, // estimated row height in px
  });

  return (
    &lt;div ref={parentRef} style={{ height: '400px', overflow: 'auto' }}&gt;
      {/* Total height spacer so scrollbar is accurate */}
      &lt;div style={{ height: virtualizer.getTotalSize() + 'px', position: 'relative' }}&gt;
        {virtualizer.getVirtualItems().map(virtualItem =&gt; (
          &lt;div
            key={virtualItem.key}
            style={{ position: 'absolute', top: virtualItem.start, width: '100%' }}
          &gt;
            {items[virtualItem.index].name}
          &lt;/div&gt;
        ))}
      &lt;/div&gt;
    &lt;/div&gt;
  );
}</pre>

        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q10">
          <span class="diff-medium">Medium</span>&nbsp;&nbsp;What are the main causes of unnecessary re-renders?
        </button>
      </h2>
      <div id="performance-q10" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
        <div class="accordion-body">
          The most common causes of unnecessary re-renders are: (1) New object/array/function references — <code>const style = { color: 'red' }</code> inside render creates a new object reference every render, causing memoized children to re-render; (2) Context value changes — putting a new object literal in <code>&lt;Context.Provider value={{ a, b }}&gt;</code> creates a new reference every render, causing all consumers to re-render even if a and b haven't changed; (3) State updates with the same value — <code>setState(sameValue)</code> still triggers a re-render in React 17 (React 18 bails out for same primitive values); (4) Parent re-renders propagating to children — children without memo re-render whenever the parent does; (5) Improper key usage — changing keys causes components to unmount and remount. Use React DevTools Profiler to diagnose which of these applies.
        </div>
      </div>
    </div>

        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q11">
          <span class="diff-medium">Medium</span>&nbsp;&nbsp;How do you memoize expensive calculations?

      <div id="performance-q11" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
        <div class="accordion-body">
          <code>useMemo(() =&gt; expensiveCalculation(input), [input])</code> memoizes the result — the calculation only re-runs when <code>input</code> changes, returning the cached value on other renders. This is useful for: sorting/filtering large arrays, complex mathematical computations, creating derived data from large datasets, and building objects for Context.Provider values. The key question is "is this actually expensive?" Sorting 1000 items takes ~1ms — likely not worth memoizing. Parsing and transforming a 50,000-row dataset might be. Always measure with the Profiler before adding useMemo — premature memoization adds complexity and the overhead of the dependency comparison.
          <pre class="code-block">function ProductList({ products, searchTerm, sortBy }) {
  // Memoize expensive filter + sort operation
  const filteredAndSorted = React.useMemo(() =&gt; {
    console.log('Computing filtered+sorted list...'); // only logs when deps change
    return products
      .filter(p =&gt; p.name.toLowerCase().includes(searchTerm.toLowerCase()))
      .sort((a, b) =&gt; {
        if (sortBy === 'price') return a.price - b.price;
        if (sortBy === 'name') return a.name.localeCompare(b.name);
        return 0;
      });
  }, [products, searchTerm, sortBy]);
  // filteredAndSorted is the same array reference as long as dependencies don't change

  return &lt;ul&gt;{filteredAndSorted.map(p =&gt; &lt;ProductItem key={p.id} product={p} /&gt;)}&lt;/ul&gt;;

        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q13">
          <span class="diff-medium">Medium</span>&nbsp;&nbsp;How do you optimize Context to avoid unnecessary re-renders?
        </button>
      </h2>
      <div id="performance-q13" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
        <div class="accordion-body">
          Every component that calls <code>useContext(MyContext)</code> re-renders whenever the Context's <code>value</code> changes (by reference). Common optimization strategies: (1) Split contexts — separate frequently-changing values from stable values into different context providers; (2) Memoize the value — <code>const value = useMemo(() =&gt; ({ user, logout }), [user])</code> to prevent new object references; (3) Context + reducer — use <code>useReducer</code> for the state and expose the dispatch separately (dispatch is stable, so a separate dispatch context prevents re-renders on state changes); (4) Use a context selector — libraries like <code>use-context-selector</code> add React Redux-like selector support to Context. The most scalable solution for complex shared state is usually Redux or Zustand rather than Context.
          <pre class="code-block">// Split context: stable values vs frequently-changing values
const UserContext = React.createContext(null);
const UserDispatchContext = React.createContext(null);

function UserProvider({ children }) {
  const [user, dispatch] = React.useReducer(userReducer, initialUser);

  // user changes often; dispatch is stable (never changes)
  return (
    &lt;UserDispatchContext.Provider value={dispatch}&gt;
      &lt;UserContext.Provider value={user}&gt;
        {children}
      &lt;/UserContext.Provider&gt;
    &lt;/UserDispatchContext.Provider&gt;
  );
}

// Components that only dispatch won't re-render when user state changes
function LogoutButton() {
  const dispatch = React.useContext(UserDispatchContext); // stable!
  return &lt;button onClick={() =&gt; dispatch({ type: 'LOGOUT' })}&gt;Logout&lt;/button&gt;;
}</pre>

      <div id="performance-q14" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
        <div class="accordion-body">
          Lazy loading images defers loading images that are off-screen until the user scrolls near them. This reduces initial page load time and network bandwidth for images the user may never see. The modern approach uses the native <code>loading="lazy"</code> attribute on <code>&lt;img&gt;</code> tags — supported by all modern browsers, no JavaScript required. For custom lazy loading (with custom thresholds, custom placeholders, or blur-up effects), use the <code>IntersectionObserver</code> API in a custom hook. Libraries like <code>react-lazyload</code> and <code>react-intersection-observer</code> provide React abstractions. For Next.js, the built-in <code>&lt;Image&gt;</code> component handles lazy loading, size optimization, and modern format conversion automatically.
          <pre class="code-block">// Native lazy loading (simplest — use this first)
&lt;img src={product.imageUrl} alt={product.name} loading="lazy" /&gt;

// Custom lazy loading with IntersectionObserver
function LazyImage({ src, alt, placeholder }) {
  const imgRef = React.useRef(null);
  const [isVisible, setIsVisible] = React.useState(false);

  React.useEffect(() =&gt; {
    const observer = new IntersectionObserver(
      ([entry]) =&gt; { if (entry.isIntersecting) { setIsVisible(true); observer.disconnect(); } },
      { rootMargin: '200px' } // start loading 200px before visible
    );
    if (imgRef.current) observer.observe(imgRef.current);
    return () =&gt; observer.disconnect();
  }, []);

  return (
    &lt;img
      ref={imgRef}
      src={isVisible ? src : placeholder ?? 'data:image/svg+xml,...'}
      alt={alt}
    /&gt;
  );
}</pre>

        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q15">
          <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is Concurrent Mode in React 18?
        </button>
      </h2>
      <div id="performance-q15" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
        <div class="accordion-body">
          Concurrent Mode is React 18's ability to prepare multiple versions of the UI simultaneously and interrupt, pause, or abandon renders that are no longer needed. Before Concurrent Mode (React 17 and earlier), rendering was synchronous — once React started rendering a component tree, it ran to completion, potentially blocking the main thread and making the UI unresponsive. In Concurrent Mode, React can pause work, work on higher-priority updates first (e.g., a keypress) before finishing lower-priority work (e.g., a heavy list re-render), and discard partial renders if the data changes again. This is the foundation for features like <code>startTransition</code>, <code>useDeferredValue</code>, and Suspense for data fetching. Concurrent Mode is opt-in via <code>ReactDOM.createRoot</code> (instead of the legacy <code>ReactDOM.render</code>).
        </div>
      </div>

        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q16">

      </h2>
      <div id="performance-q16" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
        <div class="accordion-body">
          <code>startTransition(callback)</code> (React 18+) marks state updates inside the callback as "transitions" — non-urgent updates that can be interrupted by more urgent updates (like user input). It is used when you have a state update that triggers a slow render and you want typing/clicking to remain responsive while it processes. The wrapped updates have lower priority than default updates. React can interrupt a transition render to process a higher-priority update (e.g., a new keystroke), then resume or restart the transition. This eliminates the typing lag that occurs when filtering a large list — typing is always instantly responsive, and the list update happens in the background. React 18 also provides the <code>useTransition</code> hook which gives an <code>isPending</code> boolean for showing a loading indicator during the transition.
          <pre class="code-block">import { startTransition, useTransition } from 'react';

// startTransition directly
function SearchPage() {
  const [query, setQuery] = React.useState('');
  const [results, setResults] = React.useState([]);

  const handleSearch = (e) =&gt; {
    setQuery(e.target.value); // urgent: update input immediately
    startTransition(() =&gt; {
      setResults(computeHeavyFilteredResults(e.target.value)); // non-urgent
    });
  };
  return &lt;input value={query} onChange={handleSearch} /&gt;;
}

// useTransition: also gives isPending
function SearchWithSpinner() {
  const [isPending, startTransition] = useTransition();
  const [results, setResults] = React.useState([]);

  const handleSearch = (query) =&gt; {
    startTransition(() =&gt; setResults(computeHeavyResults(query)));
  };
  return &lt;&gt;{isPending &amp;&amp; &lt;Spinner /&gt;}&lt;ResultList results={results} /&gt;&lt;/&gt;;
}</pre>

      <div id="performance-q17" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
        <div class="accordion-body">
          <code>useDeferredValue(value)</code> (React 18+) returns a deferred version of the provided value — it lags behind the actual value during high-priority renders, allowing the UI to show a slightly stale version while React processes the urgent update. It is the "pull" alternative to <code>startTransition</code> (which is "push"): instead of wrapping the setter, you wrap the value at the consuming end. Use it when you don't control the state update (e.g., a third-party component triggers the update). The deferred value stays the same reference during urgent updates, which means <code>React.memo</code>-wrapped components that receive the deferred value won't re-render during the urgent update — they update during the background render.
          <pre class="code-block">function SearchResults({ query }) {

  // deferredQuery lags behind — the heavy filter runs with the deferred value

  const isStale = deferredQuery !== query;

  const filteredItems = React.useMemo(
    () =&gt; heavyFilter(allItems, deferredQuery),
    [deferredQuery] // only re-runs when deferred value updates
  );

  return (
    &lt;div style={{ opacity: isStale ? 0.7 : 1 }}&gt; {/* visual stale indicator */}
      {filteredItems.map(item =&gt; &lt;Item key={item.id} item={item} /&gt;)}

      <div id="performance-q18" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
        <div class="accordion-body">
          Time slicing is an internal React mechanism (part of Concurrent Mode) where React breaks up large rendering work into small units ("slices") and executes them over multiple frames, yielding control back to the browser between slices. Instead of blocking the main thread for 200ms rendering a massive component tree, React renders a portion, yields to allow the browser to handle user events and paint, then resumes. This keeps the UI at 60fps (or higher) even during complex renders. Time slicing is automatic in React 18 with <code>createRoot</code> — you don't need to do anything to enable it. It works in conjunction with <code>startTransition</code> and <code>useDeferredValue</code> which designate which updates can be time-sliced (transitions) vs which must be synchronous (urgent updates).
        </div>
      </div>
    </div>

          <code>IntersectionObserver</code> allows you to detect when elements enter or leave the viewport without polling (no continuous scroll event listeners). Performance use cases: lazy loading images/components, infinite scroll pagination, triggering animations when elements become visible, and analytics (tracking which sections users see). The API is asynchronous and doesn't block the main thread. A React custom hook wrapping IntersectionObserver is the idiomatic pattern. Clean up the observer in the effect's cleanup function. For simpler cases, the <code>react-intersection-observer</code> library provides a clean hook API.
          <pre class="code-block">function useIntersectionObserver(options = {}) {
  const [isVisible, setIsVisible] = React.useState(false);
  const elementRef = React.useRef(null);

  React.useEffect(() =&gt; {
    const element = elementRef.current;
    if (!element) return;

    const observer = new IntersectionObserver(([entry]) =&gt; {
      setIsVisible(entry.isIntersecting);
    }, {
      threshold: options.threshold ?? 0,
      rootMargin: options.rootMargin ?? '0px',
    });

    observer.observe(element);
    return () =&gt; observer.unobserve(element);
  }, [options.threshold, options.rootMargin]);

  return [elementRef, isVisible];
}

// Usage: load component only when it enters viewport
function LazySection({ children }) {
  const [ref, isVisible] = useIntersectionObserver({ threshold: 0.1 });
  return (
    &lt;div ref={ref}&gt;
      {isVisible ? children : &lt;div style={{ height: '400px' }} /&gt;}
    &lt;/div&gt;
  );
}</pre>

        </button>
      </h2>
      <div id="performance-q34" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
        <div class="accordion-body">
          Tree shaking is the process where bundlers (Webpack, Vite/Rollup) analyze module imports and remove code that is never imported (dead code elimination). It works on ES modules (<code>import</code>/<code>export</code> syntax) because static analysis can determine which exports are used. For React apps, tree shaking is crucial for keeping bundle sizes small. Common tree-shaking pitfalls: importing from CommonJS modules (can't be tree-shaken); using default imports from libraries that export everything (e.g., <code>import _ from 'lodash'</code> imports ALL of lodash); side-effect imports (<code>import 'some-library'</code> defeats shaking); and libraries that don't ship ES modules (check the <code>module</code> field in package.json). Always import named exports: <code>import { debounce } from 'lodash-es'</code> imports only debounce.
        </div>
      </div>
    </div>

        </button>

Optimization strategies: (1) Uncontrolled inputs with refs — avoid React state entirely for large forms; (2) React Hook Form — designed for performance, it uses uncontrolled inputs internally and only re-renders the specific field that changed and the submit button; (3) Field-level state isolation — each field manages its own state, parent only collects values on submit; (4) Debounce validation — don't validate on every keystroke for expensive async validation; (5) Memoize field components with <code>React.memo</code> — prevent other fields from re-rendering when one field changes. React Hook Form is the best solution for complex forms as it combines minimal re-renders with a developer-friendly API.

      <h2 class="accordion-header">
        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q36">
          <span class="diff-hard">Hard</span>&nbsp;&nbsp;What is the difference between paint and layout in browser rendering?
        </button>
      </h2>
      <div id="performance-q36" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
        <div class="accordion-body">
          The browser rendering pipeline: JavaScript → Style → Layout → Paint → Composite. Layout (also called "reflow") recalculates the geometry of elements — positions and sizes. It is triggered by property reads like <code>offsetWidth</code>, <code>scrollTop</code>, <code>getBoundingClientRect()</code> or by DOM mutations that affect size/position. Paint fills in pixels for each element — colors, backgrounds, text. Compositing combines painted layers. Layout and Paint are expensive; Compositing is cheap. CSS properties that only affect compositing (<code>transform</code>, <code>opacity</code>) animate without Layout or Paint — this is why CSS transitions should use <code>transform</code> and <code>opacity</code> rather than <code>top</code>/<code>left</code>/<code>width</code>. In React, frequent style changes via inline styles or className toggling can trigger unnecessary Layout and Paint — batch DOM mutations and use CSS custom properties or CSS classes for performance.

        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q38">
          <span class="diff-hard">Hard</span>&nbsp;&nbsp;How do you implement request waterfall prevention in React?
        </button>
      </h2>
      <div id="performance-q38" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
        <div class="accordion-body">
          Request waterfalls occur when components sequentially fetch data that could be fetched in parallel — each fetch must complete before the next starts, adding latency equal to the sum of all requests. Common patterns: parent component fetches user data → renders child → child fetches user's posts → renders grandchild → grandchild fetches post comments. Solutions: (1) Parallel fetching — <code>Promise.all([fetchUser(), fetchPosts(), fetchComments()])</code> in one effect; (2) React Query's <code>useQueries</code> for parallel queries; (3) Fetch at the route level (loaders) — gather all data before rendering the component tree; (4) Relay/GraphQL — fetch the entire component tree's data requirements in one query; (5) React Server Components — server-side rendering eliminates client-server waterfalls for SSR scenarios.
          <pre class="code-block">// Bad: waterfall (3 sequential requests)
function UserProfile({ userId }) {
  const { data: user } = useQuery(['user', userId], fetchUser);
  const { data: posts } = useQuery(['posts', userId], () =&gt; fetchPosts(userId), {
    enabled: !!user, // waits for user before starting
  });
  const { data: followers } = useQuery(['followers', userId], () =&gt; fetchFollowers(userId), {
    enabled: !!user, // also waits for user
  });
}

// Good: parallel requests (posts and followers start together)
function UserProfileFixed({ userId }) {
  const { data: user } = useQuery(['user', userId], fetchUser);

  // Both start simultaneously once userId is available
  const [{ data: posts }, { data: followers }] = useQueries({
    queries: [
      { queryKey: ['posts', userId], queryFn: () =&gt; fetchPosts(userId) },
      { queryKey: ['followers', userId], queryFn: () =&gt; fetchFollowers(userId) },
    ],
  });
}</pre>
        </div>
      </div>

        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q41">
          <span class="diff-hard">Hard</span>&nbsp;&nbsp;What is the useMemo hook's relationship with referential equality?
        </button>
      </h2>
      <div id="performance-q41" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
        <div class="accordion-body">
          <code>useMemo</code> preserves referential equality of its return value between renders when dependencies haven't changed. This is the key insight: <code>useMemo(() =&gt; ({ a, b }), [a, b])</code> returns the same object reference as long as <code>a</code> and <code>b</code> are unchanged. Without useMemo, <code>{ a, b }</code> creates a new object on every render — a different reference even if the values are the same. This matters for: Context values (new reference triggers all consumer re-renders), props to memoized children (new reference defeats React.memo), and useEffect dependencies (new reference triggers effect on every render). The mental model: useMemo is not only for computation performance — it's for reference stability. Stabilizing references is often more important than skipping expensive calculations.
        </div>
      </div>
    </div>

      <h2 class="accordion-header">
        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q42">
          <span class="diff-hard">Hard</span>&nbsp;&nbsp;How does React batch state updates and why does it matter for performance?
        </button>
      </h2>
      <div id="performance-q42" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
        <div class="accordion-body">
          React batches multiple state updates in the same event handler into a single re-render — calling <code>setA(1); setB(2)</code> in a click handler triggers one re-render, not two. In React 17, batching only happened inside React event handlers (synthetic events). In React 18 with <code>createRoot</code>, batching happens automatically everywhere — inside setTimeout, Promise callbacks, and native event listeners. This is "automatic batching." For performance, this means fewer re-renders when multiple state updates happen close together. To opt out of batching (force a synchronous re-render between updates), use <code>ReactDOM.flushSync(() =&gt; setState(value))</code>. Batching is why you should never rely on reading state immediately after setting it — the update is asynchronous.
          <pre class="code-block">// React 18: automatic batching everywhere
function handleAsync() {
  setTimeout(() =&gt; {
    setCount(c =&gt; c + 1); // React 17: immediate re-render
    setFlag(true);         // React 17: second re-render
    // React 18: batched — single re-render for both!
  }, 100);
}

// Opt out of batching with flushSync
import { flushSync } from 'react-dom';
function handleClick() {
  flushSync(() =&gt; setCount(c =&gt; c + 1)); // forces immediate re-render
  // DOM is updated here before the next line
  flushSync(() =&gt; setFlag(true)); // another immediate re-render
}</pre>

The trade-off: RSCs can't use state, effects, or browser APIs. The RSC architecture in Next.js 13+ App Router (where all components are server by default) represents a significant paradigm shift in React performance optimization.

        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q46">
          <span class="diff-hard">Hard</span>&nbsp;&nbsp;What is tree shaking vs dead code elimination?
        </button>
      </h2>
      <div id="performance-q46" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
        <div class="accordion-body">
          These terms are related but distinct. Dead code elimination is the general concept of removing unused code — any bundler/minifier may do this for variables, functions, or imports that are never referenced within a module. Tree shaking is specifically about eliminating unused exports from ES modules — analyzing the import/export graph to find entire modules (or exports within modules) that are imported nowhere and removing them from the bundle. Tree shaking requires ES module syntax (<code>import</code>/<code>export</code>) because it needs static analysis (you can't analyze dynamic <code>require()</code>). Rollup pioneered tree shaking; Webpack and Vite also implement it. For React apps: import named exports from utility libraries, ensure your dependencies ship ES modules (check <code>"module"</code> or <code>"exports"</code> in package.json), and use <code>"sideEffects": false</code> in your own package.json if your library is side-effect free.
          <pre class="code-block">// Dead code elimination: minifier removes unreachable code
function getDiscount(type) {
  if (type === 'vip') return 0.2;
  // The else branch below is dead code if type is always 'vip'
  else return 0.1;
}

// Tree shaking: unused named export removed from bundle
// utils.js exports: add, subtract, multiply, divide
import { add } from './utils'; // only 'add' ends up in bundle; others are shaken out

// Ensure sideEffects: false in package.json for your library
// so bundlers know it's safe to drop unused exports
// package.json: { "sideEffects": false }</pre>
