Class Component Lifecycle — Visual Timeline

Mounting

Component appears in the DOM for the first time

constructor(props)
Initialize state, bind methods. Always call super(props) first.
getDerivedStateFromProps()
Rarely used. Derive state from props before render.
render() Required
Returns JSX. Must be pure — no side effects.
componentDidMount() Most Used
DOM is ready. Run API calls, subscriptions, timers here.
Updating

Triggered by setState() or new props from parent

getDerivedStateFromProps()
Runs again before each render. Rarely needed.
shouldComponentUpdate()
Return false to skip re-render. Performance optimization.
render()
New JSX tree computed and diffed against Fiber.
getSnapshotBeforeUpdate()
Read DOM before update (e.g., scroll position).
componentDidUpdate(prev) Common
Respond to prop/state changes. Always compare before acting!
Unmounting

Component removed from the DOM

componentWillUnmount()
Cleanup! Cancel timers, subscriptions, abort requests. Never call setState here.
Error Phases:
getDerivedStateFromError() — update UI
componentDidCatch() — log error

Full Class Component Example

A real-world DataFetcher component demonstrating all three key lifecycle methods together.

class DataFetcher extends React.Component {
  constructor(props) {
    super(props);
    this.state = { data: null, loading: true, error: null };
  }

  // Runs ONCE after first render — perfect for API calls
  async componentDidMount() {
    await this.fetchData();
  }

  // Runs after every update — check if the relevant prop changed
  async componentDidUpdate(prevProps) {
    if (prevProps.userId !== this.props.userId) {
      await this.fetchData(); // only re-fetch if userId changed
    }
  }

  // Cleanup — cancel subscriptions, clear timers, abort requests
  componentWillUnmount() {
    if (this.abortController) {
      this.abortController.abort();
    }
  }

  async fetchData() {
    this.abortController = new AbortController();
    try {
      this.setState({ loading: true, error: null });
      const res = await fetch(`/api/users/${this.props.userId}`, {
        signal: this.abortController.signal
      });
      if (!res.ok) throw new Error(`HTTP ${res.status}`);
      const data = await res.json();
      this.setState({ data, loading: false });
    } catch (error) {
      if (error.name !== 'AbortError') {
        this.setState({ error: error.message, loading: false });
      }
    }
  }

  render() {
    const { data, loading, error } = this.state;
    if (loading) return <Spinner />;
    if (error) return <ErrorMessage message={error} />;
    return <UserProfile user={data} />;
  }
}

Function Component (Hooks)
function DataFetcher({ userId }) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    const controller = new AbortController();
    setLoading(true);
    setError(null);

    api.get(`/users/${userId}`, { signal: controller.signal })
      .then(data => {
        setData(data);
        setLoading(false);
      })
      .catch(err => {
        if (err.name !== 'AbortError') {
          setError(err.message);
          setLoading(false);
        }
      });

    // This function IS componentWillUnmount + cleanup
    return () => controller.abort();
  }, [userId]); // Re-runs when userId changes

  if (loading) return <Spinner />;
  if (error) return <Error message={error} />;
  return <UserProfile user={data} />;
}
Key Insight: A single useEffect consolidates three lifecycle methods: the effect body runs like componentDidMount + componentDidUpdate, and the returned cleanup function runs like componentWillUnmount.

React Component Render Flow — Step by Step

What happens inside React each time a component renders. Understanding this helps you reason about when effects run and why re-renders occur.

1
Trigger: Parent renders with new props, or setState() / dispatch() is called — React schedules a re-render.
2
React calls your component function — the entire function body runs from top to bottom with the new state/props values.
3
Hooks execute in order — useState returns current state, useEffect checks if dependencies changed since last render.
4
JSX returned → React.createElement tree — JSX is syntactic sugar that creates a plain JavaScript object tree describing the UI.
5
Reconciliation (diffing) — React compares the new element tree against the previous Fiber tree to find the minimal set of DOM changes needed.
6
Commit phase — React applies the computed DOM mutations. This is when ref values are updated and the browser paints.
7
Cleanup runs — If a useEffect's dependencies changed, its previous cleanup function runs first to tear down the old effect.
8
Effects run — useEffect callbacks whose dependencies changed are invoked. useLayoutEffect runs synchronously before paint; useEffect runs asynchronously after.
React 18 Concurrent Mode: React can now pause, interrupt, and restart renders. This is why you should keep render functions pure — they may run more than once. Side effects belong in useEffect, not in the function body.

Error Boundaries — Class Components Only

Error Boundaries catch JavaScript errors in their child component tree, log them, and display a fallback UI instead of a blank crash screen. They still require class components in React 18.

class ErrorBoundary extends React.Component {
  state = { hasError: false, error: null };

  // Called during render phase — update state to show fallback UI
  static getDerivedStateFromError(error) {
    return { hasError: true, error };
  }

  // Called in commit phase — log the error to your error service
  componentDidCatch(error, info) {
    // info.componentStack shows where the error originated
    logErrorToService(error, info.componentStack);
    console.error('Caught by ErrorBoundary:', error);
  }

  render() {
    if (this.state.hasError) {
      return this.props.fallback || (
        <div className="text-center p-5">
          <h2>Something went wrong.</h2>
          <p className="text-muted">{this.state.error?.message}</p>
          <button onClick={() => this.setState({ hasError: false })}>
            Try Again
          </button>
        </div>
      );
    }
    return this.props.children;
  }
}

// Usage — wrap any subtree you want to protect
<ErrorBoundary fallback={<ErrorPage />}>
  <App />
</ErrorBoundary>

// Granular — protect individual sections
function Dashboard() {
  return (
    <div>
      <ErrorBoundary fallback={<p>Chart failed to load</p>}>
        <ChartWidget />
      </ErrorBoundary>
      <ErrorBoundary fallback={<p>Table failed to load</p>}>
        <DataTable />
      </ErrorBoundary>
    </div>
  );
}
What Error Boundaries do NOT catch: Errors in event handlers (use try/catch there), asynchronous code (setTimeout, fetch), errors in the boundary itself, or server-side rendering. They only catch errors during rendering, in lifecycle methods, and in constructors of child components.
