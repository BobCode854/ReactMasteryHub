01

5 Ways to Set Up React

⚡ Vite — The Modern Way

Created by Evan You (Vue creator). Uses native ES modules for near-instant dev server startup.

Quick Start Commands
# Create a new React + Vite project: npm create vite@latest my-app -- --template react # Or with TypeScript (recommended): npm create vite@latest my-app -- --template react-ts # Navigate and install: cd my-app npm install # Start dev server (port 5173 by default): npm run dev # Build for production: npm run build # Preview production build locally: npm run preview
Project Structure After Setup
my-app/ ├── public/ # Static files (favicon, etc.) │ └── vite.svg ├── src/ │ ├── assets/ # Images, fonts │ ├── App.css # App styles │ ├── App.jsx # Root component │ ├── index.css # Global styles │ └── main.jsx # Entry point (createRoot) ├── .gitignore ├── index.html # HTML entry (in root, not public!) ├── package.json └── vite.config.js # Vite configuration
vite.config.js — Common Configuration
import { defineConfig } from 'vite' import react from '@vitejs/plugin-react' export default defineConfig({ plugins: [react()], server: { port: 3000, // change default port open: true, // auto-open browser }, resolve: { alias: { '@': '/src', // path alias: import from '@/components/...' }, }, build: { outDir: 'dist', sourcemap: true, }, })
Pros
  • Cold start in <500ms (vs 30s+ for CRA)
  • Instant Hot Module Replacement (HMR)
  • Uses native ES modules — no bundling in dev
  • Small, optimized production builds (Rollup)
  • TypeScript support out of the box
  • Official React docs now recommend Vite
Cons
  • Newer — fewer Stack Overflow answers than CRA
  • Requires Node 18+
  • Some CRA-specific tutorials don't apply
Performance stat: Vite starts a fresh React project in ~300ms. Create React App takes 10-30 seconds. For large projects the difference is even more dramatic.

📦 Create React App

Legacy / Deprecated

The original zero-config React toolchain by Facebook. No longer recommended by the React team as of 2023.

Commands
# Create new CRA project: npx create-react-app my-app # With TypeScript template: npx create-react-app my-app --template typescript cd my-app npm start # Dev server on port 3000 npm run build # Production build npm test # Run Jest tests npm run eject # ⚠️ Irreversible! Exposes all config
Project Structure
my-app/ ├── public/ │ ├── index.html # HTML template (in public/) │ └── favicon.ico ├── src/ │ ├── App.js │ ├── App.test.js │ ├── index.js # Entry point │ └── setupTests.js ├── package.json └── README.md
Pros
  • Official Facebook toolchain (historically)
  • Huge number of tutorials available
  • Zero configuration to get started
  • Jest + Testing Library included
Cons
  • Extremely slow — 10-30s startup, 2s+ HMR
  • Very large node_modules (~300MB)
  • Webpack under the hood — outdated
  • Not maintained — deprecated by React team
  • Eject is irreversible and messy
Warning: The React documentation (react.dev) no longer recommends CRA. New projects should use Vite or a full framework like Next.js. Use CRA only if following an older tutorial.

🔺 Next.js — Full-Stack React Framework

Production Ready

The most popular React framework. Built by Vercel. Adds SSR, SSG, file-based routing, API routes, and more on top of React.

Quick Start
# Create Next.js app (interactive setup): npx create-next-app@latest my-app # The CLI asks you: # Would you like to use TypeScript? (Yes) # Would you like to use ESLint? (Yes) # Would you like to use Tailwind CSS? (Yes) # Would you like to use App Router? (Yes — recommended) # Would you like to use src/ directory? (Yes) cd my-app npm run dev # Dev server on port 3000 npm run build # Production build npm start # Start production server
App Router Structure (Next.js 13+)
my-app/ ├── src/ │ └── app/ │ ├── layout.tsx # Root layout (replaces _app.js) │ ├── page.tsx # Home page → route "/" │ ├── globals.css │ ├── about/ │ │ └── page.tsx # Route "/about" │ ├── blog/ │ │ ├── page.tsx # Route "/blog" │ │ └── [slug]/ │ │ └── page.tsx # Dynamic route "/blog/:slug" │ └── api/ │ └── users/ │ └── route.ts # API endpoint "/api/users" ├── public/ └── next.config.js
Pros
  • SSR/SSG built-in — great SEO
  • File-based routing — no React Router needed
  • API routes — backend in the same repo
  • Server Components — less JS to the client
  • Image/Font optimization built-in
  • Deploy to Vercel in one click
Cons
  • More complex — client vs server components
  • Opinionated structure
  • Slower local dev than Vite (but improving)
  • Overkill for simple SPAs
When to use Next.js: E-commerce, blogs, marketing sites (need SEO), full-stack apps, or any production app that needs more than just a SPA. If you're building a React app that users will find via Google search, use Next.js.

📍 Parcel — Zero-Config Bundler

Parcel is a zero-configuration bundler. No webpack.config.js, no babel.config.js — it figures everything out automatically from your files.

Setup Steps
# Initialize project: npm init -y # Install Parcel and React: npm install react react-dom npm install -D parcel # Create src/index.html: # <!DOCTYPE html> # <html><body> # <div id="root"></div> # <script type="module" src="./index.jsx"></script> # </body></html> # Add to package.json scripts: # "start": "parcel src/index.html", # "build": "parcel build src/index.html" npm start # That's it! Parcel detects JSX automatically
Pros
  • True zero-config — just works
  • Fast build times with caching
  • Great for quick prototypes
  • Auto-detects JSX, TypeScript, CSS Modules
Cons
  • Less popular — fewer resources
  • Less control over bundling config
  • Not used in most enterprise projects

🔧 Manual Setup — Learn the Internals

Setting up React manually teaches you exactly what Vite and CRA do under the hood. Recommended once as a learning exercise.

1
Initialize npm project
npm init -y
2
Install React and ReactDOM
npm install react react-dom
3
Install Babel (JSX transpiler) and Webpack (bundler)
npm install -D @babel/core @babel/preset-react @babel/preset-env babel-loader webpack webpack-cli webpack-dev-server html-webpack-plugin
4
Create .babelrc
{ "presets": [ "@babel/preset-env", ["@babel/preset-react", { "runtime": "automatic" }] ] }
5
Create webpack.config.js
const HtmlWebpackPlugin = require('html-webpack-plugin'); const path = require('path'); module.exports = { entry: './src/index.js', output: { path: path.resolve(__dirname, 'dist'), filename: 'bundle.js', }, module: { rules: [{ test: /\.(js|jsx)$/, exclude: /node_modules/, use: 'babel-loader', }], }, plugins: [ new HtmlWebpackPlugin({ template: './public/index.html' }) ], devServer: { port: 3000, hot: true }, };
6
Create entry files
// public/index.html <!DOCTYPE html> <html><body> <div id="root"></div> </body></html> // src/index.js import { createRoot } from 'react-dom/client'; import App from './App'; const root = createRoot(document.getElementById('root')); root.render(<App />); // src/App.jsx export default function App() { return <h1>Hello from manual setup!</h1>; }
7
Add scripts to package.json and run
// package.json scripts: "start": "webpack serve --mode development", "build": "webpack --mode production" # Run: npm start
Learning Value: Doing this once shows you exactly what Vite and CRA abstract away. You'll understand why Babel exists, what Webpack does, and why build tools were created. After this exercise, Vite's magic makes complete sense.

Performance Comparison

Tool Cold Start HMR Speed Bundle Size Config Best For
⚡ Vite Recommended <500ms Instant Small Minimal All new projects
📦 CRA Deprecated 10-30s ~2s Large Zero Old tutorials only
🔺 Next.js 2-5s Fast Medium Moderate Production apps, SEO
📍 Parcel Fast Fast Small Zero Quick prototypes
🔧 Manual Webpack Depends Manual Depends Full control Learning internals
Recommendation: Use Vite for SPAs, dashboards, and learning React. Use Next.js for production apps needing SEO, SSR, or full-stack capabilities. Never start a new project with CRA.

Common First-Time Setup Mistakes

Using Create React App for new projects in 2024

CRA is deprecated. Starting a new project with CRA means starting with a slow, outdated toolchain that the React team no longer recommends.

Fix: Use npm create vite@latest my-app -- --template react instead.
Forgetting to run npm install after cloning

node_modules/ is git-ignored. After cloning a React project, you must run npm install before npm run dev.

Fix: Always run npm install (or yarn / pnpm install) after cloning.
Committing node_modules/ to git

node_modules can contain 100,000+ files and 300MB+. Committing it bloats the repository and causes merge conflicts.

Fix: Ensure node_modules is in your .gitignore. Vite and CRA add this automatically.
Committing .env files with secrets

API keys, database credentials, and other secrets should never be committed to git.

Fix: Add .env to .gitignore. Use .env.example (with fake values) to show teammates what variables are needed.
Using wrong Node.js version

React 18 + Vite requires Node.js 18+. Using an older version causes cryptic errors.

Fix: Use node --version to check. Use nvm (Node Version Manager) to switch versions: nvm use 20.

Environment Variables in React

Environment variables let you have different configs for development, staging, and production without changing code. In Vite, variables must be prefixed with VITE_ to be exposed to the browser.
Vite (.env files)
# .env (all environments) VITE_API_URL=https://api.example.com VITE_APP_NAME=My App # .env.development (dev only) VITE_API_URL=http://localhost:3001 # .env.production (prod only) VITE_API_URL=https://api.production.com # Accessing in code: const apiUrl = import.meta.env.VITE_API_URL; const appName = import.meta.env.VITE_APP_NAME; # .env.example (commit this — no real secrets!) VITE_API_URL= VITE_APP_NAME=
CRA / Next.js (.env files)
# CRA — prefix with REACT_APP_ REACT_APP_API_URL=https://api.example.com REACT_APP_KEY=your-key # Access in CRA: process.env.REACT_APP_API_URL # Next.js — NEXT_PUBLIC_ for browser access NEXT_PUBLIC_API_URL=https://api.example.com DATABASE_URL=private-server-only # NOT exposed to browser # Access in Next.js: process.env.NEXT_PUBLIC_API_URL
Security: NEVER put secrets (API keys, private keys, passwords) in VITE_ or REACT_APP_ vars — they get bundled into the client-side JS and are visible to anyone who views the page source!
