Section 29 React 18+

React Features Guide

Deep dive into React 18+ features — concurrent rendering, automatic batching, server components, and the React Compiler.

1

Concurrent Features

React 18

Concurrent Mode lets React interrupt, pause, and resume renders — making apps more responsive.

useTransition — Non-Urgent Updates

Mark state updates as "non-urgent" so React can interrupt them if a more urgent update arrives (like user input).

import { useState, useTransition } from 'react'; function SearchPage() { const [inputValue, setInputValue] = useState(''); const [searchQuery, setSearchQuery] = useState(''); const [isPending, startTransition] = useTransition(); const handleSearch = (e) => { const q = e.target.value; setInputValue(q); // Urgent — updates input immediately startTransition(() => { setSearchQuery(q); // Non-urgent — can be interrupted }); }; return ( <> <input value={inputValue} onChange={handleSearch} /> {isPending && <Spinner />} <SearchResults query={searchQuery} /> // Heavy re-render </> ); }
isPending is true while the transition is in progress — use it to show a loading indicator without blocking the input.
useDeferredValue — Defer Expensive Renders

Similar to useTransition but for values you don't control (like props). React renders with the previous value while processing the new one.

import { useDeferredValue, useMemo } from 'react'; function FilteredList({ items, searchQuery }) { // deferredQuery lags behind searchQuery // UI stays responsive while filtering runs const deferredQuery = useDeferredValue(searchQuery); const results = useMemo( () => filterItems(items, deferredQuery), [items, deferredQuery] ); // Show stale-content indicator while deferred const isStale = deferredQuery !== searchQuery; return ( <ul style={{ opacity: isStale ? 0.7 : 1 }}> {results.map(item => ( <li key={item.id}>{item.name}</li> ))} </ul> ); }
useTransition vs useDeferredValue
  • useTransition: you control the state update — wrap with startTransition
  • useDeferredValue: you only have a value (from props/API) — wrap the value
2

Automatic Batching

React 18

React now batches ALL state updates together — even inside timeouts, promises, and native event handlers.

Before React 18
// Inside React event handlers: batched ✓ function onClick() { setCount(c => c + 1); // } single re-render setFlag(f => !f); // } } // But inside setTimeout: NOT batched ✗ setTimeout(() => { setCount(c => c + 1); // re-render 1 ✗ setFlag(f => !f); // re-render 2 ✗ }, 0); // Also NOT batched in promises: fetch('/api').then(() => { setData(d); // re-render 1 ✗ setLoading(false); // re-render 2 ✗ });
React 18 — Always Batched
// All contexts now batch automatically ✓ setTimeout(() => { setCount(c => c + 1); // } single re-render setFlag(f => !f); // } }, 0); fetch('/api').then(() => { setData(d); // } single re-render setLoading(false); // } }); // Opt out if needed with flushSync: import { flushSync } from 'react-dom'; flushSync(() => setCount(c => c + 1)); // immediate re-render flushSync(() => setFlag(f => !f)); // another re-render
Automatic batching is enabled by switching from ReactDOM.render() to ReactDOM.createRoot(). Apps still using the legacy render API don't get automatic batching.
3

New Root API (createRoot)

React 18

The entry point to enable all React 18 concurrent features.

// Old (React 17) — legacy blocking mode, no concurrent features import ReactDOM from 'react-dom'; ReactDOM.render(<App />, document.getElementById('root')); // New (React 18) — enables concurrent rendering, auto batching, Suspense data import { createRoot } from 'react-dom/client'; const root = createRoot(document.getElementById('root')!); root.render( <React.StrictMode> <App /> </React.StrictMode> ); // For Server-Side Rendering: hydrateRoot import { hydrateRoot } from 'react-dom/client'; hydrateRoot( document.getElementById('root')!, <App /> );

Suspense for Data Fetching

React 18

Suspense is no longer just for code splitting — React 18 fully enables Suspense for async data with frameworks.

// Suspense boundary — shows fallback while children are "suspended" function App() { return ( <Suspense fallback={<PageSkeleton />}> <UserProfile userId={1} /> </Suspense> ); } // Nested Suspense — different loading states per section function Dashboard() { return ( <> <Suspense fallback={<HeaderSkeleton />}> <Header /> </Suspense> <Suspense fallback={<ContentSkeleton />}> <MainContent /> <Suspense fallback={<SidebarSkeleton />}> <Sidebar /> </Suspense> </Suspense> </> ); } // With React Query (suspense mode) function UserProfile({ userId }) { const { data: user } = useSuspenseQuery({ queryKey: ['user', userId], queryFn: () => userApi.getById(userId), }); return <div><h1>{user.name}</h1></div>; // No loading/error check needed — handled by Suspense + ErrorBoundary }

useId — Stable IDs for SSR

React 18

Generate stable, unique IDs that match between server and client renders — critical for SSR hydration.

import { useId } from 'react'; // Accessible form fields — label htmlFor must match input id function FormField({ label, type = 'text' }) { const id = useId(); // stable: ':r0:', ':r1:', etc. return ( <div> <label htmlFor={id}>{label}</label> <input id={id} type={type} /> </div> ); } // Multiple IDs from one useId call function PasswordField() { const id = useId(); return ( <> <label htmlFor={id + '-input'}>Password</label> <input id={id + '-input'} type="password" aria-describedby={id + '-hint'} /> <p id={id + '-hint'}>Must be 8+ characters</p> </> ); }
Why Not Math.random() or a Counter?
  • Math.random() — different on server vs client → hydration mismatch error
  • Global counter — breaks with Concurrent Mode (renders can be paused/replayed)
  • useId — deterministic based on component tree position, always matches between SSR and hydration
Use Cases
  • label + input pairs (htmlFor/id)
  • aria-labelledby, aria-describedby
  • SVG filter/mask IDs (must be unique per page)
  • Accordion/tab panel IDs for ARIA

useSyncExternalStore

React 18

The official hook for subscribing to external stores (used internally by Redux, Zustand, and other state libraries).

import { useSyncExternalStore } from 'react'; // Subscribe to browser window width — updates on resize function useWindowWidth() { return useSyncExternalStore( // subscribe: called when React needs to subscribe to the store (callback) => { window.addEventListener('resize', callback); return () => window.removeEventListener('resize', callback); }, // getSnapshot: return the current value () => window.innerWidth, // getServerSnapshot: SSR fallback (optional) () => 1200 ); } // Subscribe to online/offline status function useOnlineStatus() { return useSyncExternalStore( (callback) => { window.addEventListener('online', callback); window.addEventListener('offline', callback); return () => { window.removeEventListener('online', callback); window.removeEventListener('offline', callback); }; }, () => navigator.onLine, () => true ); } // Usage in components function ResponsiveComponent() { const width = useWindowWidth(); const isOnline = useOnlineStatus(); return ( <div> {!isOnline && <p>You are offline</p>} {width < 768 ? <MobileNav /> : <DesktopNav />} </div> ); }

Server Components (RSC)

React 19 / Next.js 13+

Server Components run only on the server — zero JS sent to the browser. Game-changer for performance and data access.

Server Component
  • Runs on the server only
  • No JS sent to the browser
  • Can directly access DB, files, secrets
  • Cannot use useState, useEffect, event handlers
  • Default in Next.js 13+ App Router
  • Can be async (await data directly)
Client Component
  • Runs in browser (and during SSR)
  • JS bundle included
  • Can use useState, useEffect, events
  • Cannot access server-only resources
  • Marked with 'use client' directive
  • Receives props from server components
// ── Server Component — no 'use client', no JS sent to browser ────── async function UserProfile({ userId }: { userId: number }) { // Direct DB access — no API needed! const user = await db.user.findUnique({ where: { id: userId } }); const posts = await db.post.findMany({ where: { authorId: userId } }); return ( <div> <h1>{user.name}</h1> <p>{user.email}</p> {posts.map(post => <PostCard key={post.id} post={post} />)} <LikeButton userId={userId} /> // Client Component embedded here </div> ); } // ── Client Component — marked with 'use client' directive ─────────── 'use client'; import { useState } from 'react'; function LikeButton({ userId }: { userId: number }) { const [liked, setLiked] = useState(false); return ( <button onClick={() => setLiked(l => !l)}> {liked ? '♥ Liked' : '♡ Like'} </button> ); } // ── Server Actions — async functions that run on the server ───────── 'use server'; async function createPost(formData: FormData) { const title = formData.get('title'); await db.post.create({ data: { title } }); revalidatePath('/posts'); }
Server Component Mental Model
  • Think of the server/client boundary as a "client frontier" — mark components as client when they need interactivity
  • Push the boundary as far down the tree as possible — keep most components as server components
  • Server components can import and render client components, but NOT vice versa
  • Pass server data to client components as props — they can't access DB directly

React Compiler (React Forget)

React 19

Automatic memoization — the compiler rewrites your components to add optimal memoization. No more manual useMemo/useCallback in most cases.

Before Compiler (Manual Memoization)
// You write this: function ExpensiveList({ items, filter, onSelect }) { const filtered = useMemo( () => items.filter(i => i.type === filter), [items, filter] ); const handleSelect = useCallback( (id) => onSelect(id), [onSelect] ); return filtered.map(i => <Item key={i.id} item={i} onSelect={handleSelect} /> ); } const Item = React.memo(({ item, onSelect }) => ( <div onClick={() => onSelect(item.id)}>{item.name}</div> ));
With React Compiler
// You write this simpler version: function ExpensiveList({ items, filter, onSelect }) { const filtered = items.filter(i => i.type === filter); return filtered.map(i => <Item key={i.id} item={i} onSelect={onSelect} /> ); } function Item({ item, onSelect }) { return <div onClick={() => onSelect(item.id)}>{item.name}</div>; } // Compiler automatically adds optimal memoization // No useMemo, useCallback, or React.memo needed!
The React Compiler understands the "Rules of React" (pure functions, no mutation) and automatically memoizes only where beneficial. It requires writing idiomatic React — which you should already be doing.

Interview Questions

1. What is Concurrent Mode and how does it improve user experience?
Concurrent Mode lets React work on multiple state updates simultaneously and interrupt low-priority renders if a high-priority update arrives. Previously, React processed each render synchronously — a heavy re-render would block user input. With Concurrent Mode (enabled via createRoot), React can pause an expensive render, process a user keystroke immediately, then resume. useTransition and useDeferredValue give you control over what's "urgent" vs "non-urgent".
2. Explain automatic batching in React 18. How does it differ from React 17?
React 17 only batched state updates inside React event handlers. Updates in setTimeout, Promises, and native event handlers each caused separate re-renders. React 18 batches ALL state updates everywhere automatically, reducing re-renders. This is enabled by createRoot. Use flushSync() to opt out when you need immediate re-renders (rare — e.g., reading DOM measurements between state updates).
3. What are React Server Components and how do they differ from SSR?
SSR renders components on the server and sends HTML + JS to the browser (JS then hydrates the page). Server Components run ONLY on the server — their output (serialized React tree) is sent to the client, but no JS is shipped for them. They can directly access databases and files. They cannot use useState, useEffect, or browser APIs. They are composable with Client Components ('use client') which handle interactivity.
4. What problem does useId solve? Why not use Math.random()?
useId generates IDs that are stable between server and client renders. Math.random() generates different values each time — so the ID generated on the server won't match what the client generates during hydration, causing a hydration mismatch error. useId derives its value from the component's position in the tree, which is deterministic on both server and client.
5. What is the React Compiler and why is it significant?
The React Compiler (formerly "React Forget") is a build-time compiler that automatically adds optimal memoization to React components. It analyzes component code and inserts useMemo/useCallback/React.memo only where they provide measurable benefit. This means developers no longer need to manually reason about memoization — the compiler handles it correctly. It requires components to follow the Rules of React (pure render functions, no mutation).
6. When would you use useTransition vs useDeferredValue?
Use useTransition when you own the state update — wrap the setter in startTransition(). Use useDeferredValue when you receive a value via props or from an external source you don't control. Both defer expensive re-renders, but the API differs: useTransition wraps the update, useDeferredValue wraps the received value. useDeferredValue is also useful inside components that can't call startTransition (e.g., component libraries).
