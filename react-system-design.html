System Design for React

How to approach frontend system design interviews — frameworks, real-world case studies, and performance budgets.

System Design Interview Framework

Use this 6-step structure for every frontend system design interview. Speak through each step explicitly.

1
Requirements Gathering
Functional Requirements
  • What are the core user flows?
  • Who are the users? (scale matters)
  • What devices/browsers must be supported?
  • Real-time features needed?
  • Offline support needed?
Non-Functional Requirements
  • Performance targets (LCP, FID, TTI)
  • Availability (99.9% uptime?)
  • Accessibility (WCAG 2.1 AA?)
  • Security (auth, XSS, CSRF)
  • SEO requirements?
2
Capacity Estimation
Think about scale from the start. Numbers drive architecture decisions.
Users
DAU, MAU, peak concurrent users
Requests/sec
API calls per user session, WebSocket connections
Data Volume
Bundle size budget, images/media, real-time event frequency
3
High-Level Design
Draw the component tree, data flow, and system boundaries.
  • Choose rendering strategy: CSR, SSR, SSG, ISR, or hybrid
  • Identify major sections / page regions
  • Map user flows to component hierarchies
  • Identify shared state vs local state
4
Component Design (Deep Dive)
Pick 2-3 critical components and design them in detail.
  • Props interface / TypeScript types
  • Internal state and side effects
  • Child component breakdown
  • Performance considerations (memoization, virtualization)
5
Performance & Scalability
How does it hold up under load and on slow connections?
  • Code splitting and lazy loading strategy
  • CDN for static assets (images, fonts, JS chunks)
  • Caching: React Query stale time, HTTP cache headers
  • Virtualization for large lists
  • Service Worker / offline support if needed
6
Trade-offs Discussion
No design is perfect. Discuss what you'd change with different constraints.
  • CSR vs SSR vs SSG — SEO vs simplicity vs cost
  • REST vs GraphQL vs WebSocket — use case fit
  • Redux vs React Query — complexity vs capability
  • Monorepo vs separate repos — coordination overhead
Case Study 1

Real-time Chat Application

Functional Requirements
  • Send and receive messages in real time
  • Group conversations and DMs
  • Typing indicators
  • Online/offline presence
  • Message history with infinite scroll
  • File/image sharing
Technical Decisions
  • WebSocket (Socket.IO) for real-time events
  • REST for initial data load and history
  • Normalized Redux state for messages
  • Optimistic updates for sent messages
  • Virtual scroll for message thread
  • Debounced typing indicator events
Frontend Architecture (Component Tree)
ChatApp ├── AuthProvider JWT context — user identity ├── SocketProvider WebSocket singleton — single connection shared app-wide ├── ConversationList Virtualized list, sorted by lastMessage.createdAt │ ├── ConversationItem Optimistic unread count, last message preview │ └── ConversationSearch Debounced local search ├── MessageThread │ ├── VirtualizedMessages react-virtual, batch renders on scroll │ ├── MessageBubble Memo'd, only re-renders on edit/delete │ ├── MessageInput Draft persistence in localStorage │ └── TypingIndicator Debounced WebSocket events, auto-timeout 3s └── UserPresence Online/offline/away via heartbeat ping
State Shape (Normalized)
// Normalized state — avoid nested arrays, use byId/allIds pattern { conversations: { byId: { 'conv_1': { id, name, lastMessage, unreadCount } }, allIds: ['conv_1', 'conv_2'], activeId: 'conv_1' }, messages: { 'conv_1': [{ id, senderId, body, createdAt, status }], 'conv_2': [...] }, presences: { 'user_42': 'online', 'user_99': 'typing', 'user_7': 'offline' }, drafts: { 'conv_1': 'Hey, did you see...' } }
WebSocket Provider
function SocketProvider({ children }) { const socketRef = useRef(null); const dispatch = useDispatch(); useEffect(() => { socketRef.current = io(SOCKET_URL, { auth: { token: getToken() } }); socketRef.current.on('message:new', (msg) => { dispatch(addMessage(msg)); dispatch(updateConversationLastMessage(msg)); }); socketRef.current.on('user:typing', ({ userId, conversationId }) => { dispatch(setTyping({ userId, conversationId })); }); socketRef.current.on('user:presence', ({ userId, status }) => { dispatch(updatePresence({ userId, status })); }); return () => socketRef.current?.disconnect(); }, []); return <SocketContext.Provider value={socketRef.current}>{children}</SocketContext.Provider>; }
Key Performance Considerations
  • Virtual scroll for message thread — never render 10,000 messages at once
  • Debounce typing events — emit at most every 300ms, auto-clear after 3s silence
  • Optimistic sends — show message immediately, mark with "sending" status, confirm on ACK
  • Batch Redux updates — use unstable_batchedUpdates for multiple dispatches in one WebSocket event
  • Singleton WebSocket — one connection per app, not one per component

Key Design Decisions
  • Cart state: localStorage + server sync. Optimistic updates keep UI fast. Reconcile on login.
  • Search: Debounce 300ms. Faceted filters in URL params. Cache recent queries in React Query.
  • Checkout: Multi-step wizard with React Hook Form. Each step validates before progression.
  • Images: WebP format, CDN delivery, lazy loading, responsive srcset.
  • Product catalog: Algolia or ElasticSearch for faceted search at scale.
  • Payment: Stripe Elements — never handle raw card data client-side.

Web Performance Budgets

Define performance targets up front. These are the Core Web Vitals and key metrics used by Google for SEO ranking.

Metric Good Needs Work Poor Measurement Tool
First Contentful Paint (FCP) < 1.8s 1.8–3.0s > 3.0s Lighthouse, PageSpeed Insights
Largest Contentful Paint (LCP) < 2.5s 2.5–4.0s > 4.0s Lighthouse, CrUX, Search Console
First Input Delay (FID) < 100ms 100–300ms > 300ms Chrome UX Report (CrUX)
Interaction to Next Paint (INP) < 200ms 200–500ms > 500ms Lighthouse, PerformanceObserver
Cumulative Layout Shift (CLS) < 0.1 0.1–0.25 > 0.25 Lighthouse, Chrome DevTools
Time to Interactive (TTI) < 3.8s 3.8–7.3s > 7.3s Lighthouse
Total Blocking Time (TBT) < 200ms 200–600ms > 600ms Lighthouse
JS Bundle Size (gzipped) < 200KB 200–500KB > 500KB webpack-bundle-analyzer, bundlesize
Core Web Vitals (LCP, INP, CLS) directly affect Google Search rankings. Prioritize these for any SEO-important pages. Monitor with PageSpeed Insights and the Chrome UX Report.

Interview Questions

1. How would you design a real-time collaborative document editor (like Google Docs)?
Key challenges: conflict resolution with CRDT (Conflict-free Replicated Data Types) or OT (Operational Transformation), real-time presence via WebSocket, cursor positions for each collaborator, autosave with debounce, undo/redo history, and offline support with sync-on-reconnect. Use Yjs or Automerge for the CRDT layer, Tiptap or ProseMirror for the editor.
2. How does ISR (Incremental Static Regeneration) work in Next.js and when would you use it?
ISR pre-renders pages at build time (like SSG) but regenerates them in the background on a schedule. Set revalidate: 60 to rebuild every 60 seconds. The first request after expiry gets the stale page and triggers a rebuild; subsequent requests get the new page. Best for product pages, blog posts, dashboards — content that changes but doesn't need real-time freshness.
3. Explain optimistic updates. How do you handle rollback if the server rejects the update?
Optimistic updates apply state changes immediately before server confirmation, making the UI feel instant. In React Query, use onMutate to update the cache and capture the previous state. On onError, restore context.previousData. On onSettled, invalidate the query to sync with the server's ground truth.
4. How do you handle WebSocket reconnection and state resync after a disconnect?
Socket.IO handles reconnection automatically with exponential backoff. On reconnect: (1) re-authenticate with JWT, (2) fetch missed events since last cursor/timestamp via REST, (3) rehydrate local state with the delta. Keep a sequence number or last-event timestamp to detect gaps. Show a "Reconnecting..." UI during the gap to avoid displaying stale data.
5. How would you implement infinite scroll vs pagination? When to choose each?
Use cursor-based infinite scroll for feeds (social, chat history) — better UX for browsing. Use page-number pagination for search results, tables, and data grids — users need to jump to specific pages. Implement infinite scroll with Intersection Observer on a sentinel element at the bottom. Use TanStack Virtual for large virtualized lists to avoid DOM bloat.
6. What are Core Web Vitals and how would you improve LCP on a React app?
LCP measures when the largest above-the-fold element renders. To improve: (1) SSR/SSG for above-the-fold HTML, (2) CDN-serve images in WebP, (3) preload the hero image with <link rel="preload">, (4) eliminate render-blocking resources (defer non-critical JS/CSS), (5) reduce server response time with caching, (6) avoid lazy loading above-the-fold images.
7. How do you approach security in a React frontend? (XSS, CSRF, auth)
XSS: React escapes JSX by default — never use dangerouslySetInnerHTML with user input. Sanitize with DOMPurify if needed. CSRF: Use httpOnly cookies for tokens (can't be read by JS) + SameSite=Strict. Store JWTs in memory (not localStorage). HTTPS everywhere. CSP headers to restrict script sources. Validate and sanitize on the server too — frontend validation is UX, not security.
8. Design the state management architecture for a multi-tenant SaaS dashboard.
Separate concerns: tenant config in Context (theme, feature flags, permissions), server data in React Query (per tenant queryKey prefix), user session in a small Redux slice or Zustand, form state in React Hook Form. Per-tenant API scoping: inject tenantId into Axios interceptor. Role-based rendering: AuthGuard HOC checks permissions from JWT claims before rendering sensitive UI.
