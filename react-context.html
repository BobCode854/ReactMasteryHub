Context API

createContext, custom providers, useReducer pattern, performance gotchas, and Context vs Redux

Context Deep Dive

Context provides a way to pass data through the component tree without prop-drilling. The pattern: create a context, wrap the tree in a Provider, consume via a custom hook.

Four-step pattern: 1. createContext with default, 2. Provider component with state, 3. Custom hook with guard, 4. Consume the hook anywhere in the tree. 
// 1. Create context with typed default value
const AuthContext = createContext({
  user: null,
  login: () => {},
  logout: () => {},
});

// 2. Custom Provider with state + logic
export function AuthProvider({ children }) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    const token = localStorage.getItem('token');
    if (token) {
      verifyToken(token).then(setUser).finally(() => setLoading(false));
    } else {
      setLoading(false);
    }
  }, []);

  const login = async (credentials) => {
    const userData = await loginAPI(credentials);
    localStorage.setItem('token', userData.token);
    setUser(userData.user);
  };

  const logout = () => {
    localStorage.removeItem('token');
    setUser(null);
  };

  if (loading) return <GlobalSpinner />;

  return (
    <AuthContext.Provider value={{ user, login, logout }}>
      {children}
    </AuthContext.Provider>
  );
}

// 3. Custom hook — clean API + safety guard
export function useAuth() {
  const ctx = useContext(AuthContext);
  if (!ctx) throw new Error('useAuth must be inside AuthProvider');
  return ctx;
}

// 4. Usage — anywhere inside the Provider tree
function ProfileButton() {
  const { user, logout } = useAuth();
  return user ? (
    <button onClick={logout}>{user.name} — Logout</button>
  ) : (
    <Link to="/login">Login</Link>
  );
}

Composing Multiple Contexts

Large apps typically have multiple contexts. Compose them in a single AppProviders component to keep main.tsx clean and make it easy to add or reorder providers.

// Compose all providers at app root
function AppProviders({ children }) {
  return (
    <AuthProvider>
      <ThemeProvider>
        <CartProvider>
          <NotificationProvider>
            {children}
          </NotificationProvider>
        </CartProvider>
      </ThemeProvider>
    </AuthProvider>
  );
}

// main.tsx — clean entry point
root.render(
  <AppProviders>
    <App />
  </AppProviders>
);
Order matters: Providers listed later are nested deeper — they can consume providers listed earlier. Put foundational providers (Auth, Theme) outermost.

Context + useReducer (Mini-Redux)

Combining Context with useReducer gives you Redux-like state management without the library overhead — great for feature-level or medium-complexity state.

const CartContext = createContext();

function cartReducer(state, action) {
  switch (action.type) {
    case 'ADD':
      return { ...state, items: [...state.items, action.item] };
    case 'REMOVE':
      return { ...state, items: state.items.filter(i => i.id !== action.id) };
    case 'CLEAR':
      return { items: [], total: 0 };
    default:
      return state;
  }
}

export function CartProvider({ children }) {
  const [state, dispatch] = useReducer(cartReducer, { items: [], total: 0 });
  return (
    <CartContext.Provider value={{ state, dispatch }}>
      {children}
    </CartContext.Provider>
  );
}

export const useCart = () => useContext(CartContext);

// Usage
function CartButton({ product }) {
  const { dispatch } = useCart();
  return (
    <button onClick={() => dispatch({ type: 'ADD', item: product })}>
      Add to Cart
    </button>
  );
}

Context vs Redux

Feature Context API Redux Toolkit
Setup Zero config, built-in Install + configure store
Boilerplate Low Low (RTK greatly reduces it)
Performance Re-renders all consumers Selective re-renders via selectors
DevTools React DevTools only Redux DevTools (time-travel, diffs)
Middleware No built-in support Thunks, saga, listener middleware
Best for Auth, theme, locale, low-change global state Complex app-wide state, frequent updates
Learning curve Low Medium

Performance Gotcha: Context Re-renders All Consumers

The problem: Every time the context value changes, all components that call useContext for that context re-render — even if the specific value they use hasn't changed. 
// Problem: user or theme change triggers ALL consumers to re-render
const AppContext = createContext();
function AppProvider({ children }) {
  const [user, setUser] = useState(null);
  const [theme, setTheme] = useState('light');
  const [cart, setCart] = useState([]);

  // Changing cart re-renders ALL components using this context!
  return <AppContext.Provider value={{ user, theme, cart, setCart }}>{children}</AppContext.Provider>;
}

// Solution: split into separate focused contexts
function AppProviders({ children }) {
  return (
    <AuthProvider>       // only re-renders auth consumers
      <ThemeProvider>    // only re-renders theme consumers
        <CartProvider>   // only re-renders cart consumers
          {children}
        </CartProvider>
      </ThemeProvider>
    </AuthProvider>
  );
}
Tip: For high-frequency updates (form input, cursor position, real-time data), Context is the wrong tool. Use Zustand, Jotai, or Redux with selectors which support granular subscriptions.

Interview Questions

What is prop drilling and how does Context solve it?
Prop drilling is passing props through multiple intermediate components that don't use them — only to get data to a deeply nested child. Context solves this by making data available to any component in the subtree via useContext, without explicitly passing props through each level.
Why should you create a custom hook for context instead of using useContext directly?
A custom hook (e.g., useAuth()) provides three benefits: 1. It throws a meaningful error if used outside the Provider, catching bugs early. 2. It hides the context import — consumers don't need to know which context object is used. 3. It allows adding derived values or utilities without changing the context shape.
What is the performance issue with Context and how do you fix it?
When a Context value changes, every component subscribed via useContext re-renders regardless of which part of the value changed. Fix: split one large context into multiple smaller focused contexts so only the relevant consumers re-render. Alternative: use useMemo to stabilise the context value object.
When would you choose Context over Redux?
Context is ideal for low-frequency global state: auth user, theme, locale, feature flags. It requires no extra dependency and has minimal setup. Choose Redux when: the app has complex state logic with many actions, you need DevTools for debugging, or multiple high-frequency state slices need selective subscriptions to avoid performance issues.
How does useReducer with Context compare to Redux?
The combination provides the same conceptual pattern (actions, reducer, dispatch) without the Redux library. However it lacks Redux DevTools, middleware support (thunks, sagas), and the performance optimisations of useSelector. It's a good choice for feature-level state in medium complexity apps but doesn't scale as well as Redux for large app-wide state.
What default value should you pass to createContext?
Pass a sensible typed default matching the shape of the real value. The default is only used when a component calls useContext outside of any Provider — typically a bug. By adding a guard in your custom hook (if (!ctx) throw new Error(...)), you can pass null or undefined as the default and catch misuse early in development.
