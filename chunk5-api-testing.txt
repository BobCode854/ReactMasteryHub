    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#api-q1">
      <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is React Query (TanStack Query)?
    </button>
  </h2>
  <div id="api-q1" class="accordion-collapse collapse" data-bs-parent="#acc-api">
    <div class="accordion-body">
      React Query (now TanStack Query) is a data-fetching and server-state management library for React. It handles all the complexity of fetching, caching, synchronizing, and updating server state. Unlike Redux which manages client state, React Query manages server state — data that lives on the server and needs to be fetched, cached, and kept fresh. Key features: automatic caching with configurable stale times, background refetching, query invalidation, optimistic updates, infinite queries, and request deduplication. It eliminates the need for useEffect-based fetching patterns and dramatically reduces boilerplate.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#api-q2">
      <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is the difference between server state and client state?
    </button>
  </h2>
  <div id="api-q2" class="accordion-collapse collapse" data-bs-parent="#acc-api">
    <div class="accordion-body">
      Server state is data that lives on the server — it's asynchronous, can become stale, needs to be fetched, and may be modified by other users (product inventory, user profiles, posts). Client state is local UI state that exists only in the browser — synchronous, always up-to-date, owned entirely by the client (modal open/closed, form input values, selected tab). Most state management libraries (Redux, Zustand, Context) are designed for client state. React Query, SWR, and Apollo Client are designed for server state. The mistake of managing server state in Redux leads to complex async middleware, manual cache invalidation, and stale data problems that these dedicated libraries solve automatically.
    </div>
  </div>
</div>

  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#api-q3">
      <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is the useQuery hook?
    </button>
  </h2>
  <div id="api-q3" class="accordion-collapse collapse" data-bs-parent="#acc-api">
    <div class="accordion-body">
      <code>useQuery({ queryKey, queryFn })</code> fetches and caches data from an API. <code>queryKey</code> is an array that uniquely identifies the query — it's used for caching, deduplication, and invalidation. <code>queryFn</code> is an async function that fetches and returns the data. It returns <code>{ data, isLoading, isError, error, refetch, status }</code>. React Query automatically: fetches on mount, caches the result, refetches on window focus (configurable), deduplicates concurrent requests with the same key, and retries failed requests. The query key is like a cache key — changing any element in the array triggers a new fetch.
    </div>
  </div>
</div>

  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#api-q4">
      <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is useMutation in React Query?
    </button>
  </h2>
  <div id="api-q4" class="accordion-collapse collapse" data-bs-parent="#acc-api">
    <div class="accordion-body">
      <code>useMutation({ mutationFn, onSuccess, onError, onSettled })</code> manages write operations — creating, updating, or deleting data. Unlike <code>useQuery</code> (which runs automatically), mutations are triggered manually by calling <code>mutate(variables)</code>. It tracks <code>isLoading</code>, <code>isError</code>, <code>data</code>, and <code>error</code> state. Callbacks: <code>onSuccess(data, variables)</code> runs after success, <code>onError(error, variables)</code> after failure, <code>onSettled</code> after either. The most powerful pattern: <code>onSuccess</code> invalidates related queries (<code>queryClient.invalidateQueries(['posts'])</code>) so the UI reflects the mutation result. This is how React Query keeps server state synchronized after mutations.
    </div>
  </div>
</div>

  <div id="api-q5" class="accordion-collapse collapse" data-bs-parent="#acc-api">
    <div class="accordion-body">
      SWR (Stale-While-Revalidate) is Vercel's data-fetching library for React. The name comes from the HTTP cache-control strategy: return cached (stale) data immediately, then revalidate in the background. <code>useSWR(key, fetcher)</code> returns <code>{ data, error, isLoading, mutate }</code>. Key features: global cache, request deduplication, focus revalidation, interval polling, error retry, and optimistic updates via <code>mutate</code>. SWR is simpler and lighter than React Query but has fewer features (no mutations API, no query cancellation, less control over cache behavior). SWR is popular in Next.js projects (same creator — Vercel).

  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#api-q6">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;How does React Query's caching work?
    </button>
  </h2>
  <div id="api-q6" class="accordion-collapse collapse" data-bs-parent="#acc-api">
    <div class="accordion-body">
      React Query maintains an in-memory cache keyed by <code>queryKey</code>. When a query is fetched, the result is cached with a <code>staleTime</code> (default: 0 — immediately stale) and <code>gcTime</code> (formerly cacheTime, default: 5 minutes). <code>staleTime</code> controls how long cached data is considered fresh — during this period, components receive cached data without refetching. After staleTime, data is "stale" but still shown while a background refetch occurs. <code>gcTime</code> controls how long unused query data stays in cache before garbage collection. Understanding the distinction: fresh data is never refetched; stale data is shown immediately but refetched in the background; inactive data stays in cache for gcTime then is removed.
      <pre class="code-block">import { useQuery, QueryClient, QueryClientProvider } from '@tanstack/react-query';

const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 60 * 1000,       // data is fresh for 1 minute — no background refetch
      gcTime: 5 * 60 * 1000,      // cache kept for 5 minutes after component unmounts
      retry: 2,                   // retry failed requests twice
      refetchOnWindowFocus: true, // refetch when user returns to the tab
    },
  },
});

    <div class="accordion-body">
      <code>queryClient.invalidateQueries({ queryKey: ['posts'] })</code> marks matching queries as stale and triggers an immediate background refetch if the query is actively subscribed. Invalidation patterns: (1) Exact match: <code>{ queryKey: ['posts', postId], exact: true }</code>; (2) Prefix match: <code>{ queryKey: ['posts'] }</code> matches all queries starting with 'posts' (including ['posts', 1], ['posts', { status: 'published' }]); (3) Invalidate all: <code>queryClient.invalidateQueries()</code>. Best practice: after a mutation that creates/modifies a post, invalidate the posts list so it refetches with the latest data.
      <pre class="code-block">const queryClient = useQueryClient();

    method: 'POST',
    body: JSON.stringify(newPost)
  }).then(r =&gt; r.json()),
  onSuccess: (createdPost) =&gt; {
    // Invalidate the posts list (triggers background refetch)
    queryClient.invalidateQueries({ queryKey: ['posts'] });

    // Optionally pre-populate the cache for the individual post
    queryClient.setQueryData(['post', createdPost.id], createdPost);
  },
  onError: (error) =&gt; {
    console.error('Failed to create post:', error);
  },
});

      <pre class="code-block">function useInfinitePosts() {
  return useInfiniteQuery({
    queryKey: ['posts'],
    queryFn: ({ pageParam = 1 }) =&gt;
      fetch(`/api/posts?page=${pageParam}&amp;limit=10`).then(r =&gt; r.json()),
    getNextPageParam: (lastPage, allPages) =&gt;
      lastPage.hasMore ? allPages.length + 1 : undefined,
    initialPageParam: 1,
  });
}

    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#api-q9">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;How do you implement optimistic updates with React Query?
    </button>
  </h2>
  <div id="api-q9" class="accordion-collapse collapse" data-bs-parent="#acc-api">
    <div class="accordion-body">
      Optimistic updates in React Query update the cache immediately before the mutation completes, providing instant feedback. If the mutation fails, the previous cache value is restored. The pattern uses <code>onMutate</code> (to save the current cache and apply the optimistic update), <code>onError</code> (to rollback), and <code>onSettled</code> (to invalidate and refetch for the true server state). This gives users instant UI feedback while ensuring eventual consistency with the server.
      <pre class="code-block">const toggleLike = useMutation({
  mutationFn: (postId) =&gt;
    fetch(`/api/posts/${postId}/like`, { method: 'POST' }).then(r =&gt; r.json()),

  onMutate: async (postId) =&gt; {
    // Cancel outgoing refetches to avoid overwriting the optimistic update
    await queryClient.cancelQueries({ queryKey: ['post', postId] });

    // Snapshot the current value for rollback
    const previousPost = queryClient.getQueryData(['post', postId]);

    // Optimistically update the cache
    queryClient.setQueryData(['post', postId], (old) =&gt; ({
      ...old,
      liked: !old.liked,
      likeCount: old.liked ? old.likeCount - 1 : old.likeCount + 1,
    }));

    return { previousPost, postId };
  },

  onError: (err, postId, context) =&gt; {
    // Rollback to the snapshot on error
    queryClient.setQueryData(['post', context.postId], context.previousPost);
  },

  onSettled: (data, error, postId) =&gt; {
    // Always refetch to get the real server state
    queryClient.invalidateQueries({ queryKey: ['post', postId] });
  },
});</pre>
    </div>

      staleTime: 10 * 1000, // don't prefetch if already fresh
    });
  }, [queryClient]);

  return prefetchPost;
}

    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#api-q14">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;How do you handle race conditions in data fetching?

  </h2>
  <div id="api-q14" class="accordion-collapse collapse" data-bs-parent="#acc-api">
    <div class="accordion-body">
      Race conditions occur when multiple fetch requests are in-flight simultaneously and responses arrive out of order — an earlier request's response overwrites the more recent request's result. In React Query, race conditions are handled automatically — it tracks the latest request per query key and ignores responses from superseded requests. For manual <code>useEffect</code> fetching, use either a <code>cancelled</code> flag or <code>AbortController</code>. React Query v5 additionally uses <code>AbortController</code> internally to cancel superseded requests at the network level.
      <pre class="code-block">// Manual fix: cancelled flag

  let cancelled = false;
  setLoading(true);
  fetchData(query)
    .then(data =&gt; { if (!cancelled) setData(data); })
    .finally(() =&gt; { if (!cancelled) setLoading(false); });
  return () =&gt; { cancelled = true; };
}, [query]);

// Manual fix: AbortController
React.useEffect(() =&gt; {
  const controller = new AbortController();

    .catch(err =&gt; { if (err.name !== 'AbortError') setError(err); });
  return () =&gt; controller.abort();
}, [query]);

// React Query: handled automatically

    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#api-q15">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is request deduplication in React Query?
    </button>
  </h2>
  <div id="api-q15" class="accordion-collapse collapse" data-bs-parent="#acc-api">
    <div class="accordion-body">
      Request deduplication means that if multiple components mount at the same time and call <code>useQuery</code> with the same <code>queryKey</code>, only ONE network request is made — all components receive the same data when it arrives. Without deduplication, 10 components on a page could each trigger the same API request. React Query handles this by checking if a query with that key is already in-flight before starting a new fetch. This dramatically reduces redundant network requests on initial page load. The deduplication window is during the initial fetch — once data is in cache, new subscribers just read from cache instantly.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#api-q16">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;How do you implement pagination with React Query?
    </button>
  </h2>
  <div id="api-q16" class="accordion-collapse collapse" data-bs-parent="#acc-api">
    <div class="accordion-body">
      Standard pagination uses a page number parameter in the query key. Each page is a separate cache entry. <code>placeholderData: keepPreviousData</code> (React Query v5) maintains the previous page's data visible while the next page loads — preventing a loading flash on page change. Prefetching the next page when the current page loads gives users instant navigation.
      <pre class="code-block">import { useQuery, keepPreviousData } from '@tanstack/react-query';

function PostsPage() {
  const [page, setPage] = React.useState(1);

  const { data, isPlaceholderData, isFetching } = useQuery({
    queryKey: ['posts', page],
    queryFn: () =&gt; fetch(`/api/posts?page=${page}&amp;limit=10`).then(r =&gt; r.json()),
    placeholderData: keepPreviousData,
  });

  const queryClient = useQueryClient();
  React.useEffect(() =&gt; {
    if (!isPlaceholderData &amp;&amp; data?.hasMore) {
      queryClient.prefetchQuery({

      <span class="diff-hard">Hard</span>&nbsp;&nbsp;What is the staleTime vs gcTime difference in React Query?
    </button>
  </h2>
  <div id="api-q21" class="accordion-collapse collapse" data-bs-parent="#acc-api">
    <div class="accordion-body">
      <code>staleTime</code> (default: 0) determines how long data is considered "fresh" — during this window React Query serves cached data without any background refetch. During gcTime the data can be served instantly if a component remounts; after gcTime the data is garbage collected and the next mount triggers a fresh fetch with a loading state. Key rule: staleTime should always be less than or equal to gcTime, since there is no point keeping data in cache longer than it can be considered usable.

  <div id="api-q32" class="accordion-collapse collapse" data-bs-parent="#acc-api">
    <div class="accordion-body">

        {isRefetching ? 'Refreshing...' : 'Refresh'}
      &lt;/button&gt;
      {data?.posts.map(post =&gt; &lt;Post key={post.id} post={post} /&gt;)}
    &lt;/div&gt;
  );
}

// Polling
const { data } = useQuery({
  queryKey: ['live-metrics'],
  queryFn: fetchMetrics,
  refetchInterval: 5000,
  refetchIntervalInBackground: false,
});</pre>

    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#api-q34">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is query persistence in React Query?
    </button>
  </h2>
  <div id="api-q34" class="accordion-collapse collapse" data-bs-parent="#acc-api">
    <div class="accordion-body">
      Query persistence serializes the React Query cache to a storage layer (localStorage, IndexedDB, AsyncStorage) so data survives page reloads. On startup the cache is hydrated from storage before any network requests, giving users instant data while a background refetch confirms currency. The <code>@tanstack/query-persist-client-core</code> package and <code>PersistQueryClientProvider</code> enable this pattern. Only queries with <code>gcTime &gt; 0</code> that match a persistence filter are serialized. This is especially powerful for mobile-first and PWA apps where perceived performance and offline resilience both matter.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#api-q35">
      <span class="diff-hard">Hard</span>&nbsp;&nbsp;How do you implement retry with exponential backoff in React Query?
    </button>
  </h2>
  <div id="api-q35" class="accordion-collapse collapse" data-bs-parent="#acc-api">
    <div class="accordion-body">
      The <code>retry</code> option accepts a number or a predicate <code>(failureCount, error) =&gt; boolean</code>. The <code>retryDelay</code> option accepts a number or a function receiving the attempt index. The smart pattern skips retries on 4xx errors (client mistakes that won't succeed on retry) while retrying 5xx errors with capped exponential backoff.
      <pre class="code-block">const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      retry: (failureCount, error) =&gt; {
        if (error?.status &gt;= 400 &amp;&amp; error?.status &lt; 500) return false;
        return failureCount &lt; 3;
      },
      retryDelay: (attemptIndex) =&gt;
        Math.min(1000 * 2 ** attemptIndex, 30000), // 1s, 2s, 4s … cap 30s
    },
  },
});</pre>
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#api-q36">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is background refetching and how does it affect UX?
    </button>
  </h2>
  <div id="api-q36" class="accordion-collapse collapse" data-bs-parent="#acc-api">
    <div class="accordion-body">
      Background refetching shows stale cached data immediately while React Query silently fetches fresh data in the background. When fresh data arrives the UI updates in place — no loading spinner, no blank state. This is the stale-while-revalidate pattern: users always see something instantly, and content updates smoothly when the server responds. Triggers include window focus (<code>refetchOnWindowFocus</code>), network reconnection (<code>refetchOnReconnect</code>), manual invalidation, and interval polling (<code>refetchInterval</code>). The <code>isRefetching</code> flag (true when fetching but data already exists) lets you add a subtle corner indicator without hiding existing content behind a full loading overlay.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#api-q37">
      <span class="diff-scenario">Scenario</span>&nbsp;&nbsp;How would you architect the API layer for a large React application?
    </button>
  </h2>
  <div id="api-q37" class="accordion-collapse collapse" data-bs-parent="#acc-api">
    <div class="accordion-body">
      A scalable API architecture has four layers. (1) API client — one Axios or fetch instance configured with base URL, auth interceptors, error normalisation, and request IDs. (2) Service functions — pure async functions per domain (<code>getUser(id)</code>, <code>createPost(data)</code>) that call the client and return typed data; independently testable with mocked HTTP. (3) React Query hooks — custom hooks wrapping <code>useQuery</code>/<code>useMutation</code> that call service functions and expose component-friendly state. (4) Query key factory — centralised definitions preventing typos and enabling consistent prefix invalidation. This layering means you can swap the HTTP client without touching hooks, change caching strategy without touching service functions, and test each layer in isolation.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#api-q38">
      <span class="diff-scenario">Scenario</span>&nbsp;&nbsp;A user reports they see stale data after updating their profile. How do you fix this?
    </button>
  </h2>
  <div id="api-q38" class="accordion-collapse collapse" data-bs-parent="#acc-api">
    <div class="accordion-body">
      This is a cache invalidation bug. The mutation succeeds on the server but the React Query cache still holds the pre-update snapshot. Fix: add <code>queryClient.invalidateQueries({ queryKey: ['user', userId] })</code> in the mutation's <code>onSuccess</code> callback. This marks the query stale and triggers an immediate background refetch. For instant feedback without a loading flash, also apply an optimistic update in <code>onMutate</code> — write the new values to cache immediately, then let the server refetch confirm the final state. If the user's name or avatar appears in other queries (post authors, comments), invalidate those keys as well or use a broad prefix invalidation covering all user-related data.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#api-q39">
      <span class="diff-hard">Hard</span>&nbsp;&nbsp;What is the initialData vs placeholderData option in React Query?
    </button>
  </h2>
  <div id="api-q39" class="accordion-collapse collapse" data-bs-parent="#acc-api">
    <div class="accordion-body">
      Both provide data before the query fetches, but are treated fundamentally differently. <code>initialData</code> is stored in the cache as real data — it obeys staleTime, may trigger a background refetch if already stale, and is shared between components via the cache. Use it for SSR-hydrated data or data derived from a parent query. <code>placeholderData</code> is never stored in cache and never considered fresh — it is a pure UI placeholder shown while the real fetch runs, then discarded. Use it for the <code>keepPreviousData</code> pagination pattern (pass the built-in <code>keepPreviousData</code> helper) or for showing a skeletal content shape while loading. The key distinction: <code>initialData</code> can eliminate a network request; <code>placeholderData</code> never prevents one.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#api-q40">
      <span class="diff-scenario">Scenario</span>&nbsp;&nbsp;How do you handle offline support in a React Query app?
    </button>
  </h2>
  <div id="api-q40" class="accordion-collapse collapse" data-bs-parent="#acc-api">
    <div class="accordion-body">
      React Query v5 has built-in offline support via network mode options. <code>networkMode: 'online'</code> (default) pauses queries and mutations when offline and resumes them automatically on reconnect. <code>networkMode: 'always'</code> runs queries regardless of connectivity, useful for cache-only or service-worker scenarios. <code>networkMode: 'offlineFirst'</code> attempts one run even offline. For full offline support combine three things: (1) cache persistence to IndexedDB via <code>PersistQueryClientProvider</code> so data survives reloads; (2) a service worker to cache API responses at the network layer; (3) mutation queuing — mutations that fail offline get <code>isPaused: true</code> and are retried automatically on reconnect. The mutation <code>isPaused</code> state lets you surface a "changes pending sync" indicator to users.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#testing-q2">
      <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is the difference between getBy, queryBy, and findBy queries?
    </button>
  </h2>
  <div id="testing-q2" class="accordion-collapse collapse" data-bs-parent="#acc-testing">
    <div class="accordion-body">
      These three prefixes differ in timing and failure behavior. <code>getBy*</code> throws immediately if the element is not found — use it when the element must be present right now; it also throws if multiple matches exist (use <code>getAllBy*</code> for multiples). <code>queryBy*</code> returns <code>null</code> if not found without throwing — use it to assert an element is NOT present: <code>expect(screen.queryByText('Error')).not.toBeInTheDocument()</code>. <code>findBy*</code> returns a Promise and waits (via <code>waitFor</code> internally) for the element to appear — use it for elements that render asynchronously after data loads. Using <code>getBy</code> where <code>findBy</code> is needed is one of the most common RTL mistakes, causing tests to fail before async operations complete.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#testing-q3">
      <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is userEvent vs fireEvent?
    </button>
  </h2>
  <div id="testing-q3" class="accordion-collapse collapse" data-bs-parent="#acc-testing">
    <div class="accordion-body">
      <code>fireEvent.click(element)</code> dispatches a single synthetic DOM click event. <code>userEvent.click(element)</code> from <code>@testing-library/user-event</code> simulates the full browser event sequence a real user generates: pointerover, pointerenter, mouseover, mouseenter, pointermove, mousemove, pointerdown, mousedown, focus, pointerup, mouseup, click. <code>userEvent.type(input, 'hello')</code> fires keydown, keypress, input, and keyup for each character and respects <code>maxLength</code>. Always prefer <code>userEvent</code> over <code>fireEvent</code> — it catches bugs that <code>fireEvent</code> misses (handlers listening to <code>keydown</code> rather than <code>change</code>, focus management issues). Set up with <code>const user = userEvent.setup()</code> at the top of each test.
    </div>
  </div>
</div>

    </button>
  </h2>
  <div id="testing-q14" class="accordion-collapse collapse" data-bs-parent="#acc-testing">
    <div class="accordion-body">
      Snapshot tests capture a component's rendered output as a serialized string and compare it against a saved snapshot on future runs. If the output changes, the test fails until you update the snapshot with <code>--updateSnapshot</code>. RTL's <code>asFragment()</code> returns a DocumentFragment for snapshotting. Snapshots are useful for detecting unintentional UI regressions in stable component trees. The downsides: they are brittle (minor changes break them), they do not test behavior, large snapshots are meaningless in code review, and they get updated automatically without thought. Best practice: use snapshots only for small, stable, intentionally specific markup; prefer behavioral assertions for everything else.
      <pre class="code-block">test('Button snapshot', () =&gt; {
  const { asFragment } = render(
    &lt;Button variant="primary"&gt;Click me&lt;/Button&gt;
  );
  expect(asFragment()).toMatchSnapshot();
});

// Inline snapshot (stored in the test file itself)
test('Badge inline snapshot', () =&gt; {
  const { asFragment } = render(&lt;Badge count={5} /&gt;);
  expect(asFragment()).toMatchInlineSnapshot(`
    &lt;DocumentFragment&gt;
      &lt;span class="badge"&gt;5&lt;/span&gt;
    &lt;/DocumentFragment&gt;
  `);
});</pre>

screen.getByTestId('submit-button');         // data-testid — last resort
container.firstChild.children[1];            // DOM traversal

// Common role values
screen.getByRole('heading', { level: 1 });   // h1
screen.getByRole('textbox');                 // input[type=text]
screen.getByRole('combobox');                // select
screen.getByRole('dialog');                  // modal
screen.getByRole('alert');                   // error/status messages</pre>

    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#testing-q25">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is the Testing Trophy vs Testing Pyramid?
    </button>
  </h2>
  <div id="testing-q25" class="accordion-collapse collapse" data-bs-parent="#acc-testing">
    <div class="accordion-body">
      The Testing Pyramid (classic) recommends many unit tests, fewer integration tests, and few E2E tests. Kent C. Dodds's Testing Trophy (the philosophy behind RTL) recommends the largest investment in integration tests. Integration tests test how multiple units work together — a form component with validation, state, and submission — and provide more real-world confidence than unit tests that mock all dependencies. The rationale: unit tests that test implementation details (internal state, method calls) can pass even when the visible feature is broken; integration tests that test user behavior catch real bugs. The Trophy shape: static analysis (TypeScript, ESLint) at the base, a few unit tests for pure functions, many integration tests for component behavior, and a few E2E tests for critical complete flows (login, checkout).
    </div>

Use <code>beforeEach</code> to reset mocks and set default MSW handlers. Use <code>afterEach</code> to reset MSW handler overrides. Use <code>beforeAll</code>/<code>afterAll</code> for expensive one-time setup like starting/stopping test servers.

    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#testing-q30">
      <span class="diff-hard">Hard</span>&nbsp;&nbsp;What is test coverage and what are good targets for React apps?
    </button>
  </h2>
  <div id="testing-q30" class="accordion-collapse collapse" data-bs-parent="#acc-testing">
    <div class="accordion-body">
      Test coverage measures what percentage of lines, branches, functions, and statements are executed during the test suite. Run it with <code>npx vitest --coverage</code> (Vitest) or <code>jest --coverage</code>. An 80% line coverage baseline is a common target, but coverage percentage is a proxy metric — 100% coverage with weak assertions gives less confidence than 70% coverage with strong behavioral assertions. Prioritize coverage of: business-critical logic (auth, payments, data transformation), error handling paths, and complex conditional branches. Avoid chasing 100% — testing trivial getters, generated code, and pure configuration adds maintenance burden. The more useful question is: "Are my critical user flows tested end-to-end with meaningful assertions?"

    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#testing-q32">
      <span class="diff-scenario">Scenario</span>&nbsp;&nbsp;How do you prevent tests from being brittle?
    </button>
  </h2>
  <div id="testing-q32" class="accordion-collapse collapse" data-bs-parent="#acc-testing">
    <div class="accordion-body">
      Brittle tests fail for reasons unrelated to real bugs — styling changes, refactors, renamed internal methods. Prevention strategies: (1) Use semantic queries (<code>getByRole</code>, <code>getByLabelText</code>) not CSS classes or DOM structure; (2) Test behavior (what the user sees and can do), not implementation (component state, internal method calls); (3) Use <code>expect.objectContaining()</code> instead of exact object matching in <code>toHaveBeenCalledWith</code>; (4) Avoid snapshot tests for large dynamic components; (5) Don't test that React or third-party libraries work; (6) Use <code>within(container)</code> to scope queries when duplicate text exists rather than relying on DOM order; (7) Keep test data realistic and meaningful so failures are self-explanatory.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#testing-q33">
      <span class="diff-hard">Hard</span>&nbsp;&nbsp;What is Playwright and how does it complement React Testing Library?
    </button>
  </h2>
  <div id="testing-q33" class="accordion-collapse collapse" data-bs-parent="#acc-testing">
    <div class="accordion-body">
      Playwright is a modern E2E testing framework from Microsoft that drives real browsers (Chromium, Firefox, WebKit) in parallel. Key features: auto-wait (Playwright waits for elements to be visible, enabled, and stable before interacting — dramatically reducing flakiness), cross-browser coverage, visual regression testing, network interception for mocking, and parallel test execution. Playwright's <code>locator</code> API prioritizes accessible queries similar to RTL's <code>getByRole</code>. In a complete testing strategy: RTL handles fast, isolated component tests with mocked dependencies; Playwright handles slow, full-stack user journeys in real browsers. The two tools cover different levels of the testing trophy and are complementary, not competing.
    </div>
  </div>
</div>

      <span class="diff-scenario">Scenario</span>&nbsp;&nbsp;How do you structure a test suite for a large React application?
    </button>
  </h2>
  <div id="testing-q35" class="accordion-collapse collapse" data-bs-parent="#acc-testing">
    <div class="accordion-body">
      Large applications need organized, maintainable test suites. Best practices: (1) Colocate tests with source files (<code>Button.test.tsx</code> next to <code>Button.tsx</code>); (2) Use <code>describe</code> blocks to group related scenarios within a file; (3) Create a shared test utilities file — custom render functions with all providers, data factory functions (<code>createUser({ role: 'admin' })</code>), and common assertion helpers; (4) Separate unit tests from integration tests by naming convention (<code>*.unit.test.ts</code> vs <code>*.integration.test.ts</code>) to run them separately in CI; (5) Maintain a <code>__mocks__</code> directory for module-level mocks that are reused across tests; (6) Run tests in parallel in CI; (7) Keep tests independent — each test sets up its own state and does not rely on prior test execution order. The goal: writing a new test should be as easy as reading an existing one.

    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#testing-q37">
      <span class="diff-hard">Hard</span>&nbsp;&nbsp;What are the anti-patterns to avoid in React testing?
    </button>
  </h2>
  <div id="testing-q37" class="accordion-collapse collapse" data-bs-parent="#acc-testing">
    <div class="accordion-body">
      The most damaging testing anti-patterns: (1) Testing implementation details — asserting on component state, internal methods, or CSS classes not visible to users; (2) Over-mocking — mocking so heavily that tests no longer run real code paths; (3) Using <code>getByTestId</code> as the first choice rather than last resort; (4) Testing third-party library behavior instead of your own code; (5) Massive snapshot tests that get auto-updated without review; (6) Not cleaning up async operations — leaving pending timers or promises causes test pollution and "state update not wrapped in act" warnings; (7) Using <code>fireEvent</code> instead of <code>userEvent</code> for user interactions; (8) Writing tests that depend on execution order — each test must be fully independent and self-contained.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#testing-q38">
      <span class="diff-scenario">Scenario</span>&nbsp;&nbsp;A test is flaky — it sometimes passes and sometimes fails without code changes. How do you fix it?
    </button>
  </h2>
  <div id="testing-q38" class="accordion-collapse collapse" data-bs-parent="#acc-testing">
    <div class="accordion-body">
      Flaky tests typically stem from five causes. (1) Race conditions: async operations resolving in different orders — fix by using <code>waitFor</code> and <code>findBy</code> queries correctly rather than arbitrary delays. (2) Shared state pollution: one test's state bleeds into the next via a global variable, module-level mock, or MSW handler — fix with <code>beforeEach</code> resets and <code>server.resetHandlers()</code>. (3) Timer dependency: tests relying on real wall-clock timers — fix with <code>jest.useFakeTimers()</code>. (4) Test order dependency: a test passes only when run after another test that sets up some global state — each test must be self-contained. (5) Uncontrolled async side effects: effects running after test teardown causing "state update outside act" warnings — use <code>waitFor</code> to fully drain async queues. To reproduce: run the flaky test in a tight loop (<code>vitest --repeat 50 the.test.file</code>) until it fails consistently, then debug from there.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#testing-q39">
      <span class="diff-scenario">Scenario</span>&nbsp;&nbsp;How would you add testing to a legacy React codebase with no tests?
    </button>
  </h2>
  <div id="testing-q39" class="accordion-collapse collapse" data-bs-parent="#acc-testing">
    <div class="accordion-body">
      Introducing tests to an untested codebase requires pragmatic prioritisation rather than trying to achieve full coverage immediately. Step 1: set up the testing infrastructure first (Vitest/Jest, RTL, MSW, coverage reporting) so the tooling is in place. Step 2: establish a "no new code without tests" policy going forward. Step 3: add a failing test before each bug fix — this prevents regressions and builds the test suite organically. Step 4: prioritise pure utility functions and business logic first — they are easiest to test and highest value. Step 5: write integration tests for the most critical user flows (login, checkout, primary data table) before achieving unit coverage across all files. Step 6: track coverage trend (increasing over time) not absolute number. Avoid the "test debt sprint" trap — continuous incremental improvement beats a one-time effort that gets abandoned.
    </div>
  </div>
