Arrow Functions

Arrow functions are the default way to write functions in React — they're concise, and they lexically bind this, which is critical for class components and callbacks.

// Traditional function
function add(a, b) { return a + b; }

// Arrow function — concise
const add = (a, b) => a + b;

// Single param — no parens needed
const double = n => n * 2;

// In React: concise callback in .map()
const doubles = [1, 2, 3].map(n => n * 2);

// Arrow function vs regular — 'this' binding matters in class components
class Timer extends React.Component {
  // ❌ Regular function — 'this' is undefined in strict mode
  startBad() {
    setTimeout(function() {
      this.setState({ running: true }); // TypeError!
    }, 1000);
  }

  // ✅ Arrow function — inherits 'this' from enclosing scope
  startGood() {
    setTimeout(() => {
      this.setState({ running: true }); // works correctly
    }, 1000);
  }
}
React Pattern: Event handlers in function components should always be arrow functions or named functions — avoid inline function() {} when possible for readability.

Destructuring

Destructuring lets you unpack values from objects and arrays into variables. You'll use this constantly in React — props, state, and hooks all use destructuring.

// Object destructuring
const user = { name: 'John', age: 30, role: 'admin' };
const { name, age, role = 'user' } = user;  // default value for missing keys

// Rename while destructuring
const { name: userName } = user;  // userName = 'John'

// In React — props destructuring (preferred pattern)
function Button({ label, onClick, variant = 'primary', disabled = false }) {
  return (
    <button
      className={`btn btn-${variant}`}
      onClick={onClick}
      disabled={disabled}
    >
      {label}
    </button>
  );
}

// Array destructuring — useState returns an array!
const [count, setCount] = useState(0);
const [loading, setLoading] = useState(false);

// Skip elements with commas
const [, second, , fourth] = [1, 2, 3, 4];

// Nested destructuring
const { address: { city, state } } = person;

// Function parameter destructuring
const greet = ({ name, age }) => `${name} is ${age}`;
greet(user); // 'John is 30'
Why this matters for React: Every time you write const [state, setState] = useState(), that's array destructuring. When you write function MyComponent({ title, children }), that's object destructuring on the props argument.
3

Spread Operator ...

Spread copies items from an array or object. It's the foundation of immutable state updates in React — you never mutate state directly.

// Object spread — immutable updates
const user = { name: 'John', age: 30, city: 'NYC' };
const updated = { ...user, age: 31 }; // { name: 'John', age: 31, city: 'NYC' }

// In React state — ALWAYS use spread for updates
setUser(prev => ({ ...prev, age: 31 }));

// Nested update (one level at a time)
setUser(prev => ({
  ...prev,
  address: { ...prev.address, city: 'LA' }
}));

// Array spread — add item immutably
const newItems = [...existingItems, newItem];
const prepended = [newItem, ...existingItems];

// Combine arrays
const combined = [...arr1, ...arr2];

// Remove item immutably (filter is preferred)
const withoutItem = items.filter(item => item.id !== targetId);

// Update item in array immutably
const updatedItems = items.map(item =>
  item.id === targetId ? { ...item, completed: true } : item
);

// Pass all props through to child
function Wrapper(props) {
  return <div><Child {...props} /></div>;
}
Common Mistake: setState({ ...state, newProp }) is fine for shallow objects, but for deeply nested state, consider useReducer or a library like Immer to avoid messy nested spread chains.
4

Rest Parameters ...rest

Rest collects remaining items into an array or object. In React, the classic pattern is separating a component's own props from props it should pass down.

// Collect remaining array elements
const [first, ...rest] = [1, 2, 3, 4, 5];
// first = 1, rest = [2, 3, 4, 5]

// Collect remaining object properties
const { name, age, ...others } = user;
// name = 'John', age = 30, others = { city: 'NYC', role: 'admin' }

// In React: separate own props from pass-through props
function StyledButton({ variant, className = '', ...restProps }) {
  // variant and className used here, everything else passed through
  return (
    <button
      className={`btn btn-${variant} ${className}`}
      {...restProps}  // onClick, disabled, type, aria-*, data-* etc.
    />
  );
}

// Usage — any extra prop flows through automatically
<StyledButton
  variant="primary"
  onClick={handleClick}
  disabled={isLoading}
  aria-label="Submit"
/>

// Rest in function parameters
function sum(...numbers) {
  return numbers.reduce((a, b) => a + b, 0);
}
sum(1, 2, 3, 4); // 10

Short-Circuit & Ternary — JSX Conditionals

Since JSX is just JavaScript, you can't use if statements inline. React uses three patterns for conditional rendering.

Ternary ? :
Render one of two elements
AND &&
Render something or nothing
Nullish ??
Fallback for null/undefined only 
// Ternary — renders different elements
{isLoggedIn ? <Dashboard /> : <LoginPage />}

// Multi-line ternary for readability
{isLoading
  ? <Spinner />
  : error
    ? <ErrorMessage error={error} />
    : <DataDisplay data={data} />
}

// && — renders or nothing (short-circuit)
{hasError && <ErrorMessage error={error} />}
{items.length > 0 && <ItemList items={items} />}

// ⚠️ Gotcha: 0 && ... renders "0" (falsy non-boolean)
{items.length && <List />}       // ❌ renders "0" when array is empty!
{items.length > 0 && <List />}   // ✅ explicit boolean check

// Nullish coalescing — fallback only for null/undefined (not 0 or '')
const name = user?.name ?? 'Guest';
const count = data?.count ?? 0;   // 0 stays 0, null/undefined gets 0

// Optional chaining — safe property access
const city = user?.address?.city;
const firstTag = post?.tags?.[0];
const price = cart?.getTotal?.();   // safe method call

// Combined — very common in React
const displayName = user?.profile?.displayName ?? user?.email ?? 'Anonymous';
Common Bug: Using count && <Component /> where count might be 0 will render the number 0 in your UI. Always use count > 0 && or convert to boolean: !!count &&.

Array Methods — The React Toolkit

These array methods are used everywhere in React. They all return new arrays (non-mutating), which is exactly what immutable state updates require.

// .map() — transform every element, returns new array
const cards = items.map(item => (
  <Card key={item.id} title={item.title} {...item} />
));

// .filter() — keep matching elements
const activeUsers = users.filter(u => u.isActive);
const searchResults = products.filter(p =>
  p.name.toLowerCase().includes(query.toLowerCase())
);

// .reduce() — accumulate into single value
const total = cart.reduce((sum, item) => sum + item.price * item.qty, 0);

// Build an object from array (group by category)
const byCategory = items.reduce((acc, item) => ({
  ...acc,
  [item.category]: [...(acc[item.category] || []), item]
}), {});

// .find() — first match, or undefined
const selectedUser = users.find(u => u.id === selectedId);

// .findIndex() — index of first match
const idx = items.findIndex(i => i.id === targetId);

// .some() / .every() — boolean checks
const hasAdmin = users.some(u => u.role === 'admin');
const allVerified = users.every(u => u.isVerified);

// .includes() — for primitives
const isAllowed = allowedRoles.includes(user.role);

// .flatMap() — flatten + map in one pass
const allTags = posts.flatMap(post => post.tags);

// Sort — MUTATES original! Use spread first
const sorted = [...users].sort((a, b) => a.name.localeCompare(b.name));
Key insight: map, filter, and reduce all return new arrays without modifying the original — perfect for React's immutable state requirement. sort mutates in-place, so always spread first: [...arr].sort().

Promises & Async/Await

React apps constantly fetch data from APIs. You need to understand both Promise chains and async/await — the latter is what you'll write day-to-day.

// Promise chain (older style)
fetch('/api/users')
  .then(r => {
    if (!r.ok) throw new Error(`HTTP ${r.status}`);
    return r.json();
  })
  .then(data => setUsers(data))
  .catch(err => setError(err.message))
  .finally(() => setLoading(false));

// Async/Await — cleaner, use this in React
async function loadUsers() {
  try {
    setLoading(true);
    const response = await fetch('/api/users');
    if (!response.ok) throw new Error(`HTTP ${response.status}`);
    const data = await response.json();
    setUsers(data);
  } catch (err) {
    setError(err.message);
  } finally {
    setLoading(false);
  }
}

// useEffect pattern for async data fetching
useEffect(() => {
  // Cannot make the useEffect callback itself async!
  // Define async function inside, then immediately call it.
  const fetchData = async () => {
    try {
      const res = await fetch(`/api/posts/${postId}`);
      const json = await res.json();
      setPost(json);
    } catch (e) {
      setError(e.message);
    }
  };
  fetchData();
}, [postId]);

// Parallel requests — Promise.all() fires both simultaneously
const [users, posts] = await Promise.all([
  fetch('/api/users').then(r => r.json()),
  fetch('/api/posts').then(r => r.json()),
]);

// Promise.allSettled() — get all results regardless of failures
const results = await Promise.allSettled([req1, req2, req3]);
results.forEach(result => {
  if (result.status === 'fulfilled') console.log(result.value);
  else console.error(result.reason);
});
Common Mistake: Writing useEffect(async () => {}, []) — you cannot make the useEffect callback itself async because async functions return a Promise, but useEffect expects either nothing or a cleanup function.

ES Modules — import / export

React projects are fully modular — every file is a module. Understanding named vs default exports prevents a lot of import errors.

// Named Exports — multiple per file
export const PI = 3.14159;
export function add(a, b) { return a + b; }
export class Calculator { }

// Default Export — one per file
export default function App() { return <div>App</div>; }

// Importing
import App from './App';               // default — any name works
import { add, PI } from './math';       // named — must match export name
import { add as sum } from './math';   // rename on import
import * as utils from './utils';     // namespace import
import App, { helper } from './App';   // default + named together

// React / Library imports
import React, { useState, useEffect, useCallback } from 'react';
import { BrowserRouter, Routes, Route } from 'react-router-dom';
import { configureStore } from '@reduxjs/toolkit';

// Re-exporting — barrel files (src/components/index.js)
export { default as Button } from './Button';
export { default as Card } from './Card';
export { default as Modal } from './Modal';

// Then import cleanly from one place:
import { Button, Card, Modal } from '@/components';

Classes — Understanding React Class Components

Modern React uses function components, but you'll encounter class components in older codebases. Class basics also help you understand Error Boundaries, which still require them.

// ES6 Class basics
class Animal {
  constructor(name) {
    this.name = name;
  }
  speak() {
    return `${this.name} makes a sound.`;
  }
}

class Dog extends Animal {
  speak() {
    return `${this.name} barks.`;
  }
}

const dog = new Dog('Rex');
dog.speak(); // "Rex barks."

// React Class Component structure
class Counter extends React.Component {
  constructor(props) {
    super(props);                    // Always call super(props) first!
    this.state = { count: 0 };      // Initialize state in constructor
  }

  // Arrow class property — auto-binds 'this' (modern pattern)
  handleClick = () => {
    this.setState(prev => ({ count: prev.count + 1 }));
  }

  render() {
    return (
      <button onClick={this.handleClick}>
        Count: {this.state.count}
      </button>
    );
  }
}
Modern React: In 2024+, you'll almost never write class components from scratch. But you need to recognize them in existing code and know that Error Boundaries still require class components as of React 18.

Closures — The Key to Understanding Hooks

A closure is a function that remembers variables from its outer scope. React hooks are built on closures — if you don't get closures, stale state bugs will mystify you.

// What is a closure?
function makeCounter() {
  let count = 0;                   // 'count' lives in makeCounter's scope
  return function() {
    count++;                         // inner function CLOSES OVER 'count'
    return count;
  };
}
const counter = makeCounter();
counter(); // 1
counter(); // 2

// Stale Closure Bug in React
function Counter() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const timer = setInterval(() => {
      // ❌ 'count' is CLOSED OVER at effect creation time
      // It will always be 0 because count is not in deps array
      console.log('count is:', count); // Always prints 0!
    }, 1000);
    return () => clearInterval(timer);
  }, []); // empty deps = closure captures initial count = 0
}

// Fix #1: Add count to deps array
useEffect(() => {
  const timer = setInterval(() => {
    console.log('count is:', count); // ✅ always current
  }, 1000);
  return () => clearInterval(timer);
}, [count]); // ✅ re-runs when count changes

// Fix #2: Functional update — no closure needed
useEffect(() => {
  const timer = setInterval(() => {
    setCount(prev => prev + 1); // ✅ 'prev' is always latest value
  }, 1000);
  return () => clearInterval(timer);
}, []); // ✅ empty deps is fine — no stale read

// Fix #3: useRef for mutable values without re-render
const countRef = useRef(count);
useEffect(() => { countRef.current = count; }, [count]);

useEffect(() => {
  const timer = setInterval(() => {
    console.log(countRef.current); // ✅ always current — refs are mutable
  }, 1000);
  return () => clearInterval(timer);
}, []);
Mental Model: Every render creates a new "snapshot" — a closure capturing that render's state values. When a useEffect runs, it captures the state values from that specific render. If state changes, the effect's closure is stale unless you list the value in the deps array.

this, bind, and Why Arrow Functions Matter

this in JavaScript is determined by how a function is called, not where it's defined — unless you use an arrow function, which inherits this lexically from its enclosing scope.

// 'this' is dynamic in regular functions
const obj = {
  name: 'React',
  greet: function() { return this.name; }  // 'this' = obj when called as obj.greet()
};
const greetFn = obj.greet;
greetFn(); // undefined — 'this' is now global/undefined in strict mode

// bind() — lock 'this' permanently
const boundGreet = obj.greet.bind(obj);
boundGreet(); // 'React' — 'this' is always obj

// Class event handlers — 3 patterns
class MyButton extends React.Component {
  // Pattern 1: bind in constructor (classic)
  constructor(props) {
    super(props);
    this.handleClick = this.handleClick.bind(this);
  }
  handleClick() { console.log(this.props.label); }

  // Pattern 2: Arrow class field (modern, recommended)
  handleClick2 = () => { console.log(this.props.label); } // ✅ always bound

  render() {
    return (
      <>
        <button onClick={this.handleClick}>Click 1</button>
        <button onClick={this.handleClick2}>Click 2</button>
        // Pattern 3: Inline arrow — creates new fn each render (avoid)
        <button onClick={() => this.handleClick()}>Click 3</button>
      </>
    );
  }
}

// In function components — no 'this' needed at all!
function FunctionButton({ onClick, label }) {
  const handleClick = () => {
    onClick?.();
  };
  return <button onClick={handleClick}>{label}</button>;
}
Good News: In modern React with function components, you almost never have to think about this. This is actually one of the biggest reasons the React team moved away from class components.
