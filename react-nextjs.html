Why Next.js?
Plain React SPA Problem
  • Browser receives empty HTML: <div id="root"></div>
  • JavaScript runs, fetches data, renders content
  • Search engines can't index blank HTML
  • Users see white screen until JS loads
  • Slower First Contentful Paint (FCP)
Next.js Solution
  • Server renders full HTML before sending to browser
  • Search engines receive complete, indexable HTML
  • Users see content immediately (no white flash)
  • Better Core Web Vitals (LCP, FCP, TTI)
  • Built-in image optimization, code splitting, routing
Next.js Features: File-system routing, server-side rendering, static generation, incremental static regeneration, API routes, image optimization, font optimization, Edge Runtime, Middleware, Server Components, and full TypeScript support — all out of the box.
Rendering Strategies

Next.js lets you choose the right rendering strategy per page. Mix and match based on your data freshness needs and performance requirements.

SSR
Server-Side Rendering
When: Every request
Data: Always fresh
Use for: Dashboards, user-specific pages, real-time data

The page runs on the server for every request. The server fetches data, renders the full HTML, and sends it to the browser. Use cache: 'no-store'.

// app/users/page.tsx (Next.js 13+ App Router) async function UsersPage() { // This runs on the server on EVERY request const users = await fetch('https://api.example.com/users', { cache: 'no-store' // always fresh — never cached }).then(r => r.json()); return ( <main> <h1>Users</h1> {users.map(user => <UserCard key={user.id} {...user} />)} </main> ); } export default UsersPage;
Best for: User dashboards, shopping carts, pages that need real-time data. The tradeoff is slower TTFB (Time To First Byte) since every request hits the server.
SSG
Static Site Generation
When: Build time only
Data: Baked in at build
Use for: Blog posts, docs, marketing pages

HTML is generated at build time and served as a static file. Blazing fast — served from CDN with zero server computation per request. Use cache: 'force-cache'.

// Generates HTML at BUILD TIME — fastest possible async function BlogPage({ params }) { const post = await fetch(`https://api.example.com/posts/${params.slug}`, { cache: 'force-cache' // cache forever (until rebuild) }).then(r => r.json()); return <article><h1>{post.title}</h1><p>{post.content}</p></article>; } // Tell Next.js which paths to pre-generate at build time export async function generateStaticParams() { const posts = await fetch('https://api.example.com/posts').then(r => r.json()); return posts.map(post => ({ slug: post.slug })); }
Best for: Content that doesn't change often — documentation, blogs, marketing sites. Rebuilds are needed to update content.
ISR
Incremental Static Regeneration
When: Background revalidation
Data: Fresh after revalidation
Use for: E-commerce, news sites

Best of both worlds — static speed + automatic updates. Pages are pre-built, but Next.js regenerates them in the background after a set interval. Use next: { revalidate: N }.

// Regenerates page in background every 60 seconds async function ProductPage({ params }) { const product = await fetch(`https://api.example.com/products/${params.id}`, { next: { revalidate: 60 } // revalidate every 60 seconds }).then(r => r.json()); return <ProductDetail {...product} />; } // On-demand revalidation — trigger after CMS update // app/api/revalidate/route.ts import { revalidatePath } from 'next/cache'; export async function POST(request) { const { path } = await request.json(); revalidatePath(path); // purge cache for specific path return Response.json({ revalidated: true }); }
Best for: Product pages, news articles, anything that changes occasionally. Users always get cached (fast) responses while Next.js updates in the background.
CSR
Client-Side Rendering
When: After hydration
Data: Fetched client-side
Use for: User-specific data, highly interactive UIs

Opt into client-side rendering with the 'use client' directive. The component renders in the browser, enabling hooks, event handlers, and browser APIs.

'use client'; // opt into client-side rendering function Dashboard() { // Now you can use useState, useEffect, event handlers const { data } = useSWR('/api/dashboard', fetcher); const [filter, setFilter] = useState('all'); return ( <div> <select onChange={e => setFilter(e.target.value)}> <option value="all">All</option> </select> <div>{data?.stats}</div> </div> ); } export default Dashboard;
Best for: Dashboards with user-specific data, real-time updates, highly interactive UIs where SEO doesn't matter. Default in Next.js is Server Components — you opt in to CSR only where needed.
Server Components vs Client Components
Server Components (default)
  • No 'use client' directive needed
  • Run on the server — never sent to browser
  • Can be async — await directly in the component
  • Access databases, file system, environment variables
  • Cannot use hooks, browser APIs, or event handlers
  • Result: smaller client bundle, faster load
Client Components ('use client')
  • Add 'use client' at the top of the file
  • Run in the browser (hydrated from server HTML)
  • Can use useState, useEffect, all hooks
  • Can use event handlers, browser APIs
  • Cannot await directly (use hooks instead)
  • Added to client bundle
Rule of thumb: Keep components as Server Components as long as possible. Only add 'use client' when you need interactivity (state, events, browser APIs). You can pass server data as props to client components.
API Routes

Next.js API Routes let you build your backend directly inside your Next.js project. No separate Express server needed for simple APIs.

// app/api/users/route.ts — handles GET and POST import { NextResponse } from 'next/server'; export async function GET(request: Request) { const { searchParams } = new URL(request.url); const page = searchParams.get('page') || '1'; const users = await db.user.findMany({ skip: (parseInt(page) - 1) * 10, take: 10, }); return NextResponse.json(users); } export async function POST(request: Request) { const body = await request.json(); const user = await db.user.create({ data: body }); return NextResponse.json(user, { status: 201 }); } // app/api/users/[id]/route.ts — dynamic route handler export async function GET( request: Request, { params }: { params: { id: string } } ) { const user = await db.user.findUnique({ where: { id: params.id } }); if (!user) return NextResponse.json({ error: 'Not found' }, { status: 404 }); return NextResponse.json(user); } export async function DELETE( request: Request, { params }: { params: { id: string } } ) { await db.user.delete({ where: { id: params.id } }); return new Response(null, { status: 204 }); }

File System Routing — App Router

Next.js routing is based entirely on your folder structure. Each folder creates a URL segment. Special files have predefined roles.

app/ ├── layout.tsx ← root layout (wraps all) ├── page.tsx ← route: / ├── loading.tsx ← automatic Suspense wrapper ├── error.tsx ← error boundary UI ├── not-found.tsx ← 404 page ├── blog/ │ ├── page.tsx ← route: /blog │ └── [slug]/ ← dynamic segment │ └── page.tsx ← route: /blog/:slug ├── (auth)/ ← route group (no URL) │ ├── layout.tsx ← auth-only layout │ ├── login/ │ │ └── page.tsx ← route: /login │ └── register/ │ └── page.tsx ← route: /register └── api/ └── users/ └── route.ts ← API: /api/users
Special Files Explained
FilePurpose
page.tsxThe UI for a route — makes a route publicly accessible
layout.tsxShared UI for a segment and its children — persists across navigation
loading.tsxInstant loading UI shown while page data is fetching (Suspense fallback)
error.tsxError UI for a segment — must be a Client Component with 'use client'
not-found.tsxUI for 404 errors when notFound() is thrown
route.tsAPI endpoint — export GET, POST, PUT, DELETE, PATCH functions
[slug]Dynamic route segment — accessible via params.slug
(group)Route group — organizes routes without adding URL segment
@slotNamed slot for parallel routes — renders multiple pages simultaneously
Layouts are Persistent: Unlike pages, layouts don't re-render during navigation between pages that share that layout. This preserves scroll position and component state.

Middleware

Middleware runs before a request is processed — great for authentication redirects, A/B testing, geolocation, and request logging. Runs on the Edge for minimal latency.

// middleware.ts — at the root of your project import { NextResponse } from 'next/server'; import type { NextRequest } from 'next/server'; export function middleware(request: NextRequest) { const token = request.cookies.get('token')?.value; const isProtected = request.nextUrl.pathname.startsWith('/dashboard'); if (isProtected && !token) { const loginUrl = new URL('/login', request.url); loginUrl.searchParams.set('from', request.nextUrl.pathname); return NextResponse.redirect(loginUrl); } return NextResponse.next(); } // Apply only to matching paths export const config = { matcher: ['/dashboard/:path*', '/admin/:path*', '/api/:path*'], };
