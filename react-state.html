Section 7 Core Concept

State Management

Deep dive into React state — useState patterns, state lifecycle, async updates, batching in React 18, and when to choose local vs global state.

1 useState Deep Dive
What is State?
State is data that changes over time and causes component re-renders when updated. Unlike props (immutable, external), state is owned and managed by the component itself. React's useState hook returns a tuple: [currentValue, setterFunction]. Every time the setter is called with a new value, React schedules a re-render and the component function re-runs with the updated state value. 
// ─── Basic useState usage ──────────────────────────────────────────
import { useState } from 'react';

function Counter() {
  // Destructure the tuple: [value, setter]
  const [count, setCount] = useState(0);

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>Increment</button>
    </div>
  );
}

// ─── Functional update: prev => prev + 1 ──────────────────────────
// WHY: If multiple updates fire before a re-render (batching), using
// the closure value of `count` may be stale. The functional form
// receives the GUARANTEED latest state from React's queue — always
// use it when new state depends on the previous value.
<button onClick={() => setCount(prev => prev + 1)}>Safe Increment</button>

// ─── Object state — correct vs incorrect ──────────────────────────
const [user, setUser] = useState({ name: 'Alice', age: 30, role: 'admin' });

// ❌ WRONG — mutates state directly, React won't detect the change
user.name = 'Bob';
setUser(user); // same reference → no re-render!

// ✅ CORRECT — spread to create a new object reference
setUser(prev => ({ ...prev, name: 'Bob' }));

// ─── Array state — add, remove, update ───────────────────────────
const [items, setItems] = useState(['apple', 'banana']);

// Add item
setItems(prev => [...prev, 'cherry']);

// Remove by value
setItems(prev => prev.filter(item => item !== 'banana'));

// Remove by index
setItems(prev => prev.filter((_, idx) => idx !== 1));

// Update item at index — map to replace
setItems(prev => prev.map((item, idx) => idx === 0 ? 'avocado' : item));
Pro Tip: Always use functional updates (prev => ...) when new state depends on previous state. React may batch state updates and the closure over state could be stale — the functional form receives the freshest value from React's internal update queue, not the potentially-stale closure.

State updates are ASYNC
React does not immediately apply state changes. The new value is only available on the next render. Calling setState queues an update — it does not synchronously mutate the value. This is a core source of confusion for new React developers. 
// ─── The stale state issue ─────────────────────────────────────
function StaleDemo() {
  const [count, setCount] = useState(0);

  const handleClick = () => {
    setCount(count + 1);
    // ❌ count is STILL 0 here — state hasn't re-rendered yet
    console.log(count); // prints 0, not 1!
  };

  // ✅ FIX — useEffect runs AFTER re-render with fresh value
  useEffect(() => {
    // count is the NEW value here, after the re-render
    console.log('Count updated to:', count);
    if (count > 0) {
      sendAnalytics({ event: 'count_updated', value: count });
    }
  }, [count]); // runs every time count changes

  return <button onClick={handleClick}>Count: {count}</button>;
}
3 State Batching — React 18
Automatic Batching in React 18
React 18 introduced Automatic Batching — multiple setState calls in the same event handler, setTimeout, Promise, or any other context are batched into a single re-render. React 17 only batched inside native event handlers. This is a significant performance improvement that also simplifies reasoning about renders. 
// ─── React 18 automatic batching — ONE re-render regardless of context ──
function handleClick() {
  setCount(c => c + 1);   // batched
  setVisible(v => !v);     // batched
  setName('John');         // batched
  // Result: ONE re-render, not three!
}

// ─── React 17 — only batched in synthetic events ──────────────────────
// In React 17, setTimeout caused 2 separate re-renders:
setTimeout(() => {
  setCount(c => c + 1); // re-render #1 (React 17 — NOT batched!)
  setVisible(v => !v);   // re-render #2 (React 17 — NOT batched!)
  // React 18: still just ONE re-render — automatic batching applies!
}, 1000);

// ─── Same with Promises (React 18 batches, React 17 did NOT) ─────────
fetchData().then(() => {
  setData(result);     // React 18: batched
  setLoading(false);  // React 18: batched — ONE re-render!
});

// ─── Opt OUT of batching (rare — use sparingly!) ─────────────────────
import { flushSync } from 'react-dom';

// flushSync forces an immediate, synchronous re-render
flushSync(() => setCount(c => c + 1)); // forces immediate re-render #1
flushSync(() => setVisible(v => !v));   // forces immediate re-render #2
// Use case: reading updated DOM layout between state updates
React 17
Only batches inside synthetic event handlers. setTimeout / Promise / native events each trigger separate re-renders per setState call.
React 18
Batches everywhere — event handlers, setTimeout, Promises, native events. Opt out with flushSync when needed.
4 State Updates Are Async — Stale Closures & Fixes
Common Pitfall: Reading State After setState
Reading state immediately after calling setState returns the OLD value. State updates are scheduled — the closure still holds the stale value from the render in which the handler was created. React will apply the update on the next render, not synchronously. 
// ─── WRONG — reading state right after setState ─────────────────────
const handleClick = () => {
  setCount(count + 1);
  console.log(count);       // ❌ logs OLD value — still the stale closure!
  sendAnalytics(count);     // ❌ sends wrong count to analytics
  saveToServer(count);      // ❌ saves stale value to server
};

// ─── CORRECT — use useEffect to react to state changes ──────────────
useEffect(() => {
  if (count > 0) {
    sendAnalytics(count); // ✅ runs after re-render with new value
    saveToServer(count);  // ✅ correct value guaranteed
  }
}, [count]); // dependency array ensures this runs when count changes

// ─── CORRECT — local variable captures the intended next value ───────
const handleClickWithLocal = () => {
  const nextCount = count + 1;
  setCount(nextCount);
  sendAnalytics(nextCount); // ✅ use the local variable, not state
};

// ─── CORRECT — functional updates chain correctly ────────────────────
const handleTripleIncrement = () => {
  setCount(prev => prev + 1); // ✅ prev is guaranteed fresh from queue
  setCount(prev => prev + 1); // ✅ chains correctly: prev = previous result
  setCount(prev => prev + 1); // ✅ result: count + 3 (not count + 1!)
  // Without functional form: setCount(count + 1) x3 = only count + 1!
};
Key Mental Model: Each render is a snapshot — the event handlers created during that render close over the state values from that specific render. setState schedules a new snapshot, it does not mutate the current one. The functional updater (prev => ...) bypasses this by receiving the queued value directly from React.
5 State Initialization Patterns
Lazy Initialization
The initial value passed to useState is only used on the first render. However, if you pass an expression (like a function call), it is evaluated on every render — even though the result is discarded after mount. Pass a function reference instead to defer the computation, so it only runs once. 
// ─── Normal initialization — expression evaluates on EVERY render ────
const [list, setList] = useState(expensiveCompute()); // ❌ runs every render!

// ─── Lazy initialization — function only called ONCE on mount ────────
const [list, setList] = useState(() => expensiveCompute()); // ✅ runs once

// ─── Reading from localStorage lazily ────────────────────────────────
const [theme, setTheme] = useState(() => {
  // This function only runs on the initial render
  const saved = localStorage.getItem('theme');
  return saved || 'light';
});

// ─── Parsing URL params lazily ────────────────────────────────────────
const [filters, setFilters] = useState(() => {
  const params = new URLSearchParams(window.location.search);
  return {
    search: params.get('q') || '',
    page: parseInt(params.get('page') || '1', 10),
    sort: params.get('sort') || 'asc',
  };
});

// ─── Complex default structure built once ────────────────────────────
const [formData, setFormData] = useState(() => ({
  id: crypto.randomUUID(),   // generates unique ID once, not every render
  createdAt: new Date().toISOString(),
  fields: {},
}));
6 Local vs Global State — When to Use What
State Type When to Use Tools Example
Local state Single component, not shared with others useState, useReducer Form input value, modal open/close toggle
Lifted state Shared between sibling components useState in parent Filter value affecting a sibling list component
Context Cross-component, low-frequency updates useContext + useState Theme, current user, locale/language setting
Redux Toolkit Complex logic, many consumers, need for time-travel debug or dev tools Redux store + slices Shopping cart, multi-step form across routes
Zustand Simple global state, less boilerplate than Redux Zustand store Global app settings, auth user object
React Query / SWR Server state — data fetching, caching, background refetching React Query, SWR API data, paginated lists, infinite scroll
Rule of Thumb: Start with local state. Lift it when siblings need it. Use Context for medium-scale sharing (theme, auth, language). Use Redux/Zustand for complex global state with many consumers. Use React Query for server state — don't store fetched API data in useState; that's exactly what React Query solves. 
// ─── When to lift state vs use Context ───────────────────────────────

// 1. Local state — stays inside the component
function SearchBar() {
  const [query, setQuery] = useState(''); // only SearchBar needs this
  return <input value={query} onChange={e => setQuery(e.target.value)} />;
}

// 2. Lifted state — parent manages, passes down to siblings
function ProductPage() {
  const [filter, setFilter] = useState('all'); // shared between children
  return (
    <>
      <FilterBar filter={filter} onFilter={setFilter} />
      <ProductList filter={filter} />
    </>
  );
}

// 3. Context — avoids prop drilling across many levels
const ThemeContext = createContext('light');

function App() {
  const [theme, setTheme] = useState('light');
  return (
    <ThemeContext.Provider value={{ theme, setTheme }}>
      <Layout /> // any descendant can useContext(ThemeContext)
    </ThemeContext.Provider>
  );
}
Q Interview Questions

State is mutable data owned by a component. When state changes, React re-renders the component to reflect the new value. State is private — only the component that owns it can change it (by calling its setter).

Props are immutable inputs passed from a parent. A component can read props but cannot modify them. Props flow down the component tree; state lives inside a single component (though it can be shared by lifting to a parent).

Key difference: State makes a component dynamic and interactive. Props make a component configurable and reusable. A common pattern is a parent holding state and passing it as props to child components.

Functional updates should be used when new state depends on the previous state because React may batch multiple setState calls. If you use the state value directly from the closure (setCount(count + 1)), all batched updates read the same stale closure value — so three calls would still only result in count + 1.

With the functional form (setCount(prev => prev + 1)), React passes the most recently queued value as prev, so three calls correctly chain to count + 3. It also avoids stale closure bugs where the closure captures an outdated snapshot of state.

In React 18, all multiple setState calls within the same synchronous event handler (or any async context, thanks to automatic batching) are batched into a single re-render. React collects all the state updates, processes them, and then triggers one render with all updates applied simultaneously.

In React 17, batching only occurred inside React synthetic event handlers. Calls inside setTimeout, Promises, or native DOM event listeners would each trigger a separate re-render.

You can opt out of batching in React 18 using flushSync from react-dom, which forces synchronous, immediate flushing of the update queue.

State batching is React's optimization of grouping multiple state updates together and applying them in a single re-render rather than re-rendering after each individual update. This reduces unnecessary renders and improves performance.

React 17 and earlier: Batching only applied inside React's own synthetic event handlers (e.g., onClick, onChange). State updates triggered from setTimeout, native event listeners, Promises, or async/await were NOT batched — each setState caused a separate re-render.

React 18: Introduced Automatic Batching — state updates are now batched in ALL contexts, including setTimeout, Promises, and native events. This makes performance more consistent and predictable without requiring any code changes. Use flushSync to opt out when you need immediate DOM feedback between updates.

React state updates are asynchronous — calling setState does not immediately mutate the state variable. It schedules an update that will be applied on the next render. The current render's state values are immutable snapshots — they do not change mid-render.

JavaScript closures are the underlying mechanism: the event handler function closes over the state variable from the render in which it was created. Calling setState doesn't break the closure — the variable in scope still holds the old value.

To act on the new state value, use useEffect with the state variable in its dependency array — this runs after the re-render with the new value. Alternatively, store the intended new value in a local variable and use that instead of reading state after the setter call.

useState is ideal for simple, independent state values — a number, a boolean, a string. The update logic is typically straightforward.

useReducer is better for complex state logic where multiple sub-values are closely related, where the next state depends on the previous in non-trivial ways, or when multiple actions can update state in different ways. It mirrors the Redux pattern: you dispatch actions to a reducer function that returns the next state.

Rule of thumb: If you find yourself writing many useState calls that all update together, or if update logic is complex with multiple cases, migrate to useReducer. useReducer also makes it easier to extract and test state logic independently of the component.

React detects state changes by reference equality. Mutating an object or array in place and passing the same reference to the setter will NOT trigger a re-render — React sees the same reference and assumes nothing changed.

Objects: Use the spread operator to create a new object: setUser(prev => ({ ...prev, name: 'Bob' })). For deeply nested objects, spread at each level or use a library like Immer.

Arrays: Use non-mutating methods — map, filter, spread, or concat. Avoid push, pop, splice, or direct index assignment. For example: add with [...prev, newItem], remove with prev.filter(...)>, update with prev.map(...).

Lifting state up means moving state from a child component to the closest common ancestor of all components that need it. The parent then passes the state down as props and passes the setter as a callback prop.

You need it when two sibling components need to share or react to the same state — for example, a filter dropdown and a data list that is filtered by it. Neither sibling can directly read the other's state, so the parent holds the filter value and passes it to both children.

The trade-off: lifting state can cause unnecessary re-renders for the parent and all its children. When state is lifted very high (e.g., to the root), consider Context or a state management library to avoid deep prop drilling.

React Context is built-in and best for low-frequency, mostly-read global state like the current theme, logged-in user, or locale. It re-renders all consumers when the context value changes, so it performs poorly for frequently-updated state (like a search query that updates on every keystroke).

Redux Toolkit is better when you have: complex update logic with many action types, state that is read by dozens of components (Redux uses selector memoization to prevent unnecessary re-renders), need for middleware (thunks, sagas), or benefit from Redux DevTools time-travel debugging. It adds boilerplate but pays off in large applications.

A practical guideline: Context for simple global values, Redux for complex shared state with performance requirements. Zustand is a middle ground — Redux-like model with much less boilerplate, good for medium-complexity apps.

Lazy state initialization is the pattern of passing a function (not a value) as the initial state argument to useState. React only calls this function on the very first render — subsequent renders ignore it entirely.

Without lazy initialization, useState(expensiveCompute()) evaluates expensiveCompute() on every render (the result is just discarded after mount). With lazy initialization, useState(() => expensiveCompute()) only calls the function once.

Common use cases: reading from localStorage, parsing URL query parameters, building a complex initial data structure (like generating a unique ID with crypto.randomUUID()), or any computation that would be expensive to repeat on every render. If the initial value is a simple literal, lazy initialization has no benefit.

! Common Mistakes to Avoid
Mistake 1: Mutating State Directly

Directly modifying state objects or arrays (e.g., state.items.push(x), state.user.name = 'Bob') does not notify React of the change. React uses reference equality — the same object reference means no change detected, so the component never re-renders.

// ❌ WRONG — mutates in place, React won't re-render
state.items.push(newItem);
setState(state); // same reference!

// ✅ CORRECT — always create new references
setState(prev => ({ ...prev, items: [...prev.items, newItem] }));
Mistake 2: Reading State Immediately After setState

State updates are scheduled — the variable in scope still holds the old value from the current render's closure. Reading state right after setState gives the stale value. Use useEffect to react to state changes, or store the intended next value in a local variable and use that.

// ❌ WRONG
setCount(count + 1);
doSomethingWith(count); // still old count!

// ✅ CORRECT
const next = count + 1;
setCount(next);
doSomethingWith(next); // use the local variable
Mistake 3: Not Using Functional Updates When Depending on Previous State

When you call setState(count + 1) multiple times in one handler (or rapid successive calls), they all close over the same stale count value. The result is only count + 1, not count + N. Use the functional updater to ensure each update builds on the latest queued value.

// ❌ WRONG — all three use the same stale `count`, result is count + 1
setCount(count + 1);
setCount(count + 1);
setCount(count + 1);

// ✅ CORRECT — chains on latest queued value, result is count + 3
setCount(prev => prev + 1);
setCount(prev => prev + 1);
setCount(prev => prev + 1);
Mistake 4: Storing Derived Data in State

If a value can be computed from existing state or props, do NOT store it in additional state. Having two sources of truth that must be kept in sync is fragile, causes bugs, and requires unnecessary updates. Compute derived values during render instead.

// ❌ WRONG — fullName is derived; two states to keep in sync
const [firstName, setFirstName] = useState('');
const [lastName, setLastName] = useState('');
const [fullName, setFullName] = useState(''); // redundant!

// ✅ CORRECT — compute during render, no extra state needed
const [firstName, setFirstName] = useState('');
const [lastName, setLastName] = useState('');
const fullName = `${firstName} ${lastName}`; // derived value
Mistake 5: Creating New Object/Array References in Render Body

Defining objects or arrays directly in the render body (or in JSX props) creates a new reference on every render. When these are passed as useEffect dependencies, the effect fires on every render — potentially causing infinite loops. Hoist constants outside the component or memoize with useMemo/useCallback.

// ❌ WRONG — new array reference every render → infinite loop
function MyComponent() {
  const options = ['a', 'b', 'c']; // new reference every render!
  useEffect(() => {
    fetchData(options);
  }, [options]); // ❌ triggers every render — infinite loop!
}

// ✅ CORRECT — option 1: hoist outside component (if static)
const OPTIONS = ['a', 'b', 'c']; // stable reference — defined once
function MyComponent() {
  useEffect(() => { fetchData(OPTIONS); }, []);
}

// ✅ CORRECT — option 2: useMemo if derived from props/state
const options = useMemo(() => [a, b, c], [a, b, c]); // stable if deps unchanged
