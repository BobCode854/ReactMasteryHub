Setting Up TypeScript
# New project with TypeScript template npm create vite@latest my-app -- --template react-ts # Add TypeScript to an existing React project npm install typescript @types/react @types/react-dom # Generate tsconfig.json npx tsc --init
Key tsconfig settings for React: Set "jsx": "react-jsx", "strict": true, "moduleResolution": "bundler". Enable "noImplicitAny": true and "strictNullChecks": true for maximum safety.
Component Props with TypeScript

Define an interface or type for your component's props. Use ? for optional props and provide defaults with destructuring.

// Interface for props interface ButtonProps { label: string; onClick: () => void; variant?: 'primary' | 'secondary' | 'danger'; // union type disabled?: boolean; loading?: boolean; className?: string; children?: React.ReactNode; } const Button: React.FC<ButtonProps> = ({ label, onClick, variant = 'primary', disabled = false, loading = false, }) => { return ( <button className={`btn btn-${variant}`} onClick={onClick} disabled={disabled || loading} > {loading ? <Spinner size="sm" /> : label} </button> ); }; // Alternative: function declaration (often preferred over React.FC) function Button({ label, onClick, variant = 'primary' }: ButtonProps) { return <button className={`btn btn-${variant}`} onClick={onClick}>{label}</button>; }
interface vs type
interface — better for component props, supports extends/merging. type — better for unions, intersections, and complex computed types. Both work; be consistent in your codebase.
React.FC vs function declaration
React.FC adds implicit children prop (pre-v18). Function declaration is more explicit and avoids this. Modern React prefers explicit function declarations.
useState & useRef with TypeScript
// Primitive types — TypeScript infers these automatically const [count, setCount] = useState<number>(0); const [name, setName] = useState<string>(''); // Object state — needs explicit interface interface User { id: number; name: string; email: string; role: 'admin' | 'user'; } // Nullable state — when user might not be loaded yet const [user, setUser] = useState<User | null>(null); // Array state const [items, setItems] = useState<User[]>([]); // useRef — specify the DOM element type const inputRef = useRef<HTMLInputElement>(null); const divRef = useRef<HTMLDivElement>(null); const formRef = useRef<HTMLFormElement>(null); const buttonRef = useRef<HTMLButtonElement>(null); // Mutable ref (not a DOM node — no null initial value needed) const timerRef = useRef<ReturnType<typeof setTimeout>>(); // Access DOM element safely const focusInput = () => { inputRef.current?.focus(); // optional chaining because ref might be null };
Custom Hook with TypeScript

Generic custom hooks are one of the most powerful TypeScript + React combinations. The hook infers the exact type of data based on usage.

interface FetchState<T> { data: T | null; loading: boolean; error: string | null; } function useFetch<T>(url: string): FetchState<T> { const [state, setState] = useState<FetchState<T>>({ data: null, loading: true, error: null, }); useEffect(() => { fetch(url) .then(r => r.json()) .then((data: T) => setState({ data, loading: false, error: null })) .catch(err => setState({ data: null, loading: false, error: err.message })); }, [url]); return state; } // Usage — fully typed! data is User[] | null const { data, loading, error } = useFetch<User[]>('/api/users'); const { data: product } = useFetch<Product>(`/api/products/${id}`);
Event Types

React provides typed event objects for every DOM event. The generic parameter specifies which element the event came from.

// Click events const handleClick = (e: React.MouseEvent<HTMLButtonElement>) => { console.log(e.currentTarget.value); }; // Input change events const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => { setValue(e.target.value); }; // Select change const handleSelect = (e: React.ChangeEvent<HTMLSelectElement>) => {}; // Form submit const handleSubmit = (e: React.FormEvent<HTMLFormElement>) => { e.preventDefault(); }; // Keyboard events const handleKeyDown = (e: React.KeyboardEvent<HTMLInputElement>) => { if (e.key === 'Enter') handleSubmit(); }; // Drag events const handleDrop = (e: React.DragEvent<HTMLDivElement>) => {}; // Focus events const handleFocus = (e: React.FocusEvent<HTMLInputElement>) => {};

Generic Components

Generic components work with any data type while retaining full type safety. Perfect for reusable data display components like lists, tables, and selects.

interface ListProps<T> { items: T[]; renderItem: (item: T) => React.ReactNode; keyExtractor: (item: T) => string | number; emptyMessage?: string; } function List<T>({ items, renderItem, keyExtractor, emptyMessage = 'No items' }: ListProps<T>) { if (items.length === 0) return <p>{emptyMessage}</p>; return ( <ul> {items.map(item => ( <li key={keyExtractor(item)}>{renderItem(item)}</li> ))} </ul> ); } // TypeScript knows exactly what type "user" is in renderItem <List<User> items={users} renderItem={user => <UserCard {...user} />} keyExtractor={user => user.id} /> // Reuse with a different type <List<Product> items={products} renderItem={product => <span>{product.name}</span>} keyExtractor={product => product.sku} emptyMessage="No products available" />
Redux with TypeScript

Redux Toolkit is built with TypeScript. Use typed hooks (useAppSelector, useAppDispatch) to get full type inference from your store.

// store.ts — set up typed hooks import { configureStore } from '@reduxjs/toolkit'; import type { TypedUseSelectorHook } from 'react-redux'; import { useDispatch, useSelector } from 'react-redux'; export const store = configureStore({ reducer: { auth, users } }); export type RootState = ReturnType<typeof store.getState>; export type AppDispatch = typeof store.dispatch; // Typed hooks — use these everywhere instead of plain useSelector/useDispatch export const useAppDispatch = () => useDispatch<AppDispatch>(); export const useAppSelector: TypedUseSelectorHook<RootState> = useSelector; // authSlice.ts — fully typed slice interface AuthState { user: User | null; loading: boolean; error: string | null; } const authSlice = createSlice({ name: 'auth', initialState: { user: null, loading: false, error: null } as AuthState, reducers: { setUser: (state, action: PayloadAction<User>) => { state.user = action.payload; }, }, }); // In a component — fully type-safe const user = useAppSelector(state => state.auth.user); // User | null const dispatch = useAppDispatch(); dispatch(setUser({ id: 1, name: 'John', email: 'j@j.com', role: 'admin' }));
TypeScript Gotchas
Using any kills type safety — const data: any = response opts out of all TypeScript benefits. Use unknown instead and narrow the type, or define the proper interface.
Non-null assertion overuse — inputRef.current!.focus() (the !) tells TS "trust me, it's not null." If you're wrong, you get a runtime error. Use optional chaining ?. instead.
Forgetting to type useReducer — Without typing, the reducer state and action types default to any. Always type the state interface and a discriminated union for actions.
Type assertions instead of type guards — const user = data as User bypasses type checking. Use a type guard function: function isUser(x: unknown): x is User { return typeof (x as User).id === 'number'; }
Missing return type on async functions — TypeScript can infer most return types, but async functions returning complex objects benefit from explicit annotations: async function getUser(): Promise<User>
