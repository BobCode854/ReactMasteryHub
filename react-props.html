Section 6 Props Deep Dive

React Props

Master how data flows in React — props basics, default props, prop drilling, PropTypes validation, children, and the Render Props pattern.

Section 6.1

Props Basics

What are Props?

Props (short for properties) are the mechanism for passing data from a parent component to a child component. Props are READ-ONLY — a child component must never modify its own props. This enforces unidirectional data flow, making React apps predictable and easier to debug.

// ─────────────────────────────────────────────────────
// Parent passes props to ProductCard
// ─────────────────────────────────────────────────────
function ShopPage() {
  const handleAdd = (id) => console.log('Added:', id);

  return (
    <ProductCard
      name="React Course"
      price={49}
      inStock={true}
      tags={['frontend', 'javascript']}
      onAddToCart={() => handleAdd(product.id)}
    />
  );
}

// ─────────────────────────────────────────────────────
// Child receives and destructures props
// ─────────────────────────────────────────────────────
function ProductCard({ name, price, inStock, tags, onAddToCart }) {
  return (
    <div className={`card ${inStock ? '' : 'out-of-stock'}`}>
      <h3>{name}</h3>
      <p>${price}</p>
      <div>
        {tags.map(tag => (
          <span key={tag} className="badge">{tag}</span>
        ))}
      </div>
      <button onClick={onAddToCart} disabled={!inStock}>
        {inStock ? 'Add to Cart' : 'Out of Stock'}
      </button>
    </div>
  );
}
Pro Tip: Props can be any JavaScript type: strings, numbers, booleans, arrays, objects, functions, even JSX elements. Functions as props are called "callback props" and enable child-to-parent communication — the child calls the function, which updates state in the parent.
Key Rules of Props
Immutable — Never mutate props directly. Use state in the parent and pass it down as updated props.
Unidirectional Flow — Data always flows from parent → child, never the other way around.
Any JS Type — Pass strings, numbers, booleans, arrays, objects, or functions.
Spread Operator — Use {...props} to forward all props from parent to child efficiently.
Section 6.2

Default Props

Default Props via Destructuring

The modern way to define default prop values is directly in the function parameter destructuring using the = assignment syntax. This is clean, explicit, and works with TypeScript. The older Component.defaultProps static property is deprecated in React 18.3+.

// ─────────────────────────────────────────────────────
// Button component with default props via destructuring
// ─────────────────────────────────────────────────────
function Button({
  label = 'Click Me',
  variant = 'primary',
  size = 'md',
  disabled = false,
  onClick = () => {}
}) {
  return (
    <button
      className={`btn btn-${variant} btn-${size}`}
      disabled={disabled}
      onClick={onClick}
    >
      {label}
    </button>
  );
}

// ─────────────────────────────────────────────────────
// Usage — only override what you need
// ─────────────────────────────────────────────────────
<Button label="Save" variant="success" />

// Uses ALL defaults: "Click Me", primary, md, enabled
<Button />

// Disabled state — other props still use their defaults
<Button label="Processing..." disabled={true} />

// ─────────────────────────────────────────────────────
// TypeScript typed version with defaults
// ─────────────────────────────────────────────────────
interface ButtonProps {
  label?: string;
  variant?: 'primary' | 'secondary' | 'success' | 'danger';
  size?: 'sm' | 'md' | 'lg';
  disabled?: boolean;
  onClick?: () => void;
}

function Button({
  label = 'Click Me',
  variant = 'primary',
  size = 'md',
  disabled = false,
  onClick = () => {}
}: ButtonProps) {
  /* same body as above */
}
Avoid: .defaultProps

The Component.defaultProps = {...} pattern is deprecated in React 18.3 and will be removed in a future version. Migrate to destructuring defaults.

Prefer: Destructuring Defaults

Defaults live right where the parameters are declared — easier to read, TypeScript-friendly, and aligned with modern React conventions.

Section 6.3

Prop Drilling — Problem & Solutions

The Problem — Prop Drilling

Passing props through multiple component layers that don't need the data, just to get it to a deeply nested component. This creates tight coupling, makes refactoring painful, and pollutes intermediate components with props they never use.

Prop Drilling — Visualized

App
Has user state
↓ passes user
Layout
Doesn't need user
↓ passes user
Sidebar
Doesn't need user
↓ passes user
UserMenu
Finally needs user

Layout and Sidebar are polluted with a prop they never use

The Solutions
  • Context API — Built-in React solution for global/shared state. Best for: auth, theme, locale.
  • Redux / Zustand / Jotai — External state management. Best for: large-scale, complex state.
  • Component Composition — Restructure to avoid the need to drill by lifting rendering closer to the consumer.
// ─────────────────────────────────────────────────────
// Solution 1: Context API — eliminates drilling entirely
// ─────────────────────────────────────────────────────
import { createContext, useContext, useState } from 'react';

const UserContext = createContext(null);

// Provider at top level — wraps the whole tree
function App() {
  const [user, setUser] = useState({ name: 'John', role: 'admin' });
  return (
    <UserContext.Provider value={user}>
      <Layout />  {/* No longer needs user prop! */}
    </UserContext.Provider>
  );
}

// Layout — clean, no user prop at all
function Layout() {
  return <Sidebar />;  {/* No user prop needed */}
}

// Consumer anywhere in the tree — no drilling!
function UserMenu() {
  const user = useContext(UserContext);  // directly from context
  return <div>Welcome, {user.name} ({user.role})</div>;
}

// ─────────────────────────────────────────────────────
// Solution 2: Component Composition — pass components, not data
// ─────────────────────────────────────────────────────
function App() {
  const user = { name: 'John', role: 'admin' };
  // Compose the full tree here, where user is available
  return (
    <Layout
      sidebar={<UserMenu user={user} />}
    />
  );
}

// Layout only passes through the already-composed sidebar
function Layout({ sidebar }) {
  return <div>{sidebar}</div>;  {/* No knowledge of user! */}
}
Section 6.4

PropTypes Validation

Runtime vs. Static Type Checking

PropTypes provide runtime type checking for props. They generate console warnings in development when wrong types are passed. TypeScript is the modern alternative — it provides static type checking at compile time, catching errors before you even run the app. New projects should prefer TypeScript.

// ─────────────────────────────────────────────────────
// PropTypes — runtime validation (JavaScript projects)
// ─────────────────────────────────────────────────────
import PropTypes from 'prop-types';

function UserProfile({ name, age, email, role, address, onUpdate, tags }) {
  return <div>...</div>;
}

UserProfile.propTypes = {
  name:     PropTypes.string.isRequired,          // required string
  age:      PropTypes.number,                      // optional number
  email:    PropTypes.string.isRequired,           // required string
  role:     PropTypes.oneOf(['admin', 'user', 'moderator']),  // enum
  address:  PropTypes.shape({                      // nested object
    street:  PropTypes.string,
    city:    PropTypes.string.isRequired,
    country: PropTypes.string.isRequired,
  }),
  onUpdate: PropTypes.func,                        // function callback
  tags:     PropTypes.arrayOf(PropTypes.string),   // array of strings
};

UserProfile.defaultProps = {
  age:      0,
  role:     'user',
  onUpdate: () => {},
  tags:     [],
};

// ─────────────────────────────────────────────────────
// TypeScript Alternative — PREFERRED for new projects
// ─────────────────────────────────────────────────────
interface UserProfileProps {
  name: string;                                    // required by default
  age?: number;                                   // optional (? suffix)
  email: string;
  role?: 'admin' | 'user' | 'moderator';        // union type
  address?: {
    street?: string;
    city: string;
    country: string;
  };
  onUpdate?: () => void;
  tags?: string[];
}

function UserProfile({
  name,
  age = 0,
  email,
  role = 'user',
  address,
  onUpdate = () => {},
  tags = [],
}: UserProfileProps) {
  return <div>...</div>;
}

// Compile-time error — caught before running!
<UserProfile name={42} />  // Error: Type 'number' is not assignable to 'string'

PropTypes

  • Runtime checking (development only)
  • Errors shown in browser console
  • Good for legacy JS codebases
  • Separate prop-types package needed

TypeScript

  • Compile-time static checking
  • Errors caught in your editor instantly
  • Recommended for all new React projects
  • Zero runtime overhead
Section 6.5

Children Props

The Special children Prop

The special children prop contains whatever is between your component's opening and closing tags. It can be text, JSX elements, or even functions. This is the foundation of composition — the React way to build flexible, reusable wrapper components without knowing in advance what they'll contain.

// ─────────────────────────────────────────────────────
// Flexible wrapper component using children
// ─────────────────────────────────────────────────────
function Card({ children, title, className, footer }) {
  return (
    <div className={`card ${className || ''}`}>
      {title && (
        <div className="card-header">
          <h3>{title}</h3>
        </div>
      )}
      <div className="card-body">
        {children}  {/* renders whatever is passed between tags */}
      </div>
      {footer && (
        <div className="card-footer">{footer}</div>
      )}
    </div>
  );
}

// Pass ANY JSX as children — Card doesn't care
<Card
  title="User Profile"
  className="shadow-lg"
  footer={<button>View All</button>}
>
  <Avatar src={user.photo} />
  <h4>{user.name}</h4>
  <p>{user.bio}</p>
  <button>Follow</button>
</Card>

// ─────────────────────────────────────────────────────
// React.Children utilities for advanced child manipulation
// ─────────────────────────────────────────────────────
import React from 'react';

function List({ children }) {
  return (
    <ul>
      {React.Children.map(children, (child, index) => (
        <li key={index}>{child}</li>
      ))}
    </ul>
  );
}

// React.Children API:
React.Children.map(children, fn)     // iterate, returns new array
React.Children.forEach(children, fn) // iterate, no return
React.Children.count(children)       // number of children
React.Children.only(children)        // assert single child
React.Children.toArray(children)     // flatten to array with keys

// ─────────────────────────────────────────────────────
// TypeScript: type the children prop explicitly
// ─────────────────────────────────────────────────────
interface CardProps {
  children: React.ReactNode;      // accepts any renderable content
  title?: string;
  footer?: React.ReactNode;
}

// React.ReactNode types:
// - React.ReactElement  — JSX element
// - React.ReactChild    — element, string, or number
// - React.ReactNode     — anything React can render (most permissive)
3 Key Use Cases for children
  1. Layout Wrappers — Modal, Card, Panel, Drawer — provide the chrome, let caller provide the content.
  2. HOC-style wrappers — AuthGuard, ErrorBoundary — wrap behavior around any child tree.
  3. Slot-based composition — Pass multiple named regions (header, footer, sidebar) as separate props, each accepting JSX.
Section 6.6

Render Props Pattern

What is the Render Props Pattern?

Render Props is a pattern where a prop is a function that returns JSX. The component calls this function to determine what to render, sharing its internal state/logic with the caller. It enables powerful code reuse without Higher-Order Components (HOCs). Today, it has been largely superseded by custom hooks — but you'll still encounter it in many codebases.

// ─────────────────────────────────────────────────────
// DataFetcher — uses render prop to share data logic
// The component handles fetch state; caller controls UI
// ─────────────────────────────────────────────────────
import { useState, useEffect } from 'react';

function DataFetcher({ url, render }) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    setLoading(true);
    fetch(url)
      .then(r => r.json())
      .then(d => { setData(d); setLoading(false); })
      .catch(err => { setError(err.message); setLoading(false); });
  }, [url]);

  // Calls the render prop function — consumer controls rendering
  return render({ data, loading, error });
}

// ─────────────────────────────────────────────────────
// Usage — consumer gets full control over what renders
// ─────────────────────────────────────────────────────
<DataFetcher
  url="/api/users"
  render={({ data, loading, error }) => {
    if (loading) return <Spinner />;
    if (error)   return <ErrorMessage message={error} />;
    return <UserList users={data} />;
  }}
/>

// Alternative: children as a function (same pattern, different prop name)
<DataFetcher url="/api/users">
  {({ data, loading, error }) => (
    loading ? <Spinner /> : <UserList users={data} />
  )}
</DataFetcher>

// ─────────────────────────────────────────────────────
// Modern Alternative: Custom Hook (preferred today)
// Same logic, much cleaner API — no wrapper component needed
// ─────────────────────────────────────────────────────
function useFetch(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    fetch(url)
      .then(r => r.json())
      .then(d => { setData(d); setLoading(false); })
      .catch(err => { setError(err.message); setLoading(false); });
  }, [url]);

  return { data, loading, error };
}

// Usage — clean, no JSX wrapper needed
function UserComponent() {
  const { data, loading, error } = useFetch('/api/users');
  if (loading) return <Spinner />;
  if (error)   return <ErrorMessage message={error} />;
  return <UserList users={data} />;
}

Render Props — When You'll See It

  • React Router v4/v5 used it extensively
  • Formik uses render props for form state
  • Downshift (autocomplete) relies on it
  • React Spring animations

Custom Hooks — Modern Approach

  • No extra JSX wrapper component
  • Easier to compose multiple behaviors
  • Better TypeScript inference
  • Avoid "wrapper hell" in React DevTools

A simple mental model: Props are like function arguments — passed in from outside. State is like local variables — managed internally by the component.

Prop drilling occurs when data must be passed through multiple intermediate components that don't use the data themselves — just to deliver it to a deeply nested component that needs it.

Problems caused:

  • Tight coupling — Intermediate components become aware of data they don't care about, making them harder to reuse independently.
  • Verbosity — Every layer requires boilerplate prop forwarding code.
  • Maintenance burden — Adding/renaming a prop requires changes across every intermediate component.
  • Readability — Component signatures become cluttered with "passthrough" props, obscuring their true purpose.

  1. React Context API — Built-in solution. Create a Context, wrap with a Provider, and consume with useContext() anywhere in the tree. Best for: auth, theme, locale, user preferences.
  2. State Management Libraries — Redux, Zustand, Jotai, Recoil store state globally. Components subscribe directly. Best for: large-scale apps with complex interdependent state.
  3. Component Composition — Restructure the component tree to move the consumer closer to where the data originates. Pass JSX elements (children / slots) instead of raw data.
  4. URL / Router State — For navigation-related data, using React Router's useParams or useSearchParams can eliminate some drilling entirely.

The children prop is a special React prop that automatically receives whatever JSX content is placed between a component's opening and closing tags. It can be a string, a React element, an array of elements, or a function.

3 Key Use Cases:

  1. Layout Wrappers — Modal, Card, Panel, Drawer. The wrapper provides structure (chrome, borders, padding, animations) while the caller provides the content. This makes the wrapper infinitely reusable.
  2. Behavioral Wrappers / Guards — AuthGuard, ErrorBoundary, Tooltip. These components add cross-cutting behavior (auth checking, error catching, hover behavior) transparently around any child tree.
  3. Render Slots / Named Areas — By accepting both children (main slot) and additional props like header, footer, sidebar (each typed as React.ReactNode), a layout component can fill multiple independent content regions.

Render Props is a pattern where a component receives a function as a prop (often named render or using children as a function). The component calls this function with its internal state/data, allowing the caller to control what is rendered. It was the primary code-sharing mechanism before hooks.

Example: <MouseTracker render={({x, y}) => <Cursor x={x} y={y} />} />

Comparison with Custom Hooks:

  • Hooks are simpler — Extract the same logic into a useXxx() hook. No wrapper component needed, no extra JSX nesting.
  • Hooks compose better — You can call multiple hooks in one component; render props create "wrapper hell" when combined.
  • Hooks have better TS inference — Return types from hooks are explicit; render prop argument types require more ceremony.
  • When to use render props still — When the shared logic is inherently tied to rendering (e.g., virtualization, drag-and-drop where you need to inject DOM callbacks), render props or HOCs can still be the right choice.

PropTypes (runtime, JS only): Install the prop-types package. Assign a propTypes static object to the component. React warns in the dev console when passed values don't match. Stripped in production builds.

TypeScript (compile-time, preferred): Define an interface or type for the props object, then annotate the component function parameter. The TypeScript compiler (and your IDE) will error immediately — before the app runs — if wrong types are passed.

Key differences:

  • When errors surface: PropTypes → browser console at runtime. TypeScript → editor and build at compile-time.
  • Coverage: TypeScript catches errors at every usage site. PropTypes only validates what actually runs.
  • DX: TypeScript enables IDE autocompletion and refactoring. PropTypes does not.
  • Recommendation: Use TypeScript for new projects. PropTypes is still useful in large JS codebases that can't fully migrate to TS yet.

In the context of form elements, this distinction describes who owns the input's current value:

Controlled Component: React state is the "single source of truth." The form element's value is set via the value prop (wired to state), and every keystroke updates state via an onChange handler. You always know the current input value from state.

const [name, setName] = useState('');
<input
  value={name}
  onChange={e => setName(e.target.value)}
/>

Uncontrolled Component: The DOM manages the value itself. React reads the current value only when needed (e.g., on form submit) using a ref (useRef / createRef). Less code for simple forms but harder to validate or conditionally update.

const nameRef = useRef(null);
<input defaultValue="" ref={nameRef} />
// Read on submit: nameRef.current.value

Recommendation: Prefer controlled components for any form that needs validation, conditional disabling, or derived UI from the input value. Use uncontrolled only for very simple, fire-and-forget inputs or third-party integrations.

Watch Out

Common Props Mistakes

Pitfalls every React developer encounters — know them before your next interview or code review.

1
Mutating props directly

Props are immutable — directly assigning to a prop property (props.user.name = 'Jane') causes silent bugs. React will not detect the mutation, re-renders won't occur predictably, and shared object references can corrupt state in sibling components.

// BAD — silently mutates parent's object
function BadChild({ user }) {
  user.name = 'Jane';  // NEVER do this!
  return <div>{user.name}</div>;
}
// GOOD — call callback to update in parent
function GoodChild({ user, onUpdate }) {
  return (
    <button onClick={() => onUpdate({ ...user, name: 'Jane' })}>
      Rename
    </button>
  );
}
2
Not providing the key prop when mapping to components

When rendering arrays of elements with .map(), every element must have a unique, stable key prop. Without it, React cannot efficiently update the list and you'll get console warnings. Keys must be unique among siblings, not globally.

// BAD — missing key
{users.map(user => <UserCard user={user} />)}

// GOOD — stable, unique key (use ID, not index if list can change)
{users.map(user => <UserCard key={user.id} user={user} />)}
3
Passing objects/arrays as props without useMemo — breaks React.memo

Object and array literals create a new reference on every render. If you wrap a child in React.memo to skip re-renders, passing style={{ color: 'red' }} or items={[1,2,3]} directly will defeat the memoization — the child re-renders every time because the reference is always new.

// BAD — new object reference every render
function Parent() {
  return <MemoChild config={{ theme: 'dark', size: 'lg' }} />;  // always new ref
}

// GOOD — stable reference with useMemo
function Parent() {
  const config = useMemo(() => ({ theme: 'dark', size: 'lg' }), []);
  return <MemoChild config={config} />;  // stable reference
}
4
Prop drilling when Context would be much cleaner

Passing the same prop through 3+ levels of components just to get it to a leaf node is a code smell. It tightly couples your component hierarchy and makes future refactors painful. Recognizing this pattern and reaching for Context API (or a state manager) is a sign of React maturity.

Rule of thumb: If you're passing the same prop through more than 2 intermediary components that don't use it, consider Context or restructuring via composition.
5
Using array index as key in dynamic lists

Using the array index as the key prop causes incorrect reconciliation behavior when list items can be reordered, inserted, or deleted. React uses keys to identify which items changed — an index-based key tells React the first item is always the first item, even if the user just deleted it, leading to subtle UI bugs (wrong input focus, stale component state).

// BAD — index key breaks reconciliation on reorder/delete
{todos.map((todo, index) => (
  <TodoItem key={index} todo={todo} />
))}

// GOOD — stable ID as key (safe for any list mutation)
{todos.map(todo => (
  <TodoItem key={todo.id} todo={todo} />
))}

// Index key IS safe for: static lists that never reorder/filter/delete
