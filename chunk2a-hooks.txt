      <h2 class="accordion-header">
        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#hooks-q1">
          <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is a React hook?
        </button>
      </h2>
      <div id="hooks-q1" class="accordion-collapse collapse" data-bs-parent="#acc-hooks">
        <div class="accordion-body">
          A React hook is a special function that lets you "hook into" React features (state, lifecycle, context, refs) from function components — before hooks existed, these features were only available in class components. Hooks were introduced in React 16.8 and completely transformed how React components are written. Hook names always start with <code>use</code> (e.g., <code>useState</code>, <code>useEffect</code>, <code>useRef</code>). You can only call hooks at the top level of a component (not inside conditions, loops, or nested functions) and only from React function components or custom hooks — these are the two Rules of Hooks. Hooks enable sharing stateful logic between components without HOC/render-prop patterns.
        </div>
      </div>
    </div>

    <div class="accordion-item">
      <h2 class="accordion-header">
        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#hooks-q2">
          <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is useState?
        </button>
      </h2>
      <div id="hooks-q2" class="accordion-collapse collapse" data-bs-parent="#acc-hooks">
        <div class="accordion-body">
          <code>useState</code> is a hook that adds state to a function component. It returns a tuple: <code>[currentValue, setterFunction]</code>. The initial value is passed as the argument and is only used on the first render. <code>setterFunction</code> triggers a re-render when called with a new value (or an updater function). Unlike class component <code>this.setState</code>, <code>useState</code> setter completely replaces the state value rather than merging — for objects, you must spread the old state manually: <code>setState(prev =&gt; ({ ...prev, newField: value }))</code>. Each <code>useState</code> call manages a single independent piece of state.
        </div>
      </div>
    </div>

    <div class="accordion-item">
      <h2 class="accordion-header">
        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#hooks-q3">
          <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is useEffect?
        </button>
      </h2>
      <div id="hooks-q3" class="accordion-collapse collapse" data-bs-parent="#acc-hooks">
        <div class="accordion-body">
          <code>useEffect(callback, deps)</code> lets you perform side effects in function components — data fetching, subscriptions, timers, DOM mutations, and anything that needs to happen "after render." The callback runs after the component renders. The <code>deps</code> array controls when the effect re-runs: <code>[]</code> means run once after mount, <code>[a, b]</code> means re-run when <code>a</code> or <code>b</code> changes, and omitting deps means run after every render. Return a cleanup function from the callback to cancel subscriptions, clear timers, or abort fetches when the component unmounts or before the effect re-runs. <code>useEffect</code> is React's escape hatch from the declarative rendering model to imperative side effects.
        </div>
      </div>
    </div>

    <div class="accordion-item">
      <h2 class="accordion-header">
        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#hooks-q4">
          <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is useRef?
        </button>
      </h2>
      <div id="hooks-q4" class="accordion-collapse collapse" data-bs-parent="#acc-hooks">
        <div class="accordion-body">
          <code>useRef(initialValue)</code> returns a mutable ref object <code>{ current: initialValue }</code> whose <code>.current</code> property persists across renders. It has two main uses: (1) accessing DOM elements directly — <code>const inputRef = useRef(null)</code> then <code>&lt;input ref={inputRef}&gt;</code> makes <code>inputRef.current</code> the DOM node; (2) storing a mutable value that doesn't trigger re-renders when changed — like an instance variable in a class component. Unlike state, mutating <code>ref.current</code> does not cause a re-render. Common uses: storing previous state values, storing interval/timeout IDs, tracking whether a component is mounted, and imperatively focusing inputs.
        </div>
      </div>
    </div>

    <div class="accordion-item">
      <h2 class="accordion-header">
        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#hooks-q5">
          <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is useContext?
        </button>
      </h2>
      <div id="hooks-q5" class="accordion-collapse collapse" data-bs-parent="#acc-hooks">
        <div class="accordion-body">
          <code>useContext(MyContext)</code> subscribes a component to a React context. It takes a context object (created by <code>React.createContext()</code>) and returns the current context value — the value from the nearest <code>&lt;MyContext.Provider value={...}&gt;</code> above in the tree. Every time the Provider's value changes, components calling <code>useContext(MyContext)</code> re-render with the new value. <code>useContext</code> replaces the <code>Context.Consumer</code> render-prop pattern with a clean hook call. It is useful for sharing data that needs to be accessible throughout the component tree without prop drilling (themes, user authentication, locale settings). However, for frequently-changing state, Context + useContext has performance limitations compared to Redux.
        </div>
      </div>
    </div>

    <div class="accordion-item">
      <h2 class="accordion-header">
        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#hooks-q6">
          <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is useReducer?
        </button>
      </h2>
      <div id="hooks-q6" class="accordion-collapse collapse" data-bs-parent="#acc-hooks">
        <div class="accordion-body">
          <code>useReducer(reducer, initialState)</code> is an alternative to <code>useState</code> for managing complex state logic. It takes a reducer function <code>(state, action) =&gt; newState</code> (same pattern as Redux) and an initial state, returning <code>[state, dispatch]</code>. Use <code>useReducer</code> over <code>useState</code> when: (1) state has multiple sub-values that change together; (2) the next state depends on the previous state; (3) the update logic is complex (multiple cases). It's also preferred when a component has many state transitions — the reducer consolidates all logic in one place. <code>useReducer</code> with Context is a common pattern for simple global state management without Redux.
        </div>
      </div>
    </div>

    <div class="accordion-item">
      <h2 class="accordion-header">
        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#hooks-q7">
          <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is useMemo?
        </button>
      </h2>
      <div id="hooks-q7" class="accordion-collapse collapse" data-bs-parent="#acc-hooks">
        <div class="accordion-body">
          <code>useMemo(() =&gt; expensiveComputation(a, b), [a, b])</code> memoizes the result of a computation — the function only re-runs when the dependencies <code>[a, b]</code> change, returning the cached value otherwise. Use it for: expensive calculations (sorting/filtering large arrays, complex transformations), creating stable object/array references for Context values or memoized children's props, and derived data that is computed from props or state. The key mental model: <code>useMemo</code> is not just for performance — it's for reference stability. Without <code>useMemo</code>, every render creates a new object reference even if the data hasn't changed, which can defeat <code>React.memo</code> and trigger unnecessary <code>useEffect</code> runs.
        </div>
      </div>
    </div>

    <div class="accordion-item">
      <h2 class="accordion-header">
        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#hooks-q8">
          <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is useCallback?
        </button>
      </h2>
      <div id="hooks-q8" class="accordion-collapse collapse" data-bs-parent="#acc-hooks">
        <div class="accordion-body">
          <code>useCallback(fn, [deps])</code> memoizes a function reference — it returns the same function object across renders as long as the dependencies haven't changed. It is equivalent to <code>useMemo(() =&gt; fn, [deps])</code>. Use <code>useCallback</code> when: passing a callback to a <code>React.memo</code>-wrapped child component (a new function reference would defeat memo), or using a function in a <code>useEffect</code> dependency array (a new reference would trigger the effect every render). Without <code>useCallback</code>, <code>const handleClick = () =&gt; doSomething(id)</code> creates a new function on every render. Caveat: <code>useCallback</code> adds overhead (storing the previous function and comparing dependencies), so only use it when the stability of the reference actually matters.
        </div>
      </div>
    </div>

      <h2 class="accordion-header">
        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#hooks-q9">
          <span class="diff-easy">Easy</span>&nbsp;&nbsp;What are the Rules of Hooks?
        </button>
      </h2>
      <div id="hooks-q9" class="accordion-collapse collapse" data-bs-parent="#acc-hooks">
        <div class="accordion-body">
          There are two Rules of Hooks: (1) Only call hooks at the top level — never inside loops, conditions, or nested functions. This ensures hooks are called in the same order on every render, which is how React tracks which state belongs to which <code>useState</code> call; (2) Only call hooks from React function components or custom hooks — not from regular JavaScript functions, class components, or event handlers. React's ESLint plugin (<code>eslint-plugin-react-hooks</code>) enforces both rules automatically. Violating Rule 1 causes bugs where state gets mixed up between renders. React relies on the call order of hooks to maintain their state — calling a hook conditionally would cause the order to vary between renders.
        </div>
      </div>
    </div>

    <div class="accordion-item">
      <h2 class="accordion-header">
        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#hooks-q10">
          <span class="diff-easy">Easy</span>&nbsp;&nbsp;What are custom hooks?
        </button>
      </h2>
      <div id="hooks-q10" class="accordion-collapse collapse" data-bs-parent="#acc-hooks">
        <div class="accordion-body">
          Custom hooks are JavaScript functions whose names start with <code>use</code> that call other hooks internally. They let you extract and reuse stateful logic across multiple components — without introducing HOC/render-prop nesting. For example, <code>useWindowSize()</code> encapsulates the logic for tracking window dimensions (adding/removing event listeners, storing state). The component uses the hook and receives the data; the logic is in one place. Custom hooks follow the same rules as built-in hooks. They don't share state between components — each component calling <code>useWindowSize()</code> gets its own independent state. Custom hooks are the React community's primary pattern for sharing non-UI logic.
        </div>
      </div>
    </div>

      <h2 class="accordion-header">
        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#hooks-q11">
          <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is useLayoutEffect and when should you use it?
        </button>
      </h2>
      <div id="hooks-q11" class="accordion-collapse collapse" data-bs-parent="#acc-hooks">
        <div class="accordion-body">
          <code>useLayoutEffect</code> has the same signature as <code>useEffect</code> but fires synchronously after all DOM mutations and before the browser paints. This gives you a chance to read DOM layout (element dimensions, positions) and synchronously make DOM mutations — without the user seeing the intermediate state. <code>useEffect</code> fires asynchronously after the browser paints, so using it for DOM measurements can cause a visible flash (the browser paints once with the initial state, then again after the effect). Use <code>useLayoutEffect</code> for: reading DOM measurements (element width/height), repositioning a tooltip based on element bounds, synchronizing scroll position. The trade-off: <code>useLayoutEffect</code> blocks the browser paint and can hurt performance if used for expensive work — prefer <code>useEffect</code> by default and only switch to <code>useLayoutEffect</code> when you observe flickering or need synchronous DOM reads.
          <pre class="code-block">// useEffect: can cause flash because paint happens before effect
function Tooltip({ target, content }) {
  const [position, setPosition] = React.useState({ top: 0, left: 0 });

  // ❌ Browser paints at top:0,left:0 first, then repaints after effect
  React.useEffect(() =&gt; {
    const rect = target.current.getBoundingClientRect();
    setPosition({ top: rect.bottom, left: rect.left });
  }, [target]);

  return &lt;div style={position}&gt;{content}&lt;/div&gt;;
}

// useLayoutEffect: no flash — DOM update happens before paint
function TooltipFixed({ target, content }) {
  const [position, setPosition] = React.useState({ top: 0, left: 0 });

  // ✅ Fires synchronously before paint — user never sees top:0,left:0
  React.useLayoutEffect(() =&gt; {
    const rect = target.current.getBoundingClientRect();
    setPosition({ top: rect.bottom, left: rect.left });
  }, [target]);

  return &lt;div style={position}&gt;{content}&lt;/div&gt;;
}</pre>

      <div id="hooks-q12" class="accordion-collapse collapse" data-bs-parent="#acc-hooks">
        <div class="accordion-body">
          <code>useImperativeHandle(ref, createHandle, [deps])</code> customizes the value that is exposed to the parent component when using <code>ref</code> with <code>forwardRef</code>. Instead of exposing the entire DOM node, you expose a custom object with only the methods and properties you want the parent to control. This is the principle of least privilege for imperative handles — parents should not have unrestricted DOM access. Common use cases: exposing a <code>focus()</code> method on a custom input component, exposing <code>scroll()</code> on a custom list component, or exposing <code>reset()</code> on a custom form. It must be used with <code>React.forwardRef</code>. Without <code>useImperativeHandle</code>, <code>forwardRef</code> would give the parent the raw DOM node.
          <pre class="code-block">// Child component: exposes only focus() and clear()
const CustomInput = React.forwardRef((props, ref) =&gt; {
  const inputRef = React.useRef(null);

  React.useImperativeHandle(ref, () =&gt; ({
    focus() {
      inputRef.current.focus();
    },
    clear() {
      inputRef.current.value = '';
    },
    // Parent does NOT get access to inputRef.current directly
  }));

  return &lt;input ref={inputRef} {...props} /&gt;;
});

// Parent: uses the custom handle
function Form() {
  const inputRef = React.useRef(null);

  return (
    &lt;&gt;
      &lt;CustomInput ref={inputRef} /&gt;
      &lt;button onClick={() =&gt; inputRef.current.focus()}&gt;Focus&lt;/button&gt;
      &lt;button onClick={() =&gt; inputRef.current.clear()}&gt;Clear&lt;/button&gt;
    &lt;/&gt;
  );
}</pre>

        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#hooks-q13">
          <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is useTransition?
        </button>
      </h2>
      <div id="hooks-q13" class="accordion-collapse collapse" data-bs-parent="#acc-hooks">
        <div class="accordion-body">
          <code>useTransition()</code> returns <code>[isPending, startTransition]</code> — a hook version of <code>startTransition</code> that also provides an <code>isPending</code> boolean you can use to show a loading indicator. It marks state updates inside <code>startTransition(callback)</code> as non-urgent (lower priority) — React can interrupt these renders to handle urgent updates (like user typing). <code>isPending</code> is <code>true</code> while React is processing the deferred transition updates, letting you show a spinner or "updating..." indicator without a full loading screen. It was introduced in React 18 as part of Concurrent Mode. Use it for: filtering/searching large lists (typing stays instant, filter update is deferred), page transitions (show previous page content while next page loads), tab switching with heavy content.
          <pre class="code-block">function SearchPage({ allItems }) {
  const [query, setQuery] = React.useState('');
  const [results, setResults] = React.useState(allItems);
  const [isPending, startTransition] = React.useTransition();

  function handleSearch(e) {
    const value = e.target.value;
    setQuery(value); // urgent: updates input immediately
    startTransition(() =&gt; {
      // non-urgent: can be interrupted by further typing
      setResults(allItems.filter(item =&gt;
        item.name.toLowerCase().includes(value.toLowerCase())
      ));
    });
  }

  return (
    &lt;div&gt;
      &lt;input value={query} onChange={handleSearch} placeholder="Search..." /&gt;
      {isPending &amp;&amp; &lt;span&gt;Updating...&lt;/span&gt;}
      &lt;ul style={{ opacity: isPending ? 0.7 : 1 }}&gt;
        {results.map(item =&gt; &lt;li key={item.id}&gt;{item.name}&lt;/li&gt;)}
      &lt;/ul&gt;
    &lt;/div&gt;
  );
}</pre>

        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#hooks-q14">
          <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is useDeferredValue?
        </button>
      </h2>
      <div id="hooks-q14" class="accordion-collapse collapse" data-bs-parent="#acc-hooks">
        <div class="accordion-body">
          <code>useDeferredValue(value)</code> accepts a value and returns a deferred copy of it — during urgent renders, the deferred copy stays at its previous value; React updates it in a background render after the urgent work is done. It is the "consumer-side" equivalent of <code>startTransition</code> — useful when you don't control the state update (e.g., the value comes from a prop or another component). The deferred value is always one step behind during fast updates, allowing expensive components that receive it to stay memoized and skip re-rendering until React has time. Combined with <code>React.memo</code>, <code>useDeferredValue</code> lets you build responsive UIs where fast-changing inputs (typing in a search box) don't block expensive downstream renders.
          <pre class="code-block">function SearchPage({ query }) {
  // deferredQuery lags behind query during rapid updates
  const deferredQuery = React.useDeferredValue(query);
  const isStale = query !== deferredQuery;

  // ExpensiveList only re-renders when deferredQuery changes (not on every keystroke)
  const results = React.useMemo(
    () =&gt; allItems.filter(item =&gt; item.name.includes(deferredQuery)),
    [deferredQuery]
  );

  return (
    &lt;div style={{ opacity: isStale ? 0.6 : 1, transition: 'opacity 0.2s' }}&gt;
      {results.map(item =&gt; &lt;Item key={item.id} item={item} /&gt;)}
    &lt;/div&gt;
  );
}</pre>

        <div class="accordion-body">
          <code>useId()</code> (React 18+) generates a unique, stable ID that is consistent between server-side rendering and client-side hydration — solving the hydration mismatch problem that occurs when IDs are generated differently on server vs client. It returns a string like <code>:r1:</code>. Use it for: linking <code>&lt;label&gt;</code> to <code>&lt;input&gt;</code> with <code>htmlFor</code>/<code>id</code>, ARIA attributes (<code>aria-describedby</code>, <code>aria-labelledby</code>), and any place where a unique HTML ID is needed. Before <code>useId</code>, developers used <code>Math.random()</code> or counter-based IDs which caused SSR hydration mismatches (server and client generated different IDs, React logged warnings). Do not use <code>useId</code> for list keys — it generates one ID per hook call, not per list item.
          <pre class="code-block">function FormField({ label, type }) {
  const id = React.useId(); // stable, unique: ":r3:"

  return (
    &lt;div&gt;
      &lt;label htmlFor={id}&gt;{label}&lt;/label&gt;
      &lt;input id={id} type={type} /&gt;
    &lt;/div&gt;
  );
}

      <h2 class="accordion-header">
        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#hooks-q16">

      <div id="hooks-q16" class="accordion-collapse collapse" data-bs-parent="#acc-hooks">
        <div class="accordion-body">
          <code>useSyncExternalStore(subscribe, getSnapshot, getServerSnapshot?)</code> is a React 18 hook for subscribing to external data stores (non-React state like browser APIs, Zustand, Redux, Valtio). It ensures the component always renders a consistent snapshot of the external store, preventing tearing in Concurrent Mode (where different parts of the tree might read different versions of external state during an interrupted render). The <code>subscribe</code> function is called with a callback that React calls to re-render when the store changes. <code>getSnapshot</code> returns the current store value. This hook replaced the <code>unstable_subscribe</code> pattern and is now the recommended way for library authors to integrate external stores with React. Redux, Zustand, and other stores use this internally.
          <pre class="code-block">// Custom hook using useSyncExternalStore
function useWindowWidth() {
  return React.useSyncExternalStore(
    // subscribe: React calls callback when store changes
    (callback) =&gt; {
      window.addEventListener('resize', callback);
      return () =&gt; window.removeEventListener('resize', callback);
    },
    // getSnapshot: returns current value
    () =&gt; window.innerWidth,
    // getServerSnapshot: SSR value
    () =&gt; 1024
  );
}

function Component() {
  const width = useWindowWidth();
  return &lt;p&gt;Window width: {width}px&lt;/p&gt;;
}</pre>

        <div class="accordion-body">
          <code>useDebugValue(value, format?)</code> displays a label for custom hooks in React DevTools. When you inspect a component in DevTools, custom hooks that use <code>useDebugValue</code> show a descriptive label next to the hook name instead of the raw state value. This makes complex custom hooks easier to debug — instead of seeing the raw state structure, you see a meaningful description. The optional <code>format</code> function is called lazily (only when DevTools is open) — useful for expensive formatting operations. <code>useDebugValue</code> is intended only for custom hooks shared in libraries or within teams; adding it to every custom hook is unnecessary overhead.
          <pre class="code-block">function useOnlineStatus() {
  const [isOnline, setIsOnline] = React.useState(navigator.onLine);

  React.useEffect(() =&gt; {
    const handleOnline = () =&gt; setIsOnline(true);
    const handleOffline = () =&gt; setIsOnline(false);
    window.addEventListener('online', handleOnline);
    window.addEventListener('offline', handleOffline);
    return () =&gt; {
      window.removeEventListener('online', handleOnline);
      window.removeEventListener('offline', handleOffline);
    };
  }, []);

  // DevTools shows: OnlineStatus: "Online" or "Offline"
  React.useDebugValue(isOnline ? 'Online' : 'Offline');

  return isOnline;
}</pre>

      <h2 class="accordion-header">
        <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#hooks-q18">
          <span class="diff-hard">Hard</span>&nbsp;&nbsp;Explain the useEffect dependency array in detail. What happens with each case?
        </button>
      </h2>
      <div id="hooks-q18" class="accordion-collapse collapse" data-bs-parent="#acc-hooks">
        <div class="accordion-body">
          The <code>useEffect</code> dependency array controls when the effect re-runs: (1) <code>useEffect(fn)</code> — no array: runs after every render; (2) <code>useEffect(fn, [])</code> — empty array: runs once after mount, cleanup runs on unmount; (3) <code>useEffect(fn, [a, b])</code> — with deps: runs after mount and whenever <code>a</code> or <code>b</code> changes (by reference/value). The comparison uses <code>Object.is</code> (strict equality) — objects and functions compare by reference, so a new object <code>{}</code> always triggers the effect even if its contents are the same. This is why you must memoize objects/functions used as deps with <code>useMemo</code>/<code>useCallback</code>. Omitting a variable from the deps array that is used inside the effect is a bug (stale closure) — always include all values from the component scope used inside the effect. The <code>exhaustive-deps</code> ESLint rule enforces this.

          <span class="diff-hard">Hard</span>&nbsp;&nbsp;What are stale closures in React hooks and how do you fix them?
        </button>
      </h2>
      <div id="hooks-q19" class="accordion-collapse collapse" data-bs-parent="#acc-hooks">
        <div class="accordion-body">
          A stale closure occurs when a function (like an effect, event handler, or memoized callback) captures a variable from its outer scope at creation time, and that variable later changes without the function being recreated. The function "closes over" the old value and uses it even though the current value is different. In React: a <code>useEffect</code> with <code>[]</code> deps captures all variables at mount time — any state/prop it uses will be stale on subsequent renders; a <code>useCallback</code> with stale deps returns the same function with the old captured values. Fixes: (1) Add all used variables to the dependency array (the linter will help); (2) Use the functional updater form of setState — <code>setState(prev =&gt; prev + 1)</code> doesn't need <code>count</code> in deps; (3) Use a ref to store the latest value of frequently-changing variables that shouldn't be in deps.
          <pre class="code-block">// Stale closure in event handler

    const id = setInterval(() =&gt; {
      console.log(count); // ❌ always logs 0 — stale closure
      setCount(count + 1); // ❌ always sets to 1 (0 + 1)
    }, 1000);
    return () =&gt; clearInterval(id);
  }, []); // [] captures count=0 forever
}

// Fix 1: functional updater (no stale closure for incrementing)

  const id = setInterval(() =&gt; {
    setCount(prev =&gt; prev + 1); // ✅ always uses current value
  }, 1000);
  return () =&gt; clearInterval(id);
}, []); // safe because we don't read count anymore

// Fix 2: use a ref for the latest callback
function useInterval(callback, delay) {
  const callbackRef = React.useRef(callback);
  React.useLayoutEffect(() =&gt; { callbackRef.current = callback; });
  React.useEffect(() =&gt; {
    const id = setInterval(() =&gt; callbackRef.current(), delay);
    return () =&gt; clearInterval(id);
  }, [delay]);
}</pre>

          <pre class="code-block">function useFetch(url) {

    loading: true,
    error: null,
  });

    if (!url) return;

    let cancelled = false;

    setState({ data: null, loading: true, error: null });

    fetch(url, { signal: controller.signal })
      .then(res =&gt; {
        if (!res.ok) throw new Error(`HTTP ${res.status}`);
        return res.json();

        if (!cancelled) setState({ data, loading: false, error: null });

      .catch(err =&gt; {

          setState({ data: null, loading: false, error: err.message });
        }
      });

    return () =&gt; {
      cancelled = true;
      controller.abort(); // cancel in-flight request when url changes or unmount
    };
  }, [url]);

  return state;

  React.useEffect(() =&gt; {
    const targetElement = element?.current ?? element;
    if (!targetElement?.addEventListener) return;

    targetElement.addEventListener(eventName, eventListener);
    return () =&gt; targetElement.removeEventListener(eventName, eventListener);
  }, [eventName, element]); // re-subscribe only if eventName or element changes

          <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is the purpose of the deps array in useCallback?
        </button>
      </h2>
      <div id="hooks-q33" class="accordion-collapse collapse" data-bs-parent="#acc-hooks">
        <div class="accordion-body">
          <code>useCallback(fn, deps)</code> returns a memoized version of <code>fn</code> — the same function reference is returned as long as <code>deps</code> haven't changed. The deps array tells React when to create a new function. If deps are stale (not included in the array), the function closes over old values. The <code>exhaustive-deps</code> ESLint rule helps by warning when you've omitted a dep. Common mistake: <code>useCallback(fn, [])</code> with an empty array — the function closes over its initial values and never gets the updated ones. The correct approach: include all values from component scope that the function reads. When the deps are frequently-changing, <code>useCallback</code> doesn't help much — the function changes as often as without it. React Compiler (React 19) automatically applies memoization, making manual <code>useCallback</code> largely unnecessary going forward.

          Throttling limits function execution to at most once per interval — unlike debounce (which delays until activity stops), throttle ensures regular execution during continuous activity. A <code>useThrottle</code> hook returns a throttled version of a value or callback. Uses: scroll event handlers, mouse move tracking, resize handlers, and any high-frequency event where you want regular but rate-limited updates.
          <pre class="code-block">// Throttle a value: only updates at most once per 200ms
function useThrottle(value, limit = 200) {
  const [throttledValue, setThrottledValue] = React.useState(value);
  const lastUpdated = React.useRef(Date.now());

  React.useEffect(() =&gt; {
    const now = Date.now();
    const remaining = limit - (now - lastUpdated.current);

    if (remaining &lt;= 0) {
      lastUpdated.current = now;
      setThrottledValue(value);
    } else {
      const timer = setTimeout(() =&gt; {
        lastUpdated.current = Date.now();
        setThrottledValue(value);
      }, remaining);
      return () =&gt; clearTimeout(timer);
    }
  }, [value, limit]);

  return throttledValue;
}

// Usage: scroll position updates throttled to 100ms
function ScrollTracker() {
  const [scrollY, setScrollY] = React.useState(0);
  useEventListener('scroll', () =&gt; setScrollY(window.scrollY));
  const throttledScrollY = useThrottle(scrollY, 100);
  return &lt;p&gt;Scroll: {throttledScrollY}px&lt;/p&gt;;
}</pre>

          <pre class="code-block">function useForm({ initialValues, validate, onSubmit }) {
  const [values, setValues] = React.useState(initialValues);
  const [errors, setErrors] = React.useState({});
  const [touched, setTouched] = React.useState({});
  const [isSubmitting, setIsSubmitting] = React.useState(false);

  const handleChange = React.useCallback((e) =&gt; {
    const { name, value } = e.target;
    setValues(prev =&gt; ({ ...prev, [name]: value }));
  }, []);

  const handleBlur = React.useCallback((e) =&gt; {
    const { name } = e.target;
    setTouched(prev =&gt; ({ ...prev, [name]: true }));
    if (validate) {
      const validationErrors = validate(values);
      setErrors(validationErrors);
    }
  }, [validate, values]);

  const handleSubmit = React.useCallback((e) =&gt; {
    e.preventDefault();
    const validationErrors = validate ? validate(values) : {};
    setErrors(validationErrors);
    setTouched(Object.keys(values).reduce((acc, key) =&gt; ({ ...acc, [key]: true }), {}));
    if (Object.keys(validationErrors).length === 0) {
      setIsSubmitting(true);
      Promise.resolve(onSubmit(values)).finally(() =&gt; setIsSubmitting(false));
    }
  }, [validate, values, onSubmit]);

  return { values, errors, touched, isSubmitting, handleChange, handleBlur, handleSubmit };
}</pre>

          This is a classic debugging challenge.     case 'BACK':
      return { ...state, step: state.step - 1 };
    case 'SUBMIT':
      return { ...state, submitting: true };

  }
