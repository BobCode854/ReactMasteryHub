Fetch, Axios, React Query, error boundaries, and loading state patterns

Fetch with useEffect

The foundational pattern — use useEffect to trigger a fetch when the component mounts. Always include an AbortController to cancel in-flight requests when the component unmounts.

Memory leak warning: Without cleanup, a fetch that resolves after unmount will call setState on an unmounted component. The AbortController prevents this. 
function UsersList() {
  const [users, setUsers] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    const controller = new AbortController();

    fetch('https://jsonplaceholder.typicode.com/users', {
      signal: controller.signal
    })
      .then(res => {
        if (!res.ok) throw new Error(`HTTP ${res.status}`);
        return res.json();
      })
      .then(data => { setUsers(data); setLoading(false); })
      .catch(err => {
        if (err.name !== 'AbortError') {
          setError(err.message);
          setLoading(false);
        }
      });

    return () => controller.abort(); // cleanup on unmount
  }, []);

  if (loading) return <div>Loading...</div>;
  if (error) return <div>Error: {error}</div>;
  return <ul>{users.map(u => <li key={u.id}>{u.name}</li>)}</ul>;
}

Axios Setup + Interceptors

Axios interceptors centralise cross-cutting concerns — auth token injection, error handling, and response transformation — so every API call benefits automatically.

// api/axios.js — shared axios instance
import axios from 'axios';

const api = axios.create({
  baseURL: process.env.REACT_APP_API_URL,
  timeout: 10000,
  headers: { 'Content-Type': 'application/json' },
});

// Request interceptor — inject auth token
api.interceptors.request.use(config => {
  const token = localStorage.getItem('token');
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

// Response interceptor — handle 401 globally
api.interceptors.response.use(
  response => response.data, // unwrap .data automatically
  error => {
    if (error.response?.status === 401) {
      localStorage.removeItem('token');
      window.location.href = '/login';
    }
    return Promise.reject(error);
  }
);

export default api;

React Query (TanStack Query) — Modern Standard

React Query handles loading, error, caching, background refetching, and optimistic updates — eliminating most manual useEffect data-fetching code.

Why React Query? It replaces ~50 lines of manual fetch boilerplate with 3 lines, adds automatic background refetching, cache invalidation, and deduplication of identical requests. 
import { useQuery, useMutation, useQueryClient } from '@tanstack/react-query';

// Read data — handles loading, error, caching, background refetch
function UsersList() {
  const { data: users, isLoading, error } = useQuery({
    queryKey: ['users'],
    queryFn: () => api.get('/users'),
    staleTime: 5 * 60 * 1000, // 5 minutes before background refetch
  });

  if (isLoading) return <Spinner />;
  if (error) return <Error message={error.message} />;
  return <UserList users={users} />;
}

// Write data — mutation with automatic cache invalidation
function CreateUser() {
  const queryClient = useQueryClient();

  const mutation = useMutation({
    mutationFn: (newUser) => api.post('/users', newUser),
    onSuccess: () => {
      queryClient.invalidateQueries(['users']); // refetch list automatically
    },
  });

  return (
    <button onClick={() => mutation.mutate({ name: 'John', email: 'j@email.com' })}>
      {mutation.isPending ? 'Creating...' : 'Create User'}
    </button>
  );
}

Error Handling Patterns

Error Boundaries catch JavaScript errors anywhere in the component tree and display a fallback UI instead of crashing the whole app. They only work with class components or the react-error-boundary library.

// Class-based Error Boundary
class ErrorBoundary extends React.Component {
  state = { hasError: false, error: null };

  static getDerivedStateFromError(error) {
    return { hasError: true, error };
  }

  componentDidCatch(error, info) {
    logErrorToService(error, info); // send to Sentry / DataDog
  }

  render() {
    if (this.state.hasError) {
      return <ErrorFallback
        error={this.state.error}
        reset={() => this.setState({ hasError: false })}
      />;
    }
    return this.props.children;
  }
}

// Modern alternative — react-error-boundary library
import { ErrorBoundary } from 'react-error-boundary';

<ErrorBoundary
  fallbackRender={({ error, resetErrorBoundary }) => (
    <div>
      <p>Something went wrong: {error.message}</p>
      <button onClick={resetErrorBoundary}>Try again</button>
    </div>
  )}
  onError={(error, info) => logError(error, info)}
>
  <MyComponent />
</ErrorBoundary>

Interview Questions

Why do you need to return a cleanup function from useEffect when fetching data?
Without cleanup, if the component unmounts while a fetch is in flight, the .then() callback will still execute and call setState on an unmounted component — causing a memory leak warning. Returning () => controller.abort() cancels the pending request on unmount.
What does staleTime do in React Query?
staleTime defines how long (in ms) cached data is considered "fresh." During this window, React Query serves the cache without a background refetch. After it expires, data becomes "stale" and will be refetched in the background on next access. Default is 0 (always stale).
What is the difference between Error Boundaries and try/catch?
try/catch handles errors in imperative code (event handlers, async functions). Error Boundaries catch errors that occur during rendering, in lifecycle methods, and in constructors of the whole child tree — which try/catch cannot do. They complement each other: use both.
How do Axios interceptors differ from writing wrapper functions?
Interceptors run automatically for every request/response through the Axios instance — you write the logic once and it applies globally. Wrapper functions require every caller to remember to call them. Interceptors also have access to request config and can modify or cancel requests before they're sent.
When should you use React Query vs useEffect for data fetching?
Use React Query for any server state — data that lives on a server and needs to be synchronised. Use useEffect only for simple, one-off fetches in small apps or when adding React Query is overhead. React Query's caching, deduplication, background refetch, and mutation support make it the better default for production apps.
How does React Query handle cache invalidation after a mutation?
In the onSuccess callback of useMutation, call queryClient.invalidateQueries(['key']). This marks the cached data as stale and triggers a background refetch for any active subscribers of that query key, keeping the UI in sync without a manual page refresh.
What is optimistic updating and how do you implement it?
Optimistic updating immediately updates the UI as if the mutation succeeded, then rolls back if the server returns an error. In React Query, use onMutate to update the cache before the request, save the previous value as a snapshot, and use onError to roll back to the snapshot with queryClient.setQueryData.
Why are skeleton screens better than spinners for perceived performance?
Skeleton screens give users an immediate visual representation of the content's structure, reducing uncertainty and perceived wait time. A blank screen with a spinner gives no indication of what will appear — the brain registers more "waiting." Skeletons also prevent layout shift when real content loads.
