      Q26. How do error boundaries interact with concurrent mode? <span class="diff-hard">Hard</span>
    </button>
  </h2>
  <div id="collapse-advanced-q26" class="accordion-collapse collapse" data-bs-parent="#acc-advanced">
    <div class="accordion-body">
      <p>In concurrent mode, React can render component trees multiple times before committing to the DOM — this has significant implications for error boundaries. When an error is thrown during a concurrent render attempt, React will catch it and look for the nearest error boundary up the tree. However, because concurrent renders are interruptible and may be retried, React may call <code>render()</code> on a component multiple times before it either commits or is discarded; error boundary lifecycle methods are only invoked after React decides to commit the erroring subtree.</p>
      <p>One key difference from legacy mode: in concurrent mode, React retries a failed render automatically before giving up and triggering the error boundary. Specifically, React will attempt to render the erroring tree a second time without the failing component, using the error boundary's fallback UI. This retry mechanism means you may see intermittent errors in development that are "swallowed" on the retry — which is intentional behavior to improve resilience.</p>
      <p>Error boundaries do NOT catch errors in event handlers, asynchronous code (setTimeout, fetch), or server-side rendering. They only catch errors during rendering, in lifecycle methods, and in constructors of child components. With concurrent features like <code>startTransition</code>, if an error occurs inside a transition, the error boundary still catches it, but React may choose to revert to the previous committed state rather than showing the fallback if the transition has not yet committed.</p>
      <p>Another subtle point: with React 18's concurrent renderer, the error may be thrown during the "render phase" (before commit), meaning the DOM has not been updated yet. The error boundary's <code>componentDidCatch</code> will still be called in the "commit phase," but the error originally occurred before any DOM mutations, making recovery cleaner. You should use error boundaries to wrap discrete UI sections so that a failure in one panel doesn't bring down the entire application.</p>
<pre class="code-block">
class ErrorBoundary extends React.Component {
  constructor(props) {
    super(props);
    this.state = { hasError: false, error: null };
  }

  // Called during render phase — used to derive fallback state
  static getDerivedStateFromError(error) {
    return { hasError: true, error };
  }

  // Called during commit phase — used for side effects (logging)
  componentDidCatch(error, errorInfo) {
    console.error('Error caught by boundary:', error);
    console.error('Component stack:', errorInfo.componentStack);
    // Report to monitoring service
    reportError(error, errorInfo);
  }

  render() {
    if (this.state.hasError) {
      return (
        &lt;div role="alert"&gt;
          &lt;h2&gt;Something went wrong.&lt;/h2&gt;
          &lt;button onClick={() =&gt; this.setState({ hasError: false, error: null })}&gt;
            Try again
          &lt;/button&gt;
        &lt;/div&gt;
      );
    }
    return this.props.children;
  }
}

// Usage with concurrent features
function App() {
  return (
    &lt;ErrorBoundary&gt;
      &lt;React.Suspense fallback={&lt;Spinner /&gt;}&gt;

    &lt;/ErrorBoundary&gt;
  );
}

      Q27. What is the difference between getDerivedStateFromError and componentDidCatch? <span class="diff-hard">Hard</span>
    </button>
  </h2>
  <div id="collapse-advanced-q27" class="accordion-collapse collapse" data-bs-parent="#acc-advanced">
    <div class="accordion-body">
      <p><code>getDerivedStateFromError</code> and <code>componentDidCatch</code> are two separate lifecycle methods that together form a complete error boundary implementation. They serve fundamentally different purposes and are called at different phases of the React rendering cycle. Understanding the distinction is critical for building robust error boundaries.</p>
      <p><code>getDerivedStateFromError(error)</code> is a static method called during the <strong>render phase</strong>, before the DOM has been updated. It receives the thrown error as its only argument and must return a state object (or null) that React uses to update the component's state — this state update is what causes the error boundary to re-render with the fallback UI. Because it runs during the render phase, it must be a pure function with no side effects. You cannot call <code>setState</code>, dispatch actions, log to external services, or perform any I/O inside this method.</p>
      <p><code>componentDidCatch(error, errorInfo)</code> is called during the <strong>commit phase</strong>, after the DOM has been updated with the fallback UI. It receives the error AND an <code>errorInfo</code> object containing a <code>componentStack</code> string that traces which components in the tree threw the error. This is where you perform side effects: logging to error monitoring services (Sentry, Datadog), dispatching analytics events, or triggering notifications. It CAN call <code>setState</code>, though this is rarely needed since <code>getDerivedStateFromError</code> handles the state update.</p>
      <p>A practical rule: use <code>getDerivedStateFromError</code> to render fallback UI, and use <code>componentDidCatch</code> to log the error. If you only implement <code>componentDidCatch</code> and call <code>setState</code> inside it to trigger the fallback, there will be a brief flash of the broken UI before the fallback appears, because <code>componentDidCatch</code> runs after the commit. <code>getDerivedStateFromError</code> prevents that flash by updating state synchronously during the render phase.</p>
<pre class="code-block">
class RobustErrorBoundary extends React.Component {
  state = { hasError: false, error: null, errorInfo: null };

  // RENDER PHASE: pure, no side effects, returns new state
  static getDerivedStateFromError(error) {
    // This prevents the broken UI from ever being committed to the DOM
    return { hasError: true, error };
  }

  // COMMIT PHASE: side effects allowed here
  componentDidCatch(error, errorInfo) {
    // errorInfo.componentStack traces the component tree
    this.setState({ errorInfo });

    // Safe to call external services here
    Sentry.captureException(error, {
      contexts: {
        react: { componentStack: errorInfo.componentStack }
      }
    });
  }

  render() {
    const { hasError, error, errorInfo } = this.state;
    if (hasError) {
      return (
        &lt;div className="error-fallback"&gt;
          &lt;h2&gt;Oops! Something broke.&lt;/h2&gt;
          {process.env.NODE_ENV === 'development' &amp;&amp; (
            &lt;details&gt;
              &lt;summary&gt;Error details&lt;/summary&gt;
              &lt;pre&gt;{error?.toString()}&lt;/pre&gt;
              &lt;pre&gt;{errorInfo?.componentStack}&lt;/pre&gt;
            &lt;/details&gt;
          )}
        &lt;/div&gt;
      );
    }
    return this.props.children;
  }
}

  <div id="collapse-advanced-q27" class="accordion-collapse collapse" data-bs-parent="#acc-advanced">
    <div class="accordion-body">
      <p>A Suspense-compatible data source must integrate with React's "throw a Promise" protocol — when data is not yet available, the component throws a Promise that React catches, registers the Suspense boundary's fallback UI, and then re-renders the component once the Promise resolves. This pattern is sometimes called "suspending" a component. Implementing it correctly requires a cache layer that tracks promise states (pending, resolved, rejected) to avoid re-triggering fetches on every render.</p>
      <p>The canonical approach involves a <strong>resource cache</strong> — a map from cache keys to resource objects. Each resource object has a <code>read()</code> method that either throws a Promise (if pending), throws an error (if rejected), or returns the data (if resolved). React will call <code>read()</code> during rendering; if it throws a Promise, React suspends the subtree and shows the nearest Suspense fallback. When the Promise resolves, React replays the render and <code>read()</code> returns the data synchronously.</p>
      <p>With React 18 and the <code>use()</code> hook (stabilized in React 19), you can pass a Promise directly to <code>use(promise)</code> and React handles the suspend/resume cycle automatically. This is cleaner than the manual "throw a Promise" approach and is the recommended API going forward. The <code>use()</code> hook can be called conditionally inside components, unlike other hooks.</p>
      <p>For production-grade implementations, libraries like SWR, React Query (TanStack Query), and Relay implement Suspense-compatible caches with features like deduplication, background refetching, cache invalidation, and stale-while-revalidate patterns. Rolling your own is educational but unnecessary for most applications. The key invariant is that a resource's <code>read()</code> method must be referentially stable across renders for the same cache key, otherwise you create infinite suspend loops.</p>
<pre class="code-block">
// Manual Suspense-compatible resource (educational)
function createResource(fetchFn) {
  let status = 'pending';
  let result;

  const promise = fetchFn().then(
    (data) => { status = 'success'; result = data; },
    (error) => { status = 'error'; result = error; }
  );

  return {
    read() {
      if (status === 'pending') throw promise;   // React catches this
      if (status === 'error') throw result;      // Error boundary catches this
      return result;                              // Data available!
    }
  };
}

// Cache to prevent recreating resources on every render
const cache = new Map();

function fetchUser(userId) {
  if (!cache.has(userId)) {
    cache.set(userId, createResource(() =>
      fetch(`/api/users/${userId}`).then(r =&gt; r.json())
    ));
  }
  return cache.get(userId);
}

// Component that suspends while data loads
function UserProfile({ userId }) {
  const resource = fetchUser(userId);
  const user = resource.read(); // Throws Promise if not ready
  return &lt;div&gt;{user.name}&lt;/div&gt;;
}

// React 19 approach using use() hook

      Q29. What is the "Suspense waterfall" problem and how do you avoid it? <span class="diff-hard">Hard</span>
    </button>
  </h2>
  <div id="collapse-advanced-q29" class="accordion-collapse collapse" data-bs-parent="#acc-advanced">
    <div class="accordion-body">
      <p>The Suspense waterfall problem occurs when multiple components that each need to fetch data are nested inside one another, causing fetches to execute sequentially rather than in parallel. When component A suspends, React shows A's fallback and doesn't render B at all — so B's fetch never starts. When A's data arrives, React renders A, which renders B, which then suspends and starts its own fetch. This creates a "waterfall" of sequential network requests: A fetch → A renders → B fetch → B renders → C fetch, and so on. Total load time becomes the sum of all individual fetch times instead of the maximum.</p>
      <p>The root cause is that React cannot know about B's data requirements until A finishes rendering and React reaches B in the tree. This is a fundamental limitation of the "fetch-on-render" pattern. The solution is to adopt a "render-as-you-fetch" or "fetch-then-render" approach where all required data is fetched in parallel before (or concurrently with) rendering begins.</p>
      <p>The primary mitigation strategy is <strong>hoisting fetches</strong>: initiate all data fetches as high up the component tree as possible (often at the route level or in a parent component), pass the resulting Promises down as props, and let child components <code>use()</code> those Promises. Since all fetches start at the same time, they run in parallel. When child components suspend, the data may already be available or will arrive sooner since all requests were fired simultaneously.</p>
      <p>A complementary approach is using <strong>parallel Suspense boundaries</strong> by placing siblings (rather than nested components) inside separate Suspense boundaries. React renders sibling boundaries concurrently, so their fetches can proceed in parallel. Another powerful technique is using a data-fetching framework (like Next.js or Remix) that uses server-side data loading to pre-fetch everything before the client renders, eliminating client-side waterfalls entirely.</p>
<pre class="code-block">
// PROBLEM: Waterfall pattern (fetch-on-render)

    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#collapse-advanced-q30" aria-expanded="false" aria-controls="collapse-advanced-q30">
      Q30. What is React's reconciliation algorithm for trees with keys vs without? <span class="diff-hard">Hard</span>
    </button>
  </h2>
  <div id="collapse-advanced-q30" class="accordion-collapse collapse" data-bs-parent="#acc-advanced">
    <div class="accordion-body">
      <p>React's reconciliation algorithm (nicknamed "Fiber diffing") compares the previous virtual DOM tree with the new one to determine the minimal set of DOM mutations needed. The algorithm operates under two key heuristics that make it O(n) instead of the theoretically optimal O(n³) for tree diffing: (1) elements of different types produce entirely different trees, and (2) the <code>key</code> prop hints to React that a specific child element should be stable across renders.</p>
      <p><strong>Without keys</strong>, React reconciles children by position. If you have a list <code>[A, B, C]</code> and it becomes <code>[X, A, B, C]</code>, React compares position 0's A with X (different type/content → destroys and recreates), position 1's B with A, position 2's C with B, and creates a new node for C. This is maximally inefficient — React destroys and recreates most of the list when only an insertion at the front occurred. State attached to those components is also destroyed.</p>
      <p><strong>With keys</strong>, React builds an identity map from key → fiber for the previous render. During the new render, each child element's key is looked up in this map. If found, React reuses the existing fiber (and its DOM node) and reconciles only the changed props — no unmount/remount occurs. If not found, React creates a new fiber. If a previous key is absent from the new render, React destroys that fiber. This makes list reordering O(n) instead of O(n²).</p>
      <p>Keys must be stable (not random), unique among siblings (not globally), and meaningful to the data (preferably a database ID). Using array indices as keys is safe ONLY when the list never reorders or filters — if items can reorder, index-based keys defeat the purpose and can cause serious bugs where component state (like form input values) follows the wrong items. Keys are not passed as props to the component; they are used solely by the reconciler.</p>
<pre class="code-block">
// WITHOUT keys — position-based reconciliation
// Adding to front: React destroys and recreates most nodes
function BadList({ items }) {
  return (
    &lt;ul&gt;
      {items.map((item, index) =&gt; (
        // Using index as key is as bad as no key when list reorders
        &lt;li key={index}&gt;
          &lt;ItemComponent data={item} /&gt;
        &lt;/li&gt;
      ))}
    &lt;/ul&gt;
  );
}

// WITH stable keys — identity-based reconciliation
// React reuses existing DOM nodes and component instances
function GoodList({ items }) {
  return (
    &lt;ul&gt;
      {items.map((item) =&gt; (
        &lt;li key={item.id}&gt;  {/* Stable ID from database */}
          &lt;ItemComponent data={item} /&gt;
        &lt;/li&gt;
      ))}
    &lt;/ul&gt;
  );
}

//    - Previous key missing: destroy fiber

      <p>The <code>Offscreen</code> component was an experimental React 18 API designed to allow components to remain mounted and preserve their state while being visually hidden or deprioritized.       <p>Offscreen would support two modes: <code>hidden</code> (component is mounted but not visible, rendered at low priority) and <code>visible</code> (component is fully active). Switching from hidden to visible would be instant since the component was never unmounted. This solves the "tab flicker" problem where switching back to a previously visited tab causes a re-fetch and re-render from scratch.</p>

Effects run differently for offscreen trees — <code>useEffect</code> cleanup is called when a component goes offscreen, and setup re-runs when it comes back, similar to a focus/blur lifecycle. This respects the "effects represent synchronization" mental model while still preserving state. The component was renamed to <code>Activity</code> in React 19.</p>

    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#collapse-advanced-q33" aria-expanded="false" aria-controls="collapse-advanced-q33">
      Q33. What is the ViewTransition component in React 19? <span class="diff-medium">Medium</span>

  <div id="collapse-advanced-q33" class="accordion-collapse collapse" data-bs-parent="#acc-advanced">
    <div class="accordion-body">
      <p>React 19 introduced the <code>ViewTransition</code> component as a first-class integration with the browser's View Transitions API. The View Transitions API allows browsers to animate between different DOM states with smooth, GPU-accelerated transitions — traditionally this required complex manual DOM manipulation or JavaScript animation libraries. React's <code>ViewTransition</code> wraps state updates in a view transition automatically, capturing the before/after DOM snapshots and applying CSS animations between them.</p>
      <p>When a state update triggers a re-render inside a <code>ViewTransition</code> boundary, React coordinates with the browser to: snapshot the current DOM state, apply the update, snapshot the new DOM state, and then animate between them using the browser's native transition engine.       <p>React's architecture separates the reconciler (diffing, scheduling, component lifecycle) from the renderer (DOM manipulation). This separation is exposed via the <code>react-reconciler</code> package, which lets you create a custom renderer targeting any "host environment" — not just the browser DOM. Custom renderers are how React Native, React Three Fiber, Ink (terminal UI), and other React platforms work.</p>
      <p>To create a custom renderer, you call <code>ReactReconciler(hostConfig)</code> with a host configuration object that defines how to create, update, and delete instances in your target environment. Key host config methods include <code>createInstance</code> (create a host element), <code>appendChildToContainer</code> (add to root), <code>commitUpdate</code> (apply prop changes), and <code>removeChild</code> (delete an element). The reconciler calls these methods during its commit phase after diffing the virtual tree.</p>
      <p>The reconciler operates in mutation mode (direct instance mutation, like the DOM) or persistent mode (create new instances on update, like React Native's bridge). Most custom renderers use mutation mode. The <code>prepareUpdate</code> method receives old and new props and returns an update payload — if null, no update is needed; if non-null, <code>commitUpdate</code> is called during the commit phase with that payload. This two-phase approach (prepare in render phase, commit in commit phase) keeps the reconciler's rendering interruptible while ensuring DOM mutations are always synchronous and complete.</p>

      <p>The <code>useState</code> hook accepts either a value or an <strong>initializer function</strong> as its argument. This is called "lazy initialization" and it's a critical performance optimization for cases where computing the initial state is expensive. React only calls the initializer function <strong>once</strong> — during the initial render — and ignores it on subsequent renders. In contrast, if you pass a computed value directly, that computation executes on every render even though React discards the result after the first render.</p>
      <p>The syntax difference is subtle but important: <code>useState(expensiveComputation())</code> — with parentheses — calls the function on every render. <code>useState(expensiveComputation)</code> — without parentheses, passing the function reference — calls it only once at mount. The same pattern applies to <code>useReducer</code>: its optional third argument is an initializer function called with the second argument as input, enabling lazy initialization of complex reducer state.</p>
      <p>Common use cases for lazy initialization include reading from localStorage or sessionStorage (synchronous I/O that blocks the main thread), parsing large JSON blobs, filtering or sorting large arrays to derive an initial view, complex object construction with many computed fields, and any computation that would visibly affect render performance. When the initial state depends on props, use an arrow function closure: <code>useState(() =&gt; computeFromProps(initialProps))</code> — the closure captures the props value at mount time, which is correct since initial state only uses initial props, not updated ones.</p>

    <div class="accordion-body">
      <p>Stale closures are one of the most pervasive bugs in React hooks, and concurrent mode amplifies the problem because renders can be interrupted, paused, and replayed. A stale closure occurs when a function (event handler, effect callback, timeout) captures a variable from its enclosing scope at creation time, but by the time the function executes, the underlying React state has been updated — yet the function still references the old value from its closed-over scope.</p>
      <p>In legacy synchronous mode, the render-to-commit cycle was uninterrupted, so stale closures mostly surfaced in async operations (setTimeout, fetch callbacks). In concurrent mode, React may start rendering a component, pause it to work on higher-priority updates, then resume — creating multiple "versions" of a component's render with different state snapshots. Closures created during one render version may be called with values from a different render version's state.</p>
      <p>The primary solutions are: (1) Use the <strong>functional update form</strong> of setState — <code>setState(prev =&gt; prev + 1)</code> — which always receives the latest state value regardless of when the closure was created. (2) Use <code>useRef</code> to hold a "latest value" ref that is mutated synchronously on every render — event handlers can read from the ref to always get the current value. (3) Use <code>useReducer</code> with action dispatching instead of direct state reads in callbacks. (4) Properly declare dependencies in <code>useEffect</code> and <code>useCallback</code> dependency arrays so closures are recreated when their dependencies change.</p>
<pre class="code-block">
// STALE CLOSURE EXAMPLE
function Counter() {
  const [count, setCount] = useState(0);

  // BAD: This closure captures count=0 at creation time
  useEffect(() =&gt; {
    const timer = setInterval(() =&gt; {
      setCount(count + 1); // Always adds 1 to the stale count=0!
    }, 1000);
    return () =&gt; clearInterval(timer);
  }, []); // Empty deps — closure never updates

  return &lt;div&gt;{count}&lt;/div&gt;;
}

// FIX 1: Functional update form (always uses latest state)
function CounterFixed1() {
  const [count, setCount] = useState(0);

  useEffect(() =&gt; {
    const timer = setInterval(() =&gt; {
      setCount(prev =&gt; prev + 1); // prev is always current state
    }, 1000);
    return () =&gt; clearInterval(timer);
  }, []);

  return &lt;div&gt;{count}&lt;/div&gt;;
}

// FIX 2: useRef to track latest value
function CounterFixed2() {
  const [count, setCount] = useState(0);
  const countRef = useRef(count);
  countRef.current = count; // Always in sync with latest render

  const handleAsync = useCallback(() =&gt; {
    setTimeout(() =&gt; {
      console.log('Current count:', countRef.current); // Always latest
    }, 2000);
  }, []); // No dependency on count needed

function ConcurrentPitfall() {
  const [items, setItems] = useState([]);

    startTransition(() =&gt; {
      // If this render is interrupted, items may be stale
      // Use functional update to always work from latest state:
      setItems(prevItems =&gt; [...prevItems, { id: Date.now() }]);

    <div class="accordion-body">
      <p>"Tearing" is a visual inconsistency bug unique to concurrent mode where different parts of the UI show different values for the same external state during a single render pass. In legacy synchronous rendering, this was impossible because the entire render was uninterruptible — all components saw the same snapshot of state. In concurrent mode, React can pause a render, yield to the browser, and then resume — and if an external store (Redux, Zustand, Jotai, or any non-React state) updates during that pause, some components will have rendered with the old value and others with the new value, resulting in visual tearing.</p>
      <p>React 18 introduced <code>useSyncExternalStore</code> specifically to solve this problem. The hook takes three arguments: a <code>subscribe</code> function (to register a listener for store updates), a <code>getSnapshot</code> function (to read the current store value), and an optional <code>getServerSnapshot</code> for SSR. React uses the snapshot to detect if the external store changed mid-render — if it did, React synchronously re-renders the affected components before committing, ensuring consistency.</p>
      <p>The key invariant is that <code>getSnapshot</code> must return a <strong>referentially stable</strong> value when the store hasn't changed. If <code>getSnapshot</code> creates a new object or array on every call (even with the same underlying data), React will enter an infinite loop of detecting "changes" and re-rendering. You must either return primitive values, or memoize the snapshot (e.g., compare deeply and return the previous reference if unchanged). All state management libraries targeting React 18 (Redux, Zustand, etc.) have been updated to use <code>useSyncExternalStore</code> internally.</p>
<pre class="code-block">
import { useSyncExternalStore } from 'react';

// Building a simple store with useSyncExternalStore
function createStore(initialState) {
  let state = initialState;
  const listeners = new Set();

  return {
    getState() { return state; },

    setState(updater) {
      state = typeof updater === 'function' ? updater(state) : updater;
      listeners.forEach(listener =&gt; listener());
    },

    subscribe(listener) {
      listeners.add(listener);
      return () =&gt; listeners.delete(listener); // Cleanup function
    }
  };
}

const counterStore = createStore({ count: 0 });

// Custom hook using useSyncExternalStore
function useCounter() {
  return useSyncExternalStore(
    counterStore.subscribe,                    // subscribe
    () =&gt; counterStore.getState().count,       // getSnapshot (client)
    () =&gt; 0                                    // getServerSnapshot (SSR)
  );
}

function Counter() {

  return (

      &lt;/button&gt;

  );

// GOOD:

      <p>The terms "controlled" and "uncontrolled" Suspense describe how a parent component manages the loading states of its Suspense-wrapped children. These terms are analogous to controlled vs uncontrolled form inputs — the distinction is about whether the parent explicitly manages the transition or defers entirely to React's automatic behavior.</p>
      <p><strong>Uncontrolled Suspense</strong> is the default and simplest form: you wrap a component in <code>&lt;Suspense fallback={...}&gt;</code> and React automatically shows the fallback whenever any child suspends. The parent has no direct control over when the fallback appears or disappears — React manages the entire lifecycle. This is appropriate for most use cases: loading routes, lazy components, and data fetching where the loading state doesn't need to be coordinated with other UI state.</p>
      <p><strong>Controlled Suspense</strong> involves using <code>startTransition</code>, <code>useDeferredValue</code>, or the <code>isPending</code> flag from <code>useTransition</code> to explicitly manage when loading states are shown. When you wrap a state update in <code>startTransition</code>, React keeps showing the previous UI until the new content is ready — the Suspense fallback is not shown for transitions. Instead, you use the <code>isPending</code> boolean to show a subtle loading indicator (like a spinner overlay or opacity reduction) while the new content loads. This provides a much smoother UX than instantly replacing content with a skeleton.</p>
<pre class="code-block">
// UNCONTROLLED: React shows fallback automatically when child suspends

      <p>Drag-and-drop is a particularly tricky interaction to implement correctly with React's concurrent rendering because it involves rapid, continuous state updates (position tracking) combined with high-priority DOM feedback (visual drag cursor) and lower-priority reordering logic (list updates). Naively updating state on every <code>mousemove</code> event at full React priority will cause UI jank — but using concurrent features strategically produces smooth 60fps drag interactions.</p>
      <p>The key architectural insight is to separate drag state into two categories: <strong>urgent state</strong> (the drag overlay position, cursor style, immediate visual feedback — must update synchronously at the highest priority) and <strong>deferred state</strong> (the target drop zone highlighting, list reordering preview — can be wrapped in <code>startTransition</code> to render at lower priority). This prevents expensive list reordering computations from blocking the drag cursor from following the mouse.</p>
      <p>For the drag position itself, avoid React state entirely — use a <code>useRef</code> and directly manipulate the drag element's CSS transform via <code>ref.current.style.transform</code>. DOM mutation via ref is synchronous and bypasses React's rendering pipeline, giving you true 60fps position updates. The <code>onDrop</code> and <code>onDragEnd</code> handlers — where the actual list reordering state update happens — should use <code>startTransition</code> to mark the potentially-expensive reorder as a non-urgent update.</p>
<pre class="code-block">
import { useState, useTransition, useRef, useCallback } from 'react';

function DragDropList({ initialItems }) {
  const [items, setItems] = useState(initialItems);
  const [dragOverId, setDragOverId] = useState(null);
  const [isPending, startTransition] = useTransition();
  const dragItemId = useRef(null);

  const handleDragStart = useCallback((e, id) =&gt; {
    dragItemId.current = id;
    e.dataTransfer.effectAllowed = 'move';
  }, []);

  const handleDragOver = useCallback((e, id) =&gt; {
    e.preventDefault();
    // Wrap in transition: drop zone highlight can be deferred
    startTransition(() =&gt; {
      setDragOverId(id);
    });
  }, []);

  const handleDrop = useCallback((e, targetId) =&gt; {
    e.preventDefault();
    const sourceId = dragItemId.current;
    if (sourceId === targetId) return;

    // Reordering is non-urgent — wrap in transition
    startTransition(() =&gt; {
      setItems(prev =&gt; {
        const sourceIdx = prev.findIndex(i =&gt; i.id === sourceId);
        const targetIdx = prev.findIndex(i =&gt; i.id === targetId);
        const next = [...prev];
        const [removed] = next.splice(sourceIdx, 1);
        next.splice(targetIdx, 0, removed);
        return next;
      });
      setDragOverId(null);
    });
  }, []);

  const handleDragEnd = useCallback(() =&gt; {
    dragItemId.current = null;
    setDragOverId(null);
  }, []);

  return (
    &lt;ul style={{ opacity: isPending ? 0.8 : 1 }}&gt;
      {items.map(item =&gt; (
        &lt;li
          key={item.id}
          draggable
          onDragStart={(e) =&gt; handleDragStart(e, item.id)}
          onDragOver={(e) =&gt; handleDragOver(e, item.id)}
          onDrop={(e) =&gt; handleDrop(e, item.id)}
          onDragEnd={handleDragEnd}
          style={{
            background: dragOverId === item.id ? '#e3f2fd' : 'white',
            border: dragOverId === item.id ? '2px dashed #2196f3' : '1px solid #ccc',
            padding: '8px',
            marginBottom: '4px',
            cursor: 'grab',
            transition: 'background 0.1s'
          }}
        &gt;
          {item.name}
        &lt;/li&gt;
      ))}
    &lt;/ul&gt;
  );
}

      <p>The "zombie child" problem is a subtle bug that occurs in concurrent React (and was already present in async Redux-connected components) where a child component reads from an external store and renders with stale or inconsistent data after its parent has already processed a state update that should have caused the child to unmount or re-render with new data. The term "zombie" refers to a component that should be dead (unmounted) but continues to render with old data.</p>
      <p>The classic scenario: a parent component renders a list of items. A store update deletes item #5. The parent re-renders and no longer renders the child for item #5. However, in concurrent mode, the parent's render is interruptible — and a child component subscribed directly to the same store might render with the store's new state (item #5 deleted) before the parent's render completes. When the parent's render finally commits, the child for item #5 is removed, but if the child tried to render with the deleted item, it may have already thrown an error or rendered incorrect data.</p>
      <p>The root cause is the mismatch between parent tree state and child store subscriptions when store updates happen during interrupted renders. The solution is using <code>useSyncExternalStore</code> which forces all store reads to use a consistent snapshot throughout a render pass. Libraries like React-Redux v8+ use <code>useSyncExternalStore</code> specifically to solve zombie children — by ensuring that the store snapshot React uses for all components in the current render is the same, preventing the parent/child inconsistency. Before React 18, this was partially mitigated by the synchronous rendering model; concurrent mode makes it a more pressing concern.</p>

      <p>In React 18's concurrent model, "internal state" refers to state managed by React itself via <code>useState</code>, <code>useReducer</code>, and component state — stored inside React's fiber data structure. "External state" refers to state managed outside React: Redux stores, Zustand atoms, MobX observables, browser APIs (localStorage, URL), or any plain JavaScript variables. This distinction is critical for concurrent safety because React's internal state and external state have different guarantees during interrupted renders.</p>
      <p>React's internal state is <strong>automatically concurrent-safe</strong>. React ensures that during a render pass, all reads to internal state are consistent — if a render is interrupted and restarted, it reads the same state snapshot for the entire restart. React batches updates and applies them atomically, so you can never observe a partial state update in a render. The fiber architecture stores both the "current" and "work-in-progress" versions of state, and React manages the transition between them atomically during commit.</p>
      <p>External state is <strong>not automatically concurrent-safe</strong>. If external state updates while React has a paused concurrent render in progress, React has no way to automatically detect this or restart the render. Components may read the store at different points in time during a single interrupted render pass, seeing different values — causing tearing. The solution is <code>useSyncExternalStore</code>, which gives React a way to detect external state changes mid-render and force a synchronous re-render. Any external state management library targeting React 18 must use <code>useSyncExternalStore</code> (or the equivalent subscription model) to guarantee concurrent safety.</p>

    <div class="accordion-body">
      <p>An accessible modal requires both React portals (for correct DOM placement) and a suite of ARIA attributes and keyboard behaviors (for screen reader and keyboard user support). Using <code>ReactDOM.createPortal(children, domNode)</code> renders the modal's DOM node as a child of <code>document.body</code> (or another container outside the app root), which prevents CSS overflow clipping and z-index stacking context issues. Despite the portal being physically outside the React component tree in the DOM, React event bubbling and context still work as if the portal were in its logical parent — clicks inside the portal bubble up through React's component tree normally.</p>
      <p>For accessibility, a compliant modal must: set <code>role="dialog"</code> and <code>aria-modal="true"</code> on the modal container, include <code>aria-labelledby</code> pointing to the modal's heading and <code>aria-describedby</code> pointing to its body text, trap focus within the modal (users tabbing must not be able to leave the modal while it's open), return focus to the previously focused element when closed, and respond to the Escape key for closure. Setting <code>aria-hidden="true"</code> on the main application content while the modal is open prevents screen readers from reading the background content.</p>
      <p>Focus trapping is the most complex part: you must find all focusable elements within the modal (buttons, links, inputs, elements with tabindex), intercept Tab and Shift+Tab key events, and wrap focus from the last element back to the first (and vice versa). The focus trap must be set up on open and torn down on close. Using a well-tested library like <code>focus-trap-react</code> or the native browser <code>inert</code> attribute on the background content is preferable to implementing focus trapping manually, since edge cases (iframes, shadow DOM, dynamically added elements) are numerous.</p>

  const modalRef = useRef(null);
  const previousFocusRef = useRef(null);

  useEffect(() =&gt; {
    if (isOpen) {
      // Store currently focused element to restore on close
      previousFocusRef.current = document.activeElement;

      // Focus the modal (or first focusable element)
      modalRef.current?.focus();

      // Hide background from screen readers
      document.getElementById('app-root')?.setAttribute('aria-hidden', 'true');

      return () =&gt; {
        document.getElementById('app-root')?.removeAttribute('aria-hidden');
        previousFocusRef.current?.focus(); // Restore focus on close
      };
    }
  }, [isOpen]);

  const handleKeyDown = useCallback((e) =&gt; {
    if (e.key === 'Escape') {
      onClose();
      return;
    }

    // Focus trap: Tab / Shift+Tab
    if (e.key === 'Tab') {
      const focusable = modalRef.current?.querySelectorAll(
        'button, [href], input, select, textarea, [tabindex]:not([tabindex="-1"])'
      );
      const first = focusable[0];
      const last = focusable[focusable.length - 1];

      if (e.shiftKey) {
        if (document.activeElement === first) {
          e.preventDefault();
          last?.focus();
        }
      } else {
        if (document.activeElement === last) {
          e.preventDefault();
          first?.focus();
        }
      }
    }
  }, [onClose]);

  if (!isOpen) return null;

  return createPortal(
    &lt;div
      role="presentation"
      style={{ position: 'fixed', inset: 0, background: 'rgba(0,0,0,0.5)', zIndex: 1000 }}
      onClick={(e) =&gt; { if (e.target === e.currentTarget) onClose(); }}
    &gt;
      &lt;div
        role="dialog"
        aria-modal="true"
        aria-labelledby="modal-title"
        aria-describedby="modal-desc"
        ref={modalRef}
        tabIndex={-1}
        onKeyDown={handleKeyDown}
        style={{ background: 'white', padding: '24px', borderRadius: '8px',
                 maxWidth: '500px', margin: '10vh auto', outline: 'none' }}
      &gt;
        &lt;h2 id="modal-title"&gt;{title}&lt;/h2&gt;
        &lt;div id="modal-desc"&gt;{children}&lt;/div&gt;
        &lt;button onClick={onClose} aria-label="Close dialog"&gt;Close&lt;/button&gt;
      &lt;/div&gt;
    &lt;/div&gt;,
    document.body
  );
}

    <div class="accordion-body">
      <p><code>act()</code> is a test utility that ensures all pending React state updates, effects, and re-renders are processed and flushed before your test's assertions run. Without <code>act()</code>, tests can make assertions against a stale DOM that hasn't yet reflected the latest state updates — a particularly common problem with async operations, <code>useEffect</code>, and <code>useLayoutEffect</code>. The name "act" comes from the testing convention of "Arrange, Act, Assert" — <code>act()</code> wraps the "Act" phase to ensure React's work is complete before "Assert".</p>
      <p>In React 18's concurrent mode, <code>act()</code> became async-aware. The async form <code>await act(async () =&gt; { ... })</code> is required when your test involves async operations (like <code>useEffect</code> with a fetch, or <code>startTransition</code>) because concurrent React schedules work asynchronously. When you wrap an interaction in <code>act()</code>, React processes all pending synchronous updates, flushes the microtask queue, runs all scheduled effects, and then processes any state updates triggered by those effects — recursively, until no more work is pending.</p>
      <p>Testing Library (React Testing Library) wraps all its user interaction utilities (<code>userEvent.click</code>, <code>fireEvent</code>, <code>render</code>) in <code>act()</code> automatically, which is one reason it's the recommended testing approach. If you see the warning "An update to X inside a test was not wrapped in act()", it means an async state update (typically from a useEffect or an unresolved Promise) completed after your test assertions ran. The fix is to <code>await act(async () =&gt; {})</code> or use <code>await waitFor(() =&gt; expect(...).toBe(...))</code> from Testing Library.</p>
<pre class="code-block">
import { act, render, screen } from '@testing-library/react';

  <div id="collapse-advanced-q48" class="accordion-collapse collapse" data-bs-parent="#acc-advanced">
    <div class="accordion-body">
      <p>React implements a synthetic event system that normalizes browser event APIs across different browsers and integrates with React's rendering model. In React 17+, React attaches all event listeners to the <strong>root container</strong> (the element passed to <code>ReactDOM.createRoot()</code>) rather than to the document or individual DOM nodes. This is a significant change from React 16, which attached listeners to <code>document</code> — the change was made to better support micro-frontends and multiple React roots on the same page.</p>
      <p>When a user event occurs (e.g., a click), the native browser event bubbles up to the React root container, where React's single delegated event listener catches it. React then walks the fiber tree from the event's target back up to the root, collecting all React event handlers (<code>onClick</code>, <code>onCapture</code>, etc.) along the way. React creates a <strong>SyntheticEvent</strong> — a cross-browser wrapper around the native event — and dispatches it through the collected handlers in the correct order (capture phase handlers top-down, bubble phase handlers bottom-up).</p>
      <p>SyntheticEvents were pooled in React 16 and earlier (reused for performance by nullifying properties after each event handler), which caused issues with async event handlers that tried to access event properties after the synchronous dispatch. React 17 removed event pooling entirely — SyntheticEvents are now regular objects that get garbage collected normally. However, the <code>nativeEvent</code> property still gives you access to the underlying DOM event if you need browser-specific properties that React's normalization doesn't expose.</p>
      <p>React handles some events specially: <code>onChange</code> on inputs fires on every keystroke (not just on blur like native <code>change</code>), events like <code>onFocus</code>/<code>onBlur</code> use the capture phase for delegation (because focus doesn't bubble natively), and <code>onScroll</code> does not bubble in React (matching browser behavior). The synthetic event system is why you should call <code>e.stopPropagation()</code> on the React synthetic event rather than the native event — doing it on the native event may not stop React's simulated propagation.</p>
<pre class="code-block">
// React 17+: events delegated to root container, not document

    console.log(e.target);        // Native DOM element

    // React 17+: event properties DO persist after handler (no pooling)
    setTimeout(() =&gt; {
      console.log(e.type); // Still 'click' — no longer null! (React 17+)

    e.stopPropagation();

    // Prevent browser default
    e.preventDefault();
  };

}

  <div id="collapse-advanced-q49" class="accordion-collapse collapse" data-bs-parent="#acc-advanced">
    <div class="accordion-body">
      <p>"Interleaving" in React's concurrent rendering refers to the ability of React's scheduler to interleave the rendering work of multiple updates — pausing one update mid-render to process a higher-priority update, then resuming the original update. This is the fundamental capability that makes concurrent mode "concurrent": rather than rendering updates sequentially to completion, React can work on multiple render trees simultaneously, yielding control between them based on priority.</p>
      <p>In practice, interleaving enables the distinction between "urgent" and "non-urgent" updates. An urgent update (user typing in an input, button click) must be processed immediately to feel responsive. A non-urgent update (filtering a long list based on input, loading new route content in a transition) can be deferred. When an urgent update arrives while a non-urgent render is in progress, React can pause ("interrupt") the non-urgent render, synchronously apply the urgent update, paint to the screen, and then resume the non-urgent render — the user sees immediate feedback without waiting for the expensive operation.</p>
      <p>Interleaving has important implications for correctness. Because a single "render pass" for a component may be split across multiple execution frames, with other renders happening in between, your render functions must be pure (no side effects that depend on execution timing). Any side effect that assumes "my render runs to completion before anything else" will break with interleaving. This is why the React team has always emphasized pure render functions — in legacy mode it was a style guideline, in concurrent mode it becomes a hard requirement for correct behavior.</p>
      <p>The React compiler (introduced as an experimental feature in React 19) analyzes component code statically to identify and memoize only the parts that truly change, reducing the cost of interleaved renders. Understanding interleaving also explains why React calls render functions twice in Strict Mode during development — to help surface impure renders that produce different results on repeated invocation, catching bugs before they manifest as subtle interleaving issues in concurrent mode.</p>
<pre class="code-block">
// Interleaving example: urgent vs deferred updates
function SearchPage() {
  const [inputValue, setInputValue] = useState('');
  const [searchQuery, setSearchQuery] = useState('');
  const [isPending, startTransition] = useTransition();

  const handleChange = (e) =&gt; {
    const value = e.target.value;

    // URGENT: Input value updates immediately (interleaves at high priority)
    setInputValue(value);

    // NON-URGENT: Search query update is deferred (interleaved at low priority)
    startTransition(() =&gt; {
      setSearchQuery(value);
    });
  };

  // React's internal scheduling with interleaving:
  // 1. User types 'r':
  //    - Urgent render starts: setInputValue('r')
  //    - Concurrent render queued: setSearchQuery('r')
  // 2. User types 'e' before concurrent render finishes:
  //    - React INTERRUPTS the searchQuery='r' render
  //    - Urgent render: setInputValue('re')
  //    - Previous concurrent render DISCARDED
  //    - New concurrent render queued: setSearchQuery('re')
  // Result: Input always feels instantaneous

  return (
    &lt;div&gt;
      &lt;input value={inputValue} onChange={handleChange} /&gt;
      {/* Input shows instantly (urgent update) */}

      &lt;div style={{ opacity: isPending ? 0.5 : 1 }}&gt;
        {/* Results update when non-urgent render completes */}
        &lt;SearchResults query={searchQuery} /&gt;
      &lt;/div&gt;
    &lt;/div&gt;
  );
}

// BAD: Side effect in render (breaks with interleaving)

  return &lt;div&gt;Renders: {renderCount}&lt;/div&gt;;
}

  const [count, setCount] = useState(0);
  useEffect(() =&gt; {
    // Side effects here, in the commit phase
    analytics.track('component_mounted');
  }, []);

      <p>Feature detection for React versions is necessary when building libraries, design systems, or applications that need to support multiple React versions simultaneously — such as a component library that targets both React 18 and React 19 consumers. Unlike browser feature detection (where you test the DOM API), React feature detection involves checking the React package version and the availability of specific APIs, hooks, and components that were added in particular versions.</p>
      <p>The most reliable approach is to check the React version string via <code>React.version</code> and parse the semver. However, version checking is fragile against pre-release versions (canary, experimental builds) that may have features from a higher version. A more robust approach is to directly check for the existence of the API: <code>typeof React.use === 'function'</code> for the <code>use()</code> hook, <code>typeof React.Activity !== 'undefined'</code> for the Activity component, <code>typeof ReactDOM.createRoot === 'function'</code> for React 18's createRoot, and so on. This "duck typing" approach is version-agnostic and handles canary builds correctly.</p>
      <p>For hook-based feature detection, you need to be careful — hooks can only be called inside components. Create dedicated compatibility hooks that detect availability at call time and return either the real implementation or a polyfill. A common pattern for libraries is to export compatibility wrappers: <code>useCompat_useActionState</code> which uses <code>useActionState</code> if available and falls back to a <code>useState</code>+<code>useEffect</code> implementation for React 18. The React team also ships the <code>react-dom/client</code> API checker which logs deprecation warnings for legacy render APIs.</p>
      <p>Build-time feature detection using bundler conditions (<code>package.json</code> <code>exports</code> field with peer dependency ranges) is the cleanest long-term approach for library authors: ship different entry points for React 18 vs React 19, and let the package manager resolve the correct version. For application developers, just upgrade to the latest React version — feature detection is primarily a library author concern.</p>

const REACT_VERSION = { major, minor, isReact18: major === 18, isReact19: major >= 19 };

// Duck-typing API detection (preferred over version strings)

const hasActivity = typeof (React as any).Activity !== 'undefined';
const hasViewTransition = typeof (React as any).ViewTransition !== 'undefined';
const hasUseActionState = typeof React.useActionState === 'function'; // React 19

// Compatibility hook: useActionState with React 18 fallback
function useCompatActionState(action, initialState) {
  // React 19: use native hook
  if (typeof React.useActionState === 'function') {
    return React.useActionState(action, initialState);
  }

  // React 18 fallback implementation
  const [state, setState] = useState(initialState);
  const [isPending, setIsPending] = useState(false);

  const dispatch = useCallback(async (...args) =&gt; {
    setIsPending(true);
    try {
      const newState = await action(state, ...args);
      setState(newState);
    } finally {
      setIsPending(false);
    }
  }, [action, state]);

  return [state, dispatch, isPending];
}

// Compatibility wrapper for createRoot vs legacy render
function compatRender(element, container) {

}
