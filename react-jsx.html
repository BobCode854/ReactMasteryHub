JSX

JavaScript XML — the syntax extension that lets you write HTML-like code inside JavaScript. The heart of every React component.

01

What is JSX?

Definition: JSX (JavaScript XML) is a syntax extension for JavaScript that looks like HTML but is actually JavaScript. JSX is not valid JavaScript on its own — Babel transpiles it into React.createElement() calls at build time. It makes React components easier to read and write.
Without JSX (what you'd have to write)
// Without JSX — verbose and unreadable: function App() { return React.createElement( 'div', { className: 'container' }, React.createElement('h1', null, 'Hello'), React.createElement( 'p', { className: 'text' }, 'Welcome to React' ) ); }
With JSX (what you actually write)
// With JSX — clean and intuitive: function App() { return ( <div className="container"> <h1>Hello</h1> <p className="text"> Welcome to React </p> </div> ); }
JSX is optional but universal: Technically you can write React without JSX (using React.createElement directly), but virtually every React developer uses JSX. It's the standard.
02

JSX Rules — Must Know

Must return a single root element
// ❌ Two root elements — invalid JSX return ( <h1>Title</h1> <p>Paragraph</p> ); // ✅ Wrap in a div: return ( <div> <h1>Title</h1> <p>Paragraph</p> </div> ); // ✅ Or use a Fragment: return ( <> <h1>Title</h1> <p>Paragraph</p> </> );
Use className instead of class
// ❌ 'class' is a JS reserved word <div class="container">...</div> // ✅ Use 'className' in JSX <div className="container">...</div> // Dynamic className with template literal: <div className={`card ${isActive ? 'active' : ''}`}> ... </div> // ❌ 'for' is also reserved — use 'htmlFor' <label for="email">Email</label> // ✅ Correct: <label htmlFor="email">Email</label>
Self-close empty tags
// ❌ HTML allows unclosed tags — JSX doesn't <img src="photo.jpg"> <input type="text"> <br> <hr> // ✅ All tags must be closed in JSX <img src="photo.jpg" /> <input type="text" /> <br /> <hr /> // Components must also be closed: <MyComponent /> // or <MyComponent></MyComponent>
Inline styles use objects
// ❌ HTML string style — invalid in JSX <div style="color: red; font-size: 16px"> // ✅ JSX style — JavaScript object <div style={{ color: 'red', fontSize: '16px' }}> // Note: camelCase CSS properties: // font-size → fontSize // background-color → backgroundColor // border-radius → borderRadius // Dynamic styles: const style = { color: isError ? 'red' : 'green' }; <p style={style}>Message</p>
JS expressions inside {}
// Any JS expression goes in {} const name = 'Alice'; return <h1>Hello, {name}!</h1>; // ✅ Valid in {}: variables, math, // function calls, ternary, array.map <p>{2 + 2}</p> // renders "4" <p>{name.toUpperCase()}</p> // ❌ Statements don't work in {}: // { if (x) ... } — use ternary instead // { for (...) } — use .map() instead
Comments in JSX
// ❌ HTML comments don't work in JSX <!-- This won't work in JSX --> // ✅ Use JS comment inside {} return ( <div> {/* This is a JSX comment */} {/* Multi-line comment here */} <p>Content</p> </div> ); // Outside JSX return — normal JS comments: // This works fine outside return()

JSX is not valid JavaScript. Babel transforms it before your code runs in the browser.

What you write
// JSX you write in your component: const element = ( <div className="card" onClick={handleClick}> <h2>{title}</h2> <p className="text-muted">{description}</p> <Button variant="primary">Read More</Button> </div> );
What Babel compiles it to (React 17+ new JSX transform)
import { jsx as _jsx, jsxs as _jsxs } from 'react/jsx-runtime'; const element = _jsxs('div', { className: 'card', onClick: handleClick, children: [ _jsx('h2', { children: title }), _jsx('p', { className: 'text-muted', children: description }), _jsx(Button, { variant: 'primary', children: 'Read More' }) ] });
The resulting React Element (plain JS object)
// React.createElement / _jsx ultimately returns a plain object: { $$typeof: Symbol.for('react.element'), type: 'div', // tag name OR component function props: { className: 'card', onClick: handleClick, children: [ { type: 'h2', props: { children: title } }, { type: 'p', props: { className: 'text-muted', children: description } }, { type: Button, props: { variant: 'primary', children: 'Read More' } } ] }, key: null, ref: null }
React 17+ New JSX Transform: Before React 17, every file using JSX needed import React from 'react' at the top (because JSX compiled to React.createElement). React 17 introduced a new transform that imports from react/jsx-runtime automatically — so you no longer need to import React just for JSX.
04

JSX vs HTML — Key Differences

Feature HTML JSX Reason
CSS class class="btn" className="btn" class is a reserved keyword in JavaScript
Label for for="email" htmlFor="email" for is a reserved keyword (for loop)
Self-closing tags <img> (optional close) <img /> (required) JSX requires all tags to be explicitly closed
Inline styles style="color:red" style={{'{'}}{'{'}color:'red'{'}'}{'}'} Styles in JSX are JS objects, not strings
CSS property names font-size fontSize camelCase in JS objects (hyphens are subtraction)
Comments <!-- comment --> {'{'}/* comment */{'}'} JSX uses JS comment syntax wrapped in {'{}'}
Event handlers onclick="fn()" onClick={'{'}fn{'}'} camelCase; pass function reference, not string
Boolean attributes disabled disabled={'{'}true{'}'} or just disabled Both work in JSX; explicit JS boolean preferred
Tab index tabindex="0" tabIndex={'{'}0{'}'} camelCase for multi-word attributes
Quick rule: If an HTML attribute name is a reserved JS keyword (class, for) or is multi-word with hyphens (font-size), JSX renames it to camelCase. For event handlers, HTML uses lowercase (onclick), JSX uses camelCase (onClick).
05

Expressions in JSX — Using {'{}'}

The {'{}'} syntax lets you embed any JavaScript expression inside JSX. Anything that evaluates to a value: variables, math, function calls, ternary operators, .map(). Statements (if, for, while) do NOT work directly inside {'{}'}.
Variables Most basic use
const name = 'Alice'; const age = 25; const isAdmin = true; return ( <div> <h1>Hello, {name}!</h1> <p>Age: {age}</p> <p>Role: {isAdmin ? 'Admin' : 'User'}</p> </div> );
Ternary Operator If/else in JSX
// Conditional rendering with ternary: return ( <div> {isLoggedIn ? <Dashboard user={user} /> : <Login onLogin={handleLogin} /> } {isLoading ? <Spinner /> : <Content data={data} /> } </div> );
Short-circuit && Show or hide
// Render only when true — short-circuit: return ( <div> {error && <ErrorMessage msg={error} />} {isAdmin && <AdminPanel />} {notifications.length > 0 && ( <NotificationBell count={notifications.length} /> )} </div> ); // ⚠️ Gotcha: don't use 0 as condition! {count && <Items />} // renders "0" when count=0! {count > 0 && <Items />} // ✅ correct
Function Calls Transform data
// Call functions to format data: function formatPrice(n) { return `$${n.toFixed(2)}`; } function formatDate(d) { return new Date(d).toLocaleDateString(); } return ( <div> <p>Price: {formatPrice(product.price)}</p> <p>Date: {formatDate(order.date)}</p> <p>Name: {user.name.toUpperCase()}</p> </div> );
Template Literals Dynamic strings
// Template literals in JSX: return ( <div> <h2>{`Welcome back, ${user.firstName}!`}</h2> <img src={`/avatars/${user.id}.jpg`} alt={`${user.name}'s avatar`} /> <div className={`card ${theme} ${size}`}> Content </div> </div> );
Array.map() Render lists
// Render a list with .map(): const products = [ { id: 1, name: 'Laptop', price: 999 }, { id: 2, name: 'Phone', price: 699 }, ]; return ( <ul> {products.map(p => ( <li key={p.id}> {p.name} — ${p.price} </li> ))} </ul> );
06

React Fragments

Problem: JSX requires a single root element, but wrapping everything in a <div> adds unnecessary DOM nodes. Solution: React Fragments — a wrapper that leaves no trace in the DOM.
Form 1: React.Fragment (long form)
import React from 'react'; function Table() { return ( <React.Fragment> <tr> <td>Alice</td> <td>25</td> </tr> </React.Fragment> ); } // No extra DOM wrapper!
Form 2: Short syntax <> (most common)
function UserInfo() { return ( <> <h1>Alice</h1> <p>Developer</p> <p>San Francisco</p> </> ); } // <></> is shorthand for Fragment // Most commonly used form
Form 3: With key prop (lists)
// Short syntax <> can't have key prop! // Use React.Fragment for keyed lists: {items.map(item => ( <React.Fragment key={item.id}> <dt>{item.term}</dt> <dd>{item.definition}</dd> </React.Fragment> ))} // or: import {Fragment} from 'react' // <Fragment key={item.id}>...
When to use Fragments: Always when you need to return multiple elements but don't want an extra <div> in the DOM. Common in table rows (<tr>/<td> can't be wrapped in a div), list items, and anywhere extra divs would break CSS flex/grid layout.
07

Conditional Rendering — 4 Patterns

Pattern 1: if/else outside return
Complex conditions
function Page({ status }) { // Decide BEFORE the return: if (status === 'loading') { return <Spinner />; } if (status === 'error') { return <ErrorPage />; } // Only reach here on success: return <Content />; }

Use when you have multiple conditions that return completely different UIs. Cleanest for early returns (guard clauses).

Pattern 2: Ternary operator
Simple if/else in JSX
function AuthButton({ isLoggedIn }) { return ( <div> <h1>Welcome{isLoggedIn ? ', Alice' : ''}!</h1> {isLoggedIn ? <button onClick={logout}>Log Out</button> : <button onClick={login}>Log In</button> } </div> ); }

Best for choosing between two components/values inline. Avoid nesting ternaries (unreadable).

Pattern 3: Short-circuit &&
Show or hide
function Notifications({ count, error }) { return ( <div> {error && ( <div className="alert alert-danger"> {error.message} </div> )} {count > 0 && ( <span className="badge">{count}</span> )} </div> ); }

Best for "show if true, show nothing if false." Always use count > 0 && not count && for numbers to avoid rendering 0.

Pattern 4: Optional chaining + ??
Safe data access
function UserCard({ user }) { return ( <div> <h2>{user?.name ?? 'Anonymous'}</h2> <img src={user?.avatar?.url ?? '/default-avatar.png'} alt={user?.name ?? 'User'} /> <p>{user?.bio ?? 'No bio provided'}</p> <span>{user?.followers ?? 0} followers</span> </div> ); }

Best for safely rendering async data that might be null/undefined while loading.

08

Lists & Keys

Complete Product List Example
function ProductList({ products }) { if (!products || products.length === 0) { return <p>No products found.</p>; } return ( <div className="product-grid"> {products.map(product => ( <div key={product.id} // ✅ Stable unique ID className="product-card" > <img src={product.image} alt={product.name} /> <h3>{product.name}</h3> <p>${product.price.toFixed(2)}</p> <span> {product.inStock ? 'In Stock' : 'Out of Stock' } </span> <button disabled={!product.inStock} onClick={() => addToCart(product.id)} > Add to Cart </button> </div> ))} </div> ); }
Why Keys Matter
Purpose: Keys help React identify which items changed, were added, or removed. Without keys, React re-renders the entire list on any change. With proper keys, React only updates what changed.
Good keys: Stable, unique IDs from your data — user.id, product.sku, post.slug.
Don't use array index as key
// ❌ Index key — causes bugs when: // — List is reordered // — Items are deleted // — Items are inserted in the middle {items.map((item, i) => <Item key={i} /> // ❌ )} // ✅ Stable ID from data: {items.map(item => <Item key={item.id} /> // ✅ )}
The index bug explained: If you have 3 items [A, B, C] with keys [0, 1, 2] and delete B, you get [A, C] with keys [0, 1]. React sees key 1 still exists and reuses the B component with C's data — but the component's internal state (like an input value) stays as B's state. This causes subtle, hard-to-debug state corruption.
Nested Lists
// Nested lists — each level needs its own unique key: function CategoryList({ categories }) { return ( <div> {categories.map(category => ( <section key={category.id}> // key on outer element <h2>{category.name}</h2> <ul> {category.items.map(item => ( <li key={item.id}> // key on inner element {item.name} </li> ))} </ul> </section> ))} </div> ); }

JSX Interview Questions

Answer: JSX (JavaScript XML) is a syntax extension for JavaScript that allows you to write HTML-like code within JavaScript. It's not valid JavaScript — Babel transforms it into React.createElement() (or _jsx() with the new transform) at build time.

JSX makes React code more readable by letting you describe the UI structure visually rather than writing nested React.createElement() calls. It's not required but is used by virtually all React developers.

// JSX const el = <h1 className="title">Hello</h1>; // Compiled to: const el = _jsx('h1', { className: 'title', children: 'Hello' });

Answer: No, JSX is not mandatory. You can write React using pure React.createElement() calls:

// Without JSX (valid but impractical): const element = React.createElement( 'div', { className: 'container' }, React.createElement('h1', null, 'Hello') );

However, writing complex UIs this way is extremely verbose and unreadable. JSX is universally used in practice because it's much more intuitive and maintainable.

Answer: Babel transforms JSX into JavaScript function calls that browsers can understand. There are two transforms:

  • Classic transform (pre-React 17): Compiles to React.createElement(type, props, ...children). Required import React from 'react' in every file.
  • Automatic/new transform (React 17+): Compiles to _jsx() from react/jsx-runtime, imported automatically. No manual React import needed.

Both transforms ultimately produce plain JavaScript objects (React Elements) that React uses to build the Virtual DOM.

Answer: Yes, since React 17. The new JSX transform automatically imports what's needed from react/jsx-runtime — you no longer need import React from 'react' at the top of files that only use JSX.

You still need to import React when using React-specific APIs: React.useState (but you'd use the named import import { useState } from 'react'), React.memo, React.forwardRef, etc.

// React 17+ — no React import needed for JSX: function Button({ onClick, children }) { return <button onClick={onClick}>{children}</button>; } export default Button;

Answer: Key differences:

  • Attribute names: JSX uses camelCase (className, htmlFor, onClick, tabIndex) instead of HTML's lowercase/hyphen style (class, for, onclick, tabindex)
  • Self-closing: All JSX tags must be explicitly closed (<br />, <img />)
  • Inline styles: JSX takes a JavaScript object with camelCase properties; HTML takes a string
  • Comments: JSX uses {'{'}/* ... */{'}'} inside the markup
  • Dynamic content: JSX uses {'{'}expression{'}'} for dynamic values; HTML has no native equivalent
  • Boolean attributes: JSX uses explicit disabled={'{'}true{'}'}

Answer: class is a reserved keyword in JavaScript (used to define ES6 classes). Since JSX is written inside JavaScript files and gets compiled to JavaScript, using class as an attribute name would create a syntax conflict with the JavaScript parser.

React uses className instead, which maps to the DOM property element.className. The same logic applies to for (reserved for loops) → htmlFor.

// ❌ Syntax error — 'class' is reserved: <div class="container"> // ❌ // ✅ Use className: <div className="container"> // ✅

Answer: React Fragments (<React.Fragment> or <></>) let you group multiple JSX elements without adding an extra DOM node.

Why use them:

  • Avoid unnecessary divs: Extra wrapper divs can break CSS (especially flex/grid layouts) and add meaningless markup
  • Required in tables: <tr> children must be <td> — a wrapper <div> would create invalid HTML
  • Cleaner DOM: Fewer DOM nodes = slightly better performance

Use <></> (short syntax) when no key is needed. Use <React.Fragment key={'{'}id{'}'}> when mapping lists and need a key.

Answer: React renders lists using Array.map() to transform a data array into an array of JSX elements. Each element in the array must have a unique key prop.

The key prop: A special React prop (not passed to the component) that helps React's reconciliation algorithm identify which items changed, were added, or removed when the list updates.

Rules for keys:

  • Must be unique among siblings (not globally unique)
  • Must be stable across re-renders — don't generate keys with Math.random()
  • Use IDs from your data — user.id, post.slug
  • Avoid array index unless the list never changes order/length
{users.map(user => ( <UserCard key={user.id} user={user} /> ))}
