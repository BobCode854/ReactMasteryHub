Section 5 9 Component Types

React Components

Master all 9 types of React components — from functional to compound, HOCs to controlled/uncontrolled. Build a mental model for when to use each pattern.

01

Functional Components

Modern Standard
What it is: A plain JavaScript function that accepts props and returns JSX. The modern standard since React 16.8 introduced Hooks. Simpler, less boilerplate, and fully capable of all React features. 
function UserCard({ name, email, avatar }) {
  return (
    <div className="card">
      <img src={avatar} alt={name} />
      <h3>{name}</h3>
      <p>{email}</p>
    </div>
  );
}
When to use:
Use functional components for everything in modern React. They can use all hooks (useState, useEffect, useContext, etc.), support async patterns, and are simpler to read and test.
Real Industry Example:
Every component in Next.js apps, Vercel's dashboard, and modern React codebases. Shopify's Polaris design system uses exclusively functional components.
02

Class Components

Legacy
What it is: ES6 class that extends React.Component. The only way to write stateful components before React 16.8. Still found in older codebases. Has lifecycle methods and this binding complexity. 
class Counter extends React.Component {
  constructor(props) {
    super(props);
    this.state = { count: 0 };
  }

  increment = () => {
    this.setState(prev => ({ count: prev.count + 1 }));
  };

  render() {
    return (
      <div>
        <p>Count: {this.state.count}</p>
        <button onClick={this.increment}>+</button>
      </div>
    );
  }
}
When to use:
When maintaining legacy codebases. Also for Error Boundaries — componentDidCatch and getDerivedStateFromError have no hook equivalents yet (as of React 18).
Common Gotchas:
this binding issues in event handlers. Must use arrow function class properties or .bind(this) in constructor. Verbose lifecycle methods vs clean hooks.
03

Pure Components & React.memo

Performance
What it is: Components that only re-render when their props or state actually change. React.PureComponent does a shallow comparison for class components. React.memo does the same for functional components. Prevents unnecessary re-renders. 
// Class Pure Component — shallow prop comparison built-in
class PureUserCard extends React.PureComponent {
  render() {
    return <div>{this.props.name}</div>;
  }
}

// Functional equivalent — React.memo
const MemoizedCard = React.memo(function UserCard({ name, score }) {
  return <div>{name}: {score}</div>;
});
// Only re-renders if name or score actually changes

// Custom comparison function
const MemoizedCard = React.memo(UserCard, (prevProps, nextProps) => {
  return prevProps.id === nextProps.id; // true = skip re-render
});
When to use:
Components that render frequently but receive the same props often. List items in long lists, table rows, cards in dashboards. Pair with useCallback to stabilize function props.
Real Industry Example:
Facebook's feed items, Twitter's tweet cards — each item is memoized so scrolling doesn't re-render all visible items when one state changes. AG Grid row components.
04

Higher Order Components (HOC)

Pattern
What it is: A function that takes a component and returns a new component with added behavior. Used for cross-cutting concerns: authentication, logging, analytics, theming. Named with with prefix by convention. 
// HOC: wraps a component and adds authentication check
function withAuthentication(WrappedComponent) {
  return function AuthenticatedComponent(props) {
    const { isLoggedIn } = useAuth();
    if (!isLoggedIn) return <Navigate to="/login" />;
    return <WrappedComponent {...props} />;
  };
}

// HOC: adds loading state
function withLoading(WrappedComponent) {
  return function WithLoadingComponent({ isLoading, ...props }) {
    if (isLoading) return <Spinner />;
    return <WrappedComponent {...props} />;
  };
}

// Usage: compose HOCs
const ProtectedDashboard = withAuthentication(withLoading(Dashboard));
When to use:
Authentication guards, permission checks, analytics tracking, feature flags. When the same logic needs to wrap many different components. Redux's connect() is a classic HOC.
Real Industry Example:
React Router's withRouter(), Redux's connect(), Relay's createFragmentContainer(). Stripe's dashboard uses HOCs for permission-based component wrapping.
05

Compound Components

Advanced Pattern
What it is: A pattern where parent and child components share state via Context. The parent manages state; children read from context. Gives consumers flexible composition without prop drilling. Used heavily in UI libraries. 
const TabsContext = createContext(null);

// Parent manages shared state
function Tabs({ children }) {
  const [active, setActive] = useState(0);
  return (
    <TabsContext.Provider value={{ active, setActive }}>
      <div className="tabs">{children}</div>
    </TabsContext.Provider>
  );
}

// Child components access shared state via context
Tabs.Tab = function Tab({ index, label }) {
  const { active, setActive } = useContext(TabsContext);
  return (
    <button
      className={active === index ? 'active' : ''}
      onClick={() => setActive(index)}
    >
      {label}
    </button>
  );
};

Tabs.Panel = function Panel({ index, children }) {
  const { active } = useContext(TabsContext);
  return active === index ? <div>{children}</div> : null;
};

// Usage — flexible composition
<Tabs>
  <Tabs.Tab index={0} label="Profile" />
  <Tabs.Tab index={1} label="Settings" />
  <Tabs.Panel index={0}>Profile content</Tabs.Panel>
  <Tabs.Panel index={1}>Settings content</Tabs.Panel>
</Tabs>
When to use:
Design systems and UI libraries. Whenever you want flexible composition without excessive props. Tabs, Accordions, Dropdowns, Modals, Select menus. Radix UI, Headless UI use this pattern extensively.
Real Industry Example:
Reach UI's Tabs, Radix UI's Dialog, Headless UI's Listbox. React Bootstrap's Nav/Tabs. Any component library where consumers need to control layout while library manages state.
06

Container vs Presentational Components

Architecture Pattern
What it is: A separation-of-concerns pattern. Container components handle data fetching and state; presentational components handle rendering UI. Also called Smart/Dumb or Stateful/Stateless split.
Aspect Container Component Presentational Component
PurposeData & logicUI rendering
StateYes — manages stateNo — stateless (usually)
Side effectsYes — API calls, subscriptionsNo — pure UI
StylingMinimalMost styling lives here
ReusabilityLowHigh — reusable across pages
TestingIntegration testsUnit / snapshot tests
ExampleUserListContainerUserCard, UserList
 
// Container — fetches data, manages state
function UserListContainer() {
  const [users, setUsers] = useState([]);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    fetch('/api/users')
      .then(r => r.json())
      .then(data => { setUsers(data); setLoading(false); });
  }, []);

  return <UserList users={users} loading={loading} />;
}

// Presentational — pure UI, no side effects
function UserList({ users, loading }) {
  if (loading) return <Spinner />;
  return (
    <ul className="user-list">
      {users.map(user => <UserCard key={user.id} {...user} />)}
    </ul>
  );
}
Real Industry Example: Redux apps split containers (connected to store) from presentational (pure render). Dan Abramov popularized this pattern at Facebook/Twitter.
07

Controlled vs Uncontrolled Components

Forms
What it is: Controlled components have their form state managed by React (via useState). Uncontrolled components let the DOM manage state, accessed via refs. React recommends controlled components for most cases. 
// CONTROLLED: React is the single source of truth
function ControlledInput() {
  const [value, setValue] = useState('');
  return (
    <input
      value={value}                         // React controls value
      onChange={e => setValue(e.target.value)} // React updates on change
    />
  );
}

// UNCONTROLLED: DOM manages value, we read it via ref
function UncontrolledInput() {
  const inputRef = useRef(null);
  const handleSubmit = () => {
    console.log(inputRef.current.value); // read on demand
  };
  return (
    <>
      <input ref={inputRef} defaultValue="initial" />
      <button onClick={handleSubmit}>Submit</button>
    </>
  );
}
Controlled — Use when:
Real-time validation, conditional rendering based on input, instant feedback (search-as-you-type), formatting input on change.
Uncontrolled — Use when:
File inputs (always uncontrolled), integrating with non-React libraries, simple forms where you only need values on submit, performance-critical forms.
08

Fragment Components

Utility
What it is: React.Fragment (or the shorthand <></>) lets you return multiple elements without adding an extra DOM node. Essential for table rows, definition lists, and keeping DOM clean. 
// Without Fragment — adds unnecessary div wrapper
function TableRow({ name, age }) {
  return (
    <div> // ❌ breaks table structure!
      <td>{name}</td>
      <td>{age}</td>
    </div>
  );
}

// With Fragment — no extra DOM node
function TableRow({ name, age }) {
  return (
    <> // shorthand Fragment
      <td>{name}</td>
      <td>{age}</td>
    </>
  );
}

// Fragment with key (must use explicit React.Fragment)
{items.map(item => (
  <React.Fragment key={item.id}>
    <dt>{item.term}</dt>
    <dd>{item.description}</dd>
  </React.Fragment>
))}
09

Error Boundary Components

Error Handling
What it is: Class components that catch JavaScript errors anywhere in their child tree, log them, and display a fallback UI. Must be class components — no hooks equivalent yet. Essential for production apps. 
class ErrorBoundary extends React.Component {
  constructor(props) {
    super(props);
    this.state = { hasError: false, error: null };
  }

  static getDerivedStateFromError(error) {
    return { hasError: true, error };
  }

  componentDidCatch(error, info) {
    // Log to error reporting service
    logErrorToService(error, info.componentStack);
  }

  render() {
    if (this.state.hasError) {
      return <div className="error-fallback">Something went wrong.</div>;
    }
    return this.props.children;
  }
}

// Usage — wrap sections of your app
<ErrorBoundary>
  <UserDashboard /> // crash here won't kill the whole app
</ErrorBoundary>
Real Industry Example: Every production React app should have ErrorBoundaries. Libraries like react-error-boundary provide a functional wrapper. Sentry's React SDK integrates with Error Boundaries for automatic crash reporting.

Interview Questions — React Components

Functional components are plain JavaScript functions that accept props and return JSX. Class components extend React.Component and use render() to return JSX.

Key differences:

  • State: Functional components use useState hook; class components use this.state and this.setState.
  • Lifecycle: Functional components use useEffect; class components have componentDidMount, componentDidUpdate, componentWillUnmount.
  • Syntax: Functional is simpler — no this, no constructor, no binding.
  • Performance: Functional components can be slightly lighter and easier to optimize with React.memo.
  • Modern standard: Since React 16.8, functional components can do everything class components can. React team recommends functional components for all new code.

There are two primary scenarios where class components are still necessary or preferred:

  1. Error Boundaries: componentDidCatch and getDerivedStateFromError have no functional/hook equivalent as of React 18. Error boundaries must be class components. (The react-error-boundary library provides a functional wrapper around a class component.)
  2. Legacy codebases: When working on older React projects that were built before hooks existed, you'll encounter class components. Refactoring is possible but not always worth the risk/time.

For all new code written today, functional components with hooks are the correct choice.

React.memo is a higher-order component that memoizes a functional component. It performs a shallow comparison of props before re-rendering — if props haven't changed, it skips the re-render and returns the cached result.

When to use it:

  • Components in long lists (e.g., table rows, feed items) where parent re-renders frequently
  • Components with expensive render logic that receive stable props
  • Pure display components in dashboards that re-render due to unrelated state changes

When NOT to use it:

  • Components that always receive new props — memoization adds overhead with no benefit
  • Simple components where the render cost is trivial
  • When props include objects/functions created inline (they fail shallow equality) — pair with useMemo/useCallback to stabilize them

A Higher Order Component (HOC) is a function that takes a component as input and returns a new enhanced component. It's a pattern for reusing component logic.

Real-world example — withAuthentication: Rather than adding an auth check inside every protected page component, you create a HOC:

function withAuthentication(WrappedComponent) {
  return function AuthenticatedComponent(props) {
    const { isLoggedIn } = useAuth();
    if (!isLoggedIn) return <Navigate to="/login" />;
    return <WrappedComponent {...props} />;
  };
}
const ProtectedDashboard = withAuthentication(Dashboard);

Other real examples: Redux's connect(), React Router's withRouter(), React.memo() itself. HOCs are less common today since custom hooks often achieve the same goal with less complexity.

Compound components are a pattern where a parent component and its child components work together to share implicit state. The parent acts as the state manager; children consume that state via React Context.

This gives consumers flexible, expressive composition — they can arrange child components in any order or structure without passing dozens of props.

How Context is used:

  1. Parent creates a Context and provides its state to it via Context.Provider
  2. Child sub-components (often attached as static properties) consume the context with useContext
  3. Consumer code simply composes the children — no explicit prop threading needed

Real examples: Radix UI's Dialog (Trigger, Content, Close), Headless UI's Listbox (Button, Options, Option), Reach UI's Tabs (Tab, TabList, TabPanel, TabPanels).

This distinction is about who owns the form element's state.

Controlled components: React is the single source of truth. The input's value is driven by React state, and every change is handled via onChange. React always knows the current value.

Uncontrolled components: The DOM manages its own state. You use a ref to read the value when needed (e.g., on form submit). React doesn't track changes in real time.

When to use which:

  • Controlled: Real-time validation, dependent fields, search-as-you-type, formatting on input
  • Uncontrolled: File inputs (always uncontrolled by spec), integrating third-party DOM libraries, simple forms that only need values on submit

React's official recommendation is to use controlled components for most form elements.

The Container/Presentational pattern (also called Smart/Dumb) separates components into two types:

  • Container: Handles data fetching, state management, side effects. Passes data down as props.
  • Presentational: Handles rendering UI. Receives data via props, has no knowledge of where data comes from.

Is it still relevant? Yes and no. The pattern's goal — separation of concerns — is still valid. But the implementation has evolved:

  • With hooks, you can extract data logic into a custom hook instead of a container component, keeping separation without the extra component layer.
  • The pattern is less formal today but the mental model remains useful — especially distinguishing "data fetching" components from "pure display" components in code reviews and testing strategy.

Dan Abramov (the pattern's popularizer) has himself noted hooks make it less necessary as a hard rule.

An Error Boundary is a React component that catches JavaScript errors anywhere in its child component tree during rendering, lifecycle methods, and constructors. Instead of crashing the whole app, it shows a fallback UI.

It works via two special lifecycle methods:

  • static getDerivedStateFromError(error) — updates state to trigger fallback UI
  • componentDidCatch(error, info) — logs the error (e.g., to Sentry)

Why must it be a class component? These two lifecycle methods have no hook equivalents in React 18. The React team has discussed adding an use(ErrorBoundary) or similar API, but it hasn't shipped. Until then, Error Boundaries require class components.

The react-error-boundary library wraps this class component into a convenient functional API with <ErrorBoundary> and useErrorBoundary().

React.Component re-renders every time setState is called or the parent re-renders — regardless of whether props or state actually changed.

React.PureComponent implements shouldComponentUpdate automatically. It performs a shallow comparison of current and next props/state. If nothing has changed shallowly, it skips the re-render.

Key caveats of shallow comparison:

  • Primitive values (string, number, boolean) are compared by value — works correctly
  • Objects and arrays are compared by reference — a new array [] always fails equality even if contents are the same
  • Deeply nested changes won't trigger re-render — can cause stale UI bugs

The functional equivalent is React.memo(). Both are optimization tools — only add them when you've identified a performance issue, not preemptively on every component.

React Fragments let you group multiple JSX elements without adding an extra DOM node. React requires a single root element per component — Fragments fulfill this requirement without polluting the DOM with wrappers like <div>.

Shorthand <></> — use when:

  • You just need to group elements with no attributes
  • Most day-to-day usage — returning multiple sibling elements, table rows, etc.

Explicit <React.Fragment> — required when:

  • You need to pass a key prop (e.g., mapping over a list where each fragment needs a key for reconciliation)
  • The shorthand syntax <> does not support attributes including key
// Shorthand — no key needed
return (
  <>
    <td>Name</td>
    <td>Age</td>
  </>
);

// Explicit — key required in list
{items.map(item => (
  <React.Fragment key={item.id}>
    <dt>{item.term}</dt>
    <dd>{item.desc}</dd>
  </React.Fragment>
))}

Common Mistakes to Avoid

1. Forgetting key prop in lists
When rendering lists with .map(), every element must have a unique key prop. Missing or non-unique keys cause React's reconciliation algorithm to make incorrect assumptions — resulting in state ending up in the wrong component, stale renders, and hard-to-debug UI bugs. Never use array index as key when the list can be reordered or filtered.
2. Mutating state directly
Always use setState (class) or the setter from useState (functional) — never mutate state directly. Direct mutation like this.state.items.push(x) or arr.push(x); setArr(arr) does not trigger a re-render because React compares object references. The reference hasn't changed, so React thinks nothing happened and the UI stays stale.
3. Calling hooks conditionally
React hooks must be called in the same order on every render — no hooks inside if statements, loops, or nested functions. Violating the Rules of Hooks causes React to misalign the internal hooks array between renders, leading to incorrect state assignments and runtime crashes. If you need conditional logic, put it inside the hook, not around it.
4. Not memoizing expensive components in lists
Rendering 100+ items in a list where each item is a complex component without React.memo means every parent state change (e.g., a filter input) re-renders all 100+ items — even if their data didn't change. Wrap list item components in React.memo and stabilize function props with useCallback to avoid cascading re-renders and maintain smooth scrolling performance.
