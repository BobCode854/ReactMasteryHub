Why Testing Matters

Testing ensures your React components behave as expected, prevents regressions when you refactor, and documents intended behavior. The testing pyramid: many unit tests, some integration tests, few E2E tests.

Unit Tests
Test isolated functions and components in isolation. Fast, cheap, abundant.
Integration Tests
Test how multiple units work together. Balance of coverage and speed.
E2E Tests
Test full user flows in a real browser. Slowest but highest confidence.
Jest Basics

Jest is the most popular JavaScript testing framework. It provides test runners, assertion libraries, spies, mocks, and code coverage out of the box.

Core Jest Structure: describe() groups related tests, it() / test() defines individual test cases, expect() creates assertions with matchers.
Simple Unit Test
// Simple unit test describe('calculateTotal', () => { it('returns correct sum of items', () => { const items = [{ price: 10 }, { price: 20 }, { price: 5 }]; expect(calculateTotal(items)).toBe(35); }); it('returns 0 for empty array', () => { expect(calculateTotal([])).toBe(0); }); });
Common Jest Matchers
Equality & Truthiness
expect(x).toBe(42); // strict === expect(obj).toEqual({ a: 1 }); // deep equality expect(x).toBeTruthy(); expect(x).toBeFalsy(); expect(x).toBeNull(); expect(x).toBeUndefined();
Arrays & Strings
expect(arr).toContain('item'); expect(arr).toHaveLength(3); expect(str).toMatch(/pattern/); expect(fn).toThrow('error msg'); expect(mock).toHaveBeenCalled(); expect(mock).toHaveBeenCalledWith(arg);
Lifecycle Hooks
beforeAll(() => { /* runs once before all tests in block */ }); afterAll(() => { /* runs once after all tests in block */ }); beforeEach(() => { /* runs before each test */ }); afterEach(() => { /* runs after each test — great for cleanup */ }); describe('UserService', () => { let db; beforeAll(async () => { db = await connectDB(); }); afterAll(async () => { await db.close(); }); afterEach(async () => { await db.clearUsers(); // reset state between tests }); });
React Testing Library

React Testing Library (RTL) encourages testing your components the way users interact with them — by querying for accessible elements, not implementation details.

Core Philosophy: "The more your tests resemble the way your software is used, the more confidence they can give you." — Kent C. Dodds. Query by role, label, text — not by class names or test IDs.
Query Priority (Best to Worst)
BestgetByRole
Accessible roles: button, heading, textbox…
GoodgetByLabelText
Form elements associated with labels
OKgetByTestId
Last resort: data-testid attribute
Testing a Counter Component
import { render, screen, fireEvent, waitFor } from '@testing-library/react'; import userEvent from '@testing-library/user-event'; // Test a Counter component describe('Counter', () => { it('renders initial count of 0', () => { render(<Counter />); expect(screen.getByText('Count: 0')).toBeInTheDocument(); }); it('increments count on button click', async () => { const user = userEvent.setup(); render(<Counter />); await user.click(screen.getByRole('button', { name: /increment/i })); expect(screen.getByText('Count: 1')).toBeInTheDocument(); }); });
Tip: Prefer userEvent over fireEvent. userEvent simulates real browser interactions (dispatches multiple events like a real user), while fireEvent only dispatches a single DOM event.
Testing Async Components with API Calls
// Test async component with API call describe('UserList', () => { it('displays users after loading', async () => { const mockUsers = [{ id: 1, name: 'John' }, { id: 2, name: 'Jane' }]; global.fetch = jest.fn().mockResolvedValueOnce({ ok: true, json: async () => mockUsers, }); render(<UserList />); expect(screen.getByText(/loading/i)).toBeInTheDocument(); await waitFor(() => { expect(screen.getByText('John')).toBeInTheDocument(); expect(screen.getByText('Jane')).toBeInTheDocument(); }); }); it('shows error state when fetch fails', async () => { global.fetch = jest.fn().mockRejectedValueOnce(new Error('Network error')); render(<UserList />); await waitFor(() => { expect(screen.getByText(/something went wrong/i)).toBeInTheDocument(); }); }); });

Testing Custom Hooks with renderHook

Custom hooks can't be tested directly (they can only be called inside React components). renderHook creates a minimal test component that calls your hook, letting you test it in isolation.

import { renderHook, act } from '@testing-library/react'; test('useCounter increments correctly', () => { const { result } = renderHook(() => useCounter(0)); expect(result.current.count).toBe(0); act(() => { result.current.increment(); }); expect(result.current.count).toBe(1); }); test('useCounter decrements and resets', () => { const { result } = renderHook(() => useCounter(10)); act(() => { result.current.decrement(); result.current.decrement(); }); expect(result.current.count).toBe(8); act(() => { result.current.reset(); }); expect(result.current.count).toBe(10); // resets to initial value });
renderHook with Providers
// When your hook uses context (like useAuth, useTheme), wrap it in the provider test('useAuth returns current user', () => { const wrapper = ({ children }) => ( <AuthProvider initialUser={{ name: 'John', role: 'admin' }}> {children} </AuthProvider> ); const { result } = renderHook(() => useAuth(), { wrapper }); expect(result.current.user.name).toBe('John'); expect(result.current.user.role).toBe('admin'); expect(result.current.isAuthenticated).toBe(true); });
When to use act(): Wrap any code that causes React state updates in act(). This ensures React has processed all state updates and re-renders before your assertions run. RTL's userEvent and waitFor wrap things in act() automatically.
Mocking Modules

Mocking replaces real dependencies with controlled fakes, letting you isolate the component under test and control what dependencies return.

Mock react-router-dom
// Mock react-router-dom const mockNavigate = jest.fn(); jest.mock('react-router-dom', () => ({ ...jest.requireActual('react-router-dom'), useNavigate: () => mockNavigate, useParams: () => ({ id: '123' }), })); test('navigates to profile on button click', async () => { const user = userEvent.setup(); render(<ProfileButton />); await user.click(screen.getByRole('button', { name: /view profile/i })); expect(mockNavigate).toHaveBeenCalledWith('/profile/123'); });
Mock API Module
// Mock API module jest.mock('../api/users', () => ({ getUsers: jest.fn().mockResolvedValue([{ id: 1, name: 'John' }]), createUser: jest.fn().mockResolvedValue({ id: 2, name: 'Jane' }), })); // Override mock for specific test it('handles API error gracefully', async () => { getUsers.mockRejectedValueOnce(new Error('Server error')); render(<UserList />); await waitFor(() => { expect(screen.getByText(/failed to load/i)).toBeInTheDocument(); }); });
Mocking Timers
// Control time in tests describe('Debounced Search', () => { beforeEach(() => { jest.useFakeTimers(); }); afterEach(() => { jest.useRealTimers(); }); it('only searches after debounce delay', async () => { const user = userEvent.setup({ delay: null }); // disable user-event delays const mockSearch = jest.fn(); render(<SearchInput onSearch={mockSearch} />); await user.type(screen.getByRole('searchbox'), 'hello'); expect(mockSearch).not.toHaveBeenCalled(); // not called yet jest.advanceTimersByTime(300); expect(mockSearch).toHaveBeenCalledWith('hello'); }); });
Cypress E2E Testing

Cypress runs tests directly in the browser, giving you full control over the application. It auto-waits for elements, handles asynchronous behavior automatically, and provides powerful debugging tools.

Cypress vs Selenium: Cypress runs in the same browser process as your app — no WebDriver. This means faster, more reliable tests with automatic retries, real-time reloading, and time-travel debugging.
Login Flow E2E Test
describe('Login Flow', () => { beforeEach(() => { cy.visit('/login'); }); it('logs in successfully with valid credentials', () => { cy.get('[data-testid="email-input"]').type('user@example.com'); cy.get('[data-testid="password-input"]').type('password123'); cy.get('[data-testid="login-button"]').click(); cy.url().should('include', '/dashboard'); cy.get('[data-testid="welcome-message"]').should('contain', 'Welcome back'); }); it('shows error with invalid credentials', () => { cy.get('[data-testid="email-input"]').type('wrong@email.com'); cy.get('[data-testid="password-input"]').type('wrongpass'); cy.get('[data-testid="login-button"]').click(); cy.get('[data-testid="error-message"]').should('be.visible'); }); });
Cypress Custom Commands & API Intercepts
// cypress/support/commands.ts — reusable login command Cypress.Commands.add('login', (email, password) => { cy.request('POST', '/api/auth/login', { email, password }) .then(({ body }) => { localStorage.setItem('token', body.token); }); }); // Intercept API calls in tests it('displays products from API', () => { cy.intercept('GET', '/api/products', { fixture: 'products.json' }).as('getProducts'); cy.login('admin@example.com', 'password'); cy.visit('/products'); cy.wait('@getProducts'); cy.get('[data-testid="product-card"]').should('have.length', 5); });
Testing Best Practices Checklist
What to Do
  • Query by role, label, or text — not class names
  • Use userEvent instead of fireEvent for interactions
  • Test behavior, not implementation details
  • Use data-testid only as last resort for E2E
  • Mock external dependencies (APIs, routers, timers)
  • Clean up mocks and state between tests with afterEach
  • Write tests before fixing bugs (TDD for bugs)
  • Aim for 70-80% coverage — cover critical paths 100%
What to Avoid
  • Don't test internal state directly (component.state)
  • Don't snapshot test large components (brittle)
  • Don't mock things you don't own (like React internals)
  • Avoid act() warnings — they signal real issues
  • Don't write tests just to hit coverage numbers
  • Avoid coupling tests to component structure/markup
  • Don't use setTimeout in tests — use fake timers
  • Avoid testing 3rd-party library behavior
