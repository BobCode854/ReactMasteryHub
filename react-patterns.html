What Are Design Patterns?

Design patterns are reusable solutions to commonly occurring problems in software design. In React, patterns help you share logic between components, manage state, and compose UI in a maintainable way. Knowing which pattern to apply — and when — is a hallmark of senior React developers.

Pattern 1
Higher-Order Component (HOC)

A HOC is a function that takes a component and returns a new enhanced component. It's a way to reuse component logic — like a decorator pattern for components.

Signature: const Enhanced = withSomething(WrappedComponent) — The HOC wraps the component, adds behavior, and passes props through.
// withLoading HOC — adds loading state to any component function withLoading(WrappedComponent) { return function WithLoading({ isLoading, ...props }) { if (isLoading) return <div className="spinner">Loading...</div>; return <WrappedComponent {...props} />; }; } // withErrorBoundary HOC function withErrorBoundary(WrappedComponent, FallbackComponent) { return class extends React.Component { state = { hasError: false }; static getDerivedStateFromError() { return { hasError: true }; } render() { if (this.state.hasError) return <FallbackComponent />; return <WrappedComponent {...this.props} />; } }; } const SafeUserProfile = withErrorBoundary(UserProfile, ErrorFallback); const LoadingUserList = withLoading(UserList); // Usage <LoadingUserList isLoading={loading} users={users} />
Naming Convention: HOC functions are conventionally named with* (e.g., withAuth, withLoading, withAnalytics). The returned component should have a display name for debugging: WithLoading.displayName = `WithLoading(${WrappedComponent.displayName})`.
Pattern 2
Render Props Pattern

A component with a render prop accepts a function as a prop that it calls to render its output. This shares stateful logic without HOC wrapper complexity.

function MouseTracker({ render }) { const [pos, setPos] = useState({ x: 0, y: 0 }); return ( <div onMouseMove={e => setPos({ x: e.clientX, y: e.clientY })}> {render(pos)} </div> ); } // Usage <MouseTracker render={({ x, y }) => ( <p>Mouse at {x}, {y}</p> )} /> // Alternatively, use "children as a function" (same pattern, different API) function DataFetcher({ url, children }) { const [data, setData] = useState(null); const [loading, setLoading] = useState(true); useEffect(() => { fetch(url).then(r => r.json()).then(setData).finally(() => setLoading(false)); }, [url]); return children({ data, loading }); } // Clean usage with children as a function <DataFetcher url="/api/users"> {({ data, loading }) => loading ? <span>Loading...</span> : <UserList users={data} />} </DataFetcher>
Render Props vs Custom Hooks: Render props solve the same problem as custom hooks but with a different API. Custom hooks are generally preferred in modern React because they're simpler and don't add extra DOM nodes. Render props are still useful when you need render flexibility.
Pattern 3
Compound Components Pattern

Compound components work together to form a unified, flexible API. The parent manages shared state via context; child components are attached to the parent as properties. Think of HTML's <select> + <option> relationship.

const SelectContext = createContext(); function Select({ children, value, onChange }) { return ( <SelectContext.Provider value={{ value, onChange }}> <div className="select-container">{children}</div> </SelectContext.Provider> ); } Select.Option = function Option({ value, children }) { const { value: selected, onChange } = useContext(SelectContext); return ( <div className={`option ${selected === value ? 'selected' : ''}`} onClick={() => onChange(value)} > {children} </div> ); }; // Usage — clean consumer API, no config props explosion <Select value={fruit} onChange={setFruit}> <Select.Option value="apple">Apple</Select.Option> <Select.Option value="banana">Banana</Select.Option> <Select.Option value="cherry">Cherry</Select.Option> </Select>
Without Compound Components
<Select value={fruit} onChange={setFruit} options={[ { value: 'apple', label: 'Apple' }, { value: 'banana', label: 'Banana' }, ]} renderOption={...} optionClassName={...} /> // Prop explosion!
With Compound Components
<Select value={fruit} onChange={setFruit}> <Select.Option value="apple"> 🍎 Apple </Select.Option> // Flexible, extensible! </Select>
Pattern 4
Custom Hooks Pattern

Custom hooks extract and reuse stateful logic across multiple components. They're the modern replacement for render props and HOCs for logic sharing.

// useForm — reusable form logic function useForm(initialValues, validate) { const [values, setValues] = useState(initialValues); const [errors, setErrors] = useState({}); const [touched, setTouched] = useState({}); const handleChange = (e) => { const { name, value } = e.target; setValues(v => ({ ...v, [name]: value })); }; const handleBlur = (e) => { const { name } = e.target; setTouched(t => ({ ...t, [name]: true })); if (validate) setErrors(validate(values)); }; const handleSubmit = (onSubmit) => (e) => { e.preventDefault(); const errs = validate ? validate(values) : {}; if (Object.keys(errs).length === 0) onSubmit(values); else setErrors(errs); }; return { values, errors, touched, handleChange, handleBlur, handleSubmit }; } // Usage — clean, no logic duplication function LoginForm() { const { values, errors, handleChange, handleBlur, handleSubmit } = useForm( { email: '', password: '' }, (vals) => { const errs = {}; if (!vals.email) errs.email = 'Required'; if (!vals.password) errs.password = 'Required'; return errs; } ); // Use values, errors, handlers in JSX }
Pattern 5
Provider Pattern

The Provider pattern uses React Context to share data across a component tree without prop drilling. Any component in the tree can subscribe to the context.

// ThemeProvider — wraps app and provides theme to all components const ThemeContext = createContext(); function ThemeProvider({ children }) { const [theme, setTheme] = useState('light'); const toggleTheme = () => setTheme(t => t === 'light' ? 'dark' : 'light'); return ( <ThemeContext.Provider value={{ theme, toggleTheme }}> <div data-theme={theme}>{children}</div> </ThemeContext.Provider> ); } const useTheme = () => { const ctx = useContext(ThemeContext); if (!ctx) throw new Error('useTheme must be inside ThemeProvider'); return ctx; }; // Any component, any depth — no prop drilling function DarkModeToggle() { const { theme, toggleTheme } = useTheme(); return <button onClick={toggleTheme}>{theme === 'light' ? '🌙' : '☀️'}</button>; }
Pattern 6
Observer Pattern (Pub/Sub)

The Observer pattern allows components to subscribe to events without being tightly coupled. Useful for cross-component communication that doesn't warrant Redux/Context.

// Simple EventEmitter for cross-component communication class EventEmitter { constructor() { this.events = {}; } on(event, callback) { if (!this.events[event]) this.events[event] = []; this.events[event].push(callback); return () => this.off(event, callback); // returns unsubscribe fn } off(event, callback) { this.events[event] = (this.events[event] || []).filter(cb => cb !== callback); } emit(event, ...args) { (this.events[event] || []).forEach(cb => cb(...args)); } } const bus = new EventEmitter(); // Subscribe in useEffect — clean up on unmount function NotificationListener() { const [notification, setNotification] = useState(null); useEffect(() => { const unsubscribe = bus.on('notification', setNotification); return unsubscribe; // cleanup on unmount }, []); if (!notification) return null; return <div className="toast">{notification.message}</div>; } // Emit from anywhere function SaveButton() { return <button onClick={() => bus.emit('notification', { message: 'Saved!' })}>Save</button>; }
Pattern 7
Factory Pattern

A factory function creates different component variants based on configuration, avoiding repetitive boilerplate.

// Create input variants from a factory function createInput(type, defaultProps) { return function Input(props) { return ( <input type={type} className="form-control" {...defaultProps} {...props} /> ); }; } const EmailInput = createInput('email', { placeholder: 'Enter email', autoComplete: 'email', }); const PasswordInput = createInput('password', { placeholder: 'Enter password', autoComplete: 'current-password', }); // Usage <EmailInput name="email" onChange={handleChange} /> <PasswordInput name="password" onChange={handleChange} />
Pattern 8
Strategy Pattern

The strategy pattern lets you swap algorithms/behaviors at runtime without changing the component itself.

// Different sort strategies const sortStrategies = { alphabetical: (a, b) => a.name.localeCompare(b.name), byDate: (a, b) => new Date(b.createdAt) - new Date(a.createdAt), byPrice: (a, b) => a.price - b.price, }; function SortableList({ items, sortStrategy = 'alphabetical' }) { const sorted = [...items].sort(sortStrategies[sortStrategy]); return ( <ul> {sorted.map(item => ( <li key={item.id}>{item.name}</li> ))} </ul> ); } // Swap strategy without changing component <SortableList items={products} sortStrategy="byPrice" />
