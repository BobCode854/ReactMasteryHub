DevOps for React Applications

CI/CD, Docker, Nginx, Monitoring, and Deployment Strategies

~45 min read 9 topics 30+ code examples

The Three-Stage Pipeline — A production React CI/CD pipeline separates concerns into three jobs: (1) Test — lint, unit tests, coverage; (2) Build — compile TypeScript, bundle assets, inject env vars; (3) Deploy — push to CDN/server, invalidate caches. Each stage needs the previous one, so failures stop the pipeline early and save compute minutes.

Continuous Integration and Continuous Deployment (CI/CD) is the practice of automatically building, testing, and deploying your application every time code is pushed to a repository. For React applications, this means every pull request gets linted and tested automatically, every merge to the main branch produces a verified production build, and that build is deployed to your hosting infrastructure without any manual steps. This transforms deployment from a risky, manual ceremony into a routine, automated event.

GitHub Actions is GitHub's built-in CI/CD platform, and it is one of the best choices for React applications hosted on GitHub. Workflows are defined as YAML files stored in .github/workflows/ and are triggered by repository events — pushes, pull requests, scheduled runs, or manual triggers. Each workflow runs in isolated virtual machines (runners) with a fresh environment, so you never have "works on my machine" problems. GitHub provides generous free minutes for public repositories and a reasonable allocation for private ones.

The key philosophy for React CI/CD is fail fast, fail early. By ordering your pipeline stages so that quick checks (lint, type check) run before expensive ones (full test suite), and fast tests run before slow builds, you give developers immediate feedback on the most common failure modes. A lint error caught in 30 seconds is far less disruptive than one discovered after a 10-minute build. The workflow below embodies this philosophy with three independent but sequentially-gated jobs.

# .github/workflows/ci-cd.yml
name: React CI/CD Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

env:
  NODE_VERSION: '20'
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  test:
    name: Test & Lint
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Run ESLint
        run: npm run lint

      - name: Run type check
        run: npx tsc --noEmit

      - name: Run unit tests
        run: npm test -- --coverage --watchAll=false

      - name: Upload coverage to Codecov
        uses: codecov/codecov-action@v4
        with:
          token: ${{ secrets.CODECOV_TOKEN }}
          fail_ci_if_error: true

  build:
    name: Build Application
    runs-on: ubuntu-latest
    needs: test
    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Build for production
        run: npm run build
        env:
          REACT_APP_API_URL: ${{ secrets.API_URL }}
          REACT_APP_SENTRY_DSN: ${{ secrets.SENTRY_DSN }}
          GENERATE_SOURCEMAP: false

      - name: Upload build artifacts
        uses: actions/upload-artifact@v4
        with:
          name: build-${{ github.sha }}
          path: build/
          retention-days: 7

  deploy:
    name: Deploy to Production
    runs-on: ubuntu-latest
    needs: build
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    environment:
      name: production
      url: https://my-app.com
    steps:
      - name: Download build artifacts
        uses: actions/download-artifact@v4
        with:
          name: build-${{ github.sha }}
          path: build/

      - name: Deploy to S3
        run: |
          aws s3 sync build/ s3://${{ secrets.S3_BUCKET }} \
            --delete \
            --cache-control "public, max-age=31536000" \
            --exclude "index.html"
          aws s3 cp build/index.html s3://${{ secrets.S3_BUCKET }}/index.html \
            --cache-control "no-cache, no-store, must-revalidate"
        env:
          AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
          AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          AWS_DEFAULT_REGION: us-east-1

      - name: Invalidate CloudFront cache
        run: |
          aws cloudfront create-invalidation \
            --distribution-id ${{ secrets.CLOUDFRONT_DISTRIBUTION_ID }} \
            --paths "/*"
        env:
          AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
          AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
          AWS_DEFAULT_REGION: us-east-1

      - name: Notify Slack on success
        uses: slackapi/slack-github-action@v1
        with:
          payload: |
            {
              "text": "Deployment successful! :rocket:",
              "blocks": [
                {
                  "type": "section",
                  "text": {
                    "type": "mrkdwn",
                    "text": "*Deployed:* ${{ github.repository }} to production\n*Commit:* ${{ github.sha }}\n*By:* ${{ github.actor }}"
                  }
                }
              ]
            }
        env:
          SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK_URL }}
Understanding Each Pipeline Job

Job 1 — test (Test & Lint):

  • Runs ESLint to catch code quality issues and enforce style rules before anything else, giving near-instant feedback on common errors.
  • Runs tsc --noEmit to perform a full TypeScript type check without emitting output files — this catches type errors that your editor might miss in large projects.
  • Runs the full unit test suite with coverage reporting, uploading results to Codecov so coverage trends are tracked over time and regressions are immediately visible in pull request reviews.
  • Uses actions/setup-node's built-in npm caching so the dependency install is skipped when package-lock.json hasn't changed, cutting job time significantly.

Job 2 — build (Build Application):

  • Declares needs: test so it only runs after the test job passes — a failed test suite will never produce a deployable artifact.
  • Injects production environment variables from GitHub Secrets at build time, ensuring API URLs and DSNs are baked into the bundle without being exposed in the repository.
  • Sets GENERATE_SOURCEMAP: false for production builds to prevent source code from being downloadable by end users, which is both a security and performance improvement.
  • Uploads the compiled build/ directory as a named artifact tied to the commit SHA, allowing the deploy job to fetch it without re-running the build.

Job 3 — deploy (Deploy to Production):

  • Uses an if: conditional to only run on direct pushes to main, not on pull requests — preventing accidental production deployments from feature branch merges.
  • Syncs static assets to S3 with a one-year cache header (max-age=31536000) since Vite/CRA asset filenames include a content hash, while index.html is uploaded separately with no-cache headers so browsers always fetch the latest entry point.
  • Invalidates the CloudFront CDN cache after the S3 sync completes, ensuring users receive the new index.html immediately rather than serving a stale cached version from the edge nodes.
  • Sends a structured Slack notification on success with repository name, commit SHA, and deploying actor — giving the team real-time visibility into production changes.
Cache node_modules: Use actions/cache or the built-in cache in actions/setup-node to avoid reinstalling packages on every run. This can save 1–3 minutes per build.
Matrix Builds: Test against multiple Node versions with strategy: matrix: node: ['18', '20', '22'] to catch version-specific bugs before they reach production.
Advanced: Lighthouse CI Integration

Beyond correctness testing, you can enforce performance budgets in your CI pipeline using Lighthouse CI. This prevents performance regressions from shipping to users by failing the build if key metrics fall below defined thresholds — treating performance as a first-class requirement alongside correctness.

  lighthouse:
    name: Lighthouse CI
    runs-on: ubuntu-latest
    needs: build
    steps:
      - uses: actions/checkout@v4
      - name: Download build artifacts
        uses: actions/download-artifact@v4
        with:
          name: build-${{ github.sha }}
          path: build/
      - name: Run Lighthouse CI
        uses: treosh/lighthouse-ci-action@v11
        with:
          urls: |
            http://localhost:3000
            http://localhost:3000/about
          budgetPath: ./.lighthouserc.json
          uploadArtifacts: true
          temporaryPublicStorage: true

The corresponding .lighthouserc.json budget file defines the minimum acceptable scores for each Lighthouse category. The "error" level causes the CI job to fail and block the merge, while "warn" produces a warning annotation without failing:

{
  "ci": {
    "assert": {
      "assertions": {
        "categories:performance": ["error", {"minScore": 0.9}],
        "categories:accessibility": ["error", {"minScore": 0.9}],
        "categories:best-practices": ["warn", {"minScore": 0.85}],
        "categories:seo": ["warn", {"minScore": 0.8}]
      }
    }
  }
}

Docker brings environment consistency to React deployments. The infamous "it works on my machine" problem is eliminated because the application runs in an identical containerized environment on every developer's laptop, in every CI runner, and in every production server. You define the OS, Node version, build steps, and runtime server once in a Dockerfile, and Docker guarantees that definition is reproduced exactly — down to the system library versions — everywhere the image is run.

For React applications specifically, Docker's multi-stage build feature is transformative. A React app only needs Node.js at build time to compile TypeScript and bundle assets with Webpack or Vite. At runtime, the output is just static HTML, CSS, and JavaScript files that any web server can serve. Multi-stage builds let you use a full Node.js environment to build the app, then copy only the compiled output into a tiny production image based on Nginx Alpine — discarding all the Node.js tooling, source code, and development dependencies that have no business being in production.

Container isolation and reproducibility also make rollbacks trivial. Because each deployment is a tagged Docker image, rolling back to a previous version is as simple as redeploying the previous image tag. There is no need to rebuild, no risk of dependency drift, and no manual steps — the old image is identical to what was running before. Combined with container orchestration platforms like Kubernetes or AWS ECS, this makes zero-downtime deployments and canary releases straightforward operational practices rather than complex one-off procedures.

# Stage 1: Dependencies
FROM node:20-alpine AS deps
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production && \
    npm cache clean --force

# Stage 2: Builder
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
ARG REACT_APP_API_URL
ARG REACT_APP_SENTRY_DSN
ENV REACT_APP_API_URL=$REACT_APP_API_URL
ENV REACT_APP_SENTRY_DSN=$REACT_APP_SENTRY_DSN
RUN npm run build

# Stage 3: Production (Nginx)
FROM nginx:1.25-alpine AS production
# Copy custom nginx config
COPY nginx.conf /etc/nginx/nginx.conf
# Copy built assets from builder stage
COPY --from=builder /app/build /usr/share/nginx/html
# Add healthcheck
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD wget -qO- http://localhost/health || exit 1
# Use non-root user for security
RUN addgroup -g 1001 -S nginx-app && \
    adduser -S -D -H -u 1001 -h /var/cache/nginx -s /sbin/nologin -G nginx-app -g nginx-app nginx-app && \
    chown -R nginx-app:nginx-app /usr/share/nginx/html && \
    chown -R nginx-app:nginx-app /var/cache/nginx && \
    touch /var/run/nginx.pid && \
    chown nginx-app:nginx-app /var/run/nginx.pid
USER nginx-app
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
Understanding Each Build Stage

Stage 1 — deps (Dependencies):

  • Uses node:20-alpine — the Alpine Linux variant of the Node image, which is roughly 50MB compared to the ~1GB Debian-based node:20 image.
  • Runs npm ci --only=production to install only production dependencies and cleans the npm cache immediately to minimize layer size. This stage is often unused in the final image but kept as a separate layer for caching clarity.
  • The key optimization is copying package*.json first, installing dependencies, and then copying source — so the expensive install step is cached as long as package-lock.json doesn't change.

Stage 2 — builder (Build Application):

  • Installs all dependencies including devDependencies (TypeScript, Vite, testing tools) needed to compile the application.
  • Accepts build-time arguments via ARG and promotes them to environment variables with ENV so they are available during the npm run build step and baked into the JavaScript bundle.
  • Produces the final compiled output in /app/build — all TypeScript transpiled, all modules bundled, all assets optimized and content-hashed. None of the source files, node_modules, or build tooling will appear in the final image.

Stage 3 — production (Nginx Runtime):

  • Starts fresh from nginx:1.25-alpine — about 7MB base. Only the compiled static files from the builder stage are copied in with COPY --from=builder, resulting in a final image of ~25MB total.
  • Creates and runs as a non-root user (nginx-app) — a critical security practice. If the container is compromised, the attacker only has the permissions of an unprivileged user rather than root access to the host system.
  • Adds a HEALTHCHECK so container orchestrators like Docker Swarm or Kubernetes know whether the container is actually serving traffic, enabling automatic restarts and zero-downtime rolling updates.
Image Size Matters: A naive React Docker image with node:20 can be 1.2GB+. The multi-stage approach above produces an image of ~25MB (nginx:alpine base). Always use alpine variants and multi-stage builds.
Layer Caching: Copy package*.json and run npm ci BEFORE copying the rest of your source code. Docker caches layers — if only source files changed (not package.json), the expensive npm install layer is reused.
Nginx Configuration

The Nginx configuration is what makes or breaks a React SPA deployment. You need to configure proper SPA routing, aggressive caching for hashed assets, security headers, gzip compression, and a health check endpoint. Here is a production-ready configuration:

worker_processes auto;
error_log /var/log/nginx/error.log warn;
pid /var/run/nginx.pid;

events {
    worker_connections 1024;
    use epoll;
    multi_accept on;
}

http {
    include /etc/nginx/mime.types;
    default_type application/octet-stream;

    # Logging
    log_format main '$remote_addr - $remote_user [$time_local] "$request" '
                    '$status $body_bytes_sent "$http_referer" '
                    '"$http_user_agent" "$http_x_forwarded_for"';
    access_log /var/log/nginx/access.log main;

    # Performance
    sendfile on;
    tcp_nopush on;
    tcp_nodelay on;
    keepalive_timeout 65;
    gzip on;
    gzip_vary on;
    gzip_min_length 1024;
    gzip_types text/plain text/css text/xml text/javascript
               application/javascript application/xml+rss
               application/json image/svg+xml;

    server {
        listen 80;
        server_name _;
        root /usr/share/nginx/html;
        index index.html;

        # Security headers
        add_header X-Frame-Options "SAMEORIGIN" always;
        add_header X-Content-Type-Options "nosniff" always;
        add_header X-XSS-Protection "1; mode=block" always;
        add_header Referrer-Policy "strict-origin-when-cross-origin" always;
        add_header Content-Security-Policy "default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; font-src 'self' https://fonts.gstatic.com; img-src 'self' data: https:;" always;

        # SPA routing — serve index.html for all non-file routes
        location / {
            try_files $uri $uri/ /index.html;
        }

        # Cache static assets aggressively (they have content hashes)
        location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2|ttf|eot)$ {
            expires 1y;
            add_header Cache-Control "public, immutable";
            access_log off;
        }

        # Never cache index.html
        location = /index.html {
            add_header Cache-Control "no-store, no-cache, must-revalidate";
            expires 0;
        }

        # Health check endpoint
        location /health {
            return 200 'healthy\n';
            add_header Content-Type text/plain;
            access_log off;
        }
    }
}
Why try_files $uri $uri/ /index.html is Essential
React Router manages navigation entirely on the client side — the server only serves one HTML file. When a user visits /dashboard/settings directly or refreshes the page, Nginx receives an HTTP request for that path. Without try_files, Nginx would return a 404 because no file exists at that path. The directive tells Nginx: first try the literal URI, then try it as a directory, and if neither exists, serve index.html. React then reads the URL from window.location and renders the correct component. This is required for all SPA frameworks (React, Vue, Angular) deployed on any static server.
Docker Compose for Local & Staging Environments

Docker Compose lets you define and run multi-container applications with a single YAML file. For a React application, this typically means running the frontend, a backend API, and a database together in a locally reproducible environment. Environment variables are the standard mechanism for configuring application behavior across environments — development, staging, and production — without changing code. Both Create React App and Vite support .env files with a hierarchical override system, but they use different variable name prefixes to distinguish public (browser-safe) variables from potentially sensitive ones that should never be bundled. Understanding this system is foundational to any React deployment strategy.

Both CRA (REACT_APP_ prefix) and Vite (VITE_ prefix) only embed variables with the correct prefix into the browser bundle. Any variable without the prefix is silently ignored at build time. This is a safety mechanism — it prevents accidentally exposing a DATABASE_URL or JWT_SECRET that happens to be in your shell environment. At build time, these variables are statically replaced in the compiled JavaScript: process.env.REACT_APP_API_URL becomes the literal string value of that variable. There is no runtime process.env in the browser.

The .env file hierarchy gives you layered configuration with increasing specificity. Base files committed to the repository define sane defaults. Local files (gitignored) allow individual developers to override values without affecting teammates. Environment-specific files apply only when running in a particular mode. This layering means you rarely need to touch environment variable configuration for day-to-day development — the defaults just work — while still giving you full control for edge cases.

.env File Hierarchy
.env                    # Base defaults (committed)
.env.local              # Local overrides (gitignored)
.env.development        # Development mode (committed)
.env.development.local  # Dev overrides (gitignored)
.env.test               # Test mode (committed)
.env.production         # Production (committed)
.env.production.local   # Prod overrides (gitignored)

Accessing Env Vars in React Code
// CRA — accessed via process.env
const apiUrl = process.env.REACT_APP_API_URL;

// Vite — accessed via import.meta.env
const apiUrl = import.meta.env.VITE_API_URL;

// Type-safe env vars with Vite (vite-env.d.ts)
/// <reference types="vite/client" />
interface ImportMetaEnv {
  readonly VITE_API_URL: string;
  readonly VITE_GOOGLE_MAPS_KEY: string;
  readonly VITE_FEATURE_FLAG_NEW_UI: string;
}

// Config module pattern — centralize all env access
// src/config/env.ts
export const config = {
  apiUrl: import.meta.env.VITE_API_URL || 'http://localhost:3001',
  googleMapsKey: import.meta.env.VITE_GOOGLE_MAPS_KEY,
  isNewUiEnabled: import.meta.env.VITE_FEATURE_FLAG_NEW_UI === 'true',
  isDevelopment: import.meta.env.DEV,
  isProduction: import.meta.env.PROD,
} as const;

// Usage anywhere in the app
import { config } from '@/config/env';
const response = await fetch(`${config.apiUrl}/users`);
SECURITY WARNING — Frontend Env Vars Are PUBLIC
Any environment variable you embed in your React bundle is visible to ALL users — it is just JavaScript in the browser. Never put API secrets, private keys, database passwords, or any sensitive credentials in REACT_APP_* or VITE_* variables. These are for PUBLIC configuration only: API base URLs, public OAuth client IDs, feature flag endpoints, analytics tracking IDs. Keep secrets server-side only.
Runtime Environment Injection

Build-time env var substitution has a significant operational limitation: it bakes the values permanently into the JavaScript bundle. If you need different values for staging vs. production, you must build two separate Docker images. Runtime injection solves this by generating a small JavaScript file at container startup that reads actual environment variables from the container runtime, allowing one image to serve multiple environments.

#!/bin/sh
# docker-entrypoint.sh — inject runtime config into the served HTML

# Generate runtime config from environment variables
cat > /usr/share/nginx/html/runtime-config.js << EOF
window.__ENV__ = {
  API_URL: "${API_URL}",
  FEATURE_FLAGS: "${FEATURE_FLAGS}",
  ANALYTICS_ID: "${ANALYTICS_ID}"
};
EOF

# Start nginx
exec nginx -g "daemon off;"

Read the runtime configuration in your React application using a typed wrapper module that falls back gracefully to build-time env vars for local development (where no runtime-config.js is served):

// src/config/runtime.ts
declare global {
  interface Window {
    __ENV__?: {
      API_URL: string;
      FEATURE_FLAGS: string;
      ANALYTICS_ID: string;
    };
  }
}

export const runtimeConfig = {
  apiUrl: window.__ENV__?.API_URL ?? import.meta.env.VITE_API_URL ?? 'http://localhost:3001',
  featureFlags: window.__ENV__?.FEATURE_FLAGS?.split(',') ?? [],
  analyticsId: window.__ENV__?.ANALYTICS_ID,
};

Update your Dockerfile to use the entrypoint script instead of running Nginx directly:

# In your Dockerfile (add to the production stage)
COPY docker-entrypoint.sh /docker-entrypoint.sh
RUN chmod +x /docker-entrypoint.sh
ENTRYPOINT ["/docker-entrypoint.sh"]

And reference the runtime config script in your HTML entry point so it is loaded and executed before your React application bundle:

<!-- public/index.html — load runtime config before React app -->
<script src="/runtime-config.js"></script>
Runtime vs Build-time Config: Build-time env vars require a new Docker image for each environment (dev/staging/prod). Runtime injection lets you use ONE image across all environments, changing behaviour via environment variables at container startup. This is the 12-Factor App principle and is essential for proper Docker/Kubernetes workflows.

Security is Not Optional — A React application running in the browser is an inherently hostile environment. Your JavaScript is fully visible to users, your API calls are inspectable in DevTools, and every dependency you install is a potential supply-chain attack vector. The strategies in this section form the baseline security posture every production React application should have.

Cross-Site Scripting (XSS) remains the most prevalent vulnerability in web applications. In a React context, the framework's JSX escaping protects you from most accidental XSS — React automatically escapes values rendered into the DOM. The danger zones are dangerouslySetInnerHTML, raw href attributes accepting user-provided URLs, and third-party components that may render unescaped content. Any time you render HTML from an external source, sanitization is non-negotiable.

Cross-Site Request Forgery (CSRF) is less relevant for modern single-page applications that rely on Authorization headers (Bearer tokens) rather than cookie-based sessions, since CSRF attacks cannot read custom headers. However, if your application uses session cookies for authentication, CSRF tokens or the SameSite=Strict cookie attribute become critical defenses. Always confirm with your backend team which authentication mechanism is in use.

Supply-chain attacks — where a malicious package is injected into your dependency tree — are the fastest-growing attack surface in frontend development. The npm ecosystem's massive package graph means a single compromised transitive dependency can expose thousands of applications. Proactive defenses include regular npm audit runs, pinning exact versions in package-lock.json, using Subresource Integrity on CDN scripts, and adopting automated tools like Dependabot or Snyk to surface vulnerabilities before they are exploited in the wild.

Content Security Policy (CSP)

CSP is an HTTP response header that instructs the browser which content sources it is allowed to load and execute. A strict CSP is your single most powerful defense against XSS, because even if an attacker manages to inject a script tag, the browser will refuse to execute it if it violates the policy.

# nginx.conf — Content Security Policy header
add_header Content-Security-Policy "
  default-src 'self';
  script-src 'self' 'nonce-{NONCE}' https://www.googletagmanager.com;
  style-src 'self' 'unsafe-inline' https://fonts.googleapis.com;

XSS Prevention with DOMPurify

When you must render user-supplied HTML (rich text editors, CMS content, email previews), always sanitize it with DOMPurify before passing it to dangerouslySetInnerHTML. DOMPurify parses the HTML in a sandboxed environment and strips any elements or attributes not in its allowlist.

import DOMPurify from 'dompurify';

// NEVER do this with user content:
// <div dangerouslySetInnerHTML={{ __html: userContent }} />

// Always sanitize before rendering HTML
function SafeHtmlRenderer({ htmlContent }: { htmlContent: string }) {
  const sanitized = DOMPurify.sanitize(htmlContent, {
    ALLOWED_TAGS: ['b', 'i', 'em', 'strong', 'a', 'p', 'br', 'ul', 'ol', 'li'],
    ALLOWED_ATTR: ['href', 'target', 'rel'],
    FORCE_BODY: true,
    // Force all links to open safely
    ADD_ATTR: ['target'],
  });

  // Add rel="noopener noreferrer" to all links
  DOMPurify.addHook('afterSanitizeAttributes', (node) => {
    if ('target' in node) {
      node.setAttribute('target', '_blank');
      node.setAttribute('rel', 'noopener noreferrer');
    }
  });

  return <div dangerouslySetInnerHTML={{ __html: sanitized }} />;
}

// Also sanitize URL params to prevent javascript: protocol attacks
function sanitizeUrl(url: string): string {
  const allowedProtocols = ['http:', 'https:', 'mailto:'];
  try {
    const parsed = new URL(url);
    return allowedProtocols.includes(parsed.protocol) ? url : '#';
  } catch {
    return '#'; // Invalid URL
  }
}
HTTPS Enforcement and HSTS

HTTPS is mandatory for any application that handles user data or authentication. HTTP Strict Transport Security (HSTS) instructs browsers to always use HTTPS for your domain — even if the user types http:// — for the specified duration. Submitting to the HSTS preload list bakes this policy into browser binaries, providing protection even on the very first visit.

    listen 80;
    server_name example.com www.example.com;

Secure localStorage Usage

Developers frequently misuse localStorage to persist authentication tokens. Because localStorage is synchronously accessible to any JavaScript running on the page, a successful XSS attack immediately compromises every token stored there. Prefer httpOnly cookies set by the server — they cannot be read by JavaScript at all. When localStorage is unavoidable for non-sensitive data, wrap it in a utility that handles quota errors and namespaces keys to avoid collisions.

// Never store sensitive data in localStorage — it's accessible to any JS on the page
// BAD:
localStorage.setItem('authToken', jwtToken); // Vulnerable to XSS

// GOOD: Use httpOnly cookies (set by server, not accessible to JS)
// Server sets: Set-Cookie: token=xxx; HttpOnly; Secure; SameSite=Strict

// If you must use localStorage (for non-sensitive data only):
const STORAGE_PREFIX = 'myapp_v1_';

export const secureStorage = {
  set(key: string, value: unknown): void {
    try {
      localStorage.setItem(STORAGE_PREFIX + key, JSON.stringify(value));
    } catch (err) {
      console.warn('Storage quota exceeded or unavailable');
    }
  },
  get<T>(key: string): T | null {
    try {
      const item = localStorage.getItem(STORAGE_PREFIX + key);
      return item ? JSON.parse(item) : null;
    } catch {
      return null;
    }
  },
  remove(key: string): void {
    localStorage.removeItem(STORAGE_PREFIX + key);
  },
  clear(): void {
    Object.keys(localStorage)
      .filter(k => k.startsWith(STORAGE_PREFIX))
      .forEach(k => localStorage.removeItem(k));
  },
};
Subresource Integrity (SRI): When loading scripts from CDNs, add integrity and crossorigin attributes. Example: <script src='...' integrity='sha384-abc...' crossorigin='anonymous'></script>. If the CDN is compromised and the file changes, the browser will refuse to execute it.

React's rendering model is built around the virtual DOM diffing algorithm — when state or props change, React re-renders the affected component tree and reconciles the diff against the real DOM. In small applications this is invisible; in large ones with deeply nested trees or frequent state updates, unnecessary re-renders become the dominant performance bottleneck. Understanding the reconciler's decision-making is the foundation of all React-specific performance work.

The cardinal rule of performance optimization is: measure first, optimize second. React DevTools Profiler lets you record a session and see exactly which components rendered, how long each render took, and why it triggered. Without this data, you are guessing — and premature optimization frequently adds code complexity for zero user-visible gain. Focus on the critical rendering path: the 20% of components that account for 80% of render time. Once you have identified real bottlenecks, the techniques below provide targeted, high-leverage solutions across loading, rendering, code quality, and monitoring.

React.memo + useMemo + useCallback + useTransition

These four hooks and higher-order component work together to give you fine-grained control over when React re-renders components. The key insight is that React.memo does a shallow comparison of props — so if you pass a new function or object reference on every render, memo provides no benefit. useCallback and useMemo are the tools that stabilize those references.

import React, { useState, useMemo, useCallback, useTransition } from 'react';

interface Item {
  id: number;
  name: string;
  category: string;
  price: number;
}

interface ItemListProps {
  items: Item[];
  onSelect: (id: number) => void;
  filterCategory: string;
}

// React.memo — only re-renders if props actually change
const ItemList = React.memo(function ItemList({ items, onSelect, filterCategory }: ItemListProps) {
  console.log('ItemList render'); // Should only log when props change

  // useMemo — expensive filter/sort only recomputes when deps change
  const filteredItems = useMemo(() => {
    return items
      .filter(item => filterCategory === 'all' || item.category === filterCategory)
      .sort((a, b) => a.price - b.price);
  }, [items, filterCategory]);

  return (
    <ul>
      {filteredItems.map(item => (
        <ItemRow key={item.id} item={item} onSelect={onSelect} />
      ))}
    </ul>
  );
});

interface ItemRowProps {
  item: Item;
  onSelect: (id: number) => void;
}

// Memoize child component too
const ItemRow = React.memo(function ItemRow({ item, onSelect }: ItemRowProps) {
  return (
    <li onClick={() => onSelect(item.id)}>
      {item.name} — ${item.price}
    </li>
  );
});

function App() {
  const [items] = useState<Item[]>(/* large dataset */[]);
  const [selected, setSelected] = useState<number | null>(null);
  const [filter, setFilter] = useState('all');
  const [isPending, startTransition] = useTransition();

  // useCallback — stable reference so ItemList doesn't re-render when App re-renders
  const handleSelect = useCallback((id: number) => {
    setSelected(id);
  }, []); // No deps — setSelected is stable

  const handleFilterChange = (category: string) => {
    // useTransition — mark filter update as non-urgent
    // UI stays responsive; React can interrupt the render if needed
    startTransition(() => {
      setFilter(category);
    });
  };

  return (
    <div>
      <FilterBar onFilter={handleFilterChange} />
      {isPending && <span>Updating...</span>}
      <ItemList
        items={items}
        onSelect={handleSelect}
        filterCategory={filter}
      />
    </div>
  );
}
List Virtualization with react-window

Rendering thousands of DOM nodes simultaneously is one of the most common React performance killers. Virtualization libraries like react-window solve this by only rendering the items currently visible in the viewport — typically 20-50 rows — regardless of how large the underlying dataset is. The result is linear memory usage and constant render time even for 100,000-item lists.

import { FixedSizeList as List } from 'react-window';
import AutoSizer from 'react-virtualized-auto-sizer';

interface RowProps {
  index: number;
  style: React.CSSProperties;
  data: Item[];
}

// Only visible rows are rendered — handles 100,000+ items
const Row = ({ index, style, data }: RowProps) => (
  <div style={style} className="list-item">
    <span>{data[index].name}</span>
    <span>${data[index].price}</span>
  </div>
);

function VirtualizedItemList({ items }: { items: Item[] }) {
  return (
    // AutoSizer fills the container — works with responsive layouts
    <AutoSizer>
      {({ height, width }) => (
        <List
          height={height}
          width={width}
          itemCount={items.length}
          itemSize={50}   // Fixed row height in px
          itemData={items} // Passed to Row via data prop
          overscanCount={5} // Render 5 extra rows above/below viewport
        >
          {Row}
        </List>
      )}
    </AutoSizer>
  );
}

Image Optimization

Images are typically the largest assets on a web page and the primary driver of poor Largest Contentful Paint scores. Serving images in modern formats (WebP, AVIF) reduces payload by 30–80% vs JPEG/PNG. Using the <picture> element with format fallbacks ensures compatibility with older browsers while delivering the best format to modern ones.

// Responsive images with WebP fallback
function OptimizedImage({ src, alt, width, height }: ImageProps) {
  return (
    <picture>
      <source
        srcSet={`${src}.avif`}
        type="image/avif"
      />
      <source
        srcSet={`${src}.webp`}
        type="image/webp"
      />
      <img
        src={`${src}.jpg`}
        alt={alt}
        width={width}
        height={height}
        loading="lazy"
        decoding="async"
        // Prevent layout shift — always specify dimensions
        style={{ aspectRatio: `${width}/${height}` }}
      />
    </picture>
  );
}

The following questions appear frequently in frontend DevOps and senior React engineering interviews. They test your ability to connect theoretical knowledge (CI/CD pipelines, Docker internals, security models) with practical implementation decisions. Each answer is structured to give you the depth needed for a senior-level discussion — not just the surface definition but the trade-offs, edge cases, and React-specific nuances that distinguish a strong candidate.

Continuous Integration (CI) is the practice of automatically validating every code commit by running a suite of checks — linting, type-checking, unit tests, integration tests, and build verification — in an isolated environment. The goal is to detect regressions within minutes of the offending commit rather than hours or days later. For a React application, CI specifically means running tsc --noEmit to catch TypeScript errors, ESLint to enforce code style, Vitest or Jest to verify component and hook behavior, and a production build to confirm the bundle compiles without error and stays within your performance budget.

Continuous Delivery (CD) extends CI by automating the deployment of validated code to downstream environments. In a well-designed CD pipeline, every green CI run automatically deploys to a staging environment that mirrors production. Promotion to production is either fully automatic (Continuous Deployment) or requires a manual approval gate — the choice depends on organizational risk tolerance and compliance requirements. CD handles environment-specific concerns that CI does not: building with production environment variables, invalidating CDN caches, running smoke tests against the live deployment, and triggering rollback if health checks fail.

For a React application, the practical CI stages are: install dependencies → lint + typecheck → unit tests → build → bundle size check (using tools like bundlesize or size-limit). The CD stages add: push Docker image to registry → deploy to staging → run Playwright or Cypress end-to-end tests → await manual approval → deploy to production → invalidate CloudFront cache → verify health endpoints. Blue-green deployments and feature flags are two CD strategies that decouple deployment from release — you can deploy code to production without exposing it to users, then flip the flag when ready.

Trunk-based development (all engineers commit to main multiple times per day behind feature flags) produces the fastest CI/CD feedback loops and the fewest merge conflicts. GitFlow (long-lived feature branches merged via pull requests) is more common in teams with formal release cycles but introduces integration delays. For most React applications shipping to the web, trunk-based development with feature flags is the more mature approach.

The fundamental problem with naively Dockerizing a React application is that the Node.js build toolchain is enormous. A node_modules directory for a medium-sized React app frequently exceeds 300–500 MB, and that is before you account for the Node.js runtime itself, TypeScript compiler, Vite/webpack, and all associated plugins. If you package all of this into your production image, you are shipping hundreds of megabytes of build tools that will never execute at runtime — a significant security surface area and a slow deployment artifact.

Multi-stage builds solve this by using separate Docker stages for building and serving. The deps stage installs only production dependencies. The builder stage installs all dependencies (including devDependencies) and runs the build command, producing a dist/ folder of static files. The final production stage starts from nginx:alpine — a minimal ~23MB Linux image with Nginx — and uses COPY --from=builder /app/dist /usr/share/nginx/html to copy only the compiled static assets. The Node.js runtime, node_modules, TypeScript, and Vite never enter the production image at all. The result is typically a 20–30 MB production image versus 1.2 GB+ for the naive approach.

Layer caching order is critical for build performance. Because Docker invalidates a layer's cache whenever its inputs change, you should always copy package.json and package-lock.json first and run npm ci before copying your source files. This way, dependency installation (the slowest step) is cached and skipped on every build where you have only modified application code — which is the overwhelming majority of builds.

Beyond image size, the multi-stage pattern improves security posture. The production image contains only static files and the Nginx binary. There is no shell-accessible Node.js runtime, no npm command, and no package manager — dramatically reducing the attack surface if a container escape or misconfiguration occurs.

Build-time substitution is the approach used by default in Vite (VITE_*) and Create React App (REACT_APP_*). Environment variables are read by the bundler at build time and their values are inlined as string literals throughout the compiled JavaScript bundle. This approach is simple and requires no server-side logic. The trade-off is that every environment (development, staging, production) requires its own Docker image build — you cannot reuse a staging image for production because the API URL is literally baked into the JavaScript.

Runtime injection solves the one-image-per-environment problem. A docker-entrypoint.sh shell script runs when the container starts (before Nginx). It reads environment variables from the container runtime (set via docker run -e, Kubernetes env: or envFrom:, or AWS ECS task definition) and writes them to a small runtime-config.js file served as a static asset. Your React application loads this file via a <script> tag in index.html before the application bundle, making the values available on window.__ENV__. A typed wrapper module reads from window.__ENV__ with a fallback to Vite env vars for local development. This is the 12-Factor App methodology applied to frontend configuration.

The trade-offs are real. Build-time is simpler, easier to debug, and has zero runtime complexity — appropriate for small teams or applications with few environments. Runtime injection is the correct approach for organizations practicing proper Docker/Kubernetes hygiene, where a single immutable artifact should flow through all environments unchanged. Critically, neither approach should ever be used to store secrets — API keys, database credentials, or private tokens must never appear in a frontend bundle. All sensitive values belong exclusively in server-side infrastructure.

Content Security Policy is an HTTP response header — Content-Security-Policy — that declares to the browser which sources of content (scripts, styles, images, fonts, API connections) are trusted for a given page. When a browser receives a CSP header, it enforces the policy by refusing to load or execute any resource that does not match the declared allowlist. A strict CSP is the most powerful single defense against Cross-Site Scripting: even if an attacker succeeds in injecting a <script> tag into your HTML, the browser will refuse to execute it if it violates the policy.

For React applications, the most important directives are default-src 'self' (block everything not explicitly allowed), script-src (restrict script sources — ideally to 'self' plus specific CDN domains), and connect-src (restrict which origins your fetch/XMLHttpRequest calls can reach). The notorious challenge for React is style-src: many CSS-in-JS solutions and third-party libraries inject inline styles, which require 'unsafe-inline' to function. A better alternative is nonce-based CSP, where the server generates a cryptographic nonce per request, includes it in the CSP header, and adds it to each <style> tag — allowing legitimate inline styles while blocking injected ones.

The report-uri (or the modern report-to) directive allows you to deploy CSP in report-only mode initially: the browser enforces nothing but sends JSON violation reports to your endpoint. This lets you discover what your real-world application needs before switching to enforcing mode. Services like Report URI aggregate these reports into dashboards. Once in production, CSP violation reports are a real-time alert system for XSS attempts — legitimate users should never trigger them. CSP is defense-in-depth: it is not a replacement for proper input validation and output encoding, but it catches attacks that slip past the first line of defenses.

Core Web Vitals are Google's standardized set of user experience metrics that measure real-world page performance. The current three are: Largest Contentful Paint (LCP) — how long it takes for the largest visible element (usually a hero image or heading) to render, measuring perceived loading speed; Interaction to Next Paint (INP) — the latency from user interaction (click, tap, key press) to the next visual update, measuring interactivity; and Cumulative Layout Shift (CLS) — the total unexpected movement of page content during load, measuring visual stability. Google uses these metrics as a ranking signal, making them commercially significant beyond pure UX quality.

Improving LCP in a React application focuses on reducing the time to the first meaningful render. The highest-leverage techniques are: preloading the hero image with <link rel="preload" as="image">, using a CDN to serve assets from edge nodes geographically close to users, reducing server Time to First Byte (TTFB) with edge caching or server-side rendering for the initial HTML shell, and ensuring the above-the-fold content is not blocked by large JavaScript bundles. Code splitting with React.lazy and Suspense ensures that only the code needed for the initial view is downloaded before the first render. For critical API-dependent content, React 18's concurrent rendering with Suspense boundaries can keep the page interactive while data loads.

Improving CLS requires eliminating unexpected layout shifts. Always specify explicit width and height attributes on <img> elements so the browser reserves space before the image loads. Avoid inserting content above existing page content in response to user interactions or data loading — if you must show a notification or banner, reserve its space in the layout from the start. Use CSS transform and opacity for animations rather than properties like top, left, width, or height that trigger layout recalculation. In React, be particularly careful with conditional rendering of elements that affect layout — prefer hiding with visibility: hidden or opacity: 0 over conditionally mounting/unmounting.

For INP, React 18's concurrent features are specifically designed to help. useTransition marks expensive state updates as non-urgent, allowing React to interrupt them if a higher-priority user interaction arrives. Code splitting reduces JavaScript parse and execution time, which is the primary cause of slow INP on mobile. Virtualizing long lists eliminates the DOM node explosion that makes scroll events sluggish on data-heavy pages.

The core challenge for zero-downtime React SPA deployments is a cache coherence problem: when you deploy a new version, users who already have the old index.html cached in their browser will try to load the old JavaScript chunk filenames referenced in it. If you have deleted those old chunks from S3 (because the new build generated new content-addressed filenames), those requests return 404 errors, breaking the application for anyone in an active session at the moment of deployment.

The solution has four interlocking parts. First, content-addressed filenames: Vite and webpack both append a hash of the file contents to the filename by default (e.g., main.a3f9c2d1.js). This means the chunk files from the old and new builds have different names and can safely coexist in S3. Second, never cache index.html: set Cache-Control: no-store, no-cache specifically for index.html so browsers always fetch the freshest version. All other assets can be cached with Cache-Control: public, max-age=31536000, immutable since their content-addressed names guarantee freshness. Third, retain old chunks for 24–48 hours after deployment so users who were active at deploy time can complete their session without 404 errors. Fourth, after uploading the new build to S3, invalidate only /index.html in CloudFront — not the entire distribution — so new visitors get the new version immediately while old visitors are not broken.

For truly zero-downtime at the infrastructure level, a blue-green deployment switches the CloudFront origin between two S3 buckets (blue = current production, green = new version). You fully deploy and verify the green environment, then atomically update the CloudFront origin with a single API call. If post-deploy monitoring detects errors, you revert the CloudFront origin to blue — rollback takes seconds and requires no re-deployment. AWS CodeDeploy, GitHub Actions with AWS CLI, or a purpose-built pipeline can orchestrate this.

Feature flags decouple deployment from release — the highest-maturity zero-downtime strategy. You deploy new code to production behind a flag that is off for all users. QA verifies in production, then you flip the flag progressively: 1% of users, then 10%, then 100%. If metrics degrade, you flip the flag off instantly — no rollback, no redeployment. Tools like LaunchDarkly, Flagsmith, or a simple JSON endpoint suffice. This pattern means every deployment is inherently zero-risk from the user's perspective, because no new code path is executed until you deliberately enable it.

Deployment sequence for S3/CloudFront:

  1. Build new version locally or in CI (generates new content-addressed chunks)
  2. Upload all new chunk files to S3 (old chunks remain — do not delete yet)
  3. Upload new index.html to S3 (overwrites the old one)
  4. Invalidate /index.html in CloudFront (aws cloudfront create-invalidation --paths "/index.html")
  5. Monitor error rates for 30 minutes; if healthy, delete chunk files older than 48 hours
  6. If unhealthy, re-upload the previous index.html and invalidate again
