Redux Toolkit

RTK store, slices, async thunks, useSelector, useDispatch, and RTK Query

Redux Toolkit — Modern Redux

Redux Toolkit (RTK) is the official, opinionated toolset for Redux. It eliminates boilerplate by including Immer (immutable updates), createSlice, and createAsyncThunk out of the box.

RTK vs legacy Redux: RTK requires ~70% less code, uses Immer for "mutating" syntax in reducers, and ships createAsyncThunk instead of hand-rolled thunk middleware. 
// store/index.ts
import { configureStore } from '@reduxjs/toolkit';
import authReducer from './authSlice';
import cartReducer from './cartSlice';

export const store = configureStore({
  reducer: {
    auth: authReducer,
    cart: cartReducer,
  },
});

// TypeScript — export inferred types
export type RootState = ReturnType<typeof store.getState>;
export type AppDispatch = typeof store.dispatch;

// Typed hooks — use these everywhere instead of plain hooks
export const useAppSelector = useSelector.withTypes<RootState>();
export const useAppDispatch = useDispatch.withTypes<AppDispatch>();

Creating a Slice

A "slice" combines the reducer, action creators, and initial state for one domain in a single createSlice call. RTK uses Immer under the hood, so you can write direct "mutations" that are actually safe.

Immer magic: Inside reducers, you can write state.items.push(action.payload) — Immer intercepts this and produces a new immutable state. Never mutate state outside of RTK reducers. 
// store/cartSlice.js
import { createSlice } from '@reduxjs/toolkit';

const cartSlice = createSlice({
  name: 'cart',
  initialState: { items: [], total: 0 },
  reducers: {
    addItem(state, action) {
      // RTK uses Immer — "mutate" draft state directly!
      state.items.push(action.payload);
      state.total += action.payload.price;
    },
    removeItem(state, action) {
      const idx = state.items.findIndex(i => i.id === action.payload);
      if (idx !== -1) {
        state.total -= state.items[idx].price;
        state.items.splice(idx, 1);
      }
    },
    clearCart(state) {
      state.items = [];
      state.total = 0;
    },
  },
});

export const { addItem, removeItem, clearCart } = cartSlice.actions;
export default cartSlice.reducer;

Async Thunks

createAsyncThunk generates pending/fulfilled/rejected action types automatically and integrates cleanly with extraReducers.

// store/authSlice.js
import { createSlice, createAsyncThunk } from '@reduxjs/toolkit';

export const loginThunk = createAsyncThunk(
  'auth/login',
  async (credentials, { rejectWithValue }) => {
    try {
      const response = await api.post('/auth/login', credentials);
      localStorage.setItem('token', response.token);
      return response.user; // becomes action.payload in fulfilled
    } catch (error) {
      return rejectWithValue(error.response.data.message);
    }
  }
);

const authSlice = createSlice({
  name: 'auth',
  initialState: { user: null, loading: false, error: null },
  reducers: {
    logout(state) {
      state.user = null;
      localStorage.removeItem('token');
    }
  },
  extraReducers: (builder) => {
    builder
      .addCase(loginThunk.pending, (state) => {
        state.loading = true; state.error = null;
      })
      .addCase(loginThunk.fulfilled, (state, action) => {
        state.loading = false; state.user = action.payload;
      })
      .addCase(loginThunk.rejected, (state, action) => {
        state.loading = false; state.error = action.payload;
      });
  },
});

export const { logout } = authSlice.actions;
export default authSlice.reducer;

Using Redux in Components

Connect components to the store with useSelector (read state) and useDispatch (send actions). Keep selectors close to the component or colocate them in the slice file.

import { useSelector, useDispatch } from 'react-redux';
import { addItem } from '../store/cartSlice';

function ProductCard({ product }) {
  const dispatch = useDispatch();
  const cartItems = useSelector(state => state.cart.items);
  const isInCart = cartItems.some(i => i.id === product.id);

  return (
    <div className="card">
      <h3>{product.name}</h3>
      <p>${product.price}</p>
      <button
        onClick={() => dispatch(addItem(product))}
        disabled={isInCart}
      >
        {isInCart ? 'In Cart' : 'Add to Cart'}
      </button>
    </div>
  );
}

// Colocate selectors in the slice — recommended pattern
// In cartSlice.js:
export const selectCartItems = (state) => state.cart.items;
export const selectCartTotal = (state) => state.cart.total;
export const selectIsInCart = (id) => (state) => state.cart.items.some(i => i.id === id);

// Usage — cleaner and reusable
const isInCart = useSelector(selectIsInCart(product.id));

RTK Query — Data Fetching

RTK Query is a powerful data fetching and caching tool built into Redux Toolkit. It auto-generates hooks, handles caching, background refetching, and cache invalidation with tags.

RTK Query vs React Query: Both are excellent. RTK Query integrates directly with your Redux store (state lives in Redux, DevTools support) and is the best choice if you're already using Redux. React Query is lighter and framework-agnostic. 
// api/productsApi.js
import { createApi, fetchBaseQuery } from '@reduxjs/toolkit/query/react';

export const productsApi = createApi({
  reducerPath: 'productsApi',
  baseQuery: fetchBaseQuery({ baseUrl: '/api/' }),
  tagTypes: ['Products'],
  endpoints: (builder) => ({
    getProducts: builder.query({
      query: () => 'products',
      providesTags: ['Products'],
    }),
    getProductById: builder.query({
      query: (id) => `products/${id}`,
    }),
    createProduct: builder.mutation({
      query: (body) => ({ url: 'products', method: 'POST', body }),
      invalidatesTags: ['Products'], // auto-refetches getProducts
    }),
    deleteProduct: builder.mutation({
      query: (id) => ({ url: `products/${id}`, method: 'DELETE' }),
      invalidatesTags: ['Products'],
    }),
  }),
});

export const {
  useGetProductsQuery,
  useGetProductByIdQuery,
  useCreateProductMutation,
  useDeleteProductMutation,
} = productsApi;

// Usage — auto-generated hooks handle all loading/error states
function Products() {
  const { data, isLoading, error } = useGetProductsQuery();
  const [createProduct, { isLoading: isCreating }] = useCreateProductMutation();

  if (isLoading) return <Spinner />;
  if (error) return <Error />;
  return <ProductGrid products={data} onCreate={createProduct} />;
}

Redux vs Context Comparison

Dimension Redux Toolkit Context + useReducer
Performance Selective re-renders via selectors All consumers re-render
DevTools Redux DevTools (time-travel, action replay) React DevTools only
Async createAsyncThunk, RTK Query Manual useEffect / custom hooks
Bundle size ~11 KB (RTK) 0 KB (built-in)
Middleware Thunks, sagas, listeners Not supported
Best for Complex app-wide state, teams, large apps Simple global state, small apps
Decision guide: Start with Context for low-frequency global state (auth, theme). Add Redux when you need: cross-cutting state accessed by many components, complex async flows, time-travel debugging, or the team grows beyond 3 developers and shared state conventions matter.

Interview Questions

What are the three core principles of Redux?
1. Single source of truth — the entire app state lives in one store. 2. State is read-only — the only way to change state is to dispatch an action. 3. Changes are made with pure functions — reducers must be pure (same input → same output, no side effects).
How does Immer work inside RTK reducers?
RTK wraps each reducer with Immer's produce. Immer creates a "draft" proxy of the state. You can write mutations on the draft (like state.items.push(x)). When the reducer returns, Immer compares the draft to the original and produces a new immutable state object — your mutations never actually modify the original.
What is the difference between a selector and a reducer?
A reducer is a pure function that takes the current state and an action and returns a new state — it writes state. A selector is a pure function that takes the Redux state and returns a derived value — it reads state. Colocating selectors in slice files keeps reading and writing logic together.
Why use rejectWithValue in createAsyncThunk?
Without rejectWithValue, when you throw an error in a thunk, action.payload is undefined and the serialized error goes to action.error. Using rejectWithValue(message) sets action.payload to your custom error value in the rejected case, making error handling in extraReducers clean and predictable.
What are RTK Query tags and how does cache invalidation work?
Tags are labels you assign to query results (providesTags) and to mutations (invalidatesTags). When a mutation with invalidatesTags: ['Products'] succeeds, RTK Query automatically marks all cached queries with providesTags: ['Products'] as stale and triggers a background refetch — without any manual queryClient.invalidateQueries calls.
What is memoized selector (createSelector) and why is it needed?
createSelector (from Reselect, included in RTK) creates a memoized selector. If the input selectors return the same values as the previous call, the output function is not re-run — the cached result is returned. This prevents derived data computations (e.g., filtering a list) from running on every render when the relevant state hasn't changed.
How do you handle authentication state with Redux?
Create an authSlice with user, loading, and error state. Use createAsyncThunk for login/logout API calls. On app startup, dispatch a checkAuth thunk that reads the stored token and validates it. Components subscribe to selectCurrentUser and redirect via React Router's Navigate when authentication state changes.
When would you use RTK Query instead of createAsyncThunk?
Use RTK Query for standard CRUD operations against a REST or GraphQL API — it auto-generates hooks, handles caching, deduplication, polling, and cache invalidation without manual boilerplate. Use createAsyncThunk for non-data-fetching async logic: complex business workflows, multi-step processes, or operations that must coordinate multiple slices' state.
What is the Redux data flow?
Unidirectional: 1. Component calls dispatch(action). 2. The Redux store passes (currentState, action) to the root reducer. 3. Reducers return new state. 4. The store notifies all useSelector subscribers. 5. Components whose selected value changed re-render. This single direction makes state changes traceable and predictable.
How does useSelector prevent unnecessary re-renders?
After each dispatched action, useSelector runs the selector and compares the result to the previous result using strict equality (===). If the result is the same reference, the component does not re-render. For derived objects, use createSelector to ensure the same reference is returned when inputs haven't changed.
