React is a JavaScript library created by Meta for building composable, component-based user interfaces using a Virtual DOM for efficient updates.
  • Component-based: UI is split into isolated, reusable pieces called components.
  • Virtual DOM: React keeps an in-memory copy of the DOM and computes minimal real-DOM changes.
  • Declarative: You describe what the UI should look like; React handles the how.
  • Unidirectional data flow: Data flows from parent to child via props, making apps predictable.
  • Library, not framework: React handles only the view layer; routing and state management are separate concerns.
Why interviewers ask this: Gauges baseline understanding and ability to articulate React's core value proposition versus a full framework like Angular.

JSX (JavaScript XML) is a syntax extension that lets you write HTML-like markup directly inside JavaScript. Babel compiles it into React.createElement() calls at build time.
  • Not HTML: JSX uses className instead of class, htmlFor instead of for, and camelCase event names.
  • Expressions in {}: Any valid JS expression can be embedded inside curly braces.
  • Single root element: JSX must return one root element (or a Fragment).
  • Optional: You can write React without JSX using createElement() directly, but JSX is far more readable.
// JSX
const el = <h1>Hello</h1>;

// Compiled output (what Babel produces)
const el = React.createElement('h1', null, 'Hello');
Why interviewers ask this: Confirms you understand that JSX is syntactic sugar, not a separate template language, and that you know its compilation step.

The Virtual DOM is a lightweight, in-memory JavaScript representation of the real DOM tree. React uses it to compute the minimal set of actual DOM mutations needed after a state change.
  • Step 1 — Render: When state/props change, React creates a new Virtual DOM tree.
  • Step 2 — Diff: React compares (diffs) the new tree against the previous snapshot using its reconciliation algorithm.
  • Step 3 — Commit: Only the changed nodes are updated in the real DOM, keeping DOM mutations minimal and fast.
  • Why it matters: Real DOM operations are expensive; batching and minimizing them via Virtual DOM improves performance.
Why interviewers ask this: Tests understanding of React's performance model and why it's faster than naive direct DOM manipulation.

A React component is a reusable, self-contained UI building block — a JavaScript function (or class) that accepts props as input and returns JSX describing what to render.
  • Functional component: A plain JS function. function Button(props) { return <button>{props.label}</button>; }
  • Class component: Extends React.Component and implements a render() method.
  • Composable: Components can nest other components, building complex UIs from simple parts.
  • Naming: Component names must start with an uppercase letter so React distinguishes them from native HTML tags.
function Greeting({ name }) {
  return <h1>Hello, {name}!</h1>;
}

// Usage
<Greeting name="World" />
Why interviewers ask this: Fundamental building block knowledge. Interviewers want to hear "reusable", "props", and "returns JSX".

Functional components are plain JavaScript functions that use hooks for state and lifecycle. Class components extend React.Component and use this.state and lifecycle methods. Functional components are the modern standard.
  • Functional: Less boilerplate, easier to read and test, use hooks (useState, useEffect, etc.).
  • Class: Requires extending React.Component, uses this.state and this.setState(), lifecycle methods like componentDidMount.
  • Performance: Both are similar in performance; React.memo is the functional equivalent of PureComponent.
  • Modern standard: Since React 16.8 (2019), hooks replaced the need for class components. All new code should use functional components.
  • No this: Functional components avoid the confusing JavaScript this binding issues common in class components.
Why interviewers ask this: Checks whether you understand React's evolution and would choose the right component type in a modern codebase.

Props (short for properties) are read-only inputs passed from a parent component to a child component. They are immutable from the child's perspective and are used for configuration, data passing, and event callbacks.
  • Read-only: A child component must never mutate its own props.
  • Any type: Props can be strings, numbers, objects, arrays, functions, or even other React elements.
  • Data flow: Props enable the unidirectional (top-down) data flow React is built on.
  • Destructuring: function Button({ label, onClick }) is idiomatic for cleaner prop access.
// Parent passes props
<Button label="Submit" onClick={handleSubmit} />

// Child reads props (read-only)
function Button({ label, onClick }) {
  return <button onClick={onClick}>{label}</button>;
}
Why interviewers ask this: Foundational concept — interviewers check that you know props are immutable and understand the parent-child data relationship.

State is mutable data managed inside a component. When state changes, React schedules a re-render of that component and its descendants. useState is the hook used to add local state to functional components.
  • Local: State belongs to and is managed by its component — not accessible from outside unless passed as props.
  • Mutable (via setter): You never mutate state directly; you call the setter function returned by useState.
  • Re-renders: Every call to a state setter triggers a re-render if the new value differs from the current one.
  • Async updates: State updates may be batched by React for performance — don't rely on new state being available immediately after the setter call.
import { useState } from 'react';

function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}
Why interviewers ask this: State is the engine of React interactivity. Interviewers check you understand immutability, the setter pattern, and re-render triggering.

Props come from outside (parent), are read-only, and configure the component. State lives inside the component, is mutable via a setter, and triggers re-renders when changed.
  • Props — source: Passed by a parent component. The child cannot change them.
  • Props — analogy: Like function arguments — you receive them, you don't own them.
  • State — source: Declared and managed within the component itself.
  • State — analogy: Like local variables inside a function, but ones that persist across renders.
  • Re-render trigger: Both prop changes and state changes cause React to re-render the affected component.
Why interviewers ask this: A classic React question that tests whether you can clearly articulate ownership and mutability of data in a component tree.

useState is a React hook that adds local state to a functional component. It returns a pair: the current state value and a setter function. The initial value is only used on the first render.
  • Signature: const [value, setValue] = useState(initialValue);
  • Initial value: Can be a primitive, object, array, or a lazy initializer function for expensive computations.
  • Setter: Calling the setter with a new value schedules a re-render. Passing a function gives you the previous state: setCount(prev => prev + 1).
  • Functional updater: Always use the functional form when new state depends on previous state to avoid stale closure bugs.
const [name, setName] = useState('');

// Functional updater — safe with async batching
const increment = () => setCount(prev => prev + 1);
Why interviewers ask this: useState is the most used hook — interviewers check syntax knowledge and awareness of the functional updater pattern.

useEffect is a hook for running side effects (data fetching, subscriptions, DOM mutations, timers) after the component renders. An optional cleanup function handles teardown. The dependency array controls when the effect re-runs.
  • No deps array: Runs after every render.
  • Empty array []: Runs only once after the initial mount (like componentDidMount).
  • With deps [a, b]: Runs when a or b changes.
  • Cleanup: Return a function to clean up subscriptions/timers — runs before the next effect and on unmount.
useEffect(() => {
  const id = setInterval(() => tick(), 1000);
  return () => clearInterval(id); // cleanup
}, []); // empty array → run once on mount
Why interviewers ask this: useEffect is essential and commonly misused. Interviewers probe knowledge of the dependency array and cleanup patterns.

The key prop is a special string attribute that helps React identify which items in a list have changed, been added, or been removed during reconciliation. Keys must be stable and unique among siblings.
  • Purpose: Enables React to match elements across renders to avoid unnecessary re-creation.
  • Stable keys: Use unique IDs from your data (e.g., database IDs), not array indices.
  • Index as key: Using index causes bugs when items reorder or are inserted — React may mis-match elements and corrupt state.
  • Sibling-scope: Keys only need to be unique among siblings, not globally across the whole tree.
items.map(item => (
  // Good: stable unique ID
  <Item key={item.id} data={item} />
))

// Bad: index as key — causes bugs on reorder
items.map((item, i) => <Item key={i} />)
Why interviewers ask this: A common interview trap — they want to hear why index keys are problematic and see that you understand reconciliation.

React.Fragment is a wrapper component that lets you group multiple elements without adding an extra DOM node. Use the <></> shorthand or <React.Fragment> when you need to pass a key prop.
  • Problem it solves: JSX must return a single root element. Wrapping in a <div> adds unnecessary DOM nodes that can break CSS layouts (flex, grid).
  • Shorthand: <>...</> — most common form but cannot accept props.
  • Long form: <React.Fragment key={id}> — needed when using key in list rendering.
// No extra DOM node added
return (
  <>
    <h1>Title</h1>
    <p>Paragraph</p>
  </>
);
Why interviewers ask this: Tests awareness of DOM structure concerns and the Fragment pattern for clean HTML output.

React uses camelCase synthetic event names (onClick, onChange, onSubmit) on JSX elements. You pass a function reference as the handler. React's Synthetic Event system normalizes browser differences.
  • camelCase names: onClick (not onclick), onChange (not onchange), onKeyDown, etc.
  • Function reference: Pass the function itself — onClick={handleClick} — not a call result.
  • Synthetic events: React wraps native events in a SyntheticEvent that works consistently cross-browser.
  • preventDefault: Call e.preventDefault() inside the handler (React does not return false like vanilla HTML).
  • Event pooling (legacy): In React 17+ events are no longer pooled — accessing event properties asynchronously is safe.
function Form() {
  const handleSubmit = (e) => {
    e.preventDefault();
    console.log('submitted');
  };
  return <form onSubmit={handleSubmit}>...</form>;
}
Why interviewers ask this: Event handling is everyday React. They look for camelCase awareness, function references vs calls, and preventDefault usage.

Conditional rendering means showing different UI elements based on a condition. React supports multiple patterns: the ternary operator, logical &&, if/else before the return, and the nullish coalescing operator.
  • Ternary: {isLoggedIn ? <Dashboard /> : <Login />} — best for two alternatives.
  • Logical &&: {isOpen && <Modal />} — render or nothing. Caution: if condition is 0, React renders "0".
  • if/else: Outside JSX in the function body — good for complex conditions.
  • null: Return null from a component to render nothing without removing the component from the tree.
// Ternary — two outcomes
{user ? <Profile /> : <Login />}

// Logical AND — show or nothing
{hasError && <ErrorBanner />}
Why interviewers ask this: Checks that you know all common patterns and the subtle "0 renders" gotcha with logical &&.

Use Array.map() to transform a data array into an array of JSX elements. Each element must have a unique key prop so React can efficiently reconcile the list on updates.
  • map(): Returns a new array of JSX — it's a pure transformation, no side effects.
  • key prop: Required on the root JSX element returned from the map callback. Use stable IDs, not indices.
  • filter + map: Chain filter() before map() to render only a subset of items.
  • Empty state: Always handle the empty array case to avoid rendering nothing with no explanation.
const users = [{ id: 1, name: 'Alice' }, { id: 2, name: 'Bob' }];

return (
  <ul>
    {users.map(user => (
      <li key={user.id}>{user.name}</li>
    ))}
  </ul>
);
Why interviewers ask this: List rendering is ubiquitous. They check for the map() pattern and the key prop requirement together.

Prop drilling is the practice of passing props through multiple intermediate components that don't use them, just to deliver data to a deeply nested child. It makes components tightly coupled and harder to refactor.
  • Problem: Intermediate components become aware of data they don't need, increasing coupling.
  • Refactoring pain: Changing the data shape requires updating every component in the chain.
  • Solution 1: React Context API — broadcast data to any descendant without threading props.
  • Solution 2: State management libraries (Redux, Zustand, Jotai) — store state outside the component tree.
  • Solution 3: Component composition — pass children or render props to skip intermediate layers.
Why interviewers ask this: Gauges your ability to identify architectural problems and reach for appropriate solutions (Context, Redux, composition).

Lifting state up means moving shared state to the nearest common ancestor of the components that need it. The ancestor owns the state, passes it down as props, and provides update callbacks so children can request changes.
  • When to use: Two sibling components need to read or update the same piece of data.
  • Pattern: Parent holds state + handler → passes both down to children as props.
  • Single source of truth: One component controls the state, preventing sync issues between siblings.
  • Trade-off: Can lead to prop drilling if the ancestor is far up the tree — then Context may be more appropriate.
function Parent() {
  const [value, setValue] = useState('');
  return <>
    <InputA value={value} onChange={setValue} />
    <Preview value={value} />
  </>;
}
Why interviewers ask this: Tests understanding of React's unidirectional data flow and the go-to pattern for sibling communication.

A controlled component is a form element whose value is driven by React state. The input's value prop is bound to state and an onChange handler updates state on every keystroke — React is the single source of truth.
  • Pattern: value={state} + onChange={setState} on the input.
  • Benefits: Instant validation, easy conditional disabling, formatting on the fly.
  • Read-only inputs: If you set value without onChange, React will warn that it's read-only. Add onChange or use defaultValue.
  • Textarea and select: Also use value prop in React (unlike plain HTML where textarea uses children).
const [email, setEmail] = useState('');

<input
  type="email"
  value={email}
  onChange={e => setEmail(e.target.value)}
/>
Why interviewers ask this: Core form handling pattern — they want to hear "single source of truth" and understand the value+onChange binding.

An uncontrolled component lets the DOM maintain its own internal state. You access the input's current value by attaching a ref rather than binding it to React state via onChange.
  • useRef: Create a ref with useRef() and attach it to the input's ref prop.
  • On submit: Read the value from inputRef.current.value when needed (e.g., form submit).
  • defaultValue: Use defaultValue (not value) for initial values — setting value without onChange makes it a controlled component.
  • When to use: File inputs (always uncontrolled), simple forms where real-time validation isn't needed, integrating with non-React libraries.
const inputRef = useRef();

const handleSubmit = () => {
  console.log(inputRef.current.value);
};

<input ref={inputRef} defaultValue="initial" />
Why interviewers ask this: Paired with controlled component question — they check you know the distinction and when each is appropriate.

React.memo is a Higher Order Component (HOC) that memoizes a functional component. It skips re-rendering the wrapped component if its props have not changed (shallow equality comparison).
  • Use case: Expensive pure components that receive the same props frequently from a re-rendering parent.
  • Shallow comparison: By default, React.memo does a shallow prop comparison. Pass a custom comparator as the second argument for deep comparison.
  • Not always beneficial: Memoization has overhead — avoid it for cheap components or those that almost always re-render.
  • Pairing: Combine with useCallback for function props and useMemo for object/array props to prevent unnecessary re-renders.
const ExpensiveList = React.memo({ items }) => (
  <ul>{items.map(i => <li key={i.id}>{i.name}</li>)}</ul>
));
Why interviewers ask this: Checks performance optimization awareness and understanding of when memoization helps vs hurts.

Default props provide fallback values for props when none are provided by the parent. In functional components, use ES6 default parameter syntax directly in the function signature.
  • ES6 defaults (recommended): function Button({ label = 'Click me', size = 'md' })
  • Component.defaultProps (legacy): A static property on the component — works but is considered legacy for functional components.
  • Class components: Use static defaultProps = {...} inside the class body.
  • TypeScript: Default parameters serve the same role — no need for defaultProps in TS functional components.
// Modern: ES6 default parameters
function Badge({ color = 'blue', text = 'New' }) {
  return <span style={{ background: color }}>{text}</span>;
}
Why interviewers ask this: Checks whether you use modern ES6 defaults over legacy defaultProps, and that you handle missing props gracefully.

The children prop is a special built-in prop that is automatically populated with whatever JSX (elements, text, or components) is placed between a component's opening and closing tags.
  • Automatic: React passes children without you needing to explicitly forward them in the parent.
  • Any content: Can be a string, a React element, an array of elements, or a function (render prop pattern).
  • Layout components: The children prop is essential for wrapper/layout components like Card, Modal, Panel.
  • React.Children: Use React.Children utilities (map, count, forEach) when you need to introspect or transform children.
function Card({ children }) {
  return <div className="card">{children}</div>;
}

// Usage
<Card><h2>Title</h2><p>Body text</p></Card>
Why interviewers ask this: children is fundamental to composable component design — interviewers check that you understand slot-style composition.

A Single Page Application loads one HTML document and then dynamically updates the content via JavaScript as the user navigates — no full page reloads. React is ideally suited for building SPAs.
  • Navigation: URL changes are handled by a client-side router (React Router) — the browser never fully refreshes.
  • Benefits: Faster perceived navigation, smoother UX, no full-page flash between routes.
  • Drawbacks: Larger initial JS bundle, SEO challenges (mitigated with SSR/SSG via Next.js), requires JavaScript.
  • vs MPA: Multi-page applications request a fresh HTML page from the server for each route — simpler but slower transitions.
Why interviewers ask this: Establishes that you understand the architectural context React apps typically operate in and the trade-offs involved.

React is a UI library focused on the view layer with a flexible ecosystem. Angular is a full MVC framework with opinionated structure, two-way data binding, and built-in solutions for routing, forms, and HTTP.
  • React: Library (view only) — you choose your router, state management, HTTP client.
  • Angular: Full framework — everything is included and opinionated out of the box.
  • Data flow: React uses one-way data flow; Angular uses two-way data binding (ngModel).
  • Language: React uses JSX + JavaScript/TypeScript; Angular is TypeScript-first.
  • Learning curve: Angular has a steeper curve due to decorators, DI, modules. React is simpler to start.
  • DOM: React uses Virtual DOM; Angular uses Incremental DOM (Ivy).
Why interviewers ask this: Checks breadth of framework knowledge and your ability to articulate trade-offs in technology choices.

Babel is a JavaScript transpiler that converts modern JavaScript (ES2015+) and JSX syntax into backward-compatible code that older browsers can execute. React projects rely on Babel to process JSX before the browser sees it.
  • JSX transformation: Converts <MyComponent /> into React.createElement() calls (or the new JSX transform's _jsx()).
  • ES2015+ downleveling: Transforms arrow functions, classes, destructuring, optional chaining for older environments.
  • Presets: @babel/preset-react handles JSX; @babel/preset-env handles modern JS syntax targeting.
  • Modern alternatives: Vite uses esbuild (faster) for transpilation; SWC (Rust-based) is another fast alternative used by Next.js.
Why interviewers ask this: Tests understanding of the build toolchain that React depends on — separating compilation from bundling concerns.

Webpack is a static module bundler that processes your JavaScript, CSS, images, and other assets and bundles them into optimized output files for the browser. Create React App uses Webpack under the hood.
  • Entry point: Starts with your index.js and traces all imports to build a dependency graph.
  • Loaders: Transform files before bundling (babel-loader for JS/JSX, css-loader for CSS, file-loader for images).
  • Plugins: Extend Webpack's functionality (HtmlWebpackPlugin, MiniCssExtractPlugin, etc.).
  • Code splitting: Breaks the bundle into chunks loaded on demand to improve initial load performance.
  • Modern alternatives: Vite (esbuild + Rollup), Parcel, Turbopack — all faster dev experiences than Webpack.
Why interviewers ask this: Build tool knowledge signals seniority — they want to know you understand what happens between your code and the browser.

The public/ folder contains static assets that are served as-is without being processed by Webpack/Vite. The root index.html lives here along with favicons, robots.txt, and other static files.
  • index.html: The HTML shell with the <div id="root"> mount point and script tags. React injects itself here.
  • Not processed: Files in public/ are copied directly to the build output — no hashing, no Babel, no CSS processing.
  • Accessed at root: A file at public/logo.png is accessible at /logo.png in the browser.
  • vs src/assets: Files imported in JS via src/ are processed, optimized, and hash-named. Public/ files are not.
Why interviewers ask this: Clarifies that you understand the difference between processed source assets and static public assets in a React build setup.

The root element is the single DOM node — typically <div id="root"> in index.html — where React mounts and controls the entire component tree via ReactDOM.createRoot().
  • Mount point: React takes full control of everything inside the root element.
  • React 18: Use ReactDOM.createRoot(document.getElementById('root')) which enables concurrent features.
  • Legacy React 17: Used ReactDOM.render() — deprecated in React 18.
  • Multiple roots: You can have multiple root elements in the same page, useful when incrementally adopting React in a non-React app.
// React 18 — index.tsx
import { createRoot } from 'react-dom/client';
const root = createRoot(document.getElementById('root'));
root.render(<App />);
Why interviewers ask this: Fundamental React bootstrap knowledge — they check that you know React 18's createRoot API.

ReactDOM is the package responsible for rendering React elements to the browser DOM. It bridges the platform-agnostic React core library with the browser environment. ReactDOM.createRoot() is the React 18 entry point.
  • react vs react-dom: react contains the component model and reconciler; react-dom is the browser renderer. This separation enables React Native, React Native for Web, etc.
  • createRoot: React 18 API that enables concurrent rendering features.
  • createPortal: ReactDOM.createPortal() renders children into a DOM node outside the parent component hierarchy (e.g., modals, tooltips).
  • flushSync: Forces React to flush state updates synchronously — useful in rare edge cases.
Why interviewers ask this: Confirms you understand the separation between React's rendering logic and the platform-specific DOM bridge.

JSX is syntactic sugar for React.createElement(). They produce identical React element objects — JSX is simply a more readable syntax that Babel compiles into createElement calls at build time.
  • JSX: <Button color="blue">Click</Button>
  • Compiled: React.createElement(Button, { color: "blue" }, "Click")
  • createElement signature: (type, props, ...children) — type is a string tag or component reference.
  • New JSX transform: React 17+ introduced a new JSX transform that imports _jsx from 'react/jsx-runtime' instead of React.createElement — no need to import React in every file.
// These are exactly equivalent
const a = <h1 className="title">Hello</h1>;
const b = React.createElement('h1', { className: 'title' }, 'Hello');
Why interviewers ask this: Checks depth of understanding — that JSX is pure syntax sugar and you can explain the compilation step.

A Higher Order Component is a function that takes a component as an argument and returns a new enhanced component. It's a pattern for reusing component logic across multiple components (cross-cutting concerns).
  • Pattern: const Enhanced = withFeature(WrappedComponent);
  • Use cases: Authentication guards, logging, theming, data fetching wrappers, analytics tracking.
  • Naming convention: HOC names typically start with "with" (withAuth, withTheme, withRouter).
  • Modern alternatives: Custom hooks have largely replaced HOCs in modern React because they compose without nesting extra components in the tree.
  • Caveat: HOCs can create "wrapper hell" — deep nesting of HOC-wrapped components in DevTools.
function withAuth(Component) {
  return function AuthWrapper(props) {
    if (!isAuthenticated) return <Redirect to="/login" />;
    return <Component {...props} />;
  };
}
Why interviewers ask this: Classic React pattern question — they also want to see if you mention custom hooks as the modern replacement.

Event delegation is the technique of attaching a single event listener at a high-level ancestor rather than on every individual element. React uses this pattern internally — it attaches one listener at the root container and routes events to the correct handlers.
  • React 17+ change: Events are now attached to the React root DOM container (not document), making it safer to embed multiple React versions.
  • Performance benefit: One listener for thousands of list items instead of one listener per item.
  • Bubbling: Events bubble up from the target to the root where React intercepts them.
  • Synthetic events: React wraps the native event in a SyntheticEvent with a consistent cross-browser API.
Why interviewers ask this: Tests deeper event system knowledge — not just "how do onClick handlers work" but how React implements them under the hood.

The render() method in a class component returns the JSX (or null) that describes what should be displayed. It is called by React whenever state or props change and must be a pure function — no side effects.
  • Required: The only required method in a class component.
  • Pure: render() must not modify state, call setState(), or produce side effects — those belong in lifecycle methods.
  • Can return: JSX, arrays, strings, numbers, portals, booleans, or null.
  • Equivalent in functional: The entire function body of a functional component is effectively the render — it returns JSX directly.
class Greeting extends React.Component {
  render() {
    return <h1>Hello, {this.props.name}</h1>;
  }
}
Why interviewers ask this: Tests class component fundamentals and the key constraint that render() must be pure (no side effects).

Reconciliation is React's process of comparing the new Virtual DOM tree with the previous one (diffing) to determine the minimal set of actual DOM operations needed to bring the UI up to date.
  • Element type check: If the type changes (div → span), React tears down the old tree and builds a new one.
  • Same type: React updates only the changed attributes/props on the existing DOM node.
  • Keys: Keys help React match list items across renders, avoiding unnecessary element recreation.
  • Fiber: React 16+ uses the Fiber architecture — a re-implementation of the reconciler that enables interruptible, prioritized rendering (concurrent mode).
  • Heuristic: React's O(n) diffing algorithm makes assumptions (same-level comparison, keys for lists) to avoid the O(n³) cost of general tree diffing.
Why interviewers ask this: Deep knowledge question — interviewers want to hear "Virtual DOM diff", "minimal DOM mutations", and ideally "Fiber" for bonus points.

StrictMode is a development-only tool that highlights potential problems in an application. It wraps part of the component tree and runs extra checks and warnings without affecting the production build.
  • Double-invocation: In development, React intentionally calls render, useState, and effects twice to detect unexpected side effects.
  • Deprecated API warnings: Warns about deprecated lifecycle methods and legacy string refs.
  • State preservation: In React 18, StrictMode simulates component unmounting/remounting while preserving state — future Offscreen API preview.
  • No production impact: StrictMode is completely stripped from production builds.
import { StrictMode } from 'react';

root.render(
  <StrictMode>
    <App />
  </StrictMode>
);
Why interviewers ask this: Checks that you know why effects run twice in development and that it's intentional, not a bug.

Yes. JSX is optional syntactic sugar. You can write React apps entirely using React.createElement() calls. JSX simply makes the code more readable — both approaches produce identical React element objects.
  • Without Babel: If you don't use JSX you don't need a build step for JSX transformation — you can use React directly in a <script> tag.
  • Readability: For deeply nested UI, JSX is vastly more readable than chained createElement() calls.
  • When useful: Environments where a build step is unavailable (old prototypes, no-build setups with import maps).
  • New JSX transform: React 17+ also supports a new JSX transform via the automatic runtime — still JSX, just without needing to import React.
// JSX-free React
const el = React.createElement(
  'div',
  { className: 'container' },
  React.createElement('h1', null, 'Hello')
);
Why interviewers ask this: Confirms JSX understanding — that it's not a separate language but a convenience transformation over createElement.

Returning null from a component renders nothing to the DOM without errors — the component stays in the tree but contributes no visual output. Returning undefined throws an error in React — always return null explicitly if you want to render nothing.
  • null: Valid return value — renders nothing, component stays mounted (lifecycle/state preserved).
  • undefined: Invalid — React throws "Nothing was returned from render" error.
  • false and null: Both are valid in JSX expressions and render nothing — used for conditional rendering.
  • 0 caution: The number 0 is falsy but JSX renders it as the text "0". Use explicit null: {count > 0 ? <Badge /> : null}
Why interviewers ask this: The 0-renders bug and null vs undefined distinction are common gotchas that reveal attention to detail.

Pass the function by reference in JSX: <Child onClick={handleClick} />. The child calls props.onClick() when needed. This is the primary pattern for child-to-parent communication in React.
  • Callback pattern: Parent defines the function, passes it as a prop, child invokes it — enabling upward communication.
  • No parentheses: onClick={handleClick} passes a reference. onClick={handleClick()} immediately invokes it during render — a common mistake.
  • Arguments: Wrap in an arrow function to pass arguments: onClick={() => handleClick(item.id)}
  • useCallback: Wrap the function in useCallback if passing to a memoized child to prevent unnecessary re-renders.
function Parent() {
  const handleDelete = (id) => removeItem(id);
  return <Child onDelete={handleDelete} />;
}

function Child({ onDelete }) {
  return <button onClick={() => onDelete(42)}>Delete</button>;
}
Why interviewers ask this: Tests callback prop pattern knowledge — the foundation of child-to-parent communication in React.

onClick={handleClick} passes a stable reference to the same function. onClick={() => handleClick()} creates a brand-new arrow function on every render, which can cause unnecessary re-renders in memoized children.
  • Reference equality: React.memo and useCallback compare props by reference — a new function object each render breaks memoization.
  • When to use arrow: Use the arrow function form when you need to pass arguments: onClick={() => handleClick(item.id)}
  • useCallback: Stabilize the arrow function form with useCallback to fix the re-render issue when passing to memoized children.
  • Performance impact: For non-memoized leaf components the difference is negligible — only matters with React.memo or large lists.
Why interviewers ask this: A nuanced performance trap — interviewers check that you understand reference equality and its interaction with memoization.

  • Custom hooks: Functions named with "use" prefix that call hooks — they follow the same rules and allow reusable stateful logic.

Debounce delays execution until after a pause in calls. In React, the safest approach is a custom useDebounce hook that uses useEffect and setTimeout to debounce a value.

  • Debouncing search inputs reduces API calls significantly on fast typists
  • Use useCallback with lodash.debounce if debouncing the handler; wrap in useRef to keep stable reference
  • A value-based useDebounce hook is cleaner for most use cases
  • Always clean up the timeout in the useEffect return to prevent memory leaks
function useDebounce(value, delay) {
  const [debounced, setDebounced] = useState(value);
  useEffect(() => {
    const id = setTimeout(() => setDebounced(value), delay);
    return () => clearTimeout(id);
  }, [value, delay]);
  return debounced;
}

Fiber is a linked-list-based internal architecture where each component maps to a fiber node (a plain JS object). React maintains two trees simultaneously — current (rendered) and workInProgress (being built) — enabling incremental, interruptible rendering.

  • Commit phase: After reconciliation, effects (mutations, layout effects, passive effects) are applied in a single synchronous pass.
  • Incremental rendering: Work is broken into units; React yields to the browser every ~5ms to keep the frame budget, then resumes.

  return: parentFiber,   // parent
  pendingProps: {},

  memoizedState: {},     // hooks linked list lives here

React uses a heuristic O(n) diffing algorithm instead of the theoretically optimal O(n³) tree diff. It relies on two assumptions: different element types produce entirely different trees; and keys identify stable children across renders.

  • Keys for lists: Keys allow React to match children by identity across re-renders, avoiding unnecessary unmounts.

The React Scheduler assigns work to priority lanes (bit flags) and uses a MessageChannel (not setTimeout) for async scheduling, yielding to the browser every ~5ms to maintain 60fps.
  • Lanes (bit flags): SyncLane (highest, 1), InputContinuousLane (pointer events), DefaultLane (normal updates), TransitionLane (startTransition), IdleLane (lowest).
  • MessageChannel: Scheduler posts a message to itself via MessageChannel.port.postMessage(). The onmessage callback fires after the browser has had a chance to paint — better than setTimeout(fn, 0) which has a 4ms minimum.
  • Time slicing: Scheduler checks performance.now() during traversal. If more than ~5ms has elapsed, it yields (returns), letting the browser paint, then resumes in the next task.
  • Priority expiration: Low-priority work has a deadline; if it starves too long, it's promoted to prevent starvation.

The React element tree is the conceptual component hierarchy you write in JSX — plain, immutable objects describing what to render. The Fiber tree is React's internal mutable data structure used to perform and track rendering work.

React stores hook state in a linked list of "hook nodes" attached to the fiber for that component. Each useState call corresponds to one node in that list, identified by its position (call order). On re-render, React reads nodes in the same order.
  • Hook linked list: fiber.memoizedState → hookNode1 → hookNode2 → hookNode3. Each node stores the current value and an update queue.
  • First render (mount): React creates a new hook node, initializes it with the initial value, appends it to the list.
  • Re-render (update): React walks the existing list in order, reads each node's current state, processes any queued updates.
  • Setter function: Calling setState enqueues an update on that hook node's queue and schedules a re-render of the fiber.

React uses Object.is() to compare each dependency value between renders. Object.is() behaves like === except it correctly handles NaN === NaN (true) and -0 !== +0 (true). Objects and arrays are compared by reference, not by content.

  • Reference equality trap: const opts = {a: 1} inside a component creates a new object reference every render — dependency always "changed" — effect re-runs every render.
  • Array trap: Same issue — const ids = [1, 2, 3] inline creates a new array reference each render.

The cleanup function runs before the next effect execution (when deps change) and when the component unmounts. It captures the values from the render in which the effect ran — not the current values at cleanup time.
  • Cleanup timing: Render A runs Effect A → Render B (dep changed) → Cleanup A runs (with A's closed-over values) → Effect B runs.
  • Why captured values: The cleanup function is a closure over the values from when the effect was created. This is correct — it allows cleanup to cancel or undo the work of that specific effect run.
  • Subscription example: Effect subscribes to userId A's data. Cleanup unsubscribes from userId A's data (using A's value). New effect subscribes to userId B.
  • Common mistake: Expecting cleanup to see the current (latest) state — it always sees the state from when the effect ran.

When new state depends on the previous state, always use the functional update form: setCount(prev => prev + 1). This is safe in batched updates. Using setCount(count + 1) can silently lose updates if multiple setStates are batched and all capture the same stale count.

  • Async callbacks: In setTimeout or event handlers, count in the closure may be stale. Functional form always receives the latest queued value.

useDebounce stores a debounced copy of a value: useEffect sets a setTimeout to update the debounced value after a delay. The cleanup cancels the timeout on each re-render, so only the last value (after the user stops changing) goes through.
function useDebounce(value, delay = 300) {
  const [debouncedValue, setDebouncedValue] = useState(value);

  useEffect(() => {
    const timer = setTimeout(() => {
      setDebouncedValue(value);
    }, delay);

    return () => clearTimeout(timer); // cancel on each value change
  }, [value, delay]);

  return debouncedValue;
}

In React 18, all setState calls — regardless of where they originate — are batched into a single re-render. React enqueues each state update on the fiber's update queue, then processes all pending updates together at the end of the current task or microtask.

createAsyncThunk is an RTK utility for async operations. It takes an action type prefix and an async payload creator, and auto-generates pending, fulfilled, and rejected action types.

  • First arg: string action type prefix (e.g. 'users/fetchById')
  • Second arg: async function returning a value (becomes fulfilled.payload) or throws (becomes rejected)

  'users/fetchById',
  async (userId, { rejectWithValue }) => {
    try {

    }
  }
);

Handle the three auto-generated lifecycle actions in extraReducers using builder notation: pending sets loading, fulfilled sets data and clears loading, rejected sets error and clears loading.

  • Use action.error.message or action.payload (via rejectWithValue) for the error text
  • Track status as 'idle' | 'loading' | 'succeeded' | 'failed' string for granular UI control

  builder
    .addCase(fetchUser.pending, (state) => {
      state.status = 'loading';

    .addCase(fetchUser.fulfilled, (state, action) => {
      state.status = 'succeeded';
      state.data = action.payload;
    })
    .addCase(fetchUser.rejected, (state, action) => {
      state.status = 'failed';
      state.error = action.payload ?? action.error.message;
    });
}

A selector is a function that extracts and derives data from Redux state. createSelector (Reselect) creates a memoized selector that only recomputes when its input selector outputs change.

  • REDO: pop from future into present, push old present to past

  • For large documents: store diffs (operations) not full content snapshots
