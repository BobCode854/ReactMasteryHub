React Introduction

What is React, why it was created, how it works under the hood, and everything you need before writing your first line of code.

1. What is React?

React is an open-source JavaScript library (not a framework) for building user interfaces — specifically for building component-based, declarative UI. Developed and maintained by Meta (Facebook) and a large open-source community.

React focuses only on the View layer — the "V" in MVC. It does not prescribe how you handle routing, state management, or data fetching. You compose your app from small, reusable components that each own their own state and render independently.

Component-Based
Build encapsulated components that manage their own state, then compose them to make complex UIs.
Declarative
Describe what you want the UI to look like for a given state — React handles the DOM updates.
Learn Once, Write Anywhere
Same mental model for web (React DOM), mobile (React Native), and even desktop/TV.
UI Layer only — composable & declarative

2. Why React? Problems it Solves

Before React (Vanilla JS / jQuery)
  • Manual DOM manipulation — verbose and error-prone
  • No component model — code became spaghetti
  • State scattered across the DOM and JS variables
  • Full page reloads for simple data changes
  • No reusability — copy-paste everywhere
  • Hard to test UI logic in isolation
  • Performance issues with large DOM trees
With React
  • Declarative UI — describe what, not how
  • Component model — encapsulated, reusable pieces
  • State lives inside components — predictable data flow
  • Virtual DOM — only minimal DOM changes applied
  • Share components across the whole app
  • Unit-testable components with React Testing Library
  • Huge ecosystem, strong community & Meta backing
// Before React — manual DOM manipulation
const btn = document.getElementById('btn');
btn.addEventListener('click', () => {
  const count = parseInt(btn.textContent) + 1;
  btn.textContent = count;  // manual DOM update
});

// With React — declarative
function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(c => c + 1)}>{count}</button>;
}

3. React History Timeline

2011
Internal Creation at Facebook
Jordan Walke creates FaxJS (later renamed React) to solve performance problems with Facebook's News Feed UI. The key insight: instead of mutating DOM directly, describe the UI as a function of state.
2013
Open Source Release — React 0.3
React is open-sourced at JSConf US in May 2013. Initial reception was mixed — developers were skeptical of JSX mixing HTML with JavaScript.
2015
React Native + React 0.13
React Native launches — bringing React's component model to iOS and Android. React 0.13 introduces ES6 class components. Redux is created by Dan Abramov and Andrew Clark.
2017
React 16 — Fiber Rewrite
Complete rewrite of React's core (Fiber architecture). Introduces error boundaries, portals, fragments, and async rendering foundations. Major performance improvement.
2019
React 16.8 — Hooks
Hooks are introduced — the biggest paradigm shift in React's history. useState, useEffect, and custom hooks eliminate the need for class components in most cases.
2022
React 18 — Concurrent Features
React 18 ships with concurrent rendering, automatic batching, Suspense for data, startTransition, and the new root API. Introduces the React Compiler (experimental).
2024
React 19 — Server Components GA
React 19 ships Server Components, Server Actions, improved hydration, the React Compiler (stable), and new hooks: use(), useOptimistic(), useFormStatus().

4. SPA vs MPA

SPA — Single Page Application

One HTML file. JavaScript dynamically updates the DOM as the user navigates. The browser never fully reloads. Examples: Gmail, Twitter, Figma, React apps.

MPA — Multi Page Application

Each page is a separate HTML file served from the server. Full page reload on navigation. Examples: Traditional WordPress sites, e-commerce catalogs, Wikipedia.

Aspect SPA (React) MPA (Traditional)
Navigation JS routing, no reload — instant Full server round-trip, page reloads
Initial Load Slower (JS bundle must download first) Faster (minimal HTML served immediately)
SEO Needs SSR/SSG for good SEO (Next.js) Excellent — content is pre-rendered HTML
User Experience App-like, smooth transitions Traditional website feel, visible reloads
State Persistence State preserved across routes State lost on each page load
Best For Dashboards, apps, complex UI Blogs, marketing sites, content sites
Pro Tip: React + Next.js gives you the best of both worlds — component-based React with server rendering for SEO and fast initial loads.

5. Virtual DOM vs Real DOM

Real DOM — Slow
  • A browser object (tree of HTML nodes)
  • Updates are synchronous and blocking
  • Full style recalculation on change
  • Reflow & repaint are expensive
  • Large updates cause jank / dropped frames
  • Direct manipulation is verbose and risky
// Expensive! Re-triggers full layout
const el = document.getElementById('list');
el.innerHTML = newHTML; // destroys & recreates
Virtual DOM — Fast
  • Plain JavaScript object tree in memory
  • Updates are cheap — just object comparison
  • Diffing finds minimal changeset (O(n) heuristic)
  • Only changed nodes touch the real DOM
  • Batches multiple updates into one render
  • Automatic — you never manipulate it directly
// React handles this automatically
setState({items: newItems});
// React diffs → patches only changed nodes
Important Nuance: The Virtual DOM is not inherently faster than the real DOM for every operation. Its advantage is batching and minimizing real DOM mutations. Libraries like Svelte compile away the virtual DOM entirely — neither approach is universally superior.

Key Architectural Concepts
Fiber Architecture (React 16+)
React's internal reconciliation engine. Breaks render work into small units that can be paused, aborted, or reused — enabling concurrent features.
Reconciler (react package)
Platform-agnostic — handles virtual DOM, diffing, and scheduling. Shared between React DOM and React Native.
Renderer (react-dom package)
Platform-specific — translates React element tree to real DOM nodes. Separate packages exist for Native, Three.js, PDF, etc.
Scheduler
Prioritizes render work. High-priority updates (user input) interrupt low-priority work (background data refresh).

9. Interview Questions

React is a JavaScript library for building UIs — specifically focused on the View layer only. Unlike Angular (a full framework), React doesn't include built-in solutions for routing, state management, HTTP calls, or forms. You compose your own stack from the ecosystem. This gives React more flexibility but requires more decisions.

The Virtual DOM is a plain JavaScript object representation of the real DOM tree. React uses it to batch and minimize real DOM mutations. When state changes, React creates a new VDOM tree, diffs it against the previous one (reconciliation), and applies only the minimal set of changes to the real DOM. This avoids expensive full re-renders.

JSX is a syntax extension for JavaScript that looks like HTML. It's compiled by Babel to React.createElement() calls. JSX is not required — you can write React using pure React.createElement() calls, but JSX makes the code much more readable and maintainable.

Reconciliation is React's algorithm for efficiently updating the DOM when state or props change. React builds a new virtual DOM tree and compares it with the previous one using a heuristic diffing algorithm (O(n) complexity). It assumes different element types produce different trees, and uses key props to identify list items. Only the differences are applied to the real DOM.

Imperative: You describe how to achieve a result — step by step DOM mutations (vanilla JS). Declarative: You describe what the result should look like — React figures out how to make the DOM match. React's declarative model means you write less code, make fewer mistakes, and your code is easier to reason about.

Fiber (React 16+) is a complete rewrite of React's reconciliation engine. The key idea: break rendering into small units of work that can be paused, resumed, or abandoned. This enables concurrent features — React can pause rendering to handle a user event, then resume. Each Fiber node corresponds to a component instance and stores information about the work to be done.

React 18 introduced: Concurrent rendering (startTransition, useDeferredValue), Automatic batching (state updates in setTimeout, Promises are now batched), Suspense for data fetching, the new createRoot API, useId for stable IDs, and the useTransition hook.

react: The core library containing the reconciler, component model, and hooks. Platform-agnostic.
react-dom: The web-specific renderer that translates React's virtual tree to real browser DOM nodes. You import createRoot from here. For mobile you'd use react-native instead.

An SPA loads a single HTML file and uses JavaScript to dynamically update the content without full page reloads. Navigation is handled client-side by a router (React Router). Advantages: app-like feel, fast transitions, state persistence. Disadvantages: slower initial load, requires extra work for SEO (needs SSR), worse on low-end devices.

React was created by Jordan Walke, a software engineer at Facebook, around 2011 (originally called FaxJS). It was first used in Facebook's News Feed in 2011, then Instagram in 2012. It was open-sourced at JSConf US in May 2013. It's now maintained by Meta and a large open-source community.

10. Common Mistakes Beginners Make

Mistake 1: Thinking React is a full framework
React only handles the UI. You still need React Router for routing, Redux/Zustand/Context for state, React Query for data fetching. Don't expect React to provide these out of the box.
Mistake 2: Confusing JSX with HTML
JSX looks like HTML but it's JavaScript. You use className not class, htmlFor not for, and all tags must be closed (including <br />, <img />). JavaScript expressions go in {} not inside attribute values with quotes.
Mistake 3: Mutating state directly
state.items.push(newItem) — never do this. State mutations don't trigger re-renders. Always use setState or the state setter from useState. Always create a new object/array when updating state.
Mistake 4: Thinking the Virtual DOM is always faster than the real DOM
The VDOM adds overhead for diffing. For very simple UIs, direct DOM manipulation is faster. React's advantage is making complex, stateful UIs easier to maintain with acceptable performance, not being the absolute fastest possible approach.
Mistake 5: Missing keys in lists
Rendering lists without key props causes React's reconciler to be unable to efficiently diff the list. Always provide a stable, unique key per item — typically the item's database ID, not its array index (index causes bugs when list reorders).

React Entry Point & App Boot Flow

What actually happens from the moment the browser loads your app to the first pixel appearing on screen — every step explained.

1
Browser loads index.html

This is the ONLY html file in a React SPA. It has a single empty div — React will fill everything inside it.

<!-- public/index.html (Vite: index.html at root) --> <!DOCTYPE html> <html lang="en"> <head> <meta charset="UTF-8" /> <title>My React App</title> </head> <body> <!-- React mounts the ENTIRE app into this one div --> <div id="root"></div> <!-- Vite injects the bundled JS here automatically --> <script type="module" src="/src/main.jsx"></script> </body> </html>
Key insight: The <div id="root"> starts completely empty. React JavaScript takes over and fills it with all your components.
2
main.jsx — The True Entry Point

This is where React boots up. It finds the #root div and hands control to your <App /> component.

// src/main.jsx (Vite) OR src/index.js (Create React App) import React from 'react' import ReactDOM from 'react-dom/client' // React 18+ import App from './App' import './index.css' // global styles loaded here // 1. Find the #root div in index.html const rootElement = document.getElementById('root') // 2. Create a React root (enables React 18 concurrent features) const root = ReactDOM.createRoot(rootElement) // 3. Render your entire app tree into that root root.render( <React.StrictMode> {/* dev-only double-render to catch bugs */} <App /> </React.StrictMode> )
What is React.StrictMode? A dev-only wrapper that intentionally renders components twice (in development only — never in production) to help you spot side effects and deprecated patterns. It has zero impact on the built app.
React 17 vs React 18 root API:
ReactDOM.render(<App />, document.getElementById('root')) — old (React 17, deprecated)
ReactDOM.createRoot(el).render(<App />) — new (React 18, enables concurrent features)
3
App.jsx — The Root Component

App is just a regular React component — the root of the whole component tree. It wires together your providers, router, and top-level layout.

// src/App.jsx — typical real-world App component import { BrowserRouter, Routes, Route } from 'react-router-dom' import { Provider } from 'react-redux' import { store } from './state/store' import Navbar from './components/Navbar' import Home from './pages/Home' import Dashboard from './pages/Dashboard' import ProtectedRoute from './components/ProtectedRoute' function App() { return ( {/* Provider makes Redux store available to ALL children */} <Provider store={store}> {/* BrowserRouter enables URL-based routing */} <BrowserRouter> <Navbar /> {/* always visible */} <Routes> <Route path="/" element={<Home />} /> <Route path="/dashboard" element={ <ProtectedRoute><Dashboard /></ProtectedRoute> } /> </Routes> </BrowserRouter> </Provider> ) } export default App
4
JSX compiles → React.createElement() → Fiber tree

Your JSX is never HTML — Babel/Vite transpiles it into React.createElement() calls that produce plain JavaScript objects describing what to render.

// What you write (JSX): function Greeting({ name }) { return <h1 className="title">Hello, {name}!</h1> } // What Babel compiles it to: function Greeting({ name }) { return React.createElement( 'h1', // element type { className: 'title' }, // props object 'Hello, ', name, '!' // children ) } // That call returns a plain JS object (React element): { type: 'h1', props: { className: 'title', children: ['Hello, ', name, '!'] }, key: null, ref: null }
React 17+ introduced the new JSX transform — you no longer need import React from 'react' at the top of every file. Vite and CRA both use this automatically.
5
Reconciliation → Real DOM update

React builds a Fiber tree (Virtual DOM), diffs it against the previous tree, and commits only the minimum DOM changes needed.

// Simplified reconciliation mental model: // 1. RENDER PHASE (can be paused/restarted — no side effects) // React calls your component function → gets JSX objects // Builds new Fiber tree // Diffs new tree vs old tree → produces list of changes // 2. COMMIT PHASE (synchronous — cannot be interrupted) // Applies DOM mutations (insertions, updates, deletions) // Runs layout effects (useLayoutEffect) // Paints to screen // Runs passive effects (useEffect) // What React actually changes in the DOM for a counter update: // BEFORE: <button>Count: 0</button> // AFTER: <button>Count: 1</button> // React only updates the TEXT NODE — not the whole button

On every state update, React re-runs the component function, produces a new Fiber tree, diffs it against the previous one, and commits only the minimal DOM changes. index.html is never touched again.

What happens when state changes?
// 1. User clicks a button → event handler calls setCount(1) function Counter() { const [count, setCount] = useState(0) return ( <button onClick={() => setCount(count + 1)}> Count: {count} </button> ) } // What React does internally when setCount(1) is called: // ┌─────────────────────────────────────────────────────────┐ // │ STEP A: Schedule a re-render of Counter component │ // │ STEP B: Call Counter() function again with new state │ // │ STEP C: Get new JSX: <button>Count: 1</button> │ // │ STEP D: Diff vs old JSX: <button>Count: 0</button> │ // │ STEP E: Only change: text "Count: 0" → "Count: 1" │ // │ STEP F: Update ONLY that text node in the real DOM │ // │ STEP G: Browser repaints the changed pixel │ // └─────────────────────────────────────────────────────────┘ // The entire <button> element is NOT recreated — only the text changes. // This is why React's virtual DOM approach is efficient.
Re-render means re-running the function
When React re-renders a component it simply calls the component function again. The previous DOM is NOT destroyed — React figures out the minimal changes and applies only those.
Children re-render too
When a parent re-renders, all its child components also re-render by default. Use React.memo() to skip re-renders for children whose props haven't changed.
Vite vs CRA — Entry Point File Structure
Vite (modern — recommended)
my-app/ ├── index.html # root HTML — at project root ├── src/ │ ├── main.jsx # entry point │ ├── App.jsx # root component │ ├── App.css │ └── index.css ├── public/ # static assets └── vite.config.js
Create React App (legacy)
my-app/ ├── public/ │ └── index.html # root HTML — inside public/ ├── src/ │ ├── index.js # entry point (not main.jsx) │ ├── App.js │ ├── App.css │ └── index.css └── package.json
Only functional difference: In Vite, index.html is at the project root and directly references src/main.jsx with <script type="module">. In CRA, index.html is inside public/ and CRA injects the script tag during the build. The React boot sequence inside main.jsx/index.js is identical.
Provider Order in App.jsx — matters!

Providers wrap your app in a specific order. Inner providers can access outer providers. Get the order wrong and features break.

// Recommended provider order in App.jsx / main.jsx root.render( <React.StrictMode> {/* 1. outermost — dev checks */} <ErrorBoundary> {/* 2. catch any crash */} <Provider store={store}> {/* 3. Redux — global state */} <QueryClientProvider {/* 4. React Query — server state */} client={queryClient}> <ThemeProvider> {/* 5. theme — needs Redux for user prefs */} <AuthProvider> {/* 6. auth — needs Redux + Query */} <BrowserRouter> {/* 7. router — innermost, needs auth */} <App /> </BrowserRouter> </AuthProvider> </ThemeProvider> </QueryClientProvider> </Provider> </ErrorBoundary> </React.StrictMode> ) // Rule: if Provider B needs access to Provider A's context, A must be OUTER.
Common mistake: Putting <BrowserRouter> outside <AuthProvider> — this means your auth provider can't use useNavigate() to redirect after login. Router must wrap auth, not the other way around.
Interview Questions — Entry Point & Flow

The entry point is main.jsx (Vite) or src/index.js (CRA). This file imports ReactDOM, finds the #root div in index.html, calls ReactDOM.createRoot(element).render(<App />), and hands control to the React component tree. index.html itself just provides the empty <div id="root"> container and loads the JS bundle.

ReactDOM.render() is the React 17 API — it renders synchronously in legacy mode, meaning React 18 concurrent features like useTransition, useDeferredValue, and automatic batching are NOT available. ReactDOM.createRoot() is the React 18 API that opts into concurrent mode, enabling all modern features. The old API is deprecated — always use createRoot for new projects.

React.StrictMode is a development-only tool that intentionally double-invokes component functions and effects to expose side effects and deprecated lifecycle usage. It helps catch bugs early. It has zero effect in production — React strips it out during the production build. The double-render is why you might see console.log statements fire twice in development.

  1. Schedule: React schedules a re-render for that component (batched with other state updates in React 18).
  2. Render phase: React calls the component function again. useState now returns the new value 1 instead of 0. The function returns new JSX.
  3. Fiber diffing: React compares the new Fiber tree against the previous one using the reconciliation algorithm.
  4. Commit phase: React applies only the minimal DOM mutations identified by the diff — in this case just updating a text node.
  5. Layout effects: useLayoutEffect cleanup + new effect runs synchronously.
  6. Paint: Browser repaints the changed pixels.
  7. Passive effects: useEffect cleanup + new effect runs asynchronously after paint.

Providers form a nested tree — inner providers can use the context of outer providers, but not vice versa. For example, if AuthProvider needs to call useNavigate() to redirect after login, it must be rendered inside <BrowserRouter> because useNavigate requires the Router context. Similarly, if AuthProvider stores auth state in Redux, it must be inside <Provider store={store}>. Always ask: "Does this provider consume any other context?" — if yes, it goes inside that provider.
