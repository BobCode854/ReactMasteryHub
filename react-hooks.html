useState

Core State Hook
What it does: useState returns a pair — the current state value and a setter function. React re-renders the component whenever the setter is called with a new value. The initial value argument is only used on the first render.
How it works internally (React Fiber): React maintains a linked list of hook objects inside each component’s fiber node. Each useState call reserves one slot in this list, identified purely by its call order. On every render React replays the hook calls in order, reading from and writing to the matching slot. The setter enqueues an update action on that slot’s queue, which triggers a re-render and processes the queue to compute the new state. This is why the Rules of Hooks forbid calling hooks inside conditions — changing the order corrupts the slot mapping. 
import React, { useState } from 'react';

// Shopping cart with multiple independent states
function ShoppingCart() {
  const [items, setItems] = useState([]);
  const [total, setTotal] = useState(0);
  const [isOpen, setIsOpen] = useState(false);

  // Functional update -- always use when new state depends on previous
  const addItem = (product) => {
    setItems(prev => [...prev, product]);
    setTotal(prev => prev + product.price);
  };

  const removeItem = (id) => {
    setItems(prev => {
      const removed = prev.find(i => i.id === id);
      setTotal(t => t - (removed?.price ?? 0));
      return prev.filter(i => i.id !== id);
    });
  };

  // Object state -- always spread to preserve other fields
  const [user, setUser] = useState({ name: '', email: '', role: 'guest' });
  const updateName = (name) => setUser(prev => ({ ...prev, name }));

  return (
    <div>
      <button onClick={() => setIsOpen(o => !o)}>
        Cart ({items.length}) -- ${total.toFixed(2)}
      </button>
      {isOpen && (
        <ul>
          {items.map(item => (
            <li key={item.id}>
              {item.name}
              <button onClick={() => removeItem(item.id)}>Remove</button>
            </li>
          ))}
        </ul>
      )}
    </div>
  );
}
Functional Updates: Always use setState(prev => ...) when the new state depends on the previous state. Without it, you risk a stale closure — especially in event handlers set up during earlier renders.
Lazy initialization: If the initial state is expensive to compute, pass a function to useState: useState(() => JSON.parse(localStorage.getItem('cart') || '[]')). React calls this function only on the first render.

useEffect

Side Effects & Lifecycle
What it does: useEffect synchronizes a component with an external system — APIs, timers, subscriptions, DOM APIs, etc. It runs after the browser has painted. The optional cleanup function returned from the effect runs before the next effect execution and when the component unmounts. 
import { useState, useEffect } from 'react';

// Pattern 1: Run once on mount (empty deps array)
useEffect(() => {
  initAnalytics();
  return () => cleanupAnalytics();
}, []);

// Pattern 2: Run when specific dep changes
useEffect(() => {
  loadUserProfile(userId);
}, [userId]);

// Pattern 3: Timer with cleanup (no stale closure)
function Counter() {
  const [count, setCount] = useState(0);
  useEffect(() => {
    const id = setInterval(() => {
      setCount(c => c + 1); // functional update avoids stale closure
    }, 1000);
    return () => clearInterval(id);
  }, []);
  return <p>{count}</p>;
}

// Pattern 4: API call with AbortController (cancellation)
function UserProfile({ userId }) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState(null);

  useEffect(() => {
    const controller = new AbortController();
    const { signal } = controller;

    const fetchUser = async () => {
      setLoading(true);
      setError(null);
      try {
        const res = await fetch(`/api/users/${userId}`, { signal });
        if (!res.ok) throw new Error('Failed to fetch');
        setData(await res.json());
      } catch (err) {
        if (err.name !== 'AbortError') setError(err.message);
      } finally {
        setLoading(false);
      }
    };

    fetchUser();
    return () => controller.abort(); // cancel on userId change or unmount
  }, [userId]);

  if (loading) return <p>Loading...</p>;
  if (error) return <p>Error: {error}</p>;
  return <p>{data?.name}</p>;
}
Infinite Loop Trap: Putting an object literal directly in the deps array creates a new reference on every render, looping forever.
// BAD: new object reference every render
useEffect(() => {}, [{ id: userId }]);
// GOOD: primitive in deps
useEffect(() => {}, [userId]);
Dep array: [] = mount only  |  [a,b] = when a or b changes  |  no array = every render.

useContext

Global State Without Prop Drilling
What it does: useContext subscribes a component to a React context object. When the Provider’s value prop changes, ALL components consuming that context re-render. This makes context ideal for low-frequency updates like theme, locale, and auth. 
import React, { createContext, useContext, useState } from 'react';

// Step 1: Create context with sensible default
const ThemeContext = createContext({ theme: 'light', toggleTheme: () => {} });

// Step 2: Custom hook with safety guard
export function useTheme() {
  const ctx = useContext(ThemeContext);
  if (!ctx) throw new Error('useTheme must be inside ThemeProvider');
  return ctx;
}

// Step 3: Provider owns and manages state
export function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('light');
  const toggleTheme = () => setTheme(t => t === 'light' ? 'dark' : 'light');

  // Memoize to prevent consumers re-rendering on unrelated parent renders
  const value = React.useMemo(() => ({ theme, toggleTheme }), [theme]);

  return (
    <ThemeContext.Provider value={value}>
      {children}
    </ThemeContext.Provider>
  );
}

// Step 4: Consume anywhere -- no prop drilling
function ThemedButton() {
  const { theme, toggleTheme } = useTheme();
  return (
    <button
      onClick={toggleTheme}
      style={{ background: theme === 'dark' ? '#1e293b' : '#f1f5f9' }}
    >
      Theme: {theme}
    </button>
  );
}
Split contexts by update frequency. ThemeContext (rare) should be separate from CartContext (frequent). Consumers of the theme won’t re-render when the cart changes.
Passing a new object literal as value without memoization creates a new reference on every parent render, causing all consumers to re-render unnecessarily. Always wrap in useMemo.

useRef

Mutable Values & DOM Access
What it does: useRef returns a mutable object { current: value } that persists for the full lifetime of the component. Changing .current does NOT trigger a re-render. Use it for DOM references, persisting mutable values, and storing instance values like timer IDs. 
import { useRef, useEffect, useState } from 'react';

// Use Case 1: DOM Reference
function FocusInput() {
  const inputRef = useRef(null);
  useEffect(() => { inputRef.current?.focus(); }, []);
  return <input ref={inputRef} placeholder="Auto-focused" />;
}

// Use Case 2: Track Previous Value
function usePrevious(value) {
  const ref = useRef();
  useEffect(() => { ref.current = value; }); // runs after render
  return ref.current; // value from previous render
}

function Counter() {
  const [count, setCount] = useState(0);
  const prevCount = usePrevious(count);
  return <p>Now: {count} | Before: {prevCount}</p>;
}

// Use Case 3: Store Timer ID -- no re-render when cleared
function Stopwatch() {
  const [time, setTime] = useState(0);
  const [running, setRunning] = useState(false);
  const intervalRef = useRef(null);

  const start = () => {
    if (running) return;
    setRunning(true);
    intervalRef.current = setInterval(() => setTime(t => t + 1), 10);
  };
  const stop = () => {
    clearInterval(intervalRef.current);
    intervalRef.current = null;
    setRunning(false);
  };

  return (
    <div>
      <p>{(time / 100).toFixed(2)}s</p>
      <button onClick={start}>Start</button>
      <button onClick={stop}>Stop</button>
    </div>
  );
}
Mental model: Refs are like instance variables — survive renders but don’t cause them. If you need a value in the UI, use useState. If you just need it for logic, use useRef.

useMemo

Memoize Computed Values
What it does: useMemo memoizes the return value of a computation — it only recalculates when one of its dependencies changes. Use it to avoid expensive recalculations on every render, and to stabilize object/array references passed as props or used in other hooks’ dependency arrays. 
import { useState, useMemo } from 'react';

function ProductList({ products }) {
  const [search, setSearch] = useState('');
  const [sortBy, setSortBy] = useState('name');
  const [minPrice, setMinPrice] = useState(0);

  // Only recalculates when these 4 deps change -- not on unrelated renders
  const filteredProducts = useMemo(() => {
    return products
      .filter(p =>
        p.name.toLowerCase().includes(search.toLowerCase()) &&
        p.price >= minPrice
      )
      .sort((a, b) =>
        sortBy === 'price' ? a.price - b.price : a.name.localeCompare(b.name)
      );
  }, [products, search, sortBy, minPrice]);

  // Stabilize object reference -- prevents useEffect from re-firing unnecessarily
  const queryParams = useMemo(
    () => ({ search, minPrice, sortBy }),
    [search, minPrice, sortBy]
  );

  return (
    <div>
      <input value={search} onChange={e => setSearch(e.target.value)} />
      <p>{filteredProducts.length} products found</p>
      {filteredProducts.map(p => <div key={p.id}>{p.name}</div>)}
    </div>
  );
}
When to use useMemo: Expensive computations (filtering/sorting large arrays), stabilizing object refs used in hook dependency arrays, and props passed to React.memo-wrapped children. Do NOT use for trivial computations like a + b.
Don’t overuse: useMemo has its own cost — memory allocation and dependency comparison on every render. Profile first, optimize second.

useCallback

Memoize Function References
What it does: useCallback memoizes a function reference. Without it, a new function instance is created on every parent render, breaking React.memo on child components that receive the function as a prop. 
import { useState, useCallback, memo } from 'react';

// React.memo skips re-render if props unchanged (shallow comparison)
const MemoizedChild = memo(function Child({ onSearch, label }) {
  console.log('Child rendered');
  return <button onClick={onSearch}>{label}</button>;
});

function Parent() {
  const [name, setName] = useState('');
  const [count, setCount] = useState(0);

  // BAD: new reference every render -- defeats React.memo
  const handleBad = () => fetchResults(name);

  // GOOD: stable reference -- only changes when name changes
  const handleSearch = useCallback(() => {
    fetchResults(name);
  }, [name]);

  // No deps -- functional setState avoids closure over count
  const handleIncrement = useCallback(() => setCount(c => c + 1), []);

  return (
    <div>
      <input value={name} onChange={e => setName(e.target.value)} />
      <MemoizedChild onSearch={handleSearch} label="Search" />
      <button onClick={handleIncrement}>Count: {count}</button>
    </div>
  );
}
The golden rule: useCallback is useless without React.memo on the receiving component. Both pieces must be in place for the optimization to work.
Equivalence: useCallback(fn, deps) is exactly useMemo(() => fn, deps). useCallback is syntactic sugar that memoizes the function itself rather than its return value.

useReducer

Complex State Management
What it does: useReducer is an alternative to useState for managing complex state. reducer(state, action) => newState. The reducer is a pure function — receives current state and an action, returns next state without mutations. Easy to unit test in isolation. 
import { useReducer } from 'react';

const initialState = { items: [], total: 0, loading: false, error: null };

function cartReducer(state, action) {
  switch (action.type) {
    case 'ADD_ITEM': {
      const exists = state.items.find(i => i.id === action.payload.id);
      const items = exists
        ? state.items.map(i => i.id === action.payload.id ? { ...i, qty: i.qty + 1 } : i)
        : [...state.items, { ...action.payload, qty: 1 }];
      return { ...state, items, total: state.total + action.payload.price };
    }
    case 'REMOVE_ITEM': {
      const item = state.items.find(i => i.id === action.payload);
      return {
        ...state,
        items: state.items.filter(i => i.id !== action.payload),
        total: state.total - (item?.price * item?.qty ?? 0),
      };
    }
    case 'UPDATE_QUANTITY':
      return { ...state, items: state.items.map(i => i.id === action.payload.id ? { ...i, qty: action.payload.qty } : i) };
    case 'SET_LOADING': return { ...state, loading: action.payload };
    case 'SET_ERROR': return { ...state, error: action.payload, loading: false };
    case 'CLEAR': return initialState;
    default: throw new Error(`Unknown action: ${action.type}`);
  }
}

function Cart() {
  const [state, dispatch] = useReducer(cartReducer, initialState);
  const { items, total, loading, error } = state;

  return (
    <div>
      {loading && <p>Loading...</p>}
      {error && <p>{error}</p>}
      {items.map(item => (
        <div key={item.id}>
          {item.name} x{item.qty}
          <button onClick={() => dispatch({ type: 'REMOVE_ITEM', payload: item.id })}>Remove</button>
        </div>
      ))}
      <p>Total: ${total.toFixed(2)}</p>
      <button onClick={() => dispatch({ type: 'CLEAR' })}>Clear</button>
    </div>
  );
}

useLayoutEffect

Synchronous DOM Reads
What it does: Identical API to useEffect, but fires synchronously after DOM mutations and before the browser paints. Ideal for reading DOM measurements and synchronously updating layout to prevent visual flickering. 
import { useRef, useState, useLayoutEffect } from 'react';

// Tooltip that positions itself based on measured DOM dimensions
function SmartTooltip({ targetRef, children }) {
  const tooltipRef = useRef(null);
  const [style, setStyle] = useState({});

  // Runs BEFORE paint -- no flicker
  useLayoutEffect(() => {
    if (!tooltipRef.current || !targetRef.current) return;
    const targetRect = targetRef.current.getBoundingClientRect();
    const tooltipRect = tooltipRef.current.getBoundingClientRect();
    const showAbove = targetRect.top >= tooltipRect.height + 8;
    setStyle({
      position: 'fixed',
      left: targetRect.left + (targetRect.width - tooltipRect.width) / 2,
      top: showAbove ? targetRect.top - tooltipRect.height - 8 : targetRect.bottom + 8,
      opacity: 1,
    });
  }, [targetRef]);

  return (
    <div ref={tooltipRef} style={{ ...style, opacity: style.opacity ?? 0, background: '#1e293b', color: '#fff', padding: '4px 8px', borderRadius: '6px' }}>
      {children}
    </div>
  );
}

Rule of thumb: Always start with useEffect. Switch to useLayoutEffect only when you observe a visual flicker caused by the effect needing to read and synchronously update the DOM before paint.

useImperativeHandle

Control Exposed Ref API
What it does: Customizes the ref object that a parent component receives when using forwardRef. Instead of exposing the raw DOM node, you expose a controlled API with specific methods. Prefer props-based communication when possible. 
import { useRef, useImperativeHandle, forwardRef } from 'react';

// Child exposes a controlled API -- not the raw DOM video element
const VideoPlayer = forwardRef(function VideoPlayer({ src }, ref) {
  const videoRef = useRef(null);

  useImperativeHandle(ref, () => ({
    play()  { videoRef.current?.play(); },
    pause() { videoRef.current?.pause(); },
    seek(time) { if (videoRef.current) videoRef.current.currentTime = time; },
    getTime() { return videoRef.current?.currentTime ?? 0; },
    // Parent cannot access videoRef.current.src -- good encapsulation
  }), []);

  return <video ref={videoRef} src={src} />;
});

// Parent uses the exposed API -- not the raw DOM node
function MoviePage() {
  const playerRef = useRef(null);

  const handleSkipIntro = () => {
    playerRef.current?.seek(90);
    playerRef.current?.play();
  };

  return (
    <div>
      <VideoPlayer ref={playerRef} src="/movie.mp4" />
      <button onClick={() => playerRef.current?.play()}>Play</button>
      <button onClick={() => playerRef.current?.pause()}>Pause</button>
      <button onClick={handleSkipIntro}>Skip Intro</button>
    </div>
  );
}
Use for: media players (play/pause/seek), custom inputs (focus/clear/validate), canvas drawing APIs. Always prefer prop-based communication first.

Custom Hooks

Reusable Stateful Logic
What they are: Custom hooks are plain JavaScript functions whose names start with use and that may call other hooks. They extract reusable stateful logic from components — the modern replacement for Higher-Order Components and render props. Each component calling a custom hook gets its own isolated state. 
import { useState, useEffect, useCallback } from 'react';

// useFetch -- data fetching with cancellation
function useFetch(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState(null);

  useEffect(() => {
    if (!url) return;
    const controller = new AbortController();
    setLoading(true); setError(null);
    fetch(url, { signal: controller.signal })
      .then(r => r.json())
      .then(setData)
      .catch(err => { if (err.name !== 'AbortError') setError(err.message); })
      .finally(() => setLoading(false));
    return () => controller.abort();
  }, [url]);
  return { data, loading, error };
}

// useLocalStorage -- synced with localStorage
function useLocalStorage(key, initialValue) {
  const [value, setValue] = useState(() => {
    try {
      const stored = localStorage.getItem(key);
      return stored ? JSON.parse(stored) : initialValue;
    } catch { return initialValue; }
  });
  const setStoredValue = useCallback((val) => {
    try {
      const toStore = val instanceof Function ? val(value) : val;
      localStorage.setItem(key, JSON.stringify(toStore));
      setValue(toStore);
    } catch (err) { console.warn('localStorage write failed', err); }
  }, [key, value]);
  return [value, setStoredValue];
}

// useDebounce -- delays expensive operations
function useDebounce(value, delay = 300) {
  const [debouncedValue, setDebouncedValue] = useState(value);
  useEffect(() => {
    const timer = setTimeout(() => setDebouncedValue(value), delay);
    return () => clearTimeout(timer);
  }, [value, delay]);
  return debouncedValue;
}

// Usage: SearchBox combining useFetch + useDebounce
function SearchBox() {
  const [query, setQuery] = useState('');
  const debouncedQuery = useDebounce(query, 400);
  const { data, loading } = useFetch(debouncedQuery ? `/api/search?q=${debouncedQuery}` : null);

  return (
    <div>
      <input value={query} onChange={e => setQuery(e.target.value)} placeholder="Search..." />
      {loading && <p>Searching...</p>}
      {data?.results?.map(r => <p key={r.id}>{r.name}</p>)}
    </div>
  );
}
Naming rule: The use prefix signals to eslint-plugin-react-hooks to enforce the Rules of Hooks inside your custom hook. Without it, the linter won’t catch violations inside your hook function.

Rules of Hooks

Rule 1 — Only call hooks at the top level

Never call hooks inside loops, conditions, nested functions, or try/catch blocks. React identifies each hook by its order of call within a component render. Changing the order corrupts the fiber slot mapping, causing state to be assigned to the wrong hooks.

Rule 2 — Only call hooks from React functions

Call hooks only from React function components or custom hook functions (names starting with use). Never from regular JavaScript functions, class components, event handlers, or async callbacks outside a hook.

// VIOLATION -- conditional hook call
function BadComponent({ isLoggedIn }) {
  if (isLoggedIn) {
    const [data, setData] = useState(null); // skipped when false -- corrupts all slots!
  }
  const [count, setCount] = useState(0); // now occupies wrong slot
}

// CORRECT -- condition goes INSIDE the effect, not around the hook
function GoodComponent({ isLoggedIn }) {
  const [data, setData] = useState(null);   // always slot 0
  const [count, setCount] = useState(0); // always slot 1

  useEffect(() => {
    if (!isLoggedIn) return; // condition is INSIDE the effect
    fetchUserData().then(setData);
  }, [isLoggedIn]);
}
Enforcement: Install eslint-plugin-react-hooks and enable "react-hooks/rules-of-hooks": "error" and "react-hooks/exhaustive-deps": "warn". These catch violations at lint/build time.

20 Interview Questions — React Hooks

Hooks are functions that let you use state and other React features in functional components. They were introduced to solve three core problems with class components: (1) Complex lifecycle logic — related logic was split across componentDidMount, componentDidUpdate, and componentWillUnmount. (2) Stateful logic was hard to reuse — HOCs and render props caused “wrapper hell.” (3) Classes confused both people and machines — this binding was error-prone, class syntax didn’t minify well, and hot reloading was unreliable. Hooks enable cleaner, more reusable logic using only functions.

useState is simpler — ideal for independent primitive values or simple objects where each setter is self-contained. useReducer shines when state has multiple sub-values that update together, next state depends on previous state in complex ways, or you want state logic testable in isolation (the reducer is a pure function you can unit test directly without rendering a component). Rule of thumb: if you find yourself writing multiple useState setters in the same event handler to keep state consistent, that’s a sign useReducer would be cleaner.

Use useMemo to memoize the result of a computation (a value): useMemo(() => expensiveCalc(a, b), [a, b]). Use useCallback to memoize a function reference: useCallback(() => doSomething(x), [x]). They are equivalent: useCallback(fn, deps) equals useMemo(() => fn, deps). useCallback is syntactic sugar. In practice: useMemo when avoiding recomputing an expensive value; useCallback when you want a stable function reference to pass to React.memo-wrapped children or use in another hook’s dependency array. Neither is useful without a concrete performance reason — profile first.

useEffect fires asynchronously after the browser has painted — non-blocking, the default choice for data fetching, subscriptions, logging. useLayoutEffect fires synchronously after DOM mutations but before the browser paints — blocks the paint. Use it when you need to read a DOM measurement (like getBoundingClientRect) and synchronously update the DOM to prevent visual flickering. Example: a tooltip that measures its target and positions itself — useEffect would flash the tooltip in the wrong position first; useLayoutEffect positions it correctly on the same frame. Note: useLayoutEffect causes a warning in SSR since DOM isn’t available server-side.

[] (empty array): The effect runs only once after the initial mount (equivalent to componentDidMount). Cleanup runs on unmount. [dep] (with dependencies): The effect runs after mount AND after every render where any listed dependency changed (React uses Object.is for comparison). No dependency array: The effect runs after every single render — rare and dangerous, can cause infinite loops if the effect updates state. The rule: include ALL values from component scope used inside the effect. The exhaustive-deps ESLint rule helps enforce this.

Without a dependency array, the effect runs after every render — both initial mount and every subsequent update. This is rarely what you want. It can cause infinite loops if the effect updates state (state update → re-render → effect runs → state update…), or severe performance issues if it makes API calls. The only legitimate use case is effects that genuinely need to run after every render. Most of the time, omitting the array is a bug — either the developer forgot it (should be []) or hasn’t properly identified dependencies.

A custom hook is a plain JavaScript function whose name starts with use and that may call other hooks inside it. The use prefix is not just a convention — it signals to eslint-plugin-react-hooks to enforce the Rules of Hooks inside the function. Without the prefix, the linter won’t catch violations. Custom hooks don’t share state between components — each component calling a custom hook gets its own isolated state instance. They replaced HOCs and render props for logic reuse in modern React.

No. React identifies each hook by its order of calls within a component render. React maintains an internal linked list (fiber) where each hook call maps to a slot by position (slot 0, 1, 2…). If a hook is inside a condition, it might be skipped on some renders, shifting all subsequent hooks to wrong slots — state values get assigned to the wrong hooks, causing unpredictable bugs. The fix: move the condition inside the hook or effect, not around it. The eslint-plugin-react-hooks/rules-of-hooks rule catches this at build time.

Infinite loops occur when the effect updates state that is also in the dependency array. Solutions: (1) Remove the dependency if truly not needed. (2) Use functional state updates — setState(prev => prev + 1) lets you read current state without adding it to deps. (3) Memoize objects/functions with useMemo/useCallback before adding to deps — a new object reference on every render triggers the effect on every render. (4) Move static objects/arrays outside the component. (5) Use a ref for values you need to read inside an effect but shouldn’t re-run when they change.

(1) DOM references — attach the ref to a DOM element via the ref prop to access it imperatively: focusing an input, measuring its dimensions, triggering animations, or scrolling to it. (2) Persisting mutable values across renders without triggering re-renders — track the previous value of a prop or state, store a render count, or track whether a component is still mounted. (3) Storing mutable instance values like timer IDs (setInterval, setTimeout), WebSocket connections, or subscription handles — so they can be read or cleared in effects and event handlers without causing re-renders when they are set.

useContext(MyContext) subscribes a component to a React context object. When the Provider’s value prop changes by reference, React re-renders all components consuming that context — even if a consumer only uses one field from a multi-field context object. Mitigations: (1) Split contexts by update frequency — ThemeContext (rarely changes) separate from CartContext (changes often). (2) Memoize the context value with useMemo. (3) For fine-grained subscriptions at scale, consider Zustand, Jotai, or Redux.

useImperativeHandle(ref, createHandle, deps) customizes the ref object exposed to a parent when using forwardRef. Without it, forwardRef exposes the raw DOM node. With it, you expose a controlled API — only the methods you explicitly define. Use it for: media players (play/pause/seek), custom input components (focus/clear/validate), canvas drawing APIs. As a general principle: prefer props and callbacks over imperative handles. Use useImperativeHandle only when the imperative API is genuinely more natural or when you need to prevent parent access to internal DOM nodes.

React.memo wraps a component and skips re-rendering if its props haven’t changed (shallow comparison). The problem: if a parent passes a function as a prop, that function is recreated on every parent render — React.memo sees “props changed” and re-renders the child anyway. useCallback in the parent stabilizes the function reference. You need BOTH: React.memo on the child to opt into prop-based optimization, AND useCallback in the parent to ensure function props don’t break that optimization. Without React.memo, useCallback achieves nothing for rendering performance. Without useCallback, React.memo is defeated by unstable function props.

A stale closure occurs when an effect or callback captures a value from a previous render and uses it even after the value has updated. Classic example: setInterval inside useEffect(fn, []) — the interval callback closes over the initial state value and always sees it, never the updated value. 
// Stale -- count is always 0 inside the callback
useEffect(() => {
  const id = setInterval(() => console.log(count), 1000); // always 0!
  return () => clearInterval(id);
}, []);

// Fix -- functional update reads current state without closure
setCount(c => c + 1);

Return a cleanup function from useEffect — React calls it before the next effect execution and when the component unmounts.

1. Timer cleanup: return () => clearInterval(id)

2. Fetch cancellation: const controller = new AbortController(); fetch(url, { signal: controller.signal }); return () => controller.abort() — cancels in-flight requests when deps change or component unmounts, preventing setState on unmounted components.

3. Event listener: window.addEventListener('resize', handler); return () => window.removeEventListener('resize', handler) — prevents duplicate listeners and memory leaks. Failing to clean up causes memory leaks and duplicate subscriptions.

React maintains a “fiber” object for each component instance. This fiber has a linked list of hook objects — one per hook call, identified purely by call order (slot 0, 1, 2…). On initial render (“mount”), React creates a new hook object for each hook call. Each useState creates a hook object with: memoizedState (current value), queue (an update queue), and next (pointer to next hook). On re-renders, React replays hook calls in the same order, reading from existing hook objects. The setter enqueues an update action on the hook object’s queue and schedules a re-render. During re-render, React processes the queue to compute the new memoizedState. This is why hooks must be called in the same order every render — the linked list has no names, only positions.

Choose useReducer when: (1) State has multiple sub-values that update together — a shopping cart needs items, total, loading, and error to update in coordinated ways. (2) Next state depends on the previous in complex ways. (3) State logic should be testable in isolation — the reducer is a pure function; write unit tests directly without rendering any component. (4) Multiple distinct actions trigger state updates — like a finite state machine with named transitions. (5) Complex forms, wizards, or multi-step processes where many handlers update shared state. Use useState for simple, independent values like a single toggle, counter, or text input.

Rule 1 — Only call hooks at the top level: Never inside loops, conditions, nested functions, or try/catch blocks. Rule 2 — Only call hooks from React functions: Only from React function components or custom hooks (functions starting with use), never from regular JS functions or class components.

These rules exist because React relies on the stable order of hook calls to correctly map each call to its internal state slot in the fiber linked list. Hooks have no names — they are identified purely by position. If a hook is inside a condition or loop, it might be called a different number of times on different renders, changing the order. React would then read wrong state from wrong slots, causing hard-to-debug bugs. eslint-plugin-react-hooks enforces both rules at lint time.

Proper data fetching in useEffect: (1) Create an AbortController at the start. (2) Pass its signal to fetch options. (3) Return a cleanup function that calls controller.abort(). (4) In the catch block, check for AbortError separately — it is not a real error, just a cancellation signal. (5) Reset loading/error state at the start of each effect run to avoid stale states.

Without cancellation, if the user navigates away or deps change before the fetch completes, the async callback may call setState on an unmounted component, or set state with stale data from a previous request (race condition). In production, consider React Query or SWR which handle cancellation, caching, and deduplication automatically.

In controlled components, form element values are driven by React state (useState). Every keystroke calls onChange, which calls setState, which re-renders with the new value prop — React is the single source of truth. Enables real-time validation, conditional disabling, instant formatting, and sync across components.

In uncontrolled components, the DOM manages its own state. You use useRef to access the value when needed (e.g., on submit): inputRef.current.value. Uncontrolled components have less re-rendering overhead and simpler code for one-time reads, but real-time validation, programmatic reset, and keeping values in sync across components is much harder.

Rule of thumb: prefer controlled for forms that need validation, dynamic behavior, or synced state. Use uncontrolled for simple standalone inputs where you only need the value at submit time.
