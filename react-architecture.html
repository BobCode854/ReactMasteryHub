Section 22 Advanced

React Architecture

Patterns for structuring scalable React applications — folder layout, micro frontends, state decisions, API layers, and performance strategies.

Folder Structure Patterns

There is no single "correct" structure. The right pattern depends on your team size, domain complexity, and app scale.

Pattern 1
Feature-Based (Recommended for large apps)

Group code by domain/feature, not by file type. Each feature is a self-contained module with its own components, hooks, state, and API.

src/ ├── features/ │ ├── auth/ │ │ ├── components/ LoginForm, RegisterForm, ForgotPassword │ │ ├── hooks/ useAuth, useLogin, useLogout │ │ ├── store/ authSlice.ts, authSelectors.ts │ │ ├── api/ authApi.ts — login, register, refresh │ │ └── types/ auth.types.ts — User, AuthState, LoginPayload │ ├── products/ │ │ ├── components/ ProductCard, ProductGrid, ProductFilters │ │ ├── hooks/ useProducts, useProductSearch │ │ ├── store/ productsSlice.ts │ │ └── api/ productsApi.ts │ └── checkout/ │ ├── components/ CartItem, OrderSummary, PaymentForm │ ├── hooks/ useCart, useCheckout │ └── store/ cartSlice.ts ├── components/ Shared UI: Button, Modal, Input, Badge ├── hooks/ Shared: useDebounce, useLocalStorage, useFetch ├── utils/ formatDate, formatCurrency, validators ├── types/ Global TypeScript types: ApiResponse, PaginatedList ├── constants/ API_BASE_URL, ROUTES, ERROR_CODES ├── services/ api.ts (axios), tokenService.ts, analytics.ts └── App.tsx
Rule of thumb: If you can delete a feature folder and the rest of the app still compiles, your feature boundaries are clean.
Pattern 2
Type-Based (Small/Medium Apps)

Group by file type. Simple to understand but gets unwieldy as the app grows.

src/ ├── components/ ALL components — Header, ProductCard, Modal, etc. ├── pages/ Route-level components — Home, Products, Checkout ├── hooks/ ALL custom hooks ├── store/ Redux slices, selectors, store.ts ├── api/ All API functions ├── utils/ Helper functions └── types/ TypeScript types
Pattern 3
Atomic Design (Design System Approach)

Organizes UI components by complexity level. Best for teams building a shared design system.

src/ ├── atoms/ Button, Input, Badge, Icon, Spinner ├── molecules/ SearchBar, FormField, CardHeader ├── organisms/ Header, ProductCard, LoginForm, DataTable ├── templates/ PageLayout, DashboardLayout, AuthLayout └── pages/ Home, Products, Profile — assemble templates

Micro Frontend Architecture

Micro frontends extend microservices thinking to the frontend. Each team owns a vertical slice — from UI to API.

Module Federation (Webpack 5)

Each app exposes components/modules that other apps can consume at runtime.

// Host app webpack.config.js new ModuleFederationPlugin({ name: 'shell', remotes: { productApp: 'products@http://localhost:3001/remoteEntry.js', cartApp: 'cart@http://localhost:3002/remoteEntry.js', }, });
Routing Strategy

Shell app owns top-level routing. Each micro-frontend handles its own sub-routes.

// Shell app — top-level routes const ProductApp = lazy( () => import('productApp/App') ); <Route path="/products/*" element={<ProductApp />} />
Shared Dependencies

Share React, React DOM, design system — avoid loading them multiple times.

// Shared config shared: { react: { singleton: true, requiredVersion: '^18.0.0' }, 'react-dom': { singleton: true }, '@company/design-system': { singleton: true } }
When to Use Micro Frontends
Good fit:
  • Multiple teams working on the same product
  • Different deployment cadences per feature
  • Technology heterogeneity needed
  • Large, complex enterprise apps (10+ devs)
Poor fit:
  • Small teams (1-3 devs)
  • Tightly coupled features
  • Consistent UX is critical
  • Performance budget is tight (bundle overhead)

State Architecture Decision Tree

Before choosing a state solution, ask these questions in order:

1
Is it server/async data? (fetched from an API)
Use React Query (TanStack Query) or RTK Query — handles caching, loading states, refetching, stale time, and background sync automatically. Don't put server data in Redux.
2
Is it shared across many unrelated components?
Use React Context for simple cases (theme, locale, auth user). Use Redux Toolkit or Zustand for complex shared state with frequent updates.
3
Is it form state?
Use React Hook Form or Formik. Don't manage form state with useState/Redux — these libraries handle validation, dirty tracking, and submission efficiently.
4
Is it UI state? (modal open, active tab, tooltip visible)
Use local useState in the component that owns it. Co-locate state as close as possible to where it's used. Only lift when two sibling components need it.
5
Is it URL/navigation state? (filters, search query, active tab in URL)
Use React Router search params (useSearchParams). This makes state shareable via URL and survives page refreshes. Use for filters, pagination, and selected IDs.
Common mistake: Putting everything in Redux. Server data belongs in React Query, form state belongs in React Hook Form, and UI state belongs in useState. Redux is for complex global client state only.

Performance Architecture

Render Boundary Strategy (React.memo placement)
// Expensive list item — memoize const ProductCard = React.memo(function ProductCard({ product }) { return <div>{product.name}</div>; }, (prev, next) => prev.product.id === next.product.id); // Parent re-renders don't re-render cards function ProductList({ products }) { return products.map(p => <ProductCard key={p.id} product={p} /> ); }
Only memoize components that render frequently with the same props. React.memo has overhead — don't use it everywhere.
Code Splitting Boundaries
// Route-level splitting — always do this const ProductsPage = lazy(() => import('./pages/Products')); const CheckoutPage = lazy(() => import('./pages/Checkout')); // Feature-level splitting — heavy features const RichTextEditor = lazy(() => import('./components/RichTextEditor')); const ChartDashboard = lazy(() => import('./features/analytics/Dashboard')); <Suspense fallback={<PageSkeleton />}> <Routes> <Route path="/products" element={<ProductsPage />} /> <Route path="/checkout" element={<CheckoutPage />} /> </Routes> </Suspense>
Context Slicing Strategy
// BAD — one huge context causes all consumers to re-render const AppContext = { user, theme, cart, notifications, settings }; // GOOD — separate contexts, consumers only re-render for their slice const UserContext = createContext(); // auth user const ThemeContext = createContext(); // dark/light mode const CartContext = createContext(); // cart items // Each provider wraps only what needs it function Providers({ children }) { return ( <UserProvider> <ThemeProvider> <CartProvider>{children}</CartProvider> </ThemeProvider> </UserProvider> ); }
Performance Checklist
  • Route-based code splitting (lazy + Suspense)
  • Virtualize long lists (react-virtual, react-window)
  • Memoize expensive computations (useMemo)
  • Stable callbacks (useCallback for child props)
  • Small, focused contexts (avoid mega-context)
  • Image lazy loading + WebP + CDN
  • Avoid anonymous functions in JSX (creates new ref each render)
  • Normalize state to avoid deep renders
  • Use React DevTools Profiler to measure before optimizing

API Layer Architecture

Centralize all HTTP configuration in a single Axios instance. Add request/response interceptors for auth token injection, error normalization, and token refresh logic.

// services/api.ts — centralized Axios instance import axios from 'axios'; import { tokenService } from './tokenService'; const api = axios.create({ baseURL: process.env.REACT_APP_API_URL, timeout: 10000, headers: { 'Content-Type': 'application/json' }, }); // Request interceptor: inject auth token api.interceptors.request.use(config => { const token = tokenService.get(); if (token) config.headers.Authorization = `Bearer ${token}`; return config; }); // Response interceptor: handle 401 refresh api.interceptors.response.use( response => response.data, async error => { if (error.response?.status === 401 && !error.config._retry) { error.config._retry = true; const newToken = await tokenService.refresh(); error.config.headers.Authorization = `Bearer ${newToken}`; return api(error.config); } return Promise.reject(error); } ); export default api;
Domain API Modules
// api/products.ts import api from '../services/api'; export const productsApi = { getAll: (params) => api.get('/products', { params }), getById: (id) => api.get(`/products/${id}`), create: (data) => api.post('/products', data), update: (id, data) => api.put(`/products/${id}`, data), remove: (id) => api.delete(`/products/${id}`), };
React Query Integration
// hooks/useProducts.ts import { useQuery, useMutation } from '@tanstack/react-query'; import { productsApi } from '../api/products'; export function useProducts(params) { return useQuery({ queryKey: ['products', params], queryFn: () => productsApi.getAll(params), staleTime: 5 * 60 * 1000, // 5 minutes }); } export function useCreateProduct() { return useMutation({ mutationFn: productsApi.create, onSuccess: () => queryClient.invalidateQueries(['products']), }); }

Interview Questions

1. How would you structure a large React application? Explain feature-based vs type-based folders.
Feature-based groups all related code (components, hooks, state, API) by domain (auth/, products/, checkout/). Type-based groups by file type (components/, hooks/, store/). Feature-based scales better for large teams because each feature is self-contained and can be owned by a single team without cross-cutting concerns.
2. What are Micro Frontends and when would you use them?
Micro frontends apply microservices concepts to UI — multiple independently deployed frontend apps compose into one user experience. Use Module Federation (Webpack 5) to share components at runtime. Best for large organizations where multiple teams own distinct domains. Downsides: runtime overhead, shared dependency versioning, CORS complexity.
3. How do you decide between useState, Context, Redux, and React Query for state?
Server data → React Query (caching, refetching). Shared complex client state → Redux/Zustand. Simple shared state (theme, user) → Context. Form state → React Hook Form. URL-driven state → useSearchParams. UI-local state → useState. The key principle: keep state as close to where it's used as possible.
4. What is Context slicing and why does it matter for performance?
When a Context value changes, all consumers re-render. A large monolithic context (user + cart + theme + settings) causes every context consumer to re-render on any change. Slicing means separate contexts per concern — a theme change only re-renders ThemeContext consumers, not cart or user consumers. This is a critical optimization for apps with many context consumers.
5. How do you implement token refresh in an Axios interceptor?
The response interceptor catches 401 errors. It sets _retry = true on the config to prevent infinite retry loops, calls the refresh endpoint to get a new token, stores it, then retries the original request with the new Authorization header. If refresh also fails, redirect to login.
6. How does code splitting improve React app performance?
Code splitting (React.lazy + Suspense) breaks the JS bundle into chunks loaded on demand. Route-based splitting is the most impactful — users only download JS for the pages they visit. This reduces initial bundle size, improves Time-to-Interactive, and speeds up the critical rendering path. Measure with webpack-bundle-analyzer.
