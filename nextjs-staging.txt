While plain React is a UI library that handles only the view layer, Next.js is a full-stack framework that handles routing, data fetching, bundling (via Turbopack or Webpack), and deployment concerns. The framework integrates tightly with the Vercel deployment platform but can also be self-hosted on any Node.js server or exported as a static site.</p>

    </button>

  <div id="nextjs-q2" class="accordion-collapse collapse" aria-labelledby="heading-nextjs-q2" data-bs-parent="#acc-nextjs">
    <div class="accordion-body">
      <p>Server-Side Rendering (SSR) means that the HTML for a page is generated on the server on each incoming request, rather than in the browser. When a user navigates to an SSR page, the server fetches the necessary data, renders the React component tree to an HTML string, and sends the fully-formed HTML to the browser. The browser can display meaningful content immediately — before any JavaScript is downloaded or executed — which significantly improves Time to First Byte (TTFB) and First Contentful Paint (FCP) compared to a pure client-rendered Single Page Application (SPA).</p>
      <p>After the HTML is delivered, React "hydrates" the page by attaching event listeners and making it interactive. SSR is ideal for pages whose content changes frequently and must be personalized or real-time (e.g., a dashboard showing live data, a logged-in user's profile page). In the Pages Router, SSR is opted into via <code>getServerSideProps</code>. In the App Router, any Server Component that fetches data without caching is effectively SSR. The main tradeoff of SSR vs. SSG is that the server must do work on every request, increasing server load and TTFB relative to serving a pre-built static file.</p>
      <pre class="code-block">// Pages Router — SSR via getServerSideProps
// pages/dashboard.tsx
import type { GetServerSideProps, InferGetServerSidePropsType } from 'next';

interface DashboardProps { user: { name: string; email: string } }

export const getServerSideProps: GetServerSideProps&lt;DashboardProps&gt; = async (ctx) =&gt; {
  const session = await getSession(ctx.req);
  if (!session) return { redirect: { destination: '/login', permanent: false } };

  const user = await fetchUser(session.userId);
  return { props: { user } };
};

export default function Dashboard({ user }: InferGetServerSidePropsType&lt;typeof getServerSideProps&gt;) {
  return &lt;h1&gt;Welcome, {user.name}&lt;/h1&gt;;
}

// App Router — SSR is the default for Server Components with no-store cache
// app/dashboard/page.tsx
async function DashboardPage() {
  const user = await fetch('/api/user', { cache: 'no-store' }).then(r =&gt; r.json());
  return &lt;h1&gt;Welcome, {user.name}&lt;/h1&gt;;
}</pre>

    <div class="accordion-body">
      <p>Static Site Generation (SSG) means that HTML pages are pre-rendered at <strong>build time</strong> — when you run <code>next build</code> — rather than on each request. The generated HTML files are stored on disk (or in a CDN) and served directly to every user without any server-side computation per request. This makes SSG the fastest possible delivery method: a CDN serves a static file with extremely low latency, and the server never processes any logic per request. SSG is ideal for content that is the same for every user and does not change frequently — marketing pages, blog posts, documentation, product listings.</p>
      <p>In the Pages Router, a page uses SSG by default if it has no data-fetching function, or explicitly via <code>getStaticProps</code>. For pages with dynamic routes (e.g., <code>/blog/[slug]</code>), <code>getStaticPaths</code> tells Next.js which paths to pre-render. In the App Router, a Server Component that fetches data with the default <code>force-cache</code> behavior is statically generated. The practical limitation of SSG is staleness: if data changes after the build, the page shows outdated content until the next build. This is addressed by ISR (Incremental Static Regeneration).</p>
      <pre class="code-block">// Pages Router — SSG via getStaticProps
// pages/blog/[slug].tsx
import type { GetStaticProps, GetStaticPaths } from 'next';

export const getStaticPaths: GetStaticPaths = async () =&gt; {
  const posts = await fetchAllPosts();
  return {
    paths: posts.map(p =&gt; ({ params: { slug: p.slug } })),
    fallback: 'blocking', // ISR for new posts not built yet
  };
};

export const getStaticProps: GetStaticProps = async ({ params }) =&gt; {
  const post = await fetchPost(params!.slug as string);
  if (!post) return { notFound: true };
  return { props: { post }, revalidate: 60 }; // ISR: rebuild every 60s
};

// App Router — static by default
// app/blog/[slug]/page.tsx
export async function generateStaticParams() {
  const posts = await fetchAllPosts();
  return posts.map(p =&gt; ({ slug: p.slug }));
}

async function BlogPost({ params }: { params: { slug: string } }) {
  const post = await fetchPost(params.slug); // cached = static
  return &lt;article&gt;{post.content}&lt;/article&gt;;

    <div class="accordion-body">
      <p>Incremental Static Regeneration (ISR) is a hybrid rendering strategy that combines the performance of SSG with the freshness of SSR. With ISR, pages are statically generated at build time, but Next.js will regenerate individual pages in the background after a specified time interval (<code>revalidate</code> seconds) when a new request comes in. The first request after the interval expires triggers a background rebuild; the current (possibly stale) page is served to that visitor, and the freshly rebuilt page is stored and served to subsequent visitors. This is called the "stale-while-revalidate" pattern.</p>
      <p>ISR solves the primary pain point of SSG — stale content — without the per-request server cost of SSR. It is ideal for content that changes occasionally: e-commerce product pages, news articles, event listings. Next.js also supports <strong>on-demand ISR</strong>, where instead of a time-based interval, you explicitly trigger revalidation via an API call (e.g., from a CMS webhook), ensuring pages are always fresh immediately after content changes. In the App Router, on-demand ISR is handled via <code>revalidatePath()</code> and <code>revalidateTag()</code> from <code>next/cache</code>.</p>
      <pre class="code-block">// Pages Router ISR — time-based
export const getStaticProps: GetStaticProps = async ({ params }) =&gt; {
  const product = await fetchProduct(params!.id);
  return {
    props: { product },
    revalidate: 300, // regenerate at most every 5 minutes
  };
};

// On-demand ISR via API route (triggered by CMS webhook)
// pages/api/revalidate.ts
export default async function handler(req, res) {
  if (req.query.secret !== process.env.REVALIDATION_SECRET) {
    return res.status(401).json({ message: 'Invalid token' });
  }
  try {
    await res.revalidate(`/products/${req.query.id}`);
    return res.json({ revalidated: true });
  } catch (err) {
    return res.status(500).send('Error revalidating');
  }
}

// App Router — on-demand revalidation via Server Action or Route Handler
// app/api/revalidate/route.ts
import { revalidatePath, revalidateTag } from 'next/cache';
import { NextRequest } from 'next/server';

export async function POST(request: NextRequest) {
  const { path, tag } = await request.json();
  if (path) revalidatePath(path);
  if (tag) revalidateTag(tag);
  return Response.json({ revalidated: true, now: Date.now() });

    <div class="accordion-body">
      <p>Client-Side Rendering (CSR) means that the browser receives a minimal HTML shell with a JavaScript bundle, and React builds the entire page in the browser by executing JavaScript. The server sends no meaningful HTML content — typically just a <code>&lt;div id="root"&gt;&lt;/div&gt;</code> — and all rendering, data fetching, and UI construction happens on the client after the JavaScript loads. This is the model used by a standard Create React App (CRA) or Vite SPA setup. CSR results in a fast initial server response (just the shell HTML) but a slow Time to Interactive because the user sees a blank page or loading spinner until React bootstraps and fetches data.</p>
      <p>In Next.js, CSR is used selectively for portions of the page that are highly interactive, user-specific, or real-time — things like dashboards with live polling, rich form interfaces, or components that depend on browser-only APIs (<code>window</code>, <code>localStorage</code>). In the App Router, a Client Component (<code>'use client'</code>) that fetches data via <code>useEffect</code> or React Query is effectively CSR. Next.js also provides a <code>dynamic</code> wrapper with <code>ssr: false</code> to explicitly opt a component into CSR and skip it during SSR, which is necessary for components that cannot run in a Node.js environment.</p>
      <pre class="code-block">// Next.js — disable SSR for a component that uses browser-only APIs

const MapComponent = dynamic(() =&gt; import('@/components/Map'), {
  ssr: false,  // skip server rendering; render only in browser
  loading: () =&gt; &lt;div&gt;Loading map...&lt;/div&gt;,

export default function LocationPage() {
  return (

      &lt;h1&gt;Store Locations&lt;/h1&gt;
      &lt;MapComponent /&gt; {/* rendered only in the browser */}
    &lt;/div&gt;
  );
}

      <p>The App Router, introduced in Next.js 13 and stabilized in 13.4, is a completely new routing system built on top of React Server Components. It lives in the <code>app/</code> directory alongside (or instead of) the <code>pages/</code> directory. The App Router introduces several fundamental changes: components are Server Components by default (they run only on the server and never ship JavaScript to the client), data fetching happens directly inside components using async/await rather than special exported functions, layouts are defined with <code>layout.tsx</code> files that persist across navigations, and colocation of UI files (<code>loading.tsx</code>, <code>error.tsx</code>, <code>not-found.tsx</code>) is built into the routing convention.</p>
      <p>The App Router enables nested layouts, where each route segment can have its own persistent layout that does not remount on navigation. It also supports parallel routes and intercepting routes for complex UI patterns like modals that are bookmarkable. Route handlers (<code>route.ts</code>) replace API routes (<code>pages/api/</code>). The App Router is the future of Next.js and the recommended choice for new projects, but it has a steeper learning curve than the Pages Router, particularly around understanding Server vs. Client Components and the new caching model.</p>

    </button>
  </h2>
  <div id="nextjs-q8" class="accordion-collapse collapse" aria-labelledby="heading-nextjs-q8" data-bs-parent="#acc-nextjs">
    <div class="accordion-body">
      <p>React Server Components (RSC) are a new type of React component that execute exclusively on the server and never run in the browser. They were introduced by the React team as a first-class primitive (not a Next.js-specific feature), but Next.js App Router is currently the primary production implementation. Because RSCs run on the server, they can directly access databases, file systems, and backend services without an intermediate API layer. Their output — a serialized React component tree (RSC payload, not HTML) — is sent to the client, where it is stitched together with the rest of the UI without shipping the component's JavaScript code to the browser.</p>
      <p>The key insight of RSCs is the separation of "server concerns" from "client concerns." Components that display data, render markdown, query databases, or access server-only secrets can be Server Components — they have zero impact on the client JavaScript bundle size. Components that need interactivity (event handlers, state, effects, browser APIs) must be Client Components marked with <code>'use client'</code>. This enables a new architecture where the bulk of an application's logic stays on the server, and only the interactive leaves of the component tree are Client Components.</p>
      <pre class="code-block">// Server Component — runs ONLY on the server
// app/products/page.tsx (no 'use client' directive = Server Component by default)
import { db } from '@/lib/db'; // Direct DB access — impossible in client components!

async function ProductsPage() {
  // Fetch directly in the component — no useEffect, no API route needed
  const products = await db.product.findMany({
    where: { inStock: true },
    orderBy: { createdAt: 'desc' },
  });

  return (
    &lt;div&gt;
      &lt;h1&gt;Products&lt;/h1&gt;
      {/* ProductCard is also a Server Component */}
      {products.map(p =&gt; &lt;ProductCard key={p.id} product={p} /&gt;)}
      {/* AddToCartButton needs interactivity — must be Client Component */}
      &lt;AddToCartButton productId={products[0].id} /&gt;
    &lt;/div&gt;
  );
}

// Client Component — runs on server (for HTML) AND in the browser
// components/AddToCartButton.tsx
'use client';
import { useState } from 'react';

export function AddToCartButton({ productId }: { productId: string }) {
  const [added, setAdded] = useState(false);
  return (
    &lt;button onClick={() =&gt; { addToCart(productId); setAdded(true); }}&gt;
      {added ? 'Added!' : 'Add to Cart'}
    &lt;/button&gt;
  );
}</pre>
      <p>RSCs communicate with Client Components by passing serializable props (strings, numbers, plain objects, arrays). You cannot pass functions, class instances, or non-serializable values as props from a Server Component to a Client Component. Server Components can import Client Components, but Client Components cannot import Server Components (though they can receive them as <code>children</code> props — a critical pattern for composing the two).</p>

<div class="accordion-item">
  <h2 class="accordion-header" id="heading-nextjs-q9">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#nextjs-q9" aria-expanded="false" aria-controls="nextjs-q9">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;9. What is the difference between Server Components and Client Components in Next.js?
    </button>
  </h2>
  <div id="nextjs-q9" class="accordion-collapse collapse" aria-labelledby="heading-nextjs-q9" data-bs-parent="#acc-nextjs">
    <div class="accordion-body">
      <p>The distinction is about <strong>where the component executes and what capabilities it has access to</strong>. Server Components execute only on the server (at request time or build time), have access to server-side resources (databases, file system, environment variables, backend services), and their code is never included in the browser's JavaScript bundle. Client Components are marked with <code>'use client'</code>, their code ships to the browser, and they run on both the server (for the initial HTML render) and in the browser (for hydration and subsequent interactions).</p>
      <p>The practical rule is: default to Server Components, and reach for <code>'use client'</code> only when you need React state (<code>useState</code>), effects (<code>useEffect</code>), event handlers (<code>onClick</code>, <code>onChange</code>), browser APIs (<code>window</code>, <code>document</code>, <code>localStorage</code>), or third-party libraries that depend on these. Pushing the <code>'use client'</code> boundary as far down the component tree as possible maximizes the portion of your app that runs on the server, minimizing the client JavaScript bundle.</p>
      <pre class="code-block">// Feature comparison table implemented as code:

// SERVER COMPONENTS — can do:
// ✅ async/await at the component level
// ✅ Direct database queries
// ✅ Access process.env (including secrets)
// ✅ Import heavy server-only libraries (zero bundle cost)
// ✅ Fetch with automatic deduplication and caching

// SERVER COMPONENTS — cannot do:
// ❌ useState, useReducer, useRef
// ❌ useEffect, useLayoutEffect
// ❌ Event handlers (onClick, onChange)
// ❌ Browser APIs (window, document, localStorage)
// ❌ Context (createContext, useContext)

// CLIENT COMPONENTS — can do:
// ✅ All React hooks
// ✅ Event handlers
// ✅ Browser APIs
// ✅ Third-party libraries (Framer Motion, React Query, etc.)
// ✅ Real-time subscriptions

// CLIENT COMPONENTS — cannot do:
// ❌ Direct DB/filesystem access (must use API routes or Server Actions)
// ❌ Access server-only secrets directly</pre>
      <pre class="code-block">// Pattern: Server Component passes data as props to Client Component

  return &lt;ProfileEditor user={user} /&gt;; // Client Component handles interactions

// components/ProfileEditor.tsx (Client Component)

      <p><code>'use client'</code> is a React directive (a string literal at the top of a file) that marks a module as a Client Component and defines a boundary between Server and Client Component trees. import { getToken } from 'next-auth/jwt';

export async function middleware(request: NextRequest) {
  const { pathname } = request.nextUrl;

  // Authentication check for protected routes
  const token = await getToken({ req: request, secret: process.env.NEXTAUTH_SECRET });

  if (!token &amp;&amp; pathname.startsWith('/dashboard')) {
    const loginUrl = new URL('/login', request.url);
    loginUrl.searchParams.set('callbackUrl', pathname);
    return NextResponse.redirect(loginUrl);

      <p>Streaming in Next.js App Router leverages React 18's <code>renderToPipeableStream</code> / <code>renderToReadableStream</code> APIs to send HTML to the browser progressively as each piece of the UI is ready, rather than waiting for the entire page to render before sending anything. Traditionally, SSR is "all or nothing" — if one slow data fetch blocks the render, the user sees a blank screen until everything is ready. With streaming, fast parts of the page (the shell, navigation, static content) are sent and displayed immediately, while slow parts (data-heavy sections) stream in later as their data resolves.</p>
      <p>The mechanism works through React's <code>Suspense</code> component. The browser never needs to make an additional network request; everything arrives over the same initial HTTP connection via HTTP chunked transfer encoding.</p>

      <p>Both <code>layout.tsx</code> and <code>page.tsx</code> are special reserved files in the Next.js App Router, but they serve fundamentally different purposes and have different lifecycle behaviors. A <code>layout.tsx</code> is a shared UI wrapper that <strong>persists across navigations</strong> between its child routes — it renders once and does not remount or re-execute its data fetching when the user navigates between routes nested within it.</p>
      <p>This persistence is the critical distinction. Because layouts do not remount on child route navigation, state and data in layouts survives navigation. This enables patterns impossible in the Pages Router: a sidebar with its own scroll position, a chat widget that keeps its open/closed state, or a layout-level data fetch that provides data to all child pages without being repeated. The root layout (<code>app/layout.tsx</code>) must render the <code>&lt;html&gt;</code> and <code>&lt;body&gt;</code> tags and is required in every Next.js App Router project. Nested layouts are composed automatically by Next.js based on the directory structure.</p>

  <div id="nextjs-q25" class="accordion-collapse collapse" aria-labelledby="heading-nextjs-q25" data-bs-parent="#acc-nextjs">
    <div class="accordion-body">
      <p>The App Router has a multi-layer caching system that is one of its most powerful and most misunderstood features. Understanding all four cache layers and their interaction is essential for building correct, performant applications. The four caches are: <strong>Request Memoization</strong>, <strong>Data Cache</strong>, <strong>Full Route Cache</strong>, and <strong>Router Cache</strong>.</p>
      <p><strong>1. Request Memoization (in-memory, per-request):</strong> React automatically deduplicates identical <code>fetch()</code> calls within a single server render cycle. If three Server Components call <code>fetchUser(id)</code> with the same URL, only one network request is made. This memo is cleared after the request completes. It enables safe data fetching co-location — every component can fetch what it needs without coordinating with siblings.</p>
      <p><strong>2. Data Cache (persistent, cross-request):</strong> Next.js extends the native <code>fetch</code> API with a persistent server-side Data Cache. By default, <code>fetch()</code> responses are cached indefinitely (<code>force-cache</code>). <code>cache: 'no-store'</code> opts out entirely. <code>next: { revalidate: 3600 }</code> sets a time-based TTL. <code>next: { tags: ['posts'] }</code> enables on-demand invalidation via <code>revalidateTag('posts')</code>. This cache survives between server restarts in production deployments (stored in the filesystem or a distributed cache with platforms like Vercel).</p>
      <p><strong>3. Full Route Cache (static HTML, per-deploy):</strong> Routes that are statically renderable (no dynamic functions, all fetches cached) have their HTML and RSC payload pre-rendered and stored at build time. These are served from the CDN without any server execution. ISR revalidation or on-demand revalidation can update entries in this cache post-deploy.</p>
      <p><strong>4. Router Cache (client-side, per-session):</strong> The browser caches RSC payloads for visited and prefetched routes. Navigating back is instant (from cache). <code>router.refresh()</code> clears this cache for the current route and refetches from the server.</p>
      <pre class="code-block">// Controlling the Data Cache per fetch call
async function getData() {
  // Cached indefinitely (default) → Full Route Cache candidate
  const staticData = await fetch('https://api.example.com/config');

  // Never cached → Dynamic rendering, fresh on every request

  <div id="nextjs-q26" class="accordion-collapse collapse" aria-labelledby="heading-nextjs-q26" data-bs-parent="#acc-nextjs">
    <div class="accordion-body">
      <p>Partial Prerendering (PPR) is an experimental Next.js rendering model that combines static and dynamic rendering within a single route request, at the component level rather than the page level. Before PPR, a route was either entirely static (served from CDN instantly) or entirely dynamic (server-rendered on each request). PPR allows the static shell of a page (navigation, layout, static content) to be prerendered at build time and served from the CDN with the fastest possible TTFB, while dynamic "holes" (personalized content, real-time data) are streamed in from the server concurrently with the static shell delivery.</p>
      <p>PPR works by using React's <code>Suspense</code> boundaries as the signal for what is dynamic. The static shell (everything outside Suspense boundaries that has no dynamic dependencies) is built at compile time and stored as a static HTML+RSC payload. On a real request, the CDN serves the static shell immediately, and the server simultaneously begins computing the Suspense fallback content and streaming it in. The result is that users see the page shell in milliseconds (CDN speed) while personalized content arrives moments later via streaming — no hydration delay, no blank screen.</p>
      <pre class="code-block">// next.config.ts — enable PPR (experimental)
const config: NextConfig = {
  experimental: { ppr: 'incremental' }, // 'incremental' enables per-route opt-in
};

// app/product/[id]/page.tsx — opt this route into PPR
export const experimental_ppr = true;

import { Suspense } from 'react';

// Static shell — prerendered at build time, served from CDN
async function ProductDetails({ id }: { id: string }) {
  const product = await fetchProduct(id); // cached fetch → static
  return (
    &lt;div&gt;
      &lt;h1&gt;{product.name}&lt;/h1&gt;
      &lt;p&gt;{product.description}&lt;/p&gt;
      &lt;p&gt;${product.price}&lt;/p&gt;
    &lt;/div&gt;
  );
}

// Dynamic "hole" — computed at request time, streamed in
async function PersonalizedRecommendations({ productId }: { productId: string }) {
  const userId = await getCurrentUserId(); // reads cookies → dynamic
  const recommendations = await fetchRecommendations(userId, productId);
  return &lt;RecommendationList items={recommendations} /&gt;;
}

export default function ProductPage({ params }: { params: { id: string } }) {
  return (
    &lt;div&gt;
      {/* Static — served from CDN edge immediately */}
      &lt;ProductDetails id={params.id} /&gt;

      {/* Dynamic hole — skeleton shown, then streamed in */}
      &lt;Suspense fallback={&lt;RecommendationSkeleton /&gt;}&gt;
        &lt;PersonalizedRecommendations productId={params.id} /&gt;
      &lt;/Suspense&gt;
    &lt;/div&gt;
  );
}</pre>
      <p>PPR represents a paradigm shift: rather than choosing between the performance of static sites and the personalization of dynamic sites, it delivers both simultaneously. It is analogous to "static hosting with real-time islands" — similar in concept to Astro's Island Architecture but applied to React Server Components with streaming. PPR became a stable feature in Next.js 15 after extensive testing and is positioned as the default rendering model for future Next.js versions.</p>

    </button>

  <div id="nextjs-q29" class="accordion-collapse collapse" aria-labelledby="heading-nextjs-q29" data-bs-parent="#acc-nextjs">
    <div class="accordion-body">
      <p>Next.js offers two mechanisms for invalidating cached data in the Data Cache and Full Route Cache: <strong>time-based revalidation</strong> (ISR) and <strong>tag-based on-demand revalidation</strong>. They solve different problems and are often used together. Time-based revalidation sets a maximum age (TTL) for cached data — after the interval expires, the next request triggers a background regeneration using the stale-while-revalidate pattern. Tag-based revalidation allows you to explicitly and immediately invalidate specific cached data at any time, typically triggered by a CMS webhook, database mutation, or user action.</p>
      <p><strong>Time-based revalidation</strong> (<code>revalidate: N</code>) is set per fetch call or per route segment. It guarantees that data is never more than N seconds stale. It requires no external trigger — the system manages it automatically. The downside is that content remains stale until the interval expires even if the data changed seconds after the last regeneration. This is acceptable for content that updates on a schedule (daily blog posts, weekly pricing updates).</p>
      <p><strong>Tag-based revalidation</strong> (<code>revalidateTag('tag')</code>) lets you invalidate all fetches and routes that were tagged with a specific string, immediately, on demand. When a CMS publishes a new article, a webhook calls an API route that calls <code>revalidateTag('articles')</code>, which instantly marks all cached fetches tagged <code>'articles'</code> and all Full Route Cache entries that depend on them as stale. The next request to any affected route triggers a fresh regeneration. This enables true content freshness without polling overhead.</p>
      <pre class="code-block">// Setting up tagged fetches

    next: {
      tags: ['posts', 'content'],   // multiple tags for granular control
      revalidate: 3600,              // fallback TTL if tag never invalidated
    },
  }).then(r =&gt; r.json());
}

  return fetch(`https://cms.example.com/posts/${slug}`, {
    next: {
      tags: [`post:${slug}`, 'posts'],  // per-item tag for surgical invalidation
      revalidate: 86400,
    },

      <pre class="code-block">// On-demand tag revalidation — called from CMS webhook
// app/api/webhook/cms/route.ts
import { revalidateTag, revalidatePath } from 'next/cache';
import { NextRequest, NextResponse } from 'next/server';

export async function POST(request: NextRequest) {
  const secret = request.headers.get('x-webhook-secret');
  if (secret !== process.env.WEBHOOK_SECRET) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
  }

  const { event, data } = await request.json();

  switch (event) {
    case 'post.published':
    case 'post.updated':
      revalidateTag(`post:${data.slug}`); // invalidate specific post
      revalidateTag('posts');              // invalidate post listings
      revalidatePath('/blog');            // invalidate Full Route Cache for /blog
      revalidatePath(`/blog/${data.slug}`, 'page');
      break;

    case 'post.deleted':
      revalidateTag('posts');
      revalidatePath('/blog');
      break;
  }

  return NextResponse.json({ revalidated: true, timestamp: Date.now() });
