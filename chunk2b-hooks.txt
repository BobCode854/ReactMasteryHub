    <div class="accordion-body">
      A <code>useAsync</code> hook abstracts the async state machine pattern: idle &rarr; loading &rarr; success/error. It manages the <code>status</code>, <code>data</code>, and <code>error</code> state and provides an <code>execute</code> function the caller invokes with any arguments. It should handle cancellation via a mounted flag to prevent state updates after unmount, offer a reset capability, and optionally execute immediately on mount when passed <code>immediate = true</code>.       <pre class="code-block">function useAsync(asyncFn, immediate = false) {
  const [state, setState] = React.useState({
    status: 'idle', data: null, error: null,
  });

  const execute = React.useCallback(async (...args) =&gt; {
    setState({ status: 'loading', data: null, error: null });
    try {
      const data = await asyncFn(...args);
      setState({ status: 'success', data, error: null });
      return data;
    } catch (error) {
      setState({ status: 'idle', data: null, error });
      throw error;
    }
  }, [asyncFn]);

  React.useEffect(() =&gt; {
    if (immediate) execute();
  }, [execute, immediate]);

  return { ...state, execute, isLoading: state.status === 'loading' };
}

    <div class="accordion-body">
      Infinite scroll loads more content when the user approaches the bottom of the list. The pattern combines <code>IntersectionObserver</code> (to detect when a sentinel element becomes visible) with a data-fetching callback. The hook returns a ref to attach to the sentinel element placed at the bottom of the list. When the sentinel enters the viewport, the observer fires, the hook sets an <code>isFetching</code> guard to prevent concurrent fetches, calls <code>fetchMore</code>, and resets the guard when the fetch completes. Cleanup disconnects the observer to prevent memory leaks and stale callbacks on unmount or URL change.
      <pre class="code-block">function useInfiniteScroll(fetchMore, { threshold = 0.1, rootMargin = '200px' } = {}) {
  const sentinelRef = React.useRef(null);
  const [isFetching, setIsFetching] = React.useState(false);

  React.useEffect(() =&gt; {
    const sentinel = sentinelRef.current;
    if (!sentinel) return;

    const observer = new IntersectionObserver(
      async ([entry]) =&gt; {
        if (entry.isIntersecting &amp;&amp; !isFetching) {
          setIsFetching(true);
          try { await fetchMore(); }
          finally { setIsFetching(false); }
        }
      },
      { threshold, rootMargin }
    );
    observer.observe(sentinel);
    return () =&gt; observer.disconnect();
  }, [fetchMore, isFetching, threshold, rootMargin]);

  return { sentinelRef, isFetching };
}

      Optimistic updates immediately update the UI with the expected result before the server confirms, rolling back if the request fails. This creates a highly responsive feel — the UI responds instantly to user actions without waiting for network round-trips. The pattern has three steps: update local state immediately with the expected result, fire the async request in the background, and revert to the original state if the request fails. A <code>rollback</code> callback captures the previous state via closure at the time of the interaction. Libraries like React Query and SWR provide first-class optimistic update support via their <code>mutate</code> APIs, but the custom hook pattern shown here captures the essential mechanics.
      <pre class="code-block">function useOptimisticUpdate(mutationFn) {
  const [error, setError] = React.useState(null);

  const mutate = React.useCallback(async ({
    optimisticUpdate,
    rollback,
    variables,
  }) =&gt; {
    optimisticUpdate(); // immediately update UI
    setError(null);
    try {
      await mutationFn(variables);
    } catch (err) {
      rollback(); // revert on error
      setError(err);
    }
  }, [mutationFn]);

  return { mutate, error };
}

  React.useEffect(() =&gt; {
    ref.current = value; // updates AFTER render, so ref.current is previous value during render
  });

  return ref.current; // returns value from previous render
}

      <pre class="code-block">function useMutation(mutationFn, options = {}) {
  const { onSuccess, onError, onSettled } = options;
  const [state, setState] = React.useState({
    data: null, error: null, status: 'idle',
  });

  const mutate = React.useCallback(async (variables) =&gt; {
    setState({ data: null, error: null, status: 'loading' });
    try {
      const data = await mutationFn(variables);
      setState({ data, error: null, status: 'success' });
      onSuccess?.(data, variables);
      onSettled?.(data, null, variables);
      return data;
    } catch (error) {
      setState({ data: null, error, status: 'error' });
      onError?.(error, variables);
      onSettled?.(null, error, variables);
      throw error;
    }
  }, [mutationFn, onSuccess, onError, onSettled]);

  return { ...state, mutate, isLoading: state.status === 'loading' };
}

    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#hooks-q52">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is the difference between useState and useRef for tracking values?

      Both <code>useState</code> and <code>useRef</code> can persist values across renders, but they differ in one critical way: <code>useState</code> triggers a re-render when its value changes, while <code>useRef</code> does not trigger any re-render when <code>ref.current</code> is mutated. Use <code>useState</code> when a value change needs to update the rendered output — for anything the user sees. Use <code>useRef</code> when you need to track a value without causing re-renders, such as whether a component is currently mounted, an interval or timeout ID, the latest version of a callback for use inside an effect, or a DOM element reference. A common mistake is using <code>useRef</code> for values that should drive the UI — the component will never re-render when the ref changes, so the display will be stale. The opposite mistake is using <code>useState</code> for interval IDs or other bookkeeping values — this causes unnecessary re-renders and can trigger infinite loops when the state update is inside an effect.

      <code>useToggle</code> is a simple but frequently-used custom hook that manages a boolean state and provides stable <code>toggle</code>, <code>setTrue</code>, and <code>setFalse</code> callbacks. It is one of the most commonly written custom hooks in React codebases because boolean state (open/closed, visible/hidden, enabled/disabled) appears in nearly every feature. Memoizing the callbacks with <code>useCallback</code> and empty dependency arrays ensures the function references are stable across renders, preventing child components wrapped in <code>React.memo</code> from re-rendering unnecessarily when the parent re-renders for unrelated reasons. The hook returns both the value and the controls as a tuple, following the same convention as <code>useState</code>.
      <pre class="code-block">function useToggle(initialValue = false) {
  const [value, setValue] = React.useState(initialValue);

  const toggle = React.useCallback(() =&gt; setValue(v =&gt; !v), []);
  const setTrue = React.useCallback(() =&gt; setValue(true), []);
  const setFalse = React.useCallback(() =&gt; setValue(false), []);

  return [value, { toggle, setTrue, setFalse }];

      React <code>StrictMode</code> double-invokes certain functions in development — component render bodies, state initializer functions, reducer functions, and the callbacks passed to <code>useState</code> — and additionally mounts, unmounts, and remounts each component so that <code>useEffect</code> runs twice (mount, cleanup, mount again).   <div id="hooks-q67" class="accordion-collapse collapse" data-bs-parent="#acc-hooks">
    <div class="accordion-body">
      React Fiber is React's internal reconciliation engine, rewritten from scratch in React 16, which replaced the original stack-based reconciler. Hooks are implemented as a singly linked list of hook nodes attached to each Fiber node — the Fiber for a component instance holds a pointer to the first hook, and each hook node has a pointer to the next. This linked-list implementation is why the Rules of Hooks exist: on every render, React iterates the hooks linked list in order, matching the Nth call to <code>useState</code> with the Nth hook node stored on the fiber. If hooks are called conditionally, the number or order of hook calls can change between renders, causing React to incorrectly match a hook call to the wrong node — resulting in state corruption and unpredictable behavior. The Fiber architecture also enables React's concurrent features by making rendering interruptible — React can pause work on a low-priority tree, process a high-priority user interaction, and then resume the paused work, which was impossible with the synchronous call-stack model of the old reconciler.

    let frameId;

    const loop = (time) =&gt; {
      const deltaTime = time - lastTime;
      lastTime = time;
      callbackRef.current(deltaTime, time);
      frameId = requestAnimationFrame(loop);
    };

    frameId = requestAnimationFrame(loop);
    return () =&gt; cancelAnimationFrame(frameId);

  <div id="hooks-q78" class="accordion-collapse collapse" data-bs-parent="#acc-hooks">
    <div class="accordion-body">
      For a component to trigger a Suspense boundary, it must throw a Promise while loading — Suspense catches the thrown Promise, shows the fallback, and retries the component when the Promise resolves. This means the fetching mechanism must cache both the promise and its resolved result so that on re-render the component either throws the same pending promise again, throws the error, or returns the cached data. This "fetch-then-throw" pattern (sometimes called the render-as-you-fetch pattern) requires careful implementation of the cache to avoid re-fetching on every render. React 19's <code>use()</code> hook handles this automatically for any Promise. For React 18, libraries like React Query and SWR provide battle-tested Suspense integration via their <code>suspense: true</code> configuration option, which is strongly preferred over implementing the cache mechanism manually.

function createResource(promise) {
  let status = 'pending';
  let result;
  const suspender = promise.then(
    (data) =&gt; { status = 'success'; result = data; },
    (error) =&gt; { status = 'error'; result = error; }

  return {
    read() {
      if (status === 'pending') throw suspender; // Suspense catches this
      if (status === 'error') throw result;      // ErrorBoundary catches this
      return result;                             // success: return data
    }
  };
}

    <div class="accordion-body">
      A component managing multiple resources — WebSocket, polling interval, fetch abort controller, event listeners — needs every resource cleaned up when the component unmounts or when a key dependency changes. The safest pattern is to set up all related resources within a single <code>useEffect</code> and tear them all down in its cleanup function, treating them as an atomic group. Alternatively, each resource can live in its own <code>useEffect</code> with its own cleanup, which is more maintainable when each resource has a different dependency key. Always use an <code>AbortController</code> to cancel in-flight fetch requests on cleanup — attempting to update state after unmount generates React warnings and can cause subtle bugs. Using a cleanup function's local closure to flag whether the effect is still active is the other common pattern for preventing stale state updates.

    <div class="accordion-body">
      Large components with mixed concerns — data fetching, data transformations, UI interaction state, side effects, and derived values — should be refactored by identifying each distinct responsibility and extracting the state and effects for each into a focused custom hook. The process: (1) identify the distinct responsibilities in the component (product data, reviews, cart state, wishlist state, related products); (2) extract each concern into a hook named after its domain; (3) give each hook a single, clear responsibility; (4) compose the hooks in the component, which becomes a thin orchestration layer that passes data and callbacks to presentational components. The result is that each hook is independently testable, reusable across other components in the application, easy to reason about in isolation, and the component itself shrinks from 150+ lines to 15 lines. This is the "separation of concerns" principle applied to the hooks model.
