  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#state-q1">
      <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is Redux?
    </button>
  </h2>
  <div id="state-q1" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      Redux is a predictable state container for JavaScript applications, most commonly used with React. It centralizes all application state in a single JavaScript object called the store, making state changes traceable and debuggable. Redux follows three core principles: a single source of truth (one store), state is read-only (only actions can trigger changes), and changes are made with pure functions (reducers). The Redux DevTools extension enables powerful debugging features including time-travel (replaying past states). Redux was created by Dan Abramov and Andrew Clark in 2015, inspired by the Flux architecture and Elm's functional state management.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#state-q2">
      <span class="diff-easy">Easy</span>&nbsp;&nbsp;What are the three principles of Redux?
    </button>
  </h2>
  <div id="state-q2" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      The three fundamental principles of Redux are: 1) Single Source of Truth — the entire application state is stored in a single JavaScript object tree within one store, making it easy to inspect and debug the full application state at any point. 2) State is Read-Only — the only way to change state is to dispatch an action (a plain object describing what happened); you cannot directly mutate the state. 3) Changes are Made with Pure Functions — to specify how the state tree is transformed by actions, you write pure reducer functions that take the previous state and an action, and return the next state deterministically. These principles together make state changes explicit, predictable, and reversible.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#state-q3">
      <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is an action in Redux?
    </button>
  </h2>
  <div id="state-q3" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      An action in Redux is a plain JavaScript object that describes an intent to change the state. Every action must have a <code>type</code> property — a string constant that identifies the kind of change being made (e.g., <code>'ADD_TODO'</code>, <code>'INCREMENT_COUNTER'</code>). Actions can optionally carry additional data in a <code>payload</code> property (the Flux Standard Action convention) that provides the information needed to make the state change. Actions are the only way to get data into the Redux store — you dispatch actions to the store, and the reducers process them. Action creators are functions that return action objects, making it easy to create actions programmatically and keeping the action structure consistent across your codebase.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#state-q4">
      <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is a reducer in Redux?
    </button>
  </h2>
  <div id="state-q4" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      A reducer is a pure function that takes the current state and an action as arguments, and returns the next state. Reducers must be pure: given the same inputs they always produce the same output, and they produce no side effects — no API calls, no mutations of the input state, no random values. When an action is dispatched, Redux passes the current state and the action to the root reducer, which typically delegates to smaller "slice" reducers responsible for portions of the state tree. The convention is to return the previous state unchanged for unknown action types (<code>default: return state</code>). Reducers must treat state as immutable — they must return new objects/arrays rather than mutating existing ones.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#state-q5">
      <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is the Redux store?
    </button>
  </h2>
  <div id="state-q5" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      The Redux store is the central object that holds the application's state tree, provides methods to dispatch actions and subscribe to state changes, and manages the root reducer. You create a store with <code>configureStore</code> (Redux Toolkit) or the legacy <code>createStore</code>. The store exposes three key methods: <code>getState()</code> (returns the current state), <code>dispatch(action)</code> (sends an action through the reducer and updates state), and <code>subscribe(listener)</code> (registers a callback called after every state change). There is only one store in a Redux application (single source of truth). In React apps, the store is provided to the component tree via the <code>&lt;Provider store={store}&gt;</code> component from <code>react-redux</code>.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#state-q6">
      <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is Redux Toolkit and why was it created?
    </button>
  </h2>
  <div id="state-q6" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      Redux Toolkit (RTK) is the official, opinionated toolset for Redux development that solves the major pain points of vanilla Redux. The problems RTK solves: too much boilerplate (separate action type constants, action creators, and reducers for every feature), easy to accidentally mutate state (RTK uses Immer under the hood so you can write "mutating" code safely), configuring middleware and DevTools was complex, and writing normalized state and async logic required multiple additional packages. RTK provides <code>configureStore</code> (sets up store with sensible defaults), <code>createSlice</code> (generates action creators and reducers together), <code>createAsyncThunk</code> (handles async logic), and <code>createEntityAdapter</code> (normalized state management). The Redux team officially recommends RTK for all new Redux projects.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#state-q7">
      <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is createSlice in Redux Toolkit?
    </button>
  </h2>
  <div id="state-q7" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      <code>createSlice</code> is a function from Redux Toolkit that generates a Redux slice — a combination of a reducer function, action creators, and action types — all from a single configuration object. You provide a <code>name</code> (used to namespace actions), <code>initialState</code>, and a <code>reducers</code> object where each key becomes both a reducer case and an action creator. RTK's Immer integration means reducer functions inside <code>createSlice</code> can use "mutating" syntax (<code>state.value++</code>) and Immer will produce an immutable update under the hood. The returned slice object has <code>.reducer</code> (the generated reducer function to pass to <code>configureStore</code>) and <code>.actions</code> (the generated action creators).
      <pre class="code-block">import { createSlice } from '@reduxjs/toolkit';

const counterSlice = createSlice({
  name: 'counter',
  initialState: { value: 0, status: 'idle' },
  reducers: {
    increment: (state) =&gt; { state.value += 1; },   // Immer: looks like mutation, produces immutable
    decrement: (state) =&gt; { state.value -= 1; },
    incrementByAmount: (state, action) =&gt; { state.value += action.payload; },
  },
});

export const { increment, decrement, incrementByAmount } = counterSlice.actions;
export default counterSlice.reducer;</pre>
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#state-q8">
      <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is createAsyncThunk?
    </button>
  </h2>
  <div id="state-q8" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      <code>createAsyncThunk</code> is a Redux Toolkit utility for creating async action creators that automatically dispatch <code>pending</code>, <code>fulfilled</code>, and <code>rejected</code> action types based on the lifecycle of a Promise. You provide an action type string prefix and a "payload creator" — an async function that returns a Promise. RTK dispatches <code>prefix/pending</code> when the thunk starts, <code>prefix/fulfilled</code> when the Promise resolves (with the resolved value as payload), and <code>prefix/rejected</code> when it rejects (with the error as payload). You handle these lifecycle actions in <code>extraReducers</code> inside <code>createSlice</code> using the <code>builder.addCase</code> API. This eliminates the boilerplate of manually dispatching multiple actions for async operations.
      <pre class="code-block">import { createAsyncThunk, createSlice } from '@reduxjs/toolkit';

export const fetchUser = createAsyncThunk('user/fetch', async (userId) =&gt; {
  const response = await fetch(`/api/users/${userId}`);
  return response.json();
});

const userSlice = createSlice({
  name: 'user',
  initialState: { data: null, loading: false, error: null },
  reducers: {},
  extraReducers: (builder) =&gt; {
    builder
      .addCase(fetchUser.pending, (state) =&gt; { state.loading = true; })
      .addCase(fetchUser.fulfilled, (state, action) =&gt; {
        state.loading = false;
        state.data = action.payload;
      })
      .addCase(fetchUser.rejected, (state, action) =&gt; {
        state.loading = false;
        state.error = action.error.message;
      });
  },
});</pre>
    </div>
  </div>
</div>

  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#state-q9">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is the difference between Redux and the Context API?
    </button>
  </h2>
  <div id="state-q9" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      Redux and React Context both solve the problem of sharing data across components without prop drilling, but they are architecturally different. Context is a built-in React feature that provides a way to broadcast a value to all descendant consumers; it has no built-in mechanism for structured state updates, middleware, selectors, or DevTools integration. Redux is a dedicated state management library with a strict unidirectional data flow pattern: actions → reducers → new state → re-render. Redux has a rich middleware system (for async, logging, analytics), the Redux DevTools with time-travel debugging, and selectors (reselect) for derived state memoization. Context re-renders all consumers when the provided value changes; Redux (via react-redux's <code>useSelector</code>) only re-renders components when the selected slice of state changes. For complex, frequently-updated global state, Redux's optimized subscriptions outperform Context.
      <pre class="code-block">// Context: re-renders all consumers on any value change
const GlobalContext = React.createContext(null);
function Provider({ children }) {
  const [state, setState] = React.useState({ user: null, theme: 'light', count: 0 });
  return &lt;GlobalContext.Provider value={{ state, setState }}&gt;{children}&lt;/GlobalContext.Provider&gt;;
}
// Any change to state re-renders ALL consumers — even if they only use `theme`

// Redux: component only re-renders when selected slice changes
function ThemeToggle() {
  const theme = useSelector(state =&gt; state.ui.theme); // only re-renders on theme change
  return &lt;button&gt;{theme}&lt;/button&gt;;
}</pre>
    </div>
  </div>

  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#state-q10">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;When should you use Redux vs Context?
    </button>
  </h2>
  <div id="state-q10" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      Use Context for: low-frequency updates (theme, language, auth user, feature flags) that many components need but that rarely change; small-to-medium apps where the overhead of Redux isn't justified; data that's truly global but doesn't need complex update logic. Use Redux for: complex state with many types of updates and interactions; data that updates frequently and is consumed by many components (performance matters); when you need middleware for side effects (API calls, analytics, WebSockets); when time-travel debugging and action logging are important for your team; when you have complex async workflows; large teams where a strict update pattern reduces bugs. A rough heuristic: if you find yourself writing complex useEffect chains to synchronize Context state, Redux would likely be cleaner. The two are not mutually exclusive — many apps use Redux for server/feature state and Context for static configuration like themes.
    </div>
  </div>
</div>

  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#state-q11">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is Immer and how does Redux Toolkit use it?
    </button>
  </h2>
  <div id="state-q11" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      Immer is a JavaScript library that allows you to write "mutating" code that actually produces immutable updates. It uses JavaScript Proxies to intercept mutations on a draft object, recording every change, and then producing a new immutable state based on those recorded mutations. Redux Toolkit wraps all <code>createSlice</code> reducer functions with Immer automatically, so you can write <code>state.user.name = 'Alice'</code> inside a reducer and get an immutable update without manually spreading every level of nesting. This dramatically reduces the verbosity of writing reducers for deeply nested state. Immer also detects if you accidentally return a new value AND mutate the draft, throwing an error to prevent bugs. You can use Immer independently of Redux Toolkit via the <code>produce</code> function.
      <pre class="code-block">// Without Immer (verbose, error-prone)
function reducer(state, action) {
  return {
    ...state,
    users: {
      ...state.users,
      [action.userId]: {
        ...state.users[action.userId],
        profile: {
          ...state.users[action.userId].profile,
          name: action.name,
        }
      }
    }
  };
}

// With Immer (via createSlice) — same result, much simpler
updateUserName: (state, action) =&gt; {
  state.users[action.payload.userId].profile.name = action.payload.name;
}</pre>
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#state-q12">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is a selector in Redux? What is reselect?
    </button>
  </h2>
  <div id="state-q12" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      A selector is a function that derives data from the Redux store state. The simplest selectors just read a property: <code>state =&gt; state.users.list</code>. More complex selectors compute derived data: filtering, sorting, combining multiple parts of state. Reselect is a library (bundled with RTK) that creates memoized selectors using <code>createSelector</code>. A memoized selector only recomputes when its input selectors produce new values (using reference equality), caching the result otherwise. This prevents unnecessary re-renders in components using <code>useSelector</code> — if a complex derived value (like a filtered, sorted list) would produce the same output, reselect returns the cached result with the same reference, and <code>useSelector</code> skips the re-render. RTK exports <code>createSelector</code> directly from <code>@reduxjs/toolkit</code>.
      <pre class="code-block">import { createSelector } from '@reduxjs/toolkit';

// Input selectors (simple, cheap)
const selectAllTodos = (state) =&gt; state.todos.items;
const selectFilter = (state) =&gt; state.todos.filter;

// Memoized derived selector — only recomputes when todos or filter changes
export const selectFilteredTodos = createSelector(
  [selectAllTodos, selectFilter],
  (todos, filter) =&gt; {
    if (filter === 'all') return todos;
    return todos.filter(t =&gt; t.status === filter);
  }
);

// Component: only re-renders when filtered result changes
function TodoList() {
  const todos = useSelector(selectFilteredTodos);
  return &lt;ul&gt;{todos.map(t =&gt; &lt;li key={t.id}&gt;{t.text}&lt;/li&gt;)}&lt;/ul&gt;;
}</pre>
    </div>
  </div>
</div>

  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#state-q13">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is normalized state and why is it important?
    </button>
  </h2>
  <div id="state-q13" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      Normalized state means storing entities (users, posts, comments) as flat lookup tables (objects keyed by ID) instead of nested arrays of objects. For example, instead of <code>[{ id: 1, name: 'Alice', posts: [{ id: 10, title: '...' }] }]</code>, you store <code>{ users: { 1: { id: 1, name: 'Alice', postIds: [10] } }, posts: { 10: { id: 10, title: '...' } } }</code>. This avoids duplication (the same user appearing in multiple places), makes updates O(1) by ID (no array search), prevents complex cascade updates when an entity changes, and mirrors how a database organizes data. Redux Toolkit's <code>createEntityAdapter</code> provides normalized state utilities out of the box. The normalized shape (inspired by <code>normalizr</code>) is the recommended pattern for any Redux store containing relational data.
      <pre class="code-block">// Normalized state shape
{
  users: {
    ids: [1, 2, 3],
    entities: {
      1: { id: 1, name: 'Alice' },
      2: { id: 2, name: 'Bob' },
    }
  },
  posts: {
    ids: [10, 11],
    entities: {
      10: { id: 10, title: 'Hello', authorId: 1 },
    }
  }
}

// O(1) update — no need to find item in array
state.users.entities[userId].name = newName;</pre>
    </div>
  </div>

  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#state-q14">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is RTK Query?
    </button>
  </h2>
  <div id="state-q14" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      RTK Query is a powerful data fetching and caching solution built into Redux Toolkit. It eliminates the boilerplate of writing thunks, reducers, and loading/error state for every API endpoint. You define an API using <code>createApi</code> with a <code>baseQuery</code> (typically <code>fetchBaseQuery</code> wrapping fetch) and a set of <code>endpoints</code> (queries for GET-like operations, mutations for POST/PUT/DELETE). RTK Query automatically generates React hooks (e.g., <code>useGetUsersQuery</code>, <code>useCreateUserMutation</code>) that handle caching, loading states, background refetching, invalidation, and optimistic updates. Data is stored in Redux and can be inspected with DevTools. RTK Query reduces a typical CRUD feature from ~200 lines of Redux code to ~30 lines.
      <pre class="code-block">import { createApi, fetchBaseQuery } from '@reduxjs/toolkit/query/react';

const api = createApi({
  reducerPath: 'api',
  baseQuery: fetchBaseQuery({ baseUrl: '/api' }),
  tagTypes: ['User'],
  endpoints: (builder) =&gt; ({
    getUsers: builder.query({ query: () =&gt; '/users', providesTags: ['User'] }),
    createUser: builder.mutation({
      query: (user) =&gt; ({ url: '/users', method: 'POST', body: user }),
      invalidatesTags: ['User'], // automatically refetch users after creation
    }),
  }),
});

export const { useGetUsersQuery, useCreateUserMutation } = api;

// Component
function UserList() {
  const { data, isLoading } = useGetUsersQuery();
  if (isLoading) return &lt;Spinner /&gt;;
  return &lt;ul&gt;{data.map(u =&gt; &lt;li key={u.id}&gt;{u.name}&lt;/li&gt;)}&lt;/ul&gt;;
}</pre>

  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#state-q15">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is the difference between RTK Query and React Query?
    </button>
  </h2>
  <div id="state-q15" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      Both RTK Query and React Query (TanStack Query) are data-fetching libraries that solve caching, loading states, and synchronization, but they differ in integration and scope. RTK Query is built into Redux Toolkit and stores its cache in Redux — state is inspectable via Redux DevTools, and you can combine server state with Redux client state seamlessly; it is ideal if you're already using Redux. React Query is framework-agnostic (works without Redux), has its own cache independent of Redux, and has a larger ecosystem, more advanced features (infinite queries, optimistic updates, SSR hydration), and better TypeScript ergonomics. React Query is often preferred in non-Redux apps or when a standalone data-fetching solution is preferred. RTK Query wins when deep Redux integration is valuable. Both significantly outperform manual useEffect fetching patterns.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#state-q16">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is Zustand?
    </button>
  </h2>
  <div id="state-q16" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      Zustand is a small, fast, and flexible state management library for React that uses hooks as its primary API. You define a store using <code>create</code>, providing a setup function that receives <code>set</code> and <code>get</code> helpers and returns an object with state and actions. Components subscribe to the store via the generated hook, optionally passing a selector to subscribe to only a portion of the state — preventing re-renders when unrelated state changes. Zustand requires no Provider wrapper, works outside React components (accessing state from utility functions), and supports middleware for persistence, DevTools integration, and Immer-style mutations. Its tiny bundle size (~1KB) and minimal boilerplate make it popular for small-to-medium apps that need global state without Redux's complexity. The lack of enforced patterns is both a feature (flexibility) and a risk (inconsistency in larger teams).
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#state-q17">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is Jotai?
    </button>
  </h2>
  <div id="state-q17" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      Jotai is an atomic state management library for React inspired by Recoil and the Atom model. State is broken into small, independent atoms (think: individual useState values at the global level), and components subscribe to only the atoms they need. Atoms can be composed: derived atoms compute their value from other atoms, similar to reselect selectors but with automatic dependency tracking. Jotai is extremely lightweight, requires no Provider (it uses a default global store), and avoids the context re-render problem since components only re-render when their specific atoms change. It works well for fine-grained, per-entity state (e.g., per-row selected state in a table). The atom model encourages splitting state into small, focused units rather than one large state tree, making incremental adoption alongside existing solutions very easy.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#state-q18">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is Recoil?
    </button>
  </h2>
  <div id="state-q18" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      Recoil is Facebook's experimental atomic state management library for React. Like Jotai, it uses atoms (units of state) and selectors (derived state functions that auto-track atom dependencies). Components use <code>useRecoilState</code> (like useState but global), <code>useRecoilValue</code> (read-only), and <code>useSetRecoilState</code> (write-only). Recoil's key differentiator is its support for asynchronous selectors — selectors can return Promises, integrating with Suspense for data loading. It also has advanced features like atom families (parameterized atoms), snapshots (capturing full state for debugging/undo), and transactions. Recoil is still in experimental status (as of 2024) and less production-battle-tested than Redux or Zustand, but it demonstrates interesting ideas about fine-grained reactive state that influenced libraries like Jotai.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#state-q19">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is the Flux architecture?
    </button>
  </h2>
  <div id="state-q19" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      Flux is the application architecture pattern that Redux is based on, developed by Facebook as an alternative to MVC for client-side web applications. The core Flux pattern is unidirectional data flow: Action → Dispatcher → Store → View → (user interaction creates) → Action. In Flux, the Dispatcher is a central hub that receives all actions and broadcasts them to every registered store. Stores contain application state and logic and update themselves in response to dispatched actions. Views (React components) render based on store data and dispatch actions on user interaction. Redux simplified Flux by replacing the Dispatcher with pure reducer functions and the multiple stores with a single store. Flux ensures data flows in one direction, making state changes predictable and easier to debug compared to bidirectional data binding in MVC frameworks.
    </div>
  </div>
</div>

  </h2>
  <div id="state-q20" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      Redux middleware is code that sits between the dispatching of an action and the moment it reaches the reducer. Middleware can intercept actions, modify them, dispatch additional actions, perform async operations, or cancel them entirely. The middleware API is a chain of higher-order functions: <code>store =&gt; next =&gt; action =&gt; { ... next(action) ... }</code>.   console.log('State before:', store.getState());

  console.log('State after:', store.getState());
  return result;
};

// Apply middleware

  reducer: rootReducer,

  <div id="state-q22" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      Both redux-saga and redux-thunk handle async side effects in Redux, but with very different models. Redux-thunk uses plain async/await functions — simple, widely understood, good for straightforward async operations. The trade-off: sagas have a steep learning curve (generators, saga effects). Use thunks for simple async; use sagas for complex orchestration (wizards, polling, WebSocket management, task cancellation).
      <pre class="code-block">// redux-saga — generator functions, highly testable

    yield put({ type: 'user/fetchFulfilled', payload: user });
  } catch (error) {
    yield put({ type: 'user/fetchRejected', error });
  }
}

function* rootSaga() {
  yield takeEvery('user/fetchRequested', fetchUserSaga);
  // takeLatest: cancels previous if new action arrives
  // takeLeading: ignores new actions while one is in progress

  </h2>
  <div id="state-q25" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      Optimistic updates in Redux mean immediately updating the store with the expected server result before the async operation completes, then reconciling when the actual response arrives. The pattern: (1) dispatch an optimistic update action that immediately changes the UI state; (2) make the API call; (3) on success, optionally dispatch a confirmation action with server-generated data (like a real ID replacing a temp ID); (4) on failure, dispatch a rollback action that reverts the state to before the optimistic update.       <pre class="code-block">// RTK Query optimistic update with rollback
const api = createApi({
  endpoints: (builder) =&gt; ({
    updatePost: builder.mutation({
      query: ({ id, ...patch }) =&gt; ({ url: `/posts/${id}`, method: 'PATCH', body: patch }),
      async onQueryStarted({ id, ...patch }, { dispatch, queryFulfilled }) {
        // Optimistically update cache
        const patchResult = dispatch(
          api.util.updateQueryData('getPost', id, (draft) =&gt; {
            Object.assign(draft, patch); // Immer draft mutation
          })
        );
        try {
          await queryFulfilled; // wait for server response
        } catch {
          patchResult.undo(); // rollback on failure
        }
      },
    }),
  }),
});</pre>

    </button>
  </h2>
  <div id="state-q26" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">

It is appropriate for: UI state that only one component cares about (modal open/closed, input values, tab selection, hover state), state that is tightly coupled to a specific component's logic, and ephemeral state that doesn't need to survive navigation. The Provider is responsible for managing the state (via useState or useReducer) and exposing it (and any update functions) through a context value.   <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#state-q32">

  <div id="state-q32" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      The atom pattern models application state as a collection of small, independent state units (atoms) rather than a single large state tree. Each atom represents one piece of state (similar to a global useState), and components subscribe to only the atoms they need, re-rendering only when those specific atoms change. Atoms can be composed: derived atoms (called selectors in Recoil, or computed atoms in Jotai) automatically track their atom dependencies and recompute only when those dependencies change. This fine-grained reactivity means a change to a user's email doesn't cause a component that only reads the user's name to re-render. The atom model is extremely granular and scalable for large apps with many independently updating pieces of state, and it avoids the "one context re-renders everything" problem of React Context. The trade-off is managing a large number of atoms across a codebase requires discipline.

Where Redux enforces strict patterns (actions, reducers, immutable updates, dispatch), Zustand is permissive — you mutate state directly inside action functions using Immer (if configured) or just use <code>set</code>. There is no explicit action type system, no dispatch, no separate action creators; you just call functions on the store object. Zustand doesn't require a Provider component — the store is a module-level singleton. Zustand's selective subscription (components only re-render when their selected slice changes) is a first-class feature, whereas Redux requires careful selector memoization for the same result. Zustand is ideal for simple-to-medium global state; Redux scales better for large teams and complex state with extensive async workflows and debugging needs.

    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#state-q34">

    </button>
  </h2>
  <div id="state-q34" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      MobX is a reactive state management library that automatically tracks which components observe which state and updates only those components when the observed state changes. Instead of the explicit dispatch → reducer → update cycle of Redux, MobX uses observable state objects and derivations — you mark state as <code>@observable</code>, mark computed values as <code>@computed</code>, and mark UI-updating code as <code>@observer</code>. When observable state changes, MobX automatically re-runs all observer components that read that state. MobX is more "magic" (automatic dependency tracking) versus Redux's explicit data flow. MobX is more flexible (any state mutation triggers updates) while Redux enforces immutability and explicit updates. MobX with React requires fewer lines of code for equivalent functionality but is harder to debug since changes can happen from anywhere without a clear audit trail.

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#state-q35">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is the observer pattern in state management?
    </button>
  </h2>
  <div id="state-q35" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      The observer pattern is a design pattern where subjects (state stores) maintain a list of observers (subscribers/listeners) and automatically notify them when the subject's state changes. In React state management, it underpins many libraries: Redux's <code>store.subscribe()</code> notifies all subscribers on every dispatch; MobX's <code>@observer</code> decorator makes a component an observer of the observables it reads; Zustand's hook system subscribes components to the parts of the store they access. The React component that subscribes to the store is the "observer"; the store is the "subject." This pattern enables decoupled, reactive architectures where state changes automatically propagate to all interested parties without explicit event handling chains. React's own useState and useReducer are local implementations of the same idea — the component subscribes to its own state and re-renders when it changes.
    </div>
  </div>
</div>

      RTK's <code>createAsyncThunk</code> generates three action types for each async operation: <code>prefix/pending</code> (dispatched when the thunk starts), <code>prefix/fulfilled</code> (dispatched with the resolved value on success), and <code>prefix/rejected</code> (dispatched with the error on failure). In <code>createSlice</code>, you handle these in <code>extraReducers</code> using the builder API.       <pre class="code-block">const postsSlice = createSlice({

  initialState: { items: [], status: 'idle', error: null },
  reducers: {},
  extraReducers: (builder) =&gt; {
    builder
      .addCase(fetchPosts.pending, (state) =&gt; { state.status = 'loading'; state.error = null; })
      .addCase(fetchPosts.fulfilled, (state, action) =&gt; {
        state.status = 'succeeded';
        state.items = action.payload;
      })
      .addCase(fetchPosts.rejected, (state, action) =&gt; {
        state.status = 'failed';
        state.error = action.error.message;
      });
  },
});

function Posts() {

  if (status === 'failed') return &lt;Error message={error} /&gt;;

      <span class="diff-hard">Hard</span>&nbsp;&nbsp;What is optimistic locking in UI state management?
    </button>
  </h2>
  <div id="state-q37" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      Optimistic locking in UI state management is a pattern where you allow the user to make changes optimistically (immediately updating the UI without waiting for server confirmation) but detect and handle conflicts when the server rejects the update because the server-side data changed since the client last fetched it. The server uses a version number or ETag — when the client submits an update, it includes the version it last saw; if the server's current version is different, it rejects with a conflict error. The UI must then handle this by showing the user a conflict dialog, allowing them to choose between their changes and the server's current data, or automatically merging if possible. This is more complex than simple optimistic updates because you must handle the case where optimistic updates fail not due to network errors but due to concurrent modifications from other users. It is essential in collaborative editing applications.
    </div>

  <div id="state-q38" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      Undo/redo with Redux leverages immutable state history. The <code>redux-undo</code> library is the standard approach: it wraps a reducer and maintains a history stack of past and future states. Dispatching <code>ActionCreators.undo()</code> moves back one step; <code>ActionCreators.redo()</code> moves forward. Manually, you maintain <code>{ past: [state1, state2, ...], present: currentState, future: [] }</code>. On any normal action, push <code>present</code> to <code>past</code>, set <code>present</code> to the new state, and clear <code>future</code>. On undo, pop from <code>past</code> into <code>present</code> (pushing current <code>present</code> to <code>future</code>). On redo, pop from <code>future</code> into <code>present</code>. Filter which actions are undoable with <code>redux-undo</code>'s <code>filter</code> option — you typically don't want to undo navigation actions or server sync actions.
      <pre class="code-block">// Using redux-undo

const undoableCounter = undoable(counterReducer, {
  limit: 20, // max history depth
  filter: (action) =&gt; action.type !== 'sync/serverUpdate', // skip some actions
});

// Dispatch undo/redo
dispatch(ActionCreators.undo());
dispatch(ActionCreators.redo());

// State shape with redux-undo:
// { past: [...], present: currentState, future: [...], index: 2 }</pre>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#state-q39">
      <span class="diff-hard">Hard</span>&nbsp;&nbsp;Explain the concept of "derived state" vs "source of truth" state.
    </button>
  </h2>
  <div id="state-q39" class="accordion-collapse collapse" data-bs-parent="#acc-state">
    <div class="accordion-body">
      Source of truth state is the minimal, canonical set of data that everything else can be computed from. Derived state is any value that can be calculated from source of truth state — it is not stored but computed on demand. For example, in a todo app: the source of truth is the list of todos and the current filter setting. The derived state is the filtered list — it should NEVER be stored separately. Storing derived state creates dual sources of truth that can diverge: if you update <code>todos</code> but forget to recompute <code>filteredTodos</code>, the UI shows stale data. The Redux principle says: "Don't store in state what you can compute from state." Reselect's <code>createSelector</code> and <code>useMemo</code> implement this correctly — they compute derived values from source state on demand, with memoization for performance. Violating this principle is one of the most common sources of bugs in React apps.
    </div>
  </div>
</div>

  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q1">
      <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is React.memo?
    </button>
  </h2>
  <div id="performance-q1" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      <code>React.memo</code> is a higher-order component that memoizes a functional component's rendered output. When React would normally re-render a component because its parent re-rendered, <code>React.memo</code> first checks whether the component's props have changed using a shallow comparison. If none of the props have changed (compared by reference using <code>Object.is</code>), React skips re-rendering and reuses the last rendered output. This is the functional component equivalent of <code>React.PureComponent</code> for class components. React.memo is a performance optimization — it only helps when a parent re-renders frequently and the child's props often remain the same. Wrapping every component in memo is unnecessary and adds comparison overhead; profile first and apply memo where it provides measurable benefit.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q2">
      <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is the purpose of useMemo?
    </button>
  </h2>
  <div id="performance-q2" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      <code>useMemo(computeFn, deps)</code> memoizes the result of a computation, re-running <code>computeFn</code> only when its listed dependencies change. On every render where dependencies haven't changed, it returns the cached value from the last computation. Its purpose is to avoid re-running expensive calculations (filtering large arrays, complex sorting, building expensive derived data structures) on every render. <code>useMemo</code> returns a value (not a function). Common use cases: memoizing filtered/sorted lists from large datasets, creating stable object references to pass as props to memoized children, and computing expensive format transformations. The dependency array follows the same rules as useEffect — all reactive values used inside computeFn must be listed. Do not use useMemo for every computation — only when the computation is measurably expensive or when reference stability matters.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q3">
      <span class="diff-easy">Easy</span>&nbsp;&nbsp;What is the purpose of useCallback?
    </button>
  </h2>
  <div id="performance-q3" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      <code>useCallback(fn, deps)</code> returns a memoized version of a function that only changes when its listed dependencies change. On renders where dependencies haven't changed, it returns the same function object from the previous render (same reference). The primary purpose is to create stable function references so that memoized child components (wrapped in React.memo) don't unnecessarily re-render when the parent renders. Without useCallback, every render creates a new function object — even with identical logic — and React.memo's shallow comparison sees a new reference and re-renders the child. useCallback is also used to stabilize functions listed as useEffect dependencies, preventing unintended effect re-runs. <code>useCallback(fn, deps)</code> is essentially <code>useMemo(() =&gt; fn, deps)</code> — they are the same mechanism.
    </div>
  </div>
</div>

  </h2>
  <div id="performance-q4" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      Lazy loading (also called code splitting) is the practice of deferring the loading of a component or module until it is actually needed, rather than loading everything upfront when the app first starts. In React, you lazy-load components using <code>React.lazy()</code> combined with <code>Suspense</code>. When a user navigates to a route or triggers a condition that requires a lazily loaded component, React dynamically imports that component's JavaScript bundle at that moment. This reduces the initial bundle size, improving the Time to Interactive metric — users get a faster initial page load because they don't download code for features they haven't visited yet. Webpack, Vite, and other bundlers support automatic code splitting when they encounter dynamic <code>import()</code> calls.

  <div id="performance-q10" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      Tree shaking is a dead code elimination technique performed by modern JavaScript bundlers (Webpack, Vite, Rollup) that removes unused exports from your final bundle. When you import only <code>{ formatDate }</code> from a utility library, tree shaking ensures that only <code>formatDate</code>'s code (and its dependencies) is included in the bundle — all other exports are "shaken out." For tree shaking to work, the module must use ES module syntax (<code>export</code>/<code>import</code>), not CommonJS (<code>module.exports</code>/<code>require</code>), because ES modules have static structure that bundlers can analyze. Tree shaking is automatic in Vite and in Webpack with <code>mode: 'production'</code>. Pitfalls: side-effectful modules (barrel files that execute code on import) can prevent tree shaking — use <code>"sideEffects": false</code> in package.json to help bundlers.

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q11">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is virtualization and when do you need it?
    </button>
  </h2>
  <div id="performance-q11" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      Virtualization (also called windowing) is the technique of rendering only the visible portion of a large list instead of all items. Instead of creating DOM nodes for 10,000 list items (which is extremely slow and memory-intensive), a virtualized list only renders the ~20–50 items currently in the viewport, creating and recycling DOM nodes as the user scrolls. Virtualization becomes necessary when: rendering more than a few hundred items causes noticeable scroll lag; your list items are complex (images, interactions, nested components); or mobile performance is a concern (fewer DOM nodes = less memory). The trade-off is increased complexity (items must have known or estimated heights), accessibility considerations (screen readers may not see off-screen items), and the items' DOM nodes are destroyed and re-created on scroll.
    </div>
  </div>
</div>

  </h2>
  <div id="performance-q12" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">

Choose based on your list complexity and whether you need full grid (rows + columns) virtualization or just a simple scrolling list.
    </div>
  </div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q13">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is the windowing technique?
    </button>
  </h2>
  <div id="performance-q13" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      The windowing technique renders only the "window" (the visible viewport area) of a large dataset, recycling DOM nodes as the user scrolls. Conceptually: the container has a fixed height with overflow-y scroll; a tall inner container sets the total scrollable height (number of items × item height); only the items whose calculated position falls within the viewport range are actually rendered as DOM nodes; as the user scrolls, items that leave the window are removed (or recycled) and new items that come into view are rendered. Efficient windowing requires knowing the position and height of each item to calculate which items should be visible — simpler for fixed-height items, more complex for variable-height items (which require measuring and caching heights). The result is that the DOM always contains roughly the same small number of nodes regardless of dataset size.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q14">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is the cost of reconciliation and how to minimize it?
    </button>
  </h2>
  <div id="performance-q14" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      Reconciliation is React's process of diffing the previous and new virtual DOM trees to compute necessary DOM updates. The cost scales with the size of the component subtree being reconciled. Minimizing reconciliation cost: (1) Keep component trees shallow — deeply nested trees have more nodes to diff; (2) Use stable keys in lists — correct keys allow React to match and update nodes rather than unmount/remount them; (3) Use <code>React.memo</code> to skip subtrees whose inputs haven't changed; (4) Split large components into smaller ones that can independently skip reconciliation; (5) Use <code>startTransition</code> to make expensive reconciliations non-blocking; (6) Avoid creating anonymous object/array props inline (they create new references on every render); (7) Use the Profiler to find which reconciliations are most expensive.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q15">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is shouldComponentUpdate and when do you override it?
    </button>
  </h2>
  <div id="performance-q15" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      <code>shouldComponentUpdate(nextProps, nextState)</code> is a class component lifecycle method that lets you control whether the component re-renders. It receives the upcoming props and state, and returns <code>true</code> (allow re-render) or <code>false</code> (skip re-render). React calls it before every update phase; if it returns false, neither <code>render()</code> nor <code>componentDidUpdate</code> will run. Override it when <code>React.PureComponent</code>'s shallow comparison isn't sufficient — for example, when you want to compare only specific deep properties of an object prop, or when certain prop changes are irrelevant to the rendered output. In modern functional components, <code>React.memo</code> with a custom comparison function is the equivalent. Returning false incorrectly can cause stale UI — always be conservative in deciding to skip renders.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q16">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;How does React 18's automatic batching improve performance?
    </button>
  </h2>
  <div id="performance-q16" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      In React 17, state updates inside async contexts (Promises, setTimeout, native events) were not batched — each <code>setState</code> triggered a separate render cycle. React 18's automatic batching collects all <code>setState</code> calls anywhere in the same macro/microtask and processes them in a single render. For example, if a network response triggers three setState calls, React 17 would produce three renders; React 18 produces one. This directly reduces the number of renders, the number of DOM updates, and the number of <code>useEffect</code> runs, improving performance especially in data-heavy operations like handling API responses that update multiple pieces of state simultaneously. The only behavioral change to be aware of: if your code relied on multiple re-renders from separate setState calls (rare), <code>flushSync</code> is the escape hatch to opt out of batching.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q17">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is the Concurrent Mode feature and how does it help?
    </button>
  </h2>
  <div id="performance-q17" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      Concurrent Mode (now called "concurrent features" in React 18, enabled by default with <code>createRoot</code>) allows React to prepare multiple versions of the UI simultaneously and interrupt, pause, and resume rendering work. For responsiveness: expensive re-renders can be marked as low-priority (via <code>startTransition</code>) and interrupted when high-priority updates (like user input) arrive, keeping the UI responsive during heavy computation. For perceived performance: React can start rendering UI in the background before it's needed (Suspense prefetching, streaming SSR), and show the previous UI until the next one is ready (avoiding loading spinners for fast transitions). Concurrent Mode enables features like Suspense for data, streaming SSR with selective hydration, and Offscreen rendering (rendering hidden tabs in the background for instant activation).
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q18">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is startTransition?
    </button>
  </h2>
  <div id="performance-q18" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      <code>startTransition(callback)</code> marks state updates inside <code>callback</code> as non-urgent — they are "transitions" that can be interrupted by higher-priority updates. This allows React to keep the UI responsive during expensive renders triggered by the transitions. The key behavior: if a higher-priority update (like a user keystroke) comes in while a transition is being rendered, React discards the in-progress transition render and handles the urgent update first, then restarts the transition. Without startTransition, all state updates have equal priority, and an expensive render blocks user input. You get an <code>isPending</code> boolean (via <code>useTransition</code>) to show a loading indicator while the transition is in progress. Use it for: search result filtering, tab switching, route transitions, and any state update that triggers expensive re-renders.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q19">
      <span class="diff-medium">Medium</span>&nbsp;&nbsp;What is deferred rendering with useDeferredValue?
    </button>
  </h2>
  <div id="performance-q19" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      <code>useDeferredValue(value)</code> produces a deferred version of a value that may lag behind the original value during expensive renders. When the value changes, React first re-renders with the old deferred value (showing stale content), then schedules a lower-priority background render with the new value. This allows the UI that depends on the original value (like an input) to update immediately, while the expensive UI that depends on the deferred value (like a large list) updates in the background. The background render is interruptible — if the value changes again before it completes, React discards the in-progress background render and starts fresh. It is equivalent to using <code>startTransition</code> but applied to a value you receive from outside rather than controlling the state update yourself — useful when you don't control the state setter.
    </div>
  </div>
</div>

    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q22">
      <span class="diff-hard">Hard</span>&nbsp;&nbsp;What are the common causes of unnecessary re-renders?
    </button>
  </h2>
  <div id="performance-q22" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      The most common causes of unnecessary re-renders are: (1) New object/array/function references created inline in JSX props — <code>&lt;Child options={{ key: value }}&gt;</code> creates a new object every render; (2) Parent component re-renders propagating to children that don't need updating because they aren't wrapped in React.memo; (3) Context value changing (new object reference) causing all consumers to re-render; (4) Redux selector returning a new object reference on each call (should use reselect); (5) useEffect with missing or incorrectly specified dependencies triggering more state updates than needed; (6) State or refs set unnecessarily on every render inside effects; (7) Using array index as key in dynamic lists — React unmounts/remounts items on reorder; (8) Component defined inside another component — new function reference every render causes full re-mount.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q23">
      <span class="diff-hard">Hard</span>&nbsp;&nbsp;How do you prevent a parent re-render from causing all children to re-render?
    </button>
  </h2>
  <div id="performance-q23" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      The primary defense is <code>React.memo</code> on child components, which skips re-renders when props haven't changed by reference. For this to work, all props must be referentially stable: primitive values are fine (they compare by value), but objects, arrays, and functions must be memoized with <code>useMemo</code> or <code>useCallback</code>. For children passed via <code>props.children</code> (composition), the children are determined by the parent's parent — not the intermediate parent — so they are naturally stable during the intermediate parent's re-renders. Another powerful technique is "lifting content up" — moving state down to the component that needs it so only that subtree re-renders. Using <code>useDeferredValue</code> or <code>startTransition</code> can also help by making child re-renders non-blocking.

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q24">
      <span class="diff-hard">Hard</span>&nbsp;&nbsp;Explain the difference between expensive renders and too-many renders.
    </button>
  </h2>
  <div id="performance-q24" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      These are two distinct performance problems that require different solutions. "Expensive renders" means each individual render takes too long — the component's render function itself is slow, perhaps computing a sort of 10,000 items, rendering hundreds of DOM nodes, or performing complex layout calculations. The fix is to make each render cheaper: memoize expensive calculations with useMemo, virtualize long lists, simplify JSX, or split the component. "Too-many renders" means the component renders more often than necessary — perhaps on every parent render even though nothing relevant changed. The fix is to reduce render frequency: React.memo, useCallback/useMemo for stable references, moving state down to reduce the re-render scope, or using a more granular state subscription (Zustand selector, reselect). Most apps benefit more from fixing too-many renders than from optimizing individual render cost — always profile to determine which problem you have before optimizing.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q25">
      <span class="diff-hard">Hard</span>&nbsp;&nbsp;How does memoization interact with reference equality in JavaScript?
    </button>
  </h2>
  <div id="performance-q25" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      Memoization in React (React.memo, useMemo, useCallback) relies entirely on reference equality checks using <code>Object.is</code>. Primitives (numbers, strings, booleans) compare by value — <code>42 === 42</code> is true. Objects, arrays, and functions compare by reference — <code>{} === {}</code> is false even if contents are identical. This means: passing <code>&lt;Child options={{ size: 10 }}&gt;</code> creates a new object every render and React.memo always sees a "changed" prop; passing a callback defined inline always appears "changed." For memoization to work, you must ensure stable references for non-primitive props using useMemo (for objects/arrays) and useCallback (for functions). Understanding this is fundamental — almost every React performance optimization bug comes from inadvertently creating new references where stable ones are needed.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q26">
      <span class="diff-hard">Hard</span>&nbsp;&nbsp;What is the overhead of React.memo and when is it NOT worth it?
    </button>
  </h2>
  <div id="performance-q26" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      React.memo's overhead includes: running the props comparison function on every render of the parent (iterating all props with <code>Object.is</code>), maintaining a cache of the previous props, and the cognitive overhead of reasoning about memoization correctness. React.memo is NOT worth it when: the component is cheap to render (a simple div with text — the comparison costs more than re-rendering); the parent rarely re-renders (memo's benefit is zero if the parent only renders once); the props change on every render anyway (memo's comparison always returns "changed" — pure overhead); or when all props are primitive values that would be stable anyway. Measure first: React DevTools Profiler shows exact render times — if a component takes 0.1ms to render, memo's comparison overhead may be comparable. Apply memo to components whose renders you can confirm are expensive or too-frequent.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q27">
      <span class="diff-hard">Hard</span>&nbsp;&nbsp;How do you optimize a large list rendering without virtualization?
    </button>
  </h2>
  <div id="performance-q27" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      When virtualization is overkill (moderately-sized lists of 100–500 items), several techniques help: (1) React.memo on list item components to prevent re-renders when only some items change; (2) Stable keys from data IDs (not array index) to help React reconcile accurately and avoid re-mounting unchanged items; (3) Avoid inline functions/objects in list item props — create stable callbacks with useCallback; (4) Pagination or "load more" patterns to limit the number of items rendered at once; (5) CSS <code>contain: strict</code> on list items tells the browser that changes inside the item don't affect layout outside it, reducing layout recalculation scope; (6) Use <code>will-change: transform</code> sparingly for items that animate; (7) Avoid deeply nested item components — shallower trees reconcile faster.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q28">
      <span class="diff-hard">Hard</span>&nbsp;&nbsp;What is progressive hydration and selective hydration?
    </button>
  </h2>
  <div id="performance-q28" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      Progressive hydration is a technique where server-rendered HTML is hydrated in order of priority — critical interactive parts of the page are hydrated first, while less important parts are hydrated later (or lazily, on interaction). This ensures that the most important UI elements become interactive quickly without waiting for the entire page to hydrate. Selective hydration (React 18) is a related but distinct feature: React can hydrate different parts of the SSR tree independently in parallel, and if the user interacts with an unhydrated part before it's done, React prioritizes hydrating that part immediately. This is enabled by <code>&lt;Suspense&gt;</code> boundaries in SSR — each Suspense boundary is a selective hydration unit. Combined with streaming SSR (<code>renderToPipeableStream</code>), selective hydration means large pages become interactive piece by piece rather than all at once.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q29">
      <span class="diff-hard">Hard</span>&nbsp;&nbsp;How do you reduce the size of your React bundle?
    </button>
  </h2>
  <div id="performance-q29" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      Bundle size reduction strategies: (1) Code splitting at route and component level using React.lazy() — don't load code for unvisited routes; (2) Tree shaking — use named imports from libraries and ensure your build uses ES modules; (3) Replace heavy libraries — lodash-es instead of lodash, date-fns instead of moment.js (moment is ~300KB vs date-fns's tree-shakable modules), day.js for minimal date handling; (4) Externalize large libraries with CDN links (React itself, charting libraries); (5) Analyze bundle with rollup-plugin-visualizer to find unexpectedly large dependencies; (6) Use dynamic imports for rarely-used features; (7) Minification and compression (gzip/brotli) — Vite handles this in production builds; (8) Avoid importing entire UI component libraries — import individual components only.

  <div id="performance-q31" class="accordion-collapse collapse" data-bs-parent="#acc-performance">

      <pre class="code-block">// Native approach — simplest, best for most cases
function ProductImage({ src, alt }) {
  return &lt;img src={src} alt={alt} loading="lazy" /&gt;;
}

// Custom IntersectionObserver approach for full control
function LazyImage({ src, alt }) {
  const [loaded, setLoaded] = React.useState(false);
  const imgRef = React.useRef(null);

  React.useEffect(() =&gt; {
    const observer = new IntersectionObserver(([entry]) =&gt; {
      if (entry.isIntersecting) { setLoaded(true); observer.disconnect(); }
    });
    if (imgRef.current) observer.observe(imgRef.current);
    return () =&gt; observer.disconnect();
  }, []);

  return &lt;img ref={imgRef} src={loaded ? src : undefined} alt={alt} /&gt;;
}</pre>

  </h2>
  <div id="performance-q35" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q36">
      <span class="diff-hard">Hard</span>&nbsp;&nbsp;Explain the React rendering pipeline: trigger, render, commit phases.
    </button>
  </h2>
  <div id="performance-q36" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      React's rendering pipeline has three phases. <strong>Trigger:</strong> a state update (useState setter, useReducer dispatch), a context value change, or the first render triggers React to schedule work. React queues the update and yields to the browser. <strong>Render (reconciliation) phase:</strong> React calls the component function to compute the new React element tree. This phase is pure and may be paused, resumed, or restarted in Concurrent Mode — it does NOT touch the real DOM yet. After computing the new tree, React diffs it against the previous tree to determine the minimal set of changes. <strong>Commit phase:</strong> React applies the computed changes to the actual DOM. This phase is synchronous and cannot be interrupted. It has three sub-phases: "before mutation" (getSnapshotBeforeUpdate, useLayoutEffect cleanup), "mutation" (actual DOM insertions/deletions/updates), and "layout" (useLayoutEffect fires). After commit, the browser paints and useEffect fires asynchronously.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q37">
      <span class="diff-hard">Hard</span>&nbsp;&nbsp;What is the passive effects queue and how does it relate to performance?
    </button>
  </h2>
  <div id="performance-q37" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      "Passive effects" are the effects created by <code>useEffect</code> — they are called "passive" because they run asynchronously after the browser has painted. React schedules all pending passive effects (mount, update, cleanup) to run after the browser's paint using <code>MessageChannel</code> or <code>setTimeout</code> fallback, ensuring they don't block the rendering pipeline. This is why useEffect is the right place for non-visual side effects (data fetching, subscriptions, analytics) — they run without delaying what the user sees. The contrast is "layout effects" (<code>useLayoutEffect</code>) which run synchronously after DOM mutations but before paint — blocking the pipeline. The passive effects queue can accumulate if effects are very expensive or if many components mount simultaneously; in extreme cases this can cause jank after the initial paint. Keep useEffect callbacks fast or use <code>startTransition</code> for non-urgent state updates triggered from effects.
    </div>

    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q46">
      <span class="diff-hard">Hard</span>&nbsp;&nbsp;What is the "layout thrashing" problem and how does React prevent it?
    </button>
  </h2>
  <div id="performance-q46" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      Layout thrashing occurs when JavaScript alternates between reading and writing DOM properties that trigger browser layout recalculation (reflow). For example: reading <code>element.offsetWidth</code> forces the browser to recalculate layout; if you then write <code>element.style.width</code>, the browser must recalculate again; reading <code>offsetWidth</code> again in a loop causes every read to force a layout recalculation, causing exponential performance degradation. React prevents layout thrashing by batching all DOM writes (from the render/commit phase) before any reads (from useLayoutEffect). React updates the DOM in one pass during the commit phase, then lets <code>useLayoutEffect</code> read layout values — all reads happen after all writes are done. This write-then-read order avoids the interleaved read-write-read-write pattern that causes thrashing.
    </div>
  </div>
</div>

<div class="accordion-item">
  <h2 class="accordion-header">
    <button class="accordion-button collapsed" type="button" data-bs-toggle="collapse" data-bs-target="#performance-q47">
      <span class="diff-hard">Hard</span>&nbsp;&nbsp;How does React batch DOM updates and why does it matter?
    </button>
  </h2>
  <div id="performance-q47" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      During the commit phase, React applies all pending DOM mutations in a single synchronous operation. Even if 50 components updated in one render cycle (due to automatic batching of state updates), React makes all 50 DOM mutations in one pass rather than flushing after each. This matters because each DOM mutation can trigger browser layout recalculation, style computation, and repaint — expensive operations. A single batch of all mutations is far cheaper than 50 individual mutations each triggering their own browser calculations. React also separates DOM mutations from layout effects: it completes all mutations first, then runs all layout effects, ensuring layout reads happen after a fully consistent DOM. This is fundamentally different from manipulating the DOM manually in multiple event listeners, which would trigger layout recalculations between each manipulation.
    </div>
  </div>
</div>

    </button>
  </h2>
  <div id="performance-q48" class="accordion-collapse collapse" data-bs-parent="#acc-performance">
    <div class="accordion-body">
      Time slicing is the ability to split rendering work into small units and execute them across multiple frames, yielding to the browser between units to handle user input, paint frames, and other tasks. Without time slicing, a render that takes 200ms completely blocks the main thread — the browser cannot respond to clicks or update animations, causing a "frozen" feeling. With time slicing (Concurrent Mode), React processes a fiber unit, then checks if there's higher-priority work (like a user keypress) using <code>scheduler.shouldYield()</code>. If the frame budget (typically 5ms) is exhausted, React pauses its work, yields to the browser, and resumes in the next idle period. The user sees a responsive UI that's still rendering a large update in the background. React uses the <code>scheduler</code> package (a MessageChannel-based scheduler) rather than <code>requestIdleCallback</code> directly for more reliable timing.

The scheduler uses a min-heap (priority queue) to ensure highest-priority tasks run first. It uses <code>MessageChannel</code> to schedule work in between browser paint frames, checking <code>performance.now()</code> to yield when the time budget is exceeded. Developers interact with it via <code>startTransition</code> (maps to NormalPriority for transition work) and <code>flushSync</code>.
