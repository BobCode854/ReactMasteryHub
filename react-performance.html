React Performance

React.memo, useMemo, useCallback, code splitting, and bundle optimisation

React.memo

React.memo is a higher-order component that memoizes the rendered output. It prevents re-renders when the parent re-renders but props haven't changed (shallow comparison by default).

When to use: Components that render frequently with the same props — list items, chart components, or any component lower in a large tree that is expensive to render. 
// Default — shallow prop comparison
const ExpensiveChart = React.memo(function Chart({ data, title }) {
  // Only re-renders when data or title changes (shallow compare)
  const processedData = heavyComputation(data);
  return <canvas>...</canvas>;
});

// Custom comparison — full control
const UserCard = React.memo(function UserCard({ user }) {
  return <div>{user.name}</div>;
}, (prevProps, nextProps) => {
  // Return true = SKIP re-render (props are equal)
  // Return false = DO re-render (props changed)
  return prevProps.user.id === nextProps.user.id;
});
Pitfall: React.memo only memoizes props. If a callback prop is a new function reference on every render, memo does nothing — pair it with useCallback.

useMemo — Expensive Computation

useMemo caches the result of an expensive calculation between renders. React only recomputes the value when one of its dependencies changes.

Rule of thumb: Use useMemo when a calculation takes noticeably long (e.g., sorting/filtering 1000+ items) or when the result is passed as a prop to a memoized component. 
function DataTable({ items, sortField, filterText }) {
  // Only recalculates when items, sortField, or filterText changes
  const processedItems = useMemo(() => {
    return items
      .filter(item => item.name.toLowerCase().includes(filterText.toLowerCase()))
      .sort((a, b) => a[sortField] > b[sortField] ? 1 : -1);
  }, [items, sortField, filterText]);

  return <Table data={processedItems} />;
}

// Also use useMemo to stabilise object/array references
const config = useMemo(() => ({
  theme: 'dark',
  pageSize: 20,
}), []); // stable reference — won't cause child re-renders

useCallback — Stable Function References

useCallback returns a memoized version of a callback. The function is only recreated when its dependencies change — preventing memoized children from re-rendering due to new function references.

function TodoList() {
  const [todos, setTodos] = useState([]);

  // Without useCallback: new function reference on every render
  // → MemoizedTodo always re-renders even when todo data didn't change

  // With useCallback: same reference between renders
  const handleToggle = useCallback((id) => {
    setTodos(prev => prev.map(t =>
      t.id === id ? {...t, done: !t.done} : t
    ));
  }, []); // empty deps = never recreated (uses functional updater)

  const handleDelete = useCallback((id) => {
    setTodos(prev => prev.filter(t => t.id !== id));
  }, []);

  return todos.map(todo => (
    <MemoizedTodo key={todo.id} todo={todo} onToggle={handleToggle} onDelete={handleDelete} />
  ));
}

const MemoizedTodo = React.memo(Todo); // now actually skips re-renders

Code Splitting + Lazy Loading

Code splitting breaks your bundle into smaller chunks loaded on demand — the single most impactful performance optimisation for large SPAs.

// Route-based splitting (highest impact)
const Dashboard = lazy(() => import('./pages/Dashboard'));
const Analytics = lazy(() => import('./pages/Analytics'));
const Reports = lazy(() => import('./pages/Reports'));

// Component-level splitting — heavy modals, charts, editors
const HeavyModal = lazy(() => import('./components/HeavyModal'));

function Page() {
  const [showModal, setShowModal] = useState(false);
  return (
    <>
      <button onClick={() => setShowModal(true)}>Open</button>
      <Suspense fallback={null}>
        {showModal && <HeavyModal onClose={() => setShowModal(false)} />}
      </Suspense>
    </>
  );
}
Dynamic import naming: Use /* webpackChunkName: "dashboard" */ magic comments to give chunks readable names in the bundle analyser.

Bundle Analysis + Tree Shaking

Analyse your bundle to find large dependencies, duplicates, and unused code before optimising.

# Vite — rollup-plugin-visualizer
npm install --save-dev rollup-plugin-visualizer

# vite.config.ts
import { visualizer } from 'rollup-plugin-visualizer';
plugins: [react(), visualizer({ open: true, gzipSize: true })]

# Webpack — webpack-bundle-analyzer
npm install --save-dev webpack-bundle-analyzer

# Tree shaking — import only what you use
# Bad: imports entire lodash (500KB)
import _ from 'lodash';

# Good: imports only debounce (~2KB)
import debounce from 'lodash/debounce';
# or with ES modules (Vite auto tree-shakes):
import { debounce } from 'lodash-es';

Performance Optimisation Checklist

  • Done Use React.memo on pure components that receive stable props
  • Done Wrap callbacks with useCallback passed to memoized children
  • Done Cache expensive computations with useMemo
  • Done Implement route-based code splitting with React.lazy
  • Check Use virtual scrolling (react-virtual) for lists >200 items
  • Check Analyse bundle with visualiser — remove unused dependencies
  • Check Use named imports / lodash-es for proper tree shaking
  • Review Avoid creating objects/arrays inline in JSX props (breaks memo)
  • Review Don't over-memoize — measure first with React DevTools Profiler

Interview Questions

What is the difference between useMemo and useCallback?
useMemo memoizes a computed value (the result of calling a function). useCallback memoizes a function reference itself. useCallback(fn, deps) is equivalent to useMemo(() => fn, deps).
When does React.memo fail to prevent re-renders?
When props include new object/array/function references on every render. Memo uses shallow comparison — a new {} or [] literal always has a different reference. Fix with useMemo for objects and useCallback for functions passed as props.
What is the React Profiler and how do you use it?
The React DevTools Profiler records component render timings. Click "Record," interact with the app, then stop recording. It shows a flame graph of which components rendered, how long each took, and why (what props/state changed). Always profile before optimising — never optimise blindly.
What is tree shaking and how does Vite enable it?
Tree shaking eliminates dead code (unused exports) from the bundle. It requires ES module syntax (import/export). Vite uses Rollup under the hood which performs tree shaking automatically for ESM packages. Use lodash-es instead of lodash to get tree-shakeable lodash.
Why shouldn't you put useMemo/useCallback on everything?
Memoization has overhead — React still calls the function, compares deps, and stores the previous value. For cheap computations or rarely-updated props, this overhead exceeds the savings. Premature optimisation also makes code harder to read. Always profile first.
What is virtual scrolling and when is it needed?
Virtual scrolling (windowing) renders only the DOM nodes currently visible in the viewport, regardless of list size. With 10,000 list items, React renders ~20 DOM nodes at a time. Libraries: @tanstack/react-virtual, react-window. Needed when lists exceed ~200 items and scrolling feels sluggish.
How does useTransition help with performance?
useTransition marks state updates as non-urgent. React renders urgent updates (typing input) immediately, then defers lower-priority renders (filtering a large list). The UI stays responsive during heavy re-renders: const [isPending, startTransition] = useTransition().
What is the key prop's role in performance?
The key prop helps React identify which items in a list have changed, been added, or removed. Using stable, unique keys (like database IDs) allows React to reuse DOM nodes and component state. Using index as key causes unnecessary re-renders and can break component state when items reorder.
